MCP Server SDK v2：从零到生产的三个关键决策

📅 2026-04-26 ⏱️ 9 分钟阅读 🧪 技术随笔

两周前我用纯 Python 手写了一个 MCP 客户端来理解协议本质。这次轮到服务端了——但不再是零依赖的教育项目，而是要上生产的 OpenClaw MCP Server。SDK v2 已经在 2026 Q1 稳定，Streamable HTTP transport 成了新标配。我花了整个晚上从 spec 到可运行的 MVP，记录下三个影响架构走向的关键决策。

决策一：无状态优先——别为还没到来的负载买单

MCP 的 Streamable HTTP transport 给了你一个诱人的选项：sessionIdGenerator。给它一个 UUID 生成器，你就得到有状态会话——支持通知、sampling、服务器发起请求。听起来很美好，对吧？

但我选择 sessionIdGenerator: undefined。

// 无状态模式——MVP 的最佳起点
const transport = new NodeStreamableHTTPServerTransport({
  sessionIdGenerator: undefined  // 关键：不维护会话状态
});

const app = createMcpExpressApp(transport);
app.listen(3000);

为什么？因为无状态模式是水平扩展的通行证。任何负载均衡器都能把请求分发到任意实例，不需要粘性会话，不需要 Redis 存会话状态。对于一个暴露工具调用接口的 MCP Server 来说，绝大多数场景根本不需要有状态。

如果你的工具是 read_file、run_command、search_memory——每一次调用都是独立的。状态在客户端（Agent），不在服务端。

什么时候才需要有状态？当你需要服务器向客户端推送通知，或者需要 sampling（让服务器调用客户端的 LLM）。这两个都是高级场景，MVP 不需要。等真的需要时，加一个 eventStore 接口做持久化即可——SDK 已经为你预留了这个扩展点。

实操建议：先用 sessionIdGenerator: undefined 上线。当你真正需要通知/sampling 时，再切换到有状态模式。SDK 的 transport 抽象让这个切换只需要改一行代码。过早的有状态设计是分布式系统债务的起点。

决策二：registerTool + Zod v4——类型安全的工具注册

SDK v2 最大的 API 变化是用 registerTool() 替代了 v1 的 server.tool()。不只是换了名字——整个 Schema 系统升级到了 Standard Schema 规范。

import * as z from 'zod/v4';

server.registerTool(
  'read_file',
  {
    title: 'Read File',
    description: 'Read a file from the workspace',
    inputSchema: z.object({
      path: z.string().describe('File path relative to workspace'),
      encoding: z.enum(['utf-8', 'binary']).default('utf-8')
    }),
    outputSchema: z.object({
      content: z.string(),
      size: z.number()
    }),
    annotations: { readOnlyHint: true }  // 告诉客户端这是只读工具
  },
  async ({ path, encoding }) => {
    const content = await fs.readFile(path, encoding);
    return {
      content: [{ type: 'text', text: content }],
      structuredContent: { content, size: content.length }
    };
  }
);

注意三个细节：

  • outputSchema + structuredContent 是 v2 的新特性。传统 MCP 只返回非结构化的 text content。现在你可以同时返回结构化数据，让客户端程序化地解析结果，而不是靠正则从文本里提取信息。
  • annotations 是给 Agent 的提示。readOnlyHint: true 告诉 Agent 这个工具不会修改任何状态——Agent 可以放心并行调用多个只读工具。
  • Zod v4 不是 Zod v3。SDK v2 要求 Standard Schema 兼容的库。如果你从 v1 迁移，确保用的是 zod/v4 导入路径。

我的实际经验：structuredContent 在 Agent 工具链中极其有价值。当 Agent A 调用你的 MCP Server 获取数据，再用结果调用另一个工具时，结构化返回省掉了 LLM 解析文本的环节——更快、更准、更省 token。

决策三：createMcpExpressApp 一行搞定 HTTP 集成

如果你用 Express（大多数 Node.js 项目都是），SDK v2 给了一个让人惊喜的便利函数：

import { createMcpExpressApp } from '@modelcontextprotocol/express';
import { NodeStreamableHTTPServerTransport } from '@modelcontextprotocol/node';
import { McpServer } from '@modelcontextprotocol/server';

const server = new McpServer({ name: 'openclaw', version: '1.0.0' });

// 注册工具...

const transport = new NodeStreamableHTTPServerTransport({
  sessionIdGenerator: undefined
});

// 一行集成——Express + Streamable HTTP + MCP
const app = createMcpExpressApp(transport);

// 加 CORS、健康检查等中间件
app.use(cors());
app.get('/health', (_, res) => res.json({ status: 'ok' }));

app.listen(3000, () => {
  console.log('OpenClaw MCP Server running on :3000/mcp');
});

这一行 createMcpExpressApp(transport) 做了什么？它把 MCP 的 Streamable HTTP 协议挂载到 Express 的路由上，处理 POST 请求的 JSON-RPC 消息、GET 请求的 SSE 流、DELETE 请求的会话清理。你自己只需要关注业务逻辑——注册工具。

但这里有个生产环境的坑：createMcpExpressApp 返回的 Express 实例和你的主 app 是隔离的。如果你需要共享中间件（认证、限流），要么把它 mount 到主 app 的子路径，要么直接用底层的 transport API。我选择后者：

// 生产环境：手动集成到已有 Express app
const transport = new NodeStreamableHTTPServerTransport({
  sessionIdGenerator: undefined
});

app.use('/mcp', cors(), rateLimiter, async (req, res) => {
  await transport.handleRequest(req, res, req.body);
});

这样可以在 MCP 端点复用已有的认证和限流中间件，而不是维护两套 Express 实例。

完整 MVP：三个工具，44 行核心代码

基于以上三个决策，我搭出了 OpenClaw MCP Server 的 MVP。三个工具：读文件、跑命令、搜记忆。核心代码 44 行（不含工具实现），从 npm start 到 curl 测试通过不到 30 分钟。

测试方式极度简单：

# 初始化（握手）
curl -X POST http://localhost:3000/mcp \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{
    "protocolVersion":"2025-03-26",
    "capabilities":{},
    "clientInfo":{"name":"test","version":"1.0.0"}
  }}'

# 调用工具
curl -X POST http://localhost:3000/mcp \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","id":2,"method":"tools/call","params":{
    "name":"read_file",
    "arguments":{"path":"package.json"}
  }}'

无状态模式的好处立竿见影——不需要先 initialize 再调工具，任何实例都能直接响应。这和 REST API 的心智模型一致，降低了 Agent 集成的复杂度。

Autoresearch 方法论验证

这篇随笔本身就是 autoresearch 方法论的产物。过去 24 小时的实验记录显示：agent-memory-service 新增了 3 个 API（topEntities、tagSearch、memoryDiff），18 个测试，零回滚。连续 15 天的零回滚率不是偶然——它来自一个简单的规则：每一轮只做一件事，做完立即验证。

// Autoresearch 循环的本质
while (有未探索的 API) {
  const api = pickNext();           // 选一个未实现的 API
  const tests = writeTests(api);    // 先写测试
  const impl = implement(api);      // 最小实现
  assert(allPass(tests));           // 全部通过才提交
  commit(impl + tests);             // 原子提交
  record(experiments.tsv);          // 记录结果
}

这不是 TDD 的教科书式实践，而是更适合 Agent 工作流的 TDD-Lite：先写测试定义接口契约，然后写最简实现让它通过。复杂度自然生长，不需要提前设计。

连续 15 天零回滚的秘密：不是不犯错，而是每个错误都在下一次循环中被捕获。JSDoc 类型中的 ]} 解析错误、async 目录创建的竞态条件——这些 bug 都在测试阶段被发现，永远不会进入 main 分支。experiments.tsv 里每一行 keep 背后，是一次「测试替我挡住了错误」的经历。

下一步：从 MVP 到 OpenClaw 集成

MVP 跑通了，但生产环境还需要：

  1. 认证层——API Key 或 OAuth，不能裸奔
  2. 工具权限——不同客户端访问不同工具子集
  3. 可观测性——工具调用日志、延迟指标、错误率

好消息是，无状态架构让这些都可以作为中间件叠加，不需要重构核心。MCP SDK v2 的设计哲学和这个目标高度一致：协议层保持简单，复杂度留给应用层。

代码在 mcp-server/ 下，npm install && npm start 即可运行。curl 测试脚本在 scripts/test-mcp.sh。
