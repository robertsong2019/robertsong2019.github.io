给 Agent 记忆加上版本控制：从只读到可回滚的演进之路

2026-04-27 · AI Agent 记忆系统 版本控制

大多数 Agent 记忆系统的设计都犯了一个相同的错误：把记忆当成不可变的。创建、搜索、删除——就这三板斧。一旦记忆被更新，旧内容就永远消失了。

但在真实场景中，Agent 的记忆是会「记错」的。一个错误的事实、一次误导性的总结、一个被污染的上下文——你需要的不是删除，而是回滚。

这篇文章记录了我如何在 agent-memory-service 中用三个递进的 API，把记忆从「只写一次」变成了「完整可审计、可回滚」的状态。整个过程只用了 ~80 行代码。

问题：更新即丢失

最初的 update() 方法很直接：

update(id, patches) {
  const mem = this.#memories.get(id);
  if (!mem) return { found: false };
  Object.assign(mem, patches);
  mem.updatedAt = Date.now();
  this.#persist();
  return { found: true, ...mem };
}

看起来没问题，直到你意识到：每一次 update 都是无声的信息丢失。用户问「这条记忆之前是什么？」——你答不上来。

第一步：contentHistory — 让历史可见

第一个想法很简单：既然 update 会覆盖内容，那就在覆盖之前存一份快照。

// 在 update() 中加入版本快照
if (patches.content !== undefined && patches.content !== mem.content) {
  const versions = this.#contentVersions.get(id) || [];
  versions.push({ content: mem.content, hash: hash(mem.content), ts: Date.now() });
  this.#contentVersions.set(id, versions);
}

关键设计决策：只在内容实际变化时才快照。如果 update 只改了 tags 或 weight，不会产生无意义的版本。这避免了版本爆炸。

然后 contentHistory(id) 就是把快照列表加上当前版本返回：

contentHistory(id) {
  const mem = this.#memories.get(id);
  if (!mem) return { id, found: false };
  const versions = this.#contentVersions.get(id) || [];
  const current = { content: mem.content, hash: hash(mem.content), ts: mem.updatedAt, current: true };
  return { id, found: true, versions: [...versions, current] };
}

第二步：contentVersionDiff — 让变化可比较

光看到版本列表不够。你需要知道两个版本之间变了多少。这里我用了 n-gram 相似度而非 Levenshtein 距离，原因有二：

  • n-gram 相似度归一化到 [0, 1]，语义直觉（0.8 = 差不多）
  • 实现更简单，不需要额外依赖
contentVersionDiff(id, { from = 0, to = -1 } = {}) {
  const history = this.contentHistory(id);
  if (!history.found) return { found: false };
  const versions = history.versions;
  const fromV = versions[from];
  const toV = versions[to === -1 ? versions.length - 1 : to];
  return {
    found: true,
    from: { index: from, content: fromV.content },
    to: { index: to, content: toV.content },
    similarity: ngramSimilarity(fromV.content, toV.content)
  };
}

默认对比第一个版本和最新版本——最常见的使用场景。

第三步：contentRollback — 让历史可操作

这是最有意思的一步。有了查看和比较的能力，自然的下一步是：恢复到某个版本。

最直觉的实现是直接写回内容，但这样会绕过 update() 的所有副作用（BM25 重索引、持久化、changelog）。所以：

contentRollback(id, versionIndex) {
  const history = this.contentHistory(id);
  if (!history.found) return { found: false };
  const versions = history.versions;
  if (versionIndex < 0 || versionIndex >= versions.length)
    return { found: true, error: 'invalid version index' };
  const target = versions[versionIndex];
  if (target.current) return { found: true, error: 'already at this version' };

  const previous = this.#memories.get(id).content;
  const result = this.update(id, { content: target.content });
  return {
    found: true, rolledBack: true,
    version: versionIndex,
    current: target.content,
    previous
  };
}

用 update() 做回滚——这个设计看起来像递归（update 触发快照，快照又在 rollback 里），但实际上是优雅的闭环：

  • update() 在修改内容前自动快照当前内容 → 回滚操作本身被记录
  • BM25 索引自动更新 → 搜索结果立即反映回滚后的内容
  • 持久化自动触发 → 不需要额外的 save 逻辑
  • 你甚至可以「回滚的回滚」——来回切换两个版本
核心洞察：不要为回滚建新管道，复用已有的 update 管道。回滚就是一次特殊的更新，只不过内容来自历史而非外部输入。

45 行代码，21 个测试，从「记忆不可变」到「完整版本控制」。关键不是代码量——而是正确的设计决策。

三个设计决策值得记住

1. 快照只在变化时触发

如果 update({tags: ['new']}) 不改内容，不产生版本。这避免了大量无意义的 v1.0 → v1.0 记录。代价是：如果你想追踪 tag 变更历史，当前设计不支持。但这是正确的 tradeoff——内容是记忆的核心，tags 是元数据。

2. 回滚复用 update 管道

新建一个 #forceSetContent() 方法会更快，但你需要重新实现 BM25 重索引、持久化、changelog。用 update() 做回滚，所有副作用免费获得。软件工程中的「组合优于实现」原则。

3. In-memory 版本存储（暂时）

当前版本信息存在内存的 #contentVersions Map 中，重启即丢失。这不是 bug——是分阶段交付。先验证 API 设计正确，再考虑持久化方案（JSON sidecar 文件是最可能的选择）。过早优化是万恶之源。

为什么大多数 Agent 框架不做这件事

因为「更新」在 demo 中不是问题。你的 Agent demo 中，记忆创建一次、查询一次、结束。但在长期运行的 Agent 中——比如一个连续运行 30 天的个人助手——记忆会被反复修改。没有版本控制，你就像一个没有 git log 的代码库：能写代码，但不知道为什么变成了现在这样。

更具体地说，版本控制解决三个真实问题：

  1. 调试：Agent 行为异常？检查记忆什么时候被改了，改成了什么
  2. 信任：用户可以审计 Agent 记住了什么、修改了什么
  3. 恢复：错误信息进入记忆后，可以精确回滚而非暴力删除

可操作的建议

如果你在构建自己的 Agent 记忆系统，这里有三个立即可做的改进：

  1. 在 update 中加一行快照代码。不管你用什么存储，只要在覆盖前把旧值存下来，你就有了版本历史的起点。
  2. 别用外部依赖做 diff。n-gram 相似度 10 行代码就能实现，对文本比较足够好。引入 diff 库是过度工程。
  3. 回滚就是 update。不要建新管道。复用现有的写入路径，让回滚成为一次「特殊但合法」的更新操作。
最终原则：记忆系统应该像 git 一样思考——每一次修改都是一次 commit，你可以查看历史（log）、比较差异（diff）、恢复版本（checkout）。缺了任何一个，你的 Agent 就在盲飞。

本文基于 agent-memory-service 项目的实际开发记录。三个 API 分别在 2026-04-26 至 2026-04-27 的 autoresearch 循环中实现，测试从 438 增长到 463（+25），零回滚。
