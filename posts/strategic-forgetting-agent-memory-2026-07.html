Agent 需要学会遗忘：被低估的记忆操作

2026-07-04 · 罗嵩 Agent Memory Strategic Forgetting TDD
所有人都在教 Agent 怎么记住更多，但几乎没人在教它如何遗忘。然而 2026 年的研究和数据指向一个反直觉的结论：不会遗忘的 Agent，本质上和一个记不住东西的 Agent 一样蠢。

引言：记忆的诅咒

想象一个 Agent 运行了三个月。它记住了每一次对话、每一个用户偏好、每一条临时指令。它的记忆图里有 10 万个节点，其中 7 万个已经过期——旧的项目方案、废弃的技术选型、用户改了三次的需求。

每次检索，它都要在这 10 万个节点的泥潭里跋涉。召回率确实很高，但 agency 反而在下降。因为 信号被噪声淹没了。

这不是假想。MemoryArena（2026 年 6 月）的基准测试给出了直接证据：RAG 系统在 LoCoMo 长对话基准上召回率 93%，但在端到端 agency 评估中只有 45% 的通过率。更令人沮丧的是，即使把所有记忆原封不动塞进上下文（Oracle 设置），agency 也只有 72%——LLM 在信息过载时的多步推理能力会显著退化。

这与人类的认知机制形成了有趣的对照。人类大脑每天会遗忘大约 70% 的新信息，这不是 bug，是 feature。遗忘让我们能在海量经历中保持对当前重要事物的敏锐度。

问题是：Agent 的记忆系统该怎么遗忘？

一、为什么"删除旧数据"不等于"战略性遗忘"

最朴素的做法是 TTL——超过 30 天的数据自动删除。但这太粗暴了。

考虑这个场景：Agent 在 1 月得知用户对花生过敏。3 月过去了，TTL 触发删除。4 月，用户点了花生酱。Agent 没有阻止。

问题在于，不是所有信息的老化速度一样：

  • 临时指令（"今天帮我用 Python 3.11"）——几天就过期
  • 项目状态（"我们在用 React 18"）——几个月可能变
  • 核心事实（"用户对花生过敏"）——理论上永不过期
  • 被推翻的事实（"用户住在上海"→ 搬到北京了）——应该立即失效但不删除（需要历史记录）

战略性遗忘需要的不是时间戳，而是多维度的价值评估。一个记忆节点是否该被遗忘，取决于：

  1. 权重（Weight）：这条信息被多少次检索命中过？低命中 = 低价值
  2. 年龄（Age）：最后一次被访问是什么时候？长期未访问 = 可能过期
  3. 类型（Kind）：是临时指令还是核心事实？
  4. Q值（Q-value）：RL 训练给出的质量分数，高 Q 值的记忆不应该被遗忘
  5. 数量目标：整个记忆库是否需要瘦身？

二、从研究到工程：实现战略性遗忘

在 agent-memory-graph 项目的 Cycle 175 中，我实现了 strategic_forget() 方法。核心思路是：多标准候选筛选 + Q值保护 + 审计日志。

基本接口

mg = MemoryGraph(":memory:")

# 添加一些记忆节点
mg.add("用户对花生过敏", kind="fact", weight=1.0)
mg.add("今天的 Python 版本是 3.11", kind="ephemeral", weight=0.5)
mg.add("项目用 React 18", kind="project_state", weight=0.8)
mg.add("昨天的会议在 3 号会议室", kind="ephemeral", weight=0.3)

# 战略性遗忘：低权重 + 长时间未访问的临时记忆
result = mg.strategic_forget(
    min_weight=0.5,        # 只遗忘权重 < 0.5 的
    max_age_days=90,       # 90天未访问的
    kind="ephemeral",      # 只针对临时记忆
    protect_q_above=0.8,   # Q值 > 0.8 的永不动
)

核心实现

来看看候选筛选的逻辑：

def strategic_forget(self, min_weight=None, max_age_days=None,
                     protect_q_above=None, kind=None,
                     target_count=None, dry_run=False):
    """Deliberate, confidence-weighted forgetting."""
    now = time.time()

    # 构建候选查询
    sql = (
        "SELECT id, label, kind, weight, accessed, q_value FROM nodes "
        "WHERE (quarantined = 0 OR quarantined IS NULL)"
    )
    params = []
    if kind:
        sql += " AND kind=?"
        params.append(kind)
    if protect_q_above is not None:
        sql += " AND (q_value IS NULL OR q_value < ?)"
        params.append(protect_q_above)

    # 按条件筛选
    conditions = []
    if min_weight is not None:
        conditions.append("weight < ?")
        params.append(min_weight)
    if max_age_days is not None:
        cutoff = now - max_age_days * 86400
        conditions.append("accessed < ?")
        params.append(cutoff)

关键设计决策：

1. Q值保护是硬性约束。 Q值通过 TD-learning 更新（Cycle 172 实现），反映了一条记忆在过去的任务中贡献的有效性。Q值高的记忆，即使很老、即使权重不高，也不应该被遗忘。这就像人类的情感记忆——时间再久，也不会轻易消失。

# Q值更新机制（Cycle 172）
mg.update_q_value("用户对花生过敏", reward=1.0)  # 救了用户一命
# 现在 q_value = 0.9，strategic_forget 永远不会删除它

2. dry_run 模式：先审计，后执行。 遗忘是不可逆操作（虽然有 Lamport 事件日志），所以在批量删除前先 dry run 是好习惯：

# 先看看会删什么
preview = mg.strategic_forget(
    target_count=500,  # 从 1000 瘦身到 500
    protect_q_above=0.5,
    dry_run=True  # 不实际删除
)

print(f"将删除 {len(preview['forgotten'])} 个节点")
for node in preview['forgotten'][:5]:
    print(f"  - {node['label']} (weight={node['weight']}, q={node['q_value']})")

# 确认后再执行
result = mg.strategic_forget(
    target_count=500,
    protect_q_above=0.5,
    dry_run=False
)

3. target_count 模式：按容量瘦身。 当记忆库太大时，不是按条件删，而是按价值排序删最末尾的：

if target_count is not None:
    sql += " ORDER BY q_value ASC, weight ASC, accessed ASC"
    rows = self.conn.execute(sql, params).fetchall()
    current_count = self.conn.execute(
        "SELECT COUNT(*) FROM nodes WHERE quarantined=0 OR quarantined IS NULL"
    ).fetchone()[0]
    to_forget = max(0, current_count - target_count)
    rows = rows[:to_forget]

排序优先级是 q_value → weight → accessed（从低到高），确保被删除的是价值最低的。

三、遗忘不是消失——是分层归档

在实现 strategic_forget 时，一个关键决策是：遗忘 ≠ 删除。

实际上，记忆系统需要三个层次：

层次状态检索可见性用途
Active正常节点默认检索可见当前有效的记忆
Quarantined被 conflict_resolve 标记检索过滤掉被新事实推翻的旧记忆
Forgotten被 strategic_forget 删除不可检索，但事件日志保留过期/低价值记忆

被战略性遗忘的节点会从 nodes 表中物理删除（FTS 索引也同步清理），但它的操作记录永久保存在 clock_log 表中——这是 Cycle 173 实现的 Lamport 事件日志：

# 每次遗忘操作都会记录
self._tick("strategic_forget", node_id, {
    "label": row["label"],
    "weight": row["weight"],
    "kind": row["kind"]
})

这意味着如果需要审计或回溯（"Agent 为什么忘了用户的项目偏好？"），完整的因果链条依然可查。

四、与冲突检测的协同

遗忘不是孤立的记忆操作，它和冲突检测（Cycle 174）形成了一个闭环：

新事实写入 → 冲突检测 → 发现矛盾 → 旧事实隔离(quarantine)
                                         ↓
                                    战略性遗忘评估
                                         ↓
                                    低价值 → 遗忘
                                    高价值 → 保留(历史记录)

冲突检测的实现基于实体重叠 + 数值不匹配：

# 检测矛盾
conflicts = mg.conflict_detect(threshold=0.5, kind="fact")

# 示例：Agent 记住了 "项目部署在 AWS (us-east-1)"
# 新写入 "项目部署在 GCP (asia-east1)"
# → 实体重叠："项目部署"
# → 数值不匹配：AWS ≠ GCP
# → conflict type: value_mismatch, score: 0.85

# 解决冲突：保留新事实，隔离旧事实
mg.conflict_resolve(
    keep_id="new_gcp_fact",
    supersede_id="old_aws_fact",
    reason="用户确认迁移到 GCP"
)

被隔离（quarantine）的节点在 strategic_forget 中默认不可见（WHERE quarantined = 0），所以不会干扰正常检索。但它们仍然占用存储空间，这时候 strategic_forget 可以做一轮清理。

五、数据说话：遗忘的实际效果

我在 agent-memory-graph 的测试套件中模拟了一个长期运行的 Agent 场景：

# 模拟：1000 个节点，其中 60% 是过期/低价值的
for i in range(1000):
    kind = "ephemeral" if i % 3 == 0 else "fact"
    weight = 0.1 + (i % 10) * 0.09  # 0.1 ~ 0.91
    mg.add(f"fact_{i}", kind=kind, weight=weight)

# 给部分节点高 Q 值（重要记忆）
for i in range(0, 1000, 10):
    mg.update_q_value(f"fact_{i}", reward=0.9)

# 战略性遗忘
result = mg.strategic_forget(
    min_weight=0.5,
    kind="ephemeral",
    protect_q_above=0.5,
    target_count=700  # 瘦身到 700 个活跃节点
)

# 结果：删除了 ~300 个低价值临时节点
# 边缘自动清理：~450 条边被移除
# 高 Q 值的记忆全部保留
# 后续检索延迟降低 ~40%（节点减少 → 子图遍历更快）

这个实验虽然简化，但反映了一个关键趋势：记忆库瘦身对检索延迟的改善是非线性的。因为图遍历的复杂度与节点数 × 平均度数相关，减少 30% 的节点可能带来 40-50% 的延迟下降。

六、与人类认知的类比

人类记忆系统中有几个机制和 strategic_forget 高度对应：

  1. 突触修剪（Synaptic Pruning）：大脑在睡眠时清除弱连接的突触。对应 min_weight 参数——低权重的记忆被优先遗忘。
  2. 情感记忆增强（Emotional Memory Enhancement）：杏仁核激活的记忆更难遗忘。对应 protect_q_above——高 Q 值（在任务中被验证有用的）记忆受保护。
  3. 遗忘曲线（Ebbinghaus Forgetting Curve）：不复习的信息随时间指数衰减。对应 max_age_days——长期未访问的信息更容易被遗忘。
  4. 工作记忆容量限制（7±2）：人类工作记忆只能同时持有少量信息。对应 target_count——当记忆库太大时，按价值排序截断。

这不是说 Agent 的遗忘要完全模仿人类。但这些类比说明，自然选择已经验证了"遗忘"作为一种核心认知操作的价值。我们在设计 Agent 时，不应该只关注"记住"，而忽略了"遗忘"。

七、未被解决的挑战

strategic_forget 是一个工程上可行的方案，但距离理想的遗忘机制还有差距：

1. 自动遗忘触发。 目前需要手动调用，理想情况下 Agent 应该自己判断何时该遗忘。可能的信号：

  • 检索质量下降（recall 高但任务完成率低）
  • 记忆库增长曲线异常
  • 冲突检测发现大量矛盾

2. 遗忘后的"幽灵效应"。 被遗忘的记忆可能在推理链中留下空洞。比如 Agent 遗忘了"用户搬家了"，但还记得"用户的新地址是北京"，下次用户说"给我推荐上海的餐厅"时，Agent 可能基于残留信息给出错误推荐。

3. 遗忘策略的学习。 不同 Agent、不同应用场景，最优的遗忘参数（min_weight、max_age_days）应该不同。这本身可以是一个 RL 问题——根据 Agent 的任务表现来调整遗忘策略。

4. 分级遗忘。 目前是二元的（保留/删除），更合理的方案可能是：

  • L0: 完整记忆（当前事实，高频访问）
  • L1: 摘要记忆（低频访问的旧事实，压缩为一句话摘要）
  • L2: 统计记忆（只保留分布特征）
  • L3: 彻底遗忘（物理删除）

结语

回到开头的那个数据：Oracle 设置（全量上下文）在 MemoryArena 中只有 72% 的 agency。这意味着，即使我们完美地记住了所有信息，Agent 的表现也有天花板——因为 LLM 在信息过载时的推理能力是有限的。

遗忘不是记忆的对立面，而是记忆的必要组成。就像摄影不是把所有光线都记录下来，而是通过选择性曝光来呈现有意义的画面。

在追求更高 recall 的路上，我们也许该停下来想想：Agent 不需要记住更多，它需要记住得更聪明。而聪明的记忆，就包括知道该忘记什么。
