AI 正在杀死标品：从 SaaS 到 FDE，软件交付的终局重构

2026-07-11 · 罗嵩 · AI FDE SaaS 产品哲学 行业趋势
"本以为做了AI产品经理，饭碗就不会那么快被AI抢走。"——这句话背后藏着一个更深的命题：不是AI在抢饭碗，而是AI正在从根本上消灭"标品"这个概念本身。

一、SaaS 的底层假设：边际定制成本太高，所以做标品

过去二十年，软件行业有一个基本共识：做标品，不做定制。

Salesforce 卖的是同一套 CRM，SAP 卖的是同一套 ERP，Workday 卖的是同一套 HR 系统。客户买了之后需要"实施"——但实施的本质是把客户的业务流程塞进软件预设的框架里。框架是刚性的，你想改？可以，加钱，加时间，加人。

这个共识的底层逻辑只有一个：软件的边际定制成本太高了。

每多一个客户、每多一种需求，就意味着：产品经理写新的 PRD → 工程师写新的代码 → QA 做新的测试 → 运维部署新的版本。这条链路上每个环节都是人力，人力是线性扩展的。所以最优解是：把所有客户的需求收敛到一个最大公约数，做标品，让客户来适应你。

这个模型运行了二十年，非常成功。但它有一个致命的前提假设——

定制化软件的单位经济模型必须成立。也就是说，每个新功能、每个新场景的边际收入必须大于边际成本。在人力驱动的开发模式下，这个条件几乎永远不成立——除非你是 Palantir，每个合同价值数千万美元。

然后 AI 来了。

二、AI 把定制化的边际成本打到了接近零

这是整个故事的核心转折点。

当 LLM 可以写代码、可以理解业务上下文、可以自主调试和部署的时候，"定制化"的经济学方程式彻底改变了：

维度传统 SaaSAI 时代
新功能开发PM + 工程师，2-6 周Agent 生成 + 人类审核，数小时
客户适配配置 + 二次开发，数月自然语言描述需求，Agent 实现
迭代周期季度发布持续交付，甚至实时
定制成本线性增长（人力驱动）边际递减（模型复用 + 推理成本）
定制粒度租户级配置用户级、场景级、甚至会话级

当定制的边际成本趋近于零，"做标品"就不再是唯一解，甚至不再是最优解。

想想看：如果给每个客户定制一套系统，成本和做标品差不多——那为什么要强迫客户适应你的框架？为什么不给每个客户一个"为它而生"的系统？

这就是 AI 对 SaaS 的根本性冲击：不是用 AI 去做更好的标品，而是让"标品"这个概念本身过时。

三、FDE：在定制化可行的新世界里，谁把 AI 送到客户面前？

Forward Deployed Engineer（前沿部署工程师）这个岗位，最早由 Palantir 发扬光大。他们的做法很简单：把工程师直接派到客户现场，用 Palantir 的平台为客户构建定制化的数据分析和决策系统。

在 Palantir 的语境里，FDE 的逻辑是："我们的平台太强大了，客户不会用，所以我们派工程师来帮他们用。"——这本质上还是"交付工程师"的高级版本。

但在 AI 时代，FDE 的角色和意义发生了质变。

传统 FDE vs AI 时代 FDE

传统 FDE（Palantir 模式）AI 时代 FDE
核心工具低代码平台 + SQL + PythonLLM + Agent + 代码生成
交付速度数周到数月数小时到数天
定制深度数据管线 + 仪表盘完整业务流程 + 工作流自动化
可复制性低（每个客户都是手工作坊）高（Agent 能力可复用、可迁移）
单位经济需要千万级合同才能覆盖成本中小客户也能跑通
角色定位"高级交付工程师""AI 原生产品经理 + 工程师 + 客户成功"三合一

AI 时代的 FDE 不是"被派到客户现场的工程师"，而是"站在客户和 AI 之间的翻译官和编排者"。

他们的工作是：

  1. 理解客户的真实业务场景——不是听客户说要什么功能，而是理解客户每天在做什么、痛在哪里、效率瓶颈在哪里。
  2. 把业务场景拆解为 AI 可以执行的流程——哪些用 Agent 自主决策，哪些用工作流编排，哪些需要人类审核。
  3. 在客户现场快速构建和迭代——用 LLM 的代码能力，把定制化方案在数小时内变成可运行的系统。
  4. 把成功模式抽象回来——某个客户的定制方案，哪些可以泛化为通用能力？哪些就是纯粹属于这个客户的"千人格"？

四、为什么产品经理首当其冲？

回到那篇文章的核心观察：产品经理排队转型 FDE。这不是偶然的。

传统产品经理的核心工作是：在标品的约束下，找到最大公约数的需求。他们做用户调研、写 PRD、画原型、排优先级——所有这些工作的目的都是"在有限的功能槽位里，选择投入产出比最高的需求"。

但这个工作的前提是"功能槽位有限"。当 AI 让定制化的成本趋近于零，功能槽位不再是稀缺资源——"排优先级"这件事本身就没那么重要了。你不需要在 A 和 B 之间做选择，你可以两个都做，甚至为不同客户做不同版本。

传统 PM 的核心能力是收敛：在无限需求中找到最值得做的那几个。

AI 时代 FDE 的核心能力是发散+收敛：先理解客户独特的业务上下文（发散），再用 AI 快速实现定制方案（收敛），同时判断哪些模式可以跨客户复用（再收敛）。

这就是为什么"执行层、功能性较强"的 PM 最为危险——他们的工作本质上是"标品的维护者"，而当标品本身开始瓦解，他们的价值基础也随之消失。

五、从"千人一面"到"千人千面"：产品形态的根本转变

更大的图景在这里——

过去，软件产品的终态是一个完美的标品：界面统一、功能统一、流程统一。每个用户看到的都是同一个软件，只是数据不同。

AI 时代，软件产品的终态是无数个定制化的实例：每个用户看到的软件都是为他量身定制的，界面、功能、流程甚至交互方式都可能不同。就像推荐算法让每个人看到不同的信息流一样，AI 让每个人使用的软件都不同。

┌──────────────────────────────────────────────────────┐ │ 传统 SaaS 产品架构 │ │ │ │ [统一界面] → [统一功能] → [统一流程] → [统一数据模型] │ │ │ │ 客户 A ─┐ │ │ 客户 B ─┼──→ 同一套系统，通过配置实现差异化（很有限） │ │ 客户 C ─┘ │ └──────────────────────────────────────────────────────┘ ┌──────────────────────────────────────────────────────┐ │ AI 原生产品架构 │ │ │ │ [AI 核心] ──→ Agent 编排 ──→ 定制化交付 │ │ ↑ ↓ │ │ └──── 反馈回路（使用数据 → 模型优化） │ │ │ │ 客户 A ──→ 定制实例 A（独特工作流 + 独特界面） │ │ 客户 B ──→ 定制实例 B（独特工作流 + 独特界面） │ │ 客户 C ──→ 定制实例 C（独特工作流 + 独特界面） │ └──────────────────────────────────────────────────────┘

这不是简单的"个性化"。个性化是同一个系统的表面调整——换皮、换顺序、换推荐。真正的定制化是每个客户拥有一个为其业务流程量身构建的系统。

你提到的"千人千面"——包括车载场景——本质上就是这个逻辑。每辆车的驾驶习惯、行驶场景、乘客偏好都不同，当 AI 可以实时生成定制化的交互和服务时，为什么要给所有车装同一个"标品车载系统"？

六、哪些行业最先被冲击？

定制化的边际成本越接近零，标品的护城河就越浅。以下行业最先被冲击：

行业标品逻辑AI 定制化冲击时间线
企业 SaaS同一套 CRM/ERP/HR每个企业一套定制流程正在发生
车载系统同一套 OTA千人千面的驾驶体验1-2 年
医疗 IT同一套 HIS/EMR每个科室、每个医生定制工作流2-3 年
教育科技同一套课程系统每个学生一套学习路径正在发生
金融服务同一套风控/投顾每个客户定制风控策略和投顾方案1-2 年
工业软件同一套 MES/SCADA每条产线定制数字孪生2-3 年

共同特征：客户之间的业务流程差异越大，定制化的价值越高，标品的护城河越脆弱。

七、FDE 会成为"最重要"的岗位吗？

说 FDE 会成为最重要的岗位，这个判断需要更精确地定义。

FDE 不会取代所有产品经理和工程师，但会成为两者的交集和升级方向。

AI 时代的岗位光谱

┌─────────────────────────────────────────────────────────┐ │ │ │ 纯 PM ────→ 产品型 FDE ────→ 工程型 FDE ────→ 纯工程师 │ │ │ │ 需求分析 场景理解+编排 代码生成+调试 基础设施 │ │ 优先级排序 快速原型+交付 系统定制+优化 模型训练 │ │ 用户调研 客户成功 Agent 开发 算法研究 │ │ │ │ ←── 被AI替代风险高 AI放大的能力 ──→ │ │ │ └─────────────────────────────────────────────────────────┘

左端"纯 PM"的风险最高——如果工作内容主要是写文档、排优先级、画原型，AI 确实能干 70%+。右端"纯工程师"相对安全，但也在被 AI 代码生成侵蚀。

中间的 FDE 区间是最大机会带。原因有三：

  1. 最难被替代——需要同时理解业务和技术，这种"翻译"能力恰恰是 LLM 最薄弱的环节。LLM 可以写代码，但不知道"为什么这个功能在这个季度对这个客户如此重要"。
  2. 杠杆最大——一个 FDE 用 AI 工具可以做到过去 10 个人的交付量。不是 1:1 替代，而是 1:10 放大。
  3. 反馈闭环最短——FDE 直接面对客户，能第一时间感知需求变化和技术边界，这种信息优势是任何内部岗位都无法获得的。

八、挑战和反思

但这个趋势并非没有挑战：

1. 定制化的技术栈还不成熟

今天的 LLM 还不能完全可靠地为每个客户生成高质量定制系统。代码生成有幻觉，Agent 有可靠性问题，幻觉检测和质量保证体系还不够成熟。这是一个"方向确定、时间不确定"的趋势——方向没有疑问，但全面落地可能需要 3-5 年。

2. "定制化"不等于"不需要标品"

底层的存储、计算、安全、可观测性——这些基础设施永远是标品。AI 定制的是业务逻辑层和交互层，不是底层基础设施。未来不是"没有标品"，而是"标品下沉到基础设施层"。

3. 定制化带来的维护噩梦

当每个客户都有定制系统时，版本管理、升级策略、技术债务管理都会变得极其复杂。这本身就是一个巨大的工程挑战——也是 AI 必须解决的问题之一。Agent 不仅要能"创建"定制方案，还要能"维护"和"迁移"它们。

4. FDE 会不会也被 AI 替代？

长远来看（5-10 年），当 Agent 的能力足够强大，客户可能直接用自然语言描述需求，Agent 就完成了定制化构建——FDE 作为"翻译官"的角色确实会减弱。但在可预见的未来（3-5 年），FDE 是 AI 能力放大最显著的岗位——AI 把他们的交付能力提升 10 倍，而不是替代他们。

九、对从业者的建议

如果你是一个产品经理，或者团队负责人，面对这个趋势：

  1. 补技术理解力。不是让你写代码，而是要能理解 AI Agent 能做什么、不能做什么，能判断一个定制方案的技术可行性和成本。这不是"学 Python"这么简单——而是建立"AI 原生的工程直觉"。
  2. 补场景洞察力。你的不可替代性不在于"会写 PRD"，而在于"比 AI 更懂这个特定客户/行业的业务上下文"。深耕一个行业，理解它的肌理和潜规则。
  3. 做 AI 工具的早期使用者。不是观望，是现在就开始用 Cursor、Copilot、各种 Agent 框架做实际的交付工作。FDE 不是一种"头衔"，而是一种"用 AI 工具做端到端交付"的能力。
  4. 不要做"标品维护者"。如果你的工作内容可以概括为"维护一个标品的 PRD 和需求池"，这是最危险的信号。

结语：标品的黄昏，定制的黎明

SaaS 的黄金时代建立在"标品是软件交付的最优解"这个假设上。AI 正在摧毁这个假设。

当每个客户都可以拥有为其量身定制的系统，当定制的成本和标品相当甚至更低，当交付周期从月缩短到小时——"标品"就不再是一种产品策略，而变成了一种历史惯性。

FDE 的兴起，是这种转变的信号和载体。它不是一个新的岗位名称，而是一种新的工作方式：用 AI 的杠杆，以一当十地为客户创造定制价值。

这不是产品经理的危机，而是所有软件从业者的机会——前提是你愿意从"标品世界"的舒适区，走进"定制世界"的旷野。

本文基于对 AI 定制化趋势、FDE 岗位演进和 SaaS 经济模型变化的观察。欢迎讨论和反驳。
