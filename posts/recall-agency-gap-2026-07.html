Recall ≠ Agency：为什么 95% 召回率换不来一个靠谱的 Agent

2026-07-03 · 罗嵩 · 基于 autoresearch 方法论
Agent Memory RL Scoring Bi-Temporal LLM Reasoning
当你在为 Agent 的记忆系统追求更高召回率时，可能正走在一条曲线的渐近线上——方向没错，但终点远不是你以为的地方。

引言：一个不舒服的发现

2026 年的 Agent 记忆领域，"recall" 已经不是问题了。在 LoCoMo 长对话基准上，主流记忆系统都能拿到 90%+ 的召回率。如果你愿意堆够 token，95% 也不是难事。

但 MemoryArena（2026 年 6 月）给出了一个让所有人坐直的数据：同一个系统，在 LoCoMo 上 recall 95%，在 agency 评估中只有 40-60% 的通过率。

这意味着什么？你的 Agent 能"找到"信息，但不会"用"信息。能背出答案，但不会解题。

这篇文章记录了我从"追求高召回"到"理解 agency gap"的认知转变，以及在 agent-memory-graph 中实践 Q-value scoring 和 bi-temporal validity 的工程经验。

一、Recall 和 Agency 之间的鸿沟

先精确定义这个 gap。

Recall 衡量的是：给定一个查询，系统能否返回相关记忆？这是一个信息检索问题。

Agency 衡量的是：给定一个任务，Agent 能否正确使用记忆完成多步推理？这是一个决策问题。

两者的关系类似于：

Recall ≈ "你能找到参考书里的知识点"
Agency ≈ "你能用这些知识点解出一道从未见过的题"

MemoryArena 的核心贡献不是提出新算法，而是设计了端到端的 agency 评估：不是问 Agent "你还记得 X 吗"，而是给它一个需要 X 才能完成的任务，看它做不做得对。

二、为什么 Recall 高了，Agency 没跟上？

我分析了三个结构性原因。

原因 1：相关性 ≠ 有用性

传统记忆系统按"查询-记忆"的语义相似度排序。但 Agent 做多步推理时，最有用的记忆往往不是最相似的。

举个例子：Agent 被要求"帮用户预订上次提到的那家餐厅"。

  • 语义相似度最高的记忆可能是："用户喜欢意大利菜"（cosine 0.87）
  • 但真正有用的记忆是："上次提到的是 The French Laundry, Napa Valley"（cosine 0.62）

前者是偏好，后者是事实。Agent 需要的是后者，但向量检索天然偏好前者。

原因 2：记忆没有过期机制

大多数记忆系统是"只增不减"的。旧的、过时的信息和新的、正确的信息混在一起，Agent 不知道该信哪个。

用户三周前说"我用 Python"，上周改说"我在学 Rust"。两个记忆都在库里，recall 都能命中。但 Agent 该用哪个？

原因 3：没有质量信号

所有记忆被平等对待。一条被验证过的正确事实，和一条用户随口一提的猜测，权重完全相同。Agent 没有办法区分"高置信度记忆"和"低置信度记忆"。

三、三条工程路径：从 Recall 到 Agency

面对这个 gap，我在 agent-memory-graph 中实践了三个方案。每个都不复杂，但组合起来效果显著。

方案 1：Q-Value Scoring — 让记忆有"质量"维度

灵感来自强化学习。每条记忆都有一个 Q 值，表示它的"历史表现"——被检索后是否导致了成功的决策。

核心更新规则是经典的 TD-learning：

# Q(s) ← Q(s) + α · (r + γ · max_neighbor_Q − Q(s))
def update_q_value(self, node_id, reward, alpha=0.1, gamma=0.9):
    node = self.get_node(node_id)
    current_q = node.get("q_value", 0.0)
    
    # 找到相邻节点的最大 Q 值
    neighbors = self.get_neighbors(node_id)
    max_neighbor_q = max(
        (n.get("q_value", 0.0) for n in neighbors), 
        default=0.0
    )
    
    # TD 更新
    new_q = current_q + alpha * (
        reward + gamma * max_neighbor_q - current_q
    )
    self.update_node(node_id, {"q_value": new_q})
    return new_q

使用方式很直觉：

# Agent 完成任务后，奖励被使用的记忆
memory.reward("node_abc", reward=1.0)     # 这条记忆很有用
memory.penalize("node_xyz", penalty=0.5)  # 这条记忆导致了错误

# 检索时混合相关性和 Q 值
results = memory.recall_with_q(
    "预订餐厅",
    text_weight=0.6,   # 语义相关性
    q_weight=0.4        # 历史质量
)

💡 关键洞察

Q 值传播是免费的。一旦你有了图结构，TD 更新会自然把高质量的"知识簇"识别出来——高 Q 值节点会通过 gamma 传播给邻居，形成局部"可信区域"。这和 Memory-R1（2026.6）的发现一致：当 Agent 学会管理记忆时，NOOP（不做操作）是最重要的操作——Q-value 提供了"何时该 NOOP"的信号。

方案 2：Bi-Temporal Validity — 让记忆有时间维度

第二个问题是信息过期。解决方案是给每条记忆加上两个时间轴：

  • valid_from / valid_to：事实的有效时间区间
  • txn_time：记录写入时间（系统时间，用于审计和冲突解决）
def supersede(self, old_node_id, new_properties):
    """标记旧节点过期，创建新节点替代"""
    now = datetime.now(timezone.utc)
    
    # 1. 标记旧节点过期
    self.update_node(old_node_id, {
        "valid_to": now,           # 不再有效
        "superseded_by": None       # 稍后填充
    })
    
    # 2. 创建新节点
    new_node = self.add_node({
        **new_properties,
        "valid_from": now,
        "valid_to": None,           # 当前有效
        "txn_time": now
    })
    
    # 3. 链接新旧
    self.update_node(old_node_id, 
                     {"superseded_by": new_node["id"]})
    self.add_edge(old_node_id, new_node["id"], 
                  relation="superseded_by")
    return new_node

def query_valid_at(self, timestamp):
    """查询某时间点上所有有效的节点"""
    return self.search(nodes_where={
        "valid_from": {"<=": timestamp},
        "valid_to": {">": timestamp}  # null = 正无穷
    })

实际效果：

查询："用户现在用什么语言？"
- 2026-01: valid_from=2025-06, valid_to=2026-03, content="Python"  ❌ 已过期
- 2026-06: valid_from=2026-03, valid_to=null,      content="Rust"   ✅ 当前有效

不再有"旧信息和正确信息打架"的问题。Bi-temporal 只需要 3 列 + ~80 行代码，但它解决的是 Agent 记忆中最根本的时序一致性问题。Zep/Graphiti 在 2025 年就提出了双时序 KG，但依赖 Neo4j。我们证明了这个模式在 SQLite 上同样可行——不需要图数据库。

方案 3：有意识的遗忘

最被低估的记忆操作是遗忘。人类的记忆不是全量保留——我们每天睡 8 小时，大脑在 REM 睡眠中主动修剪、整合、遗忘。2026 年 6 月的 Sleep-Time Compute 研究表明，并发执行的记忆整合能减少 Agent 49% 的步数。

在工程实践中，遗忘不是 DELETE，而是分层降级：

def consolidate_memory(self):
    """定期整理：高 Q 值保留，低 Q 值降级"""
    nodes = self.get_all_nodes()
    
    for node in nodes:
        q = node.get("q_value", 0.0)
        last_access = node.get("last_accessed", datetime.min)
        days_since = (datetime.now() - last_access).days
        
        if q < -0.5 and days_since > 30:
            # Q 值持续为负 + 长期未访问 → 归档
            self.archive(node["id"])
        elif q < 0.0 and days_since > 7:
            # Q 值偏低 + 短期未访问 → 降低检索权重
            self.update_node(node["id"], 
                           {"retrieval_boost": 0.5})
        elif q > 1.0:
            # 高质量记忆 → 提升
            self.update_node(node["id"], 
                           {"retrieval_boost": 1.5})

这和 Mem0 v3 的发现一致：他们放弃了复杂的图结构，转而用简单的 entity boosting，反而达到了 SOTA。为什么？因为结构不如信号——一个简单的 Q 值信号比复杂的图谱遍历更有效地区分了有用记忆和噪声。

四、重新理解 Mem0 v3 的"反图"选择

Mem0 v3 是 2026 年最让我意外的发布。他们移除了图结构，用轻量级的 entity linking + vector search，反而在多个 benchmark 上超越了使用知识图谱的方案。

这听起来像是在开倒车，但仔细分析后我认为这是正确的权衡：

  1. 图的维护成本远高于收益：entity extraction + relation extraction + graph update 每步都消耗 LLM 调用
  2. Agent 不需要全局一致性：Agent 需要的是"当前任务最相关的几个事实"，不是"完美的知识图谱"
  3. Q 值 + 向量 > 纯图遍历：当你能用 Q 值标记记忆质量时，简单的 top-k 向量检索就已经能绕过大部分多跳推理的需求

但这不代表图记忆没有价值。我的判断是：

图结构的作用不是提升检索，而是提升推理。

当 Agent 需要"为什么"和"怎么到"的推理链时，图结构提供了因果路径。当 Agent 只需要"是什么"时，向量 + Q 值就够了。这就是为什么 agent-memory-graph 同时支持两种模式：快速路径（向量 + BM25 + Q 值）和深度路径（PPR 图遍历 + 社区检测）。不是二选一。

五、实操建议：你的 Agent 记忆系统该怎么做？

如果你正在构建 Agent 记忆系统，以下是我经过 170+ 次 TDD 循环后的建议：

1. 先解决"质量"再解决"数量"

大多数团队的第一反应是"存更多记忆"。错了。先给每条记忆一个 Q 值，让 Agent 学会区分信噪比。这是 ROI 最高的改进——3 列（q_value, last_accessed, access_count）就能实现。

2. Bi-temporal 是必需品，不是奢侈品

3 列（valid_from, valid_to, txn_time）+ 一个 supersede() 方法，总共不到 100 行代码。它解决的问题——信息过期和冲突——在任何实际场景中都会出现。

3. 检索不是 ranking，是 routing

不要试图用一个排序公式解决所有查询。不同的任务需要不同的检索策略：

  • 事实查询 → BM25 + 向量
  • 多跳推理 → PPR 图遍历
  • 时间查询 → bi-temporal filter
  • 决策支持 → Q 值 + 向量

一个 routing 层的成本远低于一个"通用排序"的追求。

4. 评估要测 Agency，不只是 Recall

如果你的 benchmark 只是"Agent 能否找到记忆"，你永远在优化错误的目标。设计端到端的任务评估：Agent 拿到记忆后，能否完成任务？

结语：从信息检索到认知架构

Recall ≠ Agency 这个发现，本质上是在说：记忆不是一个存储问题，而是一个认知问题。

你需要的不是更大的仓库，而是更好的"记忆管理系统"——知道哪些记忆可信（Q 值），知道哪些记忆还有效（bi-temporal），知道哪些记忆该遗忘（consolidation）。

这不是终点。MemoryArena 的 oracle 行告诉我们，即使解决了记忆质量、时效性和遗忘问题，Agent 的 agency 上限仍然受制于 LLM 本身的多步推理能力。72% 不是记忆系统能解决的。

但 72% 比 45% 好太多了。把记忆系统从"无差别召回"升级到"质量感知检索"，是我们在通往可靠 Agent 道路上最实际的一步。
