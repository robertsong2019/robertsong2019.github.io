AI Agent 的上下文经济学：用代码图谱省下 36.5% 的 Token

2026-05-21 · AI Agent codegraph token 优化

当代码库超过 10 万行，AI Agent 面临一个核心矛盾：它需要理解整个项目的结构才能做出正确的修改，但把所有文件塞进上下文窗口既贵又慢。上周在 agent-context-store 项目上的实测给出了一个有说服力的答案：用代码图谱替代原始文件，上下文消耗直接砍掉 36.5%。

问题：上下文窗口是 Agent 的硬通货

我们用 Codex (gpt-5.5) 做 autoresearch 已经 65 天零回滚了。随着项目膨胀，一个趋势越来越明显——Agent 的能力上限不是模型智商，而是它能"看到"多少上下文。

agent-context-store 有 419 个代码节点、416 条依赖边。传统做法是把相关源文件全部塞进 prompt：

// 传统方式：Agent 收到的 prompt 片段
// File: src/store.ts (247 lines)
// File: src/types.ts (89 lines)
// File: src/index.ts (156 lines)
// File: src/transform.ts (312 lines)
// ... 还有 20+ 个文件
// Total: ~92,779 tokens

9 万 tokens 只为了理解项目结构。Agent 真正需要的不是每一行代码，而是模块之间的调用关系和数据流向。

方案：CodeGraph 把代码变成图谱

codegraph 的核心思路：静态分析代码，提取函数/类/模块的调用关系，生成结构化的图谱索引。Agent 拿到的不再是源文件全文，而是一张"地图"。

# 安装和索引
pip install codegraph
cd your-project
codegraph build  # 生成 .codegraph/ 目录

# 对于 C 项目（我们测了 wget2）
# 264 files, 3971 nodes, 3707 edges
# 索引耗时: 14.7 秒

索引完成后，Agent 的 prompt 从原始代码变成了图谱摘要：

# CodeGraph 方式：Agent 收到的结构化摘要
# Nodes: 419 (functions: 287, classes: 89, modules: 43)
# Edges: 416
#
# Key call chains:
#   index.ts → store.createStore() → transform.normalize()
#   store.ts → types.StoreConfig → cache.invalidate()
#   ...
#
# Affected modules for task "add TTL support":
#   src/store.ts (write), src/types.ts (read), src/cache.ts (write)
#
# Total: ~58,913 tokens (↓36.5%)

36.5% 的 token 节省意味着什么？以 gpt-5.5 的定价计算，每次 Agent 调用省下的成本大约是 $0.05。一个 autoresearch 会话平均 40-60 次 Agent 调用，一个功能从调研到完成能省 $2-3。看起来不多，但连续跑 65 天就是 $130-200 的纯节省。更重要的是省下来的 token 空间可以塞更详细的指令和更多示例，直接提升输出质量。

踩坑：权限问题会让索引静默失败

codegraph 把索引写在项目根目录的 .codegraph/ 下。如果 Agent 以非 root 用户运行（比如我们创建的 coder 用户），索引文件可能只对创建者可读，导致后续 Agent 调用读不到图谱，退回原始文件模式——而且没有任何报错。

# 修复：确保所有用户可读写索引
chmod -R a+rw .codegraph/

# 更好的做法：在 CI/CD 中统一构建索引
# .github/workflows/codegraph.yml
- name: Build codegraph
  run: |
    pip install codegraph
    codegraph build
    chmod -R a+rw .codegraph/

这个问题花了我两小时排查。Agent 不会告诉你"我读不到图谱所以用了原始文件"，它只是静默地多花了 36% 的 token。这种静默退化是 Agent 系统中最危险的 bug 类型。

我的观点：图谱 > RAG > 全文

对于 AI Agent 理解代码库，我认为有三个层次：

  1. 全文塞入 — 最简单，但上下文窗口是硬上限。超过 100k tokens 的项目直接不可行。
  2. RAG 检索 — 按语义相似度检索代码片段。问题是丢失了结构信息：Agent 看到了函数体，但不知道谁调用了它、它依赖谁。
  3. 代码图谱 — 保留结构，压缩细节。Agent 看到的是"骨架"而非"血肉"，但做架构级决策时骨架才是关键。

这三者不是互斥的。理想的 Agent 编码流程应该是：先用图谱定位影响范围，再用 RAG 检索具体实现，最后只把需要修改的文件全文放入上下文。这种三层策略能把大型项目的上下文消耗从 O(n) 降到 O(log n)。

可操作建议

  • 项目超过 50 个文件就该建图谱。codegraph 的索引成本很低（万行级项目 < 15 秒），但省下的 token 是持续的。
  • 把 codegraph build 加入 CI，确保每次代码变更后图谱自动更新。过时的图谱比没有图谱更危险。
  • 监控 Agent 的实际 token 消耗。如果某个会话突然 token 飙升，大概率是图谱索引失效了。
  • 给 Agent 明确的 fallback 指令：读不到图谱时报错，而不是静默退回全文模式。静默退化是 Agent 系统的隐形杀手。
代码图谱的本质是用空间换时间：花一次索引成本，换取每次 Agent 调用的上下文压缩。在 Agent 自主编程成为常态的 2026 年，这不是优化，是基础设施。

本文基于 2026-05-20 的 codegraph v0.7.11 实测数据。项目数据来自 agent-context-store 和 wget2。
