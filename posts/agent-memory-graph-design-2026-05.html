给 Agent 造记忆：从 CRUD 到记忆图谱的 72 小时

2026-05-26 · 罗嵩 · agent-memory-graph 实战设计笔记

过去三天，我在做一个实验：给 AI Agent 构建一个真正可用的记忆系统。不是向量数据库那种"存进去、搜出来"的扁平结构，而是一个带权重的知识图谱——节点有衰减、边有关系、支持邻域提取和聚合查询。

这篇文章记录这个过程中的关键设计决策，以及为什么"先写测试再写代码"在 Agent 基础设施中比你想的更重要。

问题：为什么 Agent 需要"图"记忆？

大多数 Agent 记忆方案是 KV 存储 + 向量检索。这对于"找到相关文档"够用，但对于"理解上下文关系"不够。

考虑这个场景：Agent 记住了"张三是工程师"和"李四是张三的经理"。扁平存储里，这是两条独立记录。但当用户问"李四的下属是什么角色"，Agent 需要沿着边遍历——这在图结构里是一行代码，在扁平存储里是 N 次查询 + 应用层拼接。

更关键的是衰减。人的记忆会遗忘，Agent 也应该：长期不用的记忆权重降低，最终被清理。这需要图级别的操作——删掉节点的同时清理所有关联的边。

设计原则：三层 API 演化

agent-memory-graph 的 API 不是一开始就设计好的，而是在 72 小时内通过实验循环逐步演化。这是 autoresearch 方法论的核心：每个循环加一个小功能，跑通测试，确认保留。

第一层：核心图操作（35 → 45 tests）

最初只有 add()、recall()、link()、neighbors()。然后发现一个明显的缺口：没有单节点 CRUD。

# 之前：要获取一个节点，得用关键词匹配
results = graph.recall("张三")  # 返回模糊匹配列表

# 之后：直接 ID 查找
node = graph.get_node("user_zhangsan")
graph.update_node("user_zhangsan", weight=0.9)  # 部分更新
graph.delete_node("user_zhangsan")  # 自动清理关联边

delete_node() 的关键设计：删除节点时必须清理所有关联边（入边+出边）。这不是可选的——如果遗留悬空边，后续遍历会崩溃。

第二层：批量操作（45 → 53 tests）

Agent 处理一段对话后，可能需要一次性写入 20 条记忆。一条一条 add() 意味着 20 次事务提交。

# 批量写入：单事务，O(N) 插入
items = [
    {"label": "会议讨论了Q3路线图", "kind": "event", "tags": ["meeting", "planning"]},
    {"label": "张三负责后端重构", "kind": "task", "tags": ["backend"]},
    {"label": "李四批准了预算", "kind": "event", "tags": ["finance"]},
]
ids = graph.add_many(items)

# 批量建边：一次调用
pairs = [
    {"source": ids[0], "target": ids[1], "relation": "discussed"},
    {"source": ids[0], "target": ids[2], "relation": "approved_by"},
]
graph.link_many(pairs)
设计洞察：add_many() 用单事务而不是 N 个事务。在 SQLite 中，这意味着从 N 次 fsync 降到 1 次。对于 Agent 处理长对话后批量写入的场景，性能差异是数量级的。

第三层：邻域提取与分析（53 → 110 tests）

这是最有意思的部分。Agent 的上下文窗口有限——不能把整个记忆图谱塞进去。需要一种方式提取"与当前话题相关的局部子图"。

# 提取以某节点为中心、深度为2的邻域子图
sub = graph.subgraph("meeting_q3", depth=2)
# 返回格式兼容 export_json()，可直接导入另一个图

# 分析：所有"event"类节点的平均权重
avg = graph.aggregate("event", "weight", "avg")

# 清理：权重低于 0.3 的噪声节点 + 孤立边
result = graph.prune(min_weight=0.3)
print(f"清理了 {result['nodes_removed']} 个节点，{result['edges_removed']} 条边")

subgraph() 用 BFS 遍历，输出格式与 export_json() 完全兼容。这意味着你可以从一个 Agent 的记忆中提取子图，直接导入另一个 Agent——记忆迁移变成了一个 API 调用。

三个可操作建议

1. 记忆系统必须有衰减机制。
不衰减的记忆库会膨胀到不可用。decay_all() + prune() 的组合让记忆自然淘汰——权重低于阈值的节点被自动清理。这比手动管理更可靠，也更接近人类记忆的工作方式。
2. API 演化优于 API 设计。
不要试图一次设计完美的记忆 API。从最小可用的 4-5 个方法开始（add/recall/link/neighbors），然后在使用中发现缺口。subgraph() 不是我想出来的——是在使用中发现"Agent 需要局部上下文"后才加的。
3. 批量操作是性能关键路径。
Agent 处理一段对话可能产生 20-50 条记忆。如果没有 add_many()，每次写入都是一次完整的事务提交。单事务批量插入在 SQLite 中快 10-50 倍。这不是优化，是必需品。
