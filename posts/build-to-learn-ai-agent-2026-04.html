Build to Learn：为什么你需要的不是又一个 AI 框架，而是七个单文件

2026-04-09 · AI Agent 教育 快速原型

过去一个月，我写了九个 AI Agent 相关的项目。从任务编排到边缘运行时，从信任网络到 Prompt 编排引擎。回头看，最有价值的不是任何一个项目本身，而是一个越来越清晰的认知：AI Agent 的学习曲线问题，不是复杂性不够，而是复杂性太早。

框架疲劳症

打开 GitHub，搜索 "AI Agent framework"，你会得到上千个结果。每个都声称自己是"最简单的方式"来构建 Agent。但当你 clone 下来，面对的是：

  • 15 个依赖包
  • 3 层抽象的基类
  • 一个还没写完的插件系统
  • README 里许诺的功能比实际代码多 3 倍

这就是我说的"框架疲劳"——你还没学会走路，框架已经帮你造好了轮椅。

Tiny Agent Workshop 的实验

于是我做了一个实验：Tiny Agent Workshop。七个单文件，每个实现一个 Agent 核心模式，零依赖，每个文件不超过 200 行。

模式核心概念代码量
ReAct推理-行动循环~150 行
ToolCall工具调用协议~120 行
Memory短期/长期记忆~160 行
Router意图路由分发~130 行
Guardrail安全护栏~100 行
Chain任务链编排~140 行
EdgeAgent边缘设备推理~180 行

七个小文件，覆盖了一个 Agent 从"能思考"到"能安全地在边缘设备上运行"的完整链路。

为什么单文件有效

这不是极简主义的审美偏好。单文件有三个真实的技术优势：

1. 认知负载的上限

人脑同时能处理的变量大约是 7±2 个。一个 200 行的文件，你能在一次滚动中看到全貌。你不需要在 15 个文件之间跳转来理解"消息是怎么从 A 传到 B 的"。

2. 没有抽象隐藏 bug

框架最大的问题不是性能，而是调试时你不知道问题在哪一层。是基类的序列化逻辑？是中间件的消息转换？还是你自己的业务代码？单文件消除了这个问题——所有逻辑都看得见。

3. 可以真的"运行起来"

一个好的教学项目，必须在 30 秒内能跑起来。

node react-agent.js，你马上看到一个 Agent 在终端里思考、行动、观察。不需要配置 API key，不需要安装依赖，不需要读 50 行的快速开始指南。它就是跑起来了。

从单文件到生产系统

当然，单文件不是终点。我在 Edge Agent Runtime 项目中把类似的概念扩展到了生产级别——31 个测试，模块化设计，支持 ONNX 推理。但关键区别是：当我开始写生产代码时，我已经在单文件中踩过了所有核心坑。

这就是 "Build to Learn" 的方法论：

  1. 单文件原型 — 理解核心机制，暴露所有细节
  2. 测试驱动的扩展 — 把理解固化成测试用例
  3. 渐进式重构 — 只在真正需要时引入抽象

而不是反过来——先选框架，再学概念。

快速原型的真正速度

这几个月的开发还验证了一件事：AI 辅助编程最大的加速不在写代码，而在缩短"理解"到"验证"的回路。

传统流程：读文档 → 理解概念 → 写代码 → 调试 → 终于看到结果。可能 2-3 天。

AI 辅助流程：描述意图 → AI 生成单文件原型 → 运行验证 → 理解机制。可能 2-3 小时。

Edge Agent Mesh 就是这样诞生的。一个下午，从"边缘设备应该能自组织"这个模糊想法，到一个有核心协议、内存模型、通信层的可运行原型。不是因为我特别快，而是因为 AI 把"理解"和"实现"之间的鸿沟填平了。

给想入门的人

如果你今天想开始学 AI Agent 开发：
不要从 LangChain 开始。不要从任何框架开始。

打开一个空文件，写一个 50 行的循环：接收输入 → 调用 LLM → 解析输出 → 执行动作 → 把结果喂回去。

这就是 Agent。剩下的都是优化和工程化。

优化和工程化很重要，但它们应该在理解之后，而不是之前。

七个单文件不是终点，是起点。当你能把 ReAct 循环、工具调用、记忆系统各用 200 行讲清楚时，你已经比 90% 的"用过框架"的人理解得更深了。

这就是 Build to Learn——不是为了造轮子，而是为了理解轮子为什么是圆的。
