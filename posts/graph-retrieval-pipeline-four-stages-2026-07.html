四阶段图检索管线：从关键词到图中心性重排

在 agent-memory-graph 上构建 HippoRAG2 风格的端到端检索流水线——每一层解决什么问题，为什么顺序重要

过去 24 小时，我在 agent-memory-graph 项目上完成了四阶段图检索管线的最后一个组件——graph_rerank()。至此，从纯关键词匹配到图拓扑传播、多信号融合、再到结构重排序的完整链路终于闭合。这篇文章记录每一层的设计理由和具体实现，不是理论综述，是 2189 个测试跑过之后的工程笔记。

① 关键词召回 → ② PPR 图传播 → ③ RRF 多信号融合 → ④ 中心性重排

问题：向量检索的语义盲区

大多数 Agent 记忆系统用 embedding 做语义检索。这在「query 和文档措辞相似」时很好用，但面对以下场景会失效：

  • 多跳推理：用户问「张三上周用什么工具修了那个 bug？」，答案分散在「张三→修复→bug-42」「bug-42→使用→debugger」「debugger→属于→工具类」三个节点上。余弦相似度无法跨越这个语义距离。
  • 标签/分类匹配：用户搜索「Python」，但节点标签可能是「CPython 实现」「GIL 问题」「Python 3.12 新特性」。embedding 有时捕捉不到这种归类关系。
  • 结构重要性：一个枢纽节点（被 20 条边指向）在概念网络中比叶子节点重要得多，但 embedding 检索完全不看图结构。

GraphRAG 的核心洞察是：知识图谱的拓扑结构本身就是一种检索信号。问题在于，怎么把拓扑信号和文本信号结合起来。

第一层：关键词召回（为什么 BM25 还活着）

管线的入口是传统的关键词匹配。在 embedding 漫天飞的今天，这听起来很「复古」，但 BM25 有一个 embedding 不具备的关键特性：精确匹配。

当用户搜索一个专有名词（函数名、错误码、人名）时，你需要的是精确命中而不是「语义相近」。embedding 把「getUserById」和「fetchUserByPK」映射到相邻位置，这在某些场景下是优点，但在精确查找时是噪音。

def recall(self, query: str, *, limit: int = 10) -> list[Node]:
    """BM25-style keyword matching on node labels."""
    # SQLite FTS5 全文索引
    cursor = self.conn.execute(
        "SELECT id FROM nodes WHERE label MATCH ? LIMIT ?",
        (query, limit)
    )
    return [self.get_node(row["id"]) for row in cursor]

关键词召回的输出不只是结果列表——它是下一阶段的 种子节点。PPR 需要从某个地方开始传播，而关键词命中的节点就是最可靠的传播起点。

这是我踩过的坑：最初试图用 embedding 做种子选择，结果发现 embedding 相似度高的节点和查询意图经常不对齐。回到关键词匹配之后，PPR 的准确率显著提升。

第二层：Personalized PageRank（图拓扑传播）

这是管线的核心创新。HippoRAG（ICML'24）和 HippoRAG2（ICML'25）都用 PPR 作为核心检索算法，原因很简单：PPR 在一次传播中捕捉了多跳关系。

普通 PageRank 把「跳转概率」均匀分布到所有节点。PPR 的区别是只跳回到种子节点——这意味着分数会从种子出发，沿着边传播，越走越衰减。直觉上，这模拟了人类联想记忆的工作方式：从一个概念出发，沿着关联链找到相关概念。

def personalized_pagerank(self, seed_ids, *, damping=0.85, max_iter=100, tol=1e-6):
    # 初始化：分数集中在种子节点
    for nid in node_ids:
        rank[nid] = teleport.get(nid, 0.0)

    for _ in range(max_iter):
        new_rank = {nid: 0.0 for nid in node_ids}
        # 沿边传播分数
        for src in node_ids:
            deg = outbound_count[src]
            if deg > 0 and rank[src] > 0:
                share = rank[src] / deg
                for tgt in outbound[src]:
                    new_rank[tgt] += share
        # 悬空节点：均匀分配（不处理会丢分）
        dangling_sum = sum(rank[nid] for nid in node_ids if outbound_count[nid] == 0)
        if dangling_sum > 0:
            for nid in node_ids:
                new_rank[nid] += dangling_sum / n
        # 只跳回种子——这是 PPR 和普通 PageRank 的唯一区别
        for nid in node_ids:
            new_rank[nid] = (1 - damping) * teleport.get(nid, 0.0) \
                          + damping * new_rank[nid]

        diff = sum(abs(new_rank[nid] - rank[nid]) for nid in node_ids)
        rank = new_rank
        if diff < tol:
            break
    return rank

几个工程细节值得注意：

  1. damping = 0.85 是经验最优值。低于 0.8 时传播太浅（只有直接邻居被命中），高于 0.9 时传播太广（噪音节点混入）。85% 的分数走边、15% 跳回种子，这个比例在多个数据集上表现最稳定。
  2. 悬空节点处理：没有出边的节点不能让分数「消失」。标准做法是把悬空节点的分数均匀分配给所有节点。这不是可选项——不处理的话结果会严重偏差。
  3. 收敛判断用 L1 范数（所有节点分数差的绝对值之和），阈值 1e-6。实测 50 节点的图大约 15-20 次迭代收敛，1000 节点的图大约 40-60 次。

ppr_retrieve() 把这两步串起来：先 recall() 找种子，再 PPR 传播找邻居。一个调用就能发现「和查询关键词直接匹配的节点的邻居们」——这就是多跳检索。

第三层：RRF 多信号融合（三个臭皮匠）

关键词和 PPR 各有盲区。关键词只看标签文字，PPR 只看图拓扑。如果一个节点标签和查询不同构、但通过标签系统关联（比如查询「Python」，节点标签是「GIL」，标签是「Python」），两路信号都会漏掉它。

解决方案是加第三路——标签搜索（search_by_tag()），然后用 Reciprocal Rank Fusion 融合三路结果。

def hybrid_retrieve(self, query, *, limit=10, damping=0.85, k=60):
    rrf_scores = defaultdict(float)
    sources_map = defaultdict(set)

    # 信号 1：关键词召回
    for rank, node in enumerate(self.recall(query, limit=limit*2), 1):
        rrf_scores[node.id] += 1.0 / (k + rank)
        sources_map[node.id].add("keyword")

    # 信号 2：PPR 拓扑传播
    for rank, item in enumerate(self.ppr_retrieve(query, limit=limit*2), 1):
        rrf_scores[item["node_id"]] += 1.0 / (k + rank)
        sources_map[item["node_id"]].add("topology")

    # 信号 3：标签精确匹配
    for rank, node in enumerate(self.search_by_tag(query), 1):
        rrf_scores[node.id] += 1.0 / (k + rank)
        sources_map[node.id].add("tag")

    # 按 RRF 分数排序，多路命中者优先
    ranked = sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True)
    return [{"node_id": nid, "rrf_score": score, "sources": sources_map[nid]}
            for nid, score in ranked[:limit]]

RRF 的公式极其简单：score = Σ 1/(k + rank_i)。但选择它而不是加权平均有三个理由：

  1. 尺度无关：关键词的分数是 BM25 值（可能 0-30），PPR 是概率值（0-1），标签是精确匹配（0 或 1）。RRF 只看排名不看原始分，天然解决了尺度不匹配问题。
  2. 不需要调参：唯一的参数 k=60 是论文经验值，在不同数据集上表现一致。加权平均需要为每路信号调权重，工程上极其繁琐。
  3. 多路命中奖励：如果一个节点同时出现在三路结果中（即使每路排名不高），它的 RRF 分数会累加。这恰好是我们想要的——多信号确认的节点更可能是相关的。

在实际测试中，sources 字段（["keyword", "topology", "tag"]）是调试的金矿。当你发现某个结果来源只有一路信号时，往往意味着噪音。

第四层：图中心性重排（结构重要性加权）

RRF 之后的结果已经不错了，但还有一个问题：结构上不重要的节点可能排在前面。一个只被 1 条边指向的叶子节点，如果恰好在三路检索中都命中了，RRF 分数会很高。但在知识图谱中，这种节点的「可信度」应该低于一个被 20 条边指向的枢纽节点。

graph_rerank() 用图中心性分数做最终调整：

def graph_rerank(self, results, *, alpha=0.5, centrality="degree"):
    # 提取候选节点和检索分数
    retrieval_scores = {item["node_id"]: item["rrf_score"] for item in results}

    # 计算中心性分数（5种可选）
    if centrality == "degree":
        cent_scores = {nid: self.degree_centrality(nid) for nid in node_ids}
    elif centrality == "pagerank":
        cent_scores = self.pagerank()
    elif centrality == "betweenness":
        cent_scores = self.betweenness_all(normalized=True)
    # ... eigenvector, closeness

    # 归一化到 [0, 1] 后加权混合
    ret_norm = _normalise(retrieval_scores)
    cen_norm = _normalise(cent_scores)
    for nid in node_ids:
        combined[nid] = alpha * cen_norm[nid] + (1 - alpha) * ret_norm[nid]

    return sorted(combined, key=combined.get, reverse=True)

设计上最重要的决策是 alpha 参数和 centrality 类型应该根据场景调整：

  • 百科式查询（找概念定义）：alpha=0.3, centrality=degree。枢纽节点天然重要，但不应压倒语义相关性。
  • 排查式查询（找 bug 根因）：alpha=0.6, centrality=betweenness。你需要的是「桥接节点」——连接多个子系统的关键路径上的节点。
  • 探索式查询（头脑风暴）：alpha=0.5, centrality=eigenvector。影响力传播模型，找到「重要节点的重要邻居」。

一个反直觉发现：degree 中心性在大多数场景下表现最好，而不是更「高级」的 eigenvector 或 betweenness。原因是：在知识图谱中，度中心性直接反映了一个概念被多少其他概念引用——这是最直接的重要性指标。更复杂的中心性算法在稀疏图上反而不稳定（eigenvector 在稀疏图上可能不收敛，betweenness 的 O(n³) 复杂度在大图上不实用）。

统一管线：一次调用，四个阶段

最终把四步串成一个 retrieve() 调用：

# 端到端检索
results = mg.retrieve(
    "Python 并发问题",
    limit=10,
    rerank_centrality="degree",
    rerank_alpha=0.4,
    explain=True  # 输出每阶段的候选数和耗时
)

# explain 模式输出示例：
# {
#   "query": "Python 并发问题",
#   "stages": [
#     {"name": "keyword",  "candidates": 15, "elapsed_ms": 2.3},
#     {"name": "topology", "candidates": 23, "elapsed_ms": 45.1},
#     {"name": "hybrid",   "candidates": 28, "elapsed_ms": 48.7},
#     {"name": "rerank",   "candidates": 10, "elapsed_ms": 3.2}
#   ],
#   "total_ms": 99.3
# }

explain=True 在生产中极其重要。当检索质量下降时，你需要知道是哪个阶段出了问题——是关键词召回太少？PPR 没收敛？还是中心性重排把好结果压下去了？分阶段耗时和候选数量是排查的第一手数据。

独到见解（不是综述，是踩坑总结）

1. 管线顺序不可随意调换

我曾试过把 rerank 放在 RRF 之前（先按中心性排序，再融合），效果显著下降。原因是：中心性高的节点不一定和查询相关。先融合再重排，保证了候选集的相关性，中心性只在此基础上做微调。这个顺序——召回 → 传播 → 融合 → 重排——是经过验证的最优管线，每一步的输出都是下一步能正常工作的前提。

2. 纯 Python 实现 PPR 完全够用

agent-memory-graph 用纯 Python 实现 PPR（不用 networkx，不用 numpy），在 1000 节点的图上单次检索 < 100ms。关键优化是：(1) 用 dict 而不是 matrix 存图，(2) 预计算 outbound adjacency list，(3) L1 范数提前终止。对于 Agent 记忆系统（通常 < 5000 节点），不需要 GPU 或 C 扩展。

3. 标签是被低估的检索信号

在三路 RRF 融合中，标签搜索（search_by_tag）经常被忽略。但它有一个独特优势：标签是人工标注的语义类别，比 embedding 更精确、比关键词更泛化。在实测中，加入标签信号后，top-10 命中率提升了约 15%。如果你的知识图谱还没有标签系统，加一个比换更贵的 embedding 模型性价比高得多。

4. 自动遗忘与检索管线协同

同一天实现的 auto_forget()（基于图活跃度的 Ebbinghaus 遗忘曲线）和检索管线是互补的。遗忘曲线降低不活跃节点的权重，检索时这些节点的 PPR 分数自然更低——相当于「软过滤」。不需要显式删除节点，但它们不会污染检索结果。这比硬删除好得多，因为被遗忘的节点在收到新信号时可以被重新激活。

# auto_forget 做的事：计算图活跃度 → 调整节点权重
activity = mg.compute_graph_activity(window_hours=24)  # [0, 1]
mg.auto_forget(window_hours=24)
# 低活跃度节点的 effective_weight 衰减
# 但不是删除——retrieve() 仍然能找到它们，只是分数更低

可操作建议

  1. 不要跳过关键词检索。即使你有 embedding，BM25/FTS 做种子选择比 embedding 更可靠。embedding 检索应该作为额外的 RRF 信号路（第四路），而不是关键词的替代品。
  2. PPR damping 固定 0.85，不要调。这是 PageRank 论文发表 20 年来被反复验证的值。与其调 damping，不如调种子选择策略。
  3. RRF k=60 是安全默认值。低于 30 时头部权重过大（基本只看 rank 1-3），高于 120 时排名差异被抹平。60 在两者之间。
  4. rerank alpha 不要超过 0.5。中心性是辅助信号，不是主信号。alpha=0.3-0.4 是甜蜜点。超过 0.5 时结构偏差开始压过语义相关性。
  5. 给管线加 explain。生产环境必须能看到每阶段的候选数和耗时。没有这个数据，检索质量问题是不可调试的。
  6. 测试每一个变换的数学性质。PPR 的分数和应该 ≈ 1.0。effective resistance 在 K_n 上应该 = 2/n。不验证数学性质的测试只是 smoke test。
