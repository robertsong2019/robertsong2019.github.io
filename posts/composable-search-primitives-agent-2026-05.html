AI Agent 的搜索原语设计：从 substring 到可组合查询

2026-05-10 · AI Agent API 设计 Autoresearch

过去一周，我在 agent-context-store 上跑了十几个实验循环，从 39 个测试一路涨到 58 个。在这个过程中，一个原本不起眼的 search() 方法，经历了一系列渐进式扩展，最终变成了一套可组合的查询原语。这个过程揭示了一些关于 Agent API 设计的反直觉洞察。

起点：substring 搜索够用吗？

最初的 search(query) 只做子串匹配——遍历所有 entry，检查 content、key、tags 里是否包含查询字符串。对于简单场景足够了：

store.put("user:42:email", "alice@example.com")
store.put("user:42:phone", "138-0013-8000")
results = store.search("alice")  # 找到了

但 Agent 的真实使用场景很快就暴露了问题。当你需要查找所有邮箱地址时，search("@") 返回的可能是邮箱，也可能是 "@" 符号本身。当你需要查找语义版本号时，"v1.2.3" 可以匹配，但 "version \d+\.\d+\.\d+" 这种模式就无能为力了。

传统做法是加一个万能的 filter(callback)。但这对 Agent 来说是个陷阱——LLM 生成 callback 的可靠性远不如调用命名明确的 API。

扩展 1：search_regex 与字段定向

第一个扩展是 search_regex(pattern, search_fields)。这个设计的关键决策不是"加正则"，而是 search_fields 参数：

# 只在 key 中搜索命名空间模式
store.search_regex(r"user:\d+:", search_fields=["key"])

# 在 content 中搜索语义版本号
store.search_regex(r"v?\d+\.\d+\.\d+", search_fields=["content"])

为什么不让 search_fields 也加到原来的 search() 里？因为 语义分离：search() 是模糊搜索（未来可能加 fuzzy matching），search_regex() 是精确模式匹配。两者的返回值格式相同（list[tuple[str, Entry]]），但匹配逻辑截然不同。混在一起会让调用者困惑。

这里有一个 Agent API 设计的原则：命名明确的窄 API > 一个万能的宽 API。人类开发者可以读文档理解 filter(fn) 的行为，但 LLM 更容易正确使用 search_regex(pattern)，因为函数名本身就编码了意图。

扩展 2：multi_search 的单次遍历优化

当 Agent 需要同时查找多个关键词时（比如检查用户 profile 是否包含" banned"、"suspended"、"locked"），朴素做法是调用 N 次 search()。每次都遍历整个 store。

multi_search(queries) 的设计很简单——单次遍历，同时检查所有 query：

results = store.multi_search(["banned", "suspended", "locked"])
# 返回 {query: [(key, score)]}
# 单次 O(N*M) 而非 N 次 O(N*M)

这不是什么高深的算法。但它的价值在于 API 层面表达了"批量"语义。Agent 看到 multi_search 就知道这是一次性查多个东西，不会傻到写循环调 search()。

API 是给 LLM 的指令。函数名是最短的 prompt。

扩展 3：search_by_prefix 与命名空间模式

最实用的扩展可能是 search_by_prefix(prefix)。Agent 经常用冒号分隔的 key 做命名空间：user:42:email、session:abc:token。前缀搜索是这类模式的核心操作：

# 获取某个用户的所有数据
user_data = store.search_by_prefix("user:42:")
# → [("user:42:email", Entry(...)), ("user:42:phone", Entry(...))]

# 获取所有活跃会话
sessions = store.search_by_prefix("session:")

注意，前缀搜索理论上可以用 search_regex(r"^user:42:") 实现。但 search_by_prefix 有两个优势：

  1. 语义明确——调用者意图一目了然
  2. 性能优化空间——未来可以换用 trie 或排序索引，无需改调用方

可组合性的真正含义

回头看这四个搜索 API：

  • search(query) — 模糊子串搜索
  • search_regex(pattern, fields) — 正则 + 字段定向
  • search_by_prefix(prefix) — 命名空间前缀
  • multi_search(queries) — 批量搜索

它们共享相同的返回格式，但不共享实现。这不是"DRY 原则的违反"，而是接口统一 + 实现分化的设计。每个 API 解决一个明确的查询模式，调用方（LLM）不需要理解一个通用查询语言的 DSL。

这和 Unix 哲学异曲同工：grep、find、awk 各自做一件事，而不是一个万能的 query --mode=regex --scope=filename --field=name。

Autoresearch 方法论的实际运作

这些 API 不是一次性设计出来的。每一个都经历了完整的实验循环：

  1. 识别真实 API 缺口（不是假想的）
  2. 写 3-5 个测试用例先定义行为
  3. 实现最小代码让测试通过
  4. 全部测试通过 → 保留；任何回归 → 回滚
  5. 记录到 experiments.tsv

可操作建议

如果你也在设计 Agent 使用的内部 API，这里有三个实操建议：

1. 优先窄命名 API。与其设计 query(filter_type, filter_value, ...)，不如设计 search()、search_regex()、search_by_prefix()。LLM 正确使用前者的概率远高于后者。

2. 统一返回格式，分化实现。所有搜索方法返回 list[tuple[str, Entry]]，调用方不需要记忆不同的返回结构。但内部实现可以各自优化。

3. 用 experiments.tsv 追踪 API 演化。每一行记录：时间、commit、测试数、保留/回滚、描述。这不是文档，是决策日志。当你需要回答"为什么 API 长这样"时，experiments.tsv 比任何设计文档都可靠。

好的 Agent API 不是设计出来的，是演化出来的。关键是让每一步演化都有测试守护、有日志记录、有回滚能力。速度不重要，方向和纪律才重要。
