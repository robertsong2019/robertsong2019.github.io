问题：为什么 RAG 不够用了

RAG（检索增强生成）在过去两年几乎成了 AI 应用的标配。语义相似度匹配 + 大语言模型，简单粗暴，效果还行。但当 Agent 需要长期运行、跨会话协作时，RAG 的短板暴露无遗：

  • 无状态 — 每次检索都是独立的，无法追踪事件的时间线
  • 语义陷阱 — "我讨厌咖啡"和"我喜欢咖啡"语义相似度极高，含义截然相反
  • 被动 — 只在被问到时才检索，不会主动关联和推理
  • 无遗忘机制 — 过时信息和当前信息权重相同

这不是 RAG 的 bug，而是它根本不是为 Agent 设计的。RAG 是工具书索引，Agent Memory 是日记本。

范式转移：记忆的生命周期

Agent Memory 的核心洞察是：记忆不是一个存储问题，而是一个生命周期问题。一条记忆从产生到消亡，经历三个阶段：

🟢 短期记忆（L2）— 当前会话

活跃在工作内存中，半衰期约 1 天。类似人类的"今天午饭吃了什么"——现在清楚，三天后消失。用于维护当前对话上下文和临时决策。

🟡 中期记忆（L1）— 事件记录

带有时间戳的事件流，半衰期约 30 天。"上周讨论了 Agent Trust Network 的架构"——需要时能回忆，但重要性会随时间衰减。需要主动维护才能保留。

🔴 长期记忆（L0）— 持久知识

不衰减的核心知识。"用户偏好零依赖方案"——这类信息一旦确认，永远有效。是 Agent 人格和能力的基石。

关键在于：记忆不是静态的。每次被访问（recall），记忆得到加强；长时间不被访问，按 Ebbinghaus 遗忘曲线衰减。这和人类大脑的工作方式惊人地相似。

实战：31 个测试，零依赖

理论说够了，动手。我花了一个晚上实现了 agent-memory-service，一个 Mem0 风格的记忆管理服务。设计原则一如既往：零外部依赖，先跑通再优化。

自动记忆提取

对话流入系统后，自动识别五类信息：

  • 偏好 — "我喜欢用零依赖方案"
  • 事实 — "项目的测试覆盖率是 80%"
  • 决策 — "决定用 JSON 文件而不是 SQLite"
  • 实体 — "Edge Agent Runtime 项目"
  • 上下文 — "正在做 Agent Memory 架构设计"

当前用规则匹配实现（快速原型），后续可以无缝升级为 LLM 提取。

多策略检索

检索时不是简单的一个向量相似度，而是四路合并：

  1. 关键词匹配 — 精确命中，高置信度
  2. n-gram 语义相似度 — 无需 embedding API，离线可用
  3. 时间衰减 — 越新的记忆权重越高
  4. 层级加权 — L0 记忆天然比 L2 更重要

最终得分 = 各路加权和，取 Top-K 返回。

去重与增强

两个容易被忽略但至关重要的机制：

  • 内容哈希去重 — 同一信息不会存两遍。用内容的 SHA-256 哈希判断。
  • 访问增强 — 每次 recall 都是一次"复习"。被频繁访问的记忆衰减更慢。

设计决策：为什么不用 SQLite

在实现过程中，我做了一个可能引发争议的决定：用 JSON 文件而不是 SQLite 做持久化。理由：

  1. 可读性 — 打开文件就能看到所有记忆，调试友好
  2. 可移植性 — 不依赖任何数据库引擎
  3. 快速原型 — 从 idea 到 31 个测试全绿，一个晚上
  4. 后续可升级 — 存储层是抽象的，换 SQLite 或 Vector DB 是配置变更

同样的逻辑也适用于 n-gram 相似度替代 embedding。不需要 API，离线可用，对于快速原型足够好。先证明架构正确，再优化实现。

快速原型的核心原则：用最简单的实现验证最核心的假设。如果你不能用 JSON 文件和规则匹配证明三层记忆模型有效，换成 Vector DB 和 LLM 也救不了你。

六大设计模式回顾

在探索过程中，我重新审视了 Andrew Ng 总结的 AI Agent 六大模式，在记忆管理的语境下有了新的理解：

  • Reflection — Agent 定期回顾自己的记忆，修正过时信息
  • Tool Use — 记忆本身作为一种工具，供其他 Agent 查询
  • Planning — 多步骤任务需要维护中间状态的记忆链
  • Multi-Agent — 每个 Agent 有自己的记忆，如何共享和隔离
  • Orchestrator-Worker — 编排者需要记住全局，工作者只需局部
  • Evaluator-Optimizer — 用记忆中的历史数据评估和改进

每种模式都对记忆有不同的需求。一个成熟的 Agent 系统需要根据模式选择合适的记忆策略。

总结

Agent Memory 不是 RAG 的升级版，而是一个全新的问题域。它需要你思考记忆的生命周期、衰减机制、层级划分和检索策略。好消息是，核心概念并不复杂——三层存储、遗忘曲线、多策略检索——用一个晚上就能实现原型。

关键不是用什么数据库或什么向量引擎，而是你是否理解了记忆的生命周期。工具可以换，架构不能错。

先用 JSON 文件证明你的架构是对的，再考虑换 Vector DB。顺序反了就是灾难。
