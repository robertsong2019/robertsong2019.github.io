从单体到模块：Agent Skill 如何重塑 AI 的能力架构

2026-07-18 · 罗嵩 · Agent Skills LLM 模块化架构 SKILL.md
假设你雇了一个聪明的新人。他什么都会一点，但每次任务你都得从头到尾告诉他怎么做——先打开这个文件，然后运行这个命令，如果报错了就检查这个。你会累死，他也会。但如果你给他一本《新员工操作手册》，每个常见任务都写好了步骤、注意事项和异常处理，他就能独立完成了。Agent Skill 就是这本操作手册。2026年，它正在变成 AI 行业的标准基础设施。

一、问题：为什么单体 Agent 行不通了？

过去两年，我们习惯了把所有指令塞进一个巨大的 system prompt。一个"万能 Agent"的系统提示可能包含代码审查规则、邮件写作规范、日历管理逻辑、安全约束……堆到 60,000 tokens 是家常便饭。

这就像让一个人同时记住所有岗位的操作规程——既是程序员，又是客服，又是会计。问题显而易见：

  • 上下文膨胀：60,000 tokens 的指令开销，每个请求都得付。按 $3/1M tokens、每天 1,000 次任务算，每天白花 $180 在"自我介绍"上。
  • 指令互相干扰：正式法律文书和随意社交媒体回复的风格指令放在一起，Agent 会精神分裂。
  • 不可维护：改一条规则，可能影响十条其他规则。没有 diff，没有版本，没有依赖图。
  • 无法复用：第二个项目要"代码审查"功能？复制粘贴，祈祷别漂移。

研究者给这个现象起了个名字——"context rot"（上下文腐烂）。在 18 个主流 LLM 上的实验表明，性能随上下文增长而非线性下降，而且下降方式不可预测。

二、解法：多了一层"技能"抽象

解决方案不是更大的上下文窗口，而是一个经典的软件工程思想：关注点分离。

新的架构引入了三个层次：

┌──────────────────────────────────────────┐ │ 用户目标 (Goal) │ │ "帮我做本周的技术分享 PPT" │ └────────────────┬─────────────────────────┘ │ 语义匹配触发 ┌────────────────▼─────────────────────────┐ │ 技能层 (Skill) │ │ 编排多个工具 + 策略 + 错误处理 + 模板 │ │ "tech-share-ppt" 技能被激活 │ └────────────────┬─────────────────────────┘ │ 调用原子操作 ┌────────────────▼─────────────────────────┐ │ 工具层 (Tool) │ │ 搜索论文 · 生成大纲 · 创建幻灯片 · 通知 │ └──────────────────────────────────────────┘

关键区别在这里：工具回答一个问题；技能执行一整套流程。

一个"搜索"工具返回结果列表。但一个"深度调研"技能知道：先确定研究范围，再分多路并行搜索，然后去重、综合、结构化，最后生成带引用的报告。它不只是一个更好的搜索，它是一种完全不同的思考方式。

论文 arxiv:2602.12430 把这个区别说得很精准："工具执行并返回结果；技能重塑 agent 对任务的理解。"——它改变的不是一个动作，而是 agent 解决问题的框架。

三、渐进式披露：为什么 30 个技能只花 3000 tokens

这是整个架构中最精妙的设计。渐进式披露（Progressive Disclosure）是一种三级上下文管理协议：

层级加载内容Token 开销何时加载
Level 1 · 发现技能名 + 一句话描述~100 tokens/技能Agent 启动时
Level 2 · 激活完整 SKILL.md 指令~2,000-5,000 tokens任务匹配时
Level 3 · 引用references/ 下的领域知识按需技能执行中需要时

这意味着什么？一个部署了 30 个技能的系统，在大多数会话中只消耗 ~3,000 tokens 的 Level 1 开销。只有当某个技能被触发时，才会加载它的完整指令（Level 2）。而且技能指令中的深度领域知识（安全清单、风格指南、API 文档等）可以放在 references/ 目录里，只在需要时才读取（Level 3）。

对比：单体 system prompt 始终携带 60,000 tokens；渐进式披露日常会话只需 3,000-18,000 tokens。3-8 倍的上下文节省。每天省 $50-100。

这就像人的记忆——你知道图书馆里有这本书（Level 1），翻到目录页看到感兴趣的章节（Level 2），再具体阅读那几页（Level 3）。你不会把整个图书馆背下来。

四、SKILL.md 标准：一份文件格式如何统一行业

2025年12月18日，Anthropic 发布了 Agent Skills 开放标准。核心就是一个文件格式：

my-skill/
├── SKILL.md          # 必需：YAML frontmatter + Markdown 指令
├── scripts/          # 可选：可执行脚本
│   └── run.sh
├── references/       # 可选：领域知识，按需加载
│   └── guide.md
└── assets/           # 可选：模板、配置、示例

SKILL.md 文件本身很简单——YAML 头部提供发现所需的元数据，正文是完整的操作指令：

---
name: "weekly-report-check"
description: |
  Check team daily report completion and send reminders.
  Use when receiving lark-daily-report scheduled task.
  Triggers on: "daily report", "lark-daily-report"
---

## Instructions

When triggered, perform the following steps:
1. Fetch the list of team members...
2. Check who hasn't submitted...
3. Send a friendly reminder...

48 小时内，Microsoft 和 OpenAI 宣布支持同一格式。到 2026年3月，32 个厂商的工具全部兼容——包括 Google Gemini CLI、JetBrains Junie、AWS Kiro、GitHub Copilot、Cursor、Cline、Windsurf……

这意味着你为 Claude Code 写的技能，在 Codex CLI 和 Gemini CLI 里直接可用，无需修改。这在 AI 工具历史上是前所未有的——一个文件格式在 5 个月内实现了跨厂商互通。

社区生态的爆发更加疯狂：从 2025年12月的几千个技能，到 2026年Q2 的 800,000+。主要市场包括 SkillsMP（66K+ 精选技能）、Skills.sh（89K+ 技能）、LobeHub、ClawHub 等。

五、技能从哪来？四种获取范式

这是最有学术深度的部分。技能不是只能靠人写——研究者已经探索出四条截然不同的路径：

路径 1：人工编写——最直接也最有效

领域专家把操作流程写成 Markdown。听起来"低技术"，但效果最可控。Anthropic 的框架设计得极其平易近人——一个技能可以就是一个几十行的 Markdown 文件。Atlassian、Canva、Sentry 等公司已经用这种方式编写了生产级技能。

路径 2：强化学习——让 Agent 自己学会什么技能有用

SAGE（Skill Augmented GRPO for self-Evolution）是这条路径的代表作。它的核心创新是 Sequential Rollout：不是在孤立任务上训练，而是在相似任务链上部署 Agent。早期任务中生成的技能被保留下来，供后续任务复用。

效果？在 AppWorld 基准上，比不使用技能库的基线 提升 8.9%，同时减少 26% 的交互步数和 59% 的 token 消耗。省了一大笔钱，还做得更好。

路径 3：自主发现——Agent 探索未知软件

SEAgent 解决了一个更难的问题：面对从未见过的软件，Agent 能不能自己摸索出操作技能？它引入了"世界状态模型"来评估每一步操作的效果，以及一个"课程生成器"从软件使用手册中生成由易到难的任务序列。

在 OSWorld 的 5 个全新软件环境中，成功率从 11.3% 提升到 34.5%。——在完全陌生的环境里，从"几乎不能用"变成了"勉强能用"。

路径 4：组合合成——从零件拼装新技能

SkillComposer（arxiv:2606.32025）把技能选择形式化为一个结构化预测问题：给定任务和技能库，一次解码同时决定激活哪些技能、用几个、按什么顺序。使用受约束的自回归解码器，让三个维度自然涌现。

在 GPT-5.2-Codex 上，pass rate 提升 23.1 个百分点——达到了"黄金技能组合"上界的效果，但 token 成本更低。

一个 provocative 发现：Li (2026) 证明了多 Agent 系统通常可以"编译"为单 Agent + 技能库——token 和延迟大幅下降，准确率不变。但存在相变：技能库超过某个规模后，选择准确率急剧下降。这暗示了单个 Agent 有效管理技能数量存在根本极限。

六、四种组合模式：技能如何协作

模式 1：顺序管道

Planner ──► Worker ──► Reviewer ──► Publisher (规划) (执行) (审核) (发布)

最直观的模式。每个技能的输出是下一个技能的输入。适合内容生产、报告生成等线性工作流。错误处理通常用"责任链"策略：Stage N 失败就放宽约束重试，再不行就降级，最后才升级给人工。

模式 2：并行扇出

┌─ Research skill A ─┐ Planner ─┤─ Research skill B ─┼──► Synthesizer └─ Research skill C ─┘

当任务需要多角度调研或同时收集多源数据时，多个技能并行执行，输出最终合并。关键是使用后台执行（如 run_in_background: true），避免主循环阻塞在网络调用上。

模式 3：分形组合（技能调用技能）

技能可以调用其他技能，形成递归结构。一个"通讯桥"技能根据目标平台，委派给 telegram 子技能、飞书子技能或 Web 子技能。每个子技能封装了各自平台的怪癖（Telegram 限频、飞书 API 的各种 idiosyncrasy），而父技能提供统一接口。

模式 4：条件路由

技能内部包含决策逻辑："如果消息包含 X，走这条路径；如果包含 Y，走那条路径。"——把路由逻辑和领域逻辑放在一起，而不是散落在系统提示的各个角落。

七、安全：npm 历史正在重演

这是最需要警惕的部分。技能生态的爆发带来了熟悉的供应链安全问题。

三项独立研究（Snyk、SkillsMP 安全审计、学术分析）的数据令人不安：

  • 26.1% - 36.8% 的社区技能包含至少一个安全缺陷
  • 13.4% 包含严重级别问题
  • 76 个被确认为故意植入的恶意载荷

常见攻击模式像极了早期 npm/PyPI 的套路：

攻击类型手法类比
数据外泄读取 .env 或 SSH 密钥，POST 到攻击者服务器恶意 npm 包
反向 Shell以"远程调试"名义建立持久连接供应链后门
权限越界申请超出声明用途的工具权限过度权限 App
依赖混淆取一个常见名字（如 deploy），影子化合法技能typo-squatting

正在发展的防御措施包括：

  • 签名技能：加密签名 + 公钥钉扎，验证发布者身份
  • 能力清单：显式声明技能可调用哪些工具，运行时强制
  • 沙箱执行：技能脚本在隔离环境中运行，限制网络和文件系统访问
  • 注册表审核：自动化扫描 + 人工审核

论文 arxiv:2602.12430 提出了一个四层信任模型——从"完全可信"（第一方审计过的技能）到"完全沙箱"（未审计的社区技能），不同来源的技能获得不同级别的权限。这比简单的"信任所有/不信任所有"精细得多。

实践建议：在生产部署中，把技能来源审核当作不可协商的底线，而不是"最好这样做"。36.8% 的缺陷率意味着：不审核 ≈ 每 3 个社区技能里就有 1 个有问题。

八、循环灵活性不是免费的午餐

一个自然的冲动是给 Agent 最大灵活性——让它能回到任何步骤、重试任何工具、分支到任何节点。但 arxiv:2604.22820 的研究用完全循环子任务图（CCSG）测试了这个直觉，结果很有启发：

领域类型代表基准循环灵活性效果
需要恢复的领域ALFWorld✅ 确实有帮助——回溯能解决卡死
有先决条件链TextCraft❌ 主要增加开销——顺序不能乱跳
外部瓶颈领域Finance-Agent⚠️ 几乎无关——瓶颈在数据质量

启示：匹配工作流结构与领域特征。不要盲目追求最大灵活性。对于"必须按顺序来"的任务（编译、部署、审批流），简单的有向图比花哨的循环图更高效也更可靠。

九、可以立刻做什么

1. 如果你在构建 Agent 系统：引入"技能层"

不要把所有指令塞进一个 system prompt。按照"原子工具 → 技能 → 目标"三层重构。每个技能一个目录，包含 SKILL.md 和可选的 scripts/references/assets。你的上下文预算会立刻下降 3-8 倍。
2. 如果你在用 Claude Code / OpenClaw / Codex：优化你的技能描述

技能的 description 是它的"路由键"——决定 Agent 什么时候选它。好的描述包含三要素：
• 做什么（一句话）
• 何时触发（关键词列表）
• 何时不用（排除相邻但不适用的场景）
3. 如果你在评估社区技能：建立审核流程

至少做到：
• 阅读 SKILL.md 全文（不看代码就安装 = 裸奔）
• 检查 scripts/ 中的脚本有无网络外联
• 验证所需的工具权限是否超出声明用途
• 在沙箱中先跑一遍再上生产
4. 如果你在做技术选型：关注技能库规模的相变点

单个 Agent 管理的技能数量有极限。超过临界点，选择准确率急剧下降。如果你有 50+ 技能，考虑分域到多个 Agent（每个有更窄但更精准的技能集），而不是塞进一个巨型库。

十、展望：自进化的技能生态

把所有线索串起来，一个更大的图景浮现了：

人工编写技能 ──► 社区市场 (800K+) │ ▼ Agent 使用技能 ──► 产生执行轨迹 │ ▼ SAGE 式强化学习 ──► 自动优化技能质量 │ ▼ SEAgent 式自主发现 ──► 探索新软件，生成新技能 │ ▼ SkillComposer 式组合 ──► 拼装出新的复合能力 │ ▼ 「自进化的技能生态」

这不是科幻——每一个环节都已经有可运行的论文系统。缺的是把它们串起来的工程。而这个串联工作，正在 2026 年的某个实验室或创业公司里发生。

Agent Skill 不是又一个 buzzword。它是软件工程中"模块化"思想在 AI 领域的迟到回响。从单体到模块，从硬编码到动态加载，从无法复用到生态市场——每一次抽象层的引入都创造了数量级的增长。函数库成就了 C 语言，包管理器成就了 Node.js，App Store 成就了 iPhone。Agent Skill 正在成就下一代 AI Agent。
