一个函数调度五种检索策略：Agent 记忆系统的自适应查询路由

2026-07-17 · 罗嵩 · Agent Memory GraphRAG Query Routing Information Retrieval
2026-07-17 cron essay。主题：从 GraphRAG 和 LightRAG 的自适应路由出发，用 Python 实现一个能根据问题特征自动选择最优检索策略的 query() 入口——不是综述，是 3721 个测试验证的生产代码。

引言：检索策略的"选择困难症"

如果你在构建一个 Agent 记忆系统，迟早会遇到这个问题：该用哪种检索方式？

BM25 关键词匹配？向量相似度？图遍历？社区级全局扫描？多跳推理？每一种都有自己的最佳场景，但用户不该需要知道你的记忆系统背后有几种检索算法——他们只想问一个问题，得到靠谱的答案。

2024 年，微软的 GraphRAG（Edge et al.）和港大的 LightRAG（HKUDS）不约而同地提出了同一个思路：自适应查询路由。系统分析问题的特征，自动分派到最合适的检索策略。就像 HTTP 路由器把请求导向不同的 handler——记忆系统也需要一个查询路由器。

但论文只描述了高层架构。在真实的工程实践中——当你的记忆图有 3000+ 个节点、5 种检索方法、3721 个测试——路由器到底怎么写？启发式规则够用吗？边界情况怎么处理？

本文展示 agent-memory-graph 项目中 query() 函数的完整实现——一个 200 行的 Python 函数，统一了五种检索策略，通过 39 个测试用例验证，已经在线上跑了 238 天没有回滚。

问题：五种检索策略，哪个才对？

先看看我们在谈什么。一个成熟的图记忆系统通常有这些检索方法：

方法底层算法最佳场景代价
retrieve()BM25 + 向量相似度精确关键词查找O(n) 扫描
query_global()社区检测 + 摘要"主要主题是什么？"需要预计算社区
spread_activation()Collins & Loftus 1975 BFS 传播"X 和什么有关？"图遍历，可扩展

每种方法都有自己擅长的场景。问题是：当用户问"Alice 为什么选择微服务架构？"时，你该调用哪个？

如果没有路由层，调用者（通常是 LLM agent）需要自己判断。但这意味着：

  1. Agent 需要知道所有可用的检索方法
  2. Agent 需要理解每种方法的适用场景
  3. Agent 的判断可能出错——而且你没法追踪为什么出错了
好的抽象应该消灭决策成本。 就像你调用 sort() 不需要告诉 Python 用 quicksort 还是 mergesort——运行时自己知道。

<h3>规则 1：≤3 词 → basic</h3>



最简单的规则，但效果最好。短查询几乎总是实体查找：



<pre><code class="language-plaintext">&quot;Python&quot;        → 找名为 Python 的节点


&quot;Alice&quot;         → 找 Alice 这个人


&quot;微服务架构&quot;     → 关键词匹配


</code></pre>



BM25 + 向量检索在这种场景下又快又准。不需要启动社区扫描或图遍历——那是杀鸡用牛刀。



<strong>工程考量：</strong> 为什么是 3 个词而不是 2 个或 5 个？因为 3 个词足以表达 "Alice works on"（实体 + 动作），但不足以构成复杂问题。4 个词以上的查询，才有可能包含 "how" 或 "why" 这样的复杂标记。



<h3>规则 2：全局关键词 → community scan</h3>



当用户说 "给我一个 overview" 或 "主要主题是什么"，他们不是在找一个具体的节点——他们想看到<strong>全局画面</strong>。

<strong>独到见解：</strong> 很多人以为全局扫描等于"全表扫描"。错了。好的 query_global 实现是<strong>预计算 + 增量更新</strong>的——社区结构在写入时就计算好了，查询时只是读取。成本是 O(社区数)，不是 O(节点数)。



<h3>规则 3：复杂问题 → DRIFT（必须在 local 之前！）</h3>



这是<strong>最重要的一条规则</strong>，也是最容易搞错顺序的：



<pre><code class="language-python"># 3. 复杂多跳 → drift (必须在 local 之前检查！)

为什么顺序重要？考虑这个问题：<strong>"How is Alice connected to Bob?"</strong>



它同时包含 "how"（复杂标记）和 "connected"（关系标记）。如果 local 规则在前面，这个问题会被路由到 spread_activation——一个局部图遍历。但 "how" 暗示用户想要推理链路，不仅仅是邻居列表。



DRIFT 搜索会先做全局定位（找到 Alice 和 Bob 各自的社区），然后局部细化（在两个社区之间寻找路径），最后合并结果。这种 "全局→局部→精炼" 的三阶段流程，才是复杂问题的正确回答方式。



<strong>血的教训：</strong> 我最初把 local 放在 complex 前面。测试发现，所有包含 "connected" 的复杂问题都被误路由了。调换顺序后，复杂问题的回答质量显著提升。

<h2>dispatch：统一返回格式</h2>



路由确定后，query() 函数分派到对应的检索方法，然后把结果统一成一个格式：



<pre><code class="language-python">def query(self, question: str, *, mode: str = &quot;auto&quot;,


limit: int = 10, detail: bool = False, **kwargs) -&gt; dict:


# 确定模式


if mode == &quot;auto&quot;:


mode, rationale = self._route_query(question)


else:

if mode == &quot;basic&quot;:

&quot;mode&quot;: mode,


&quot;rationale&quot;: rationale,


&quot;results&quot;: results[:limit],


&quot;stats&quot;: {

&quot;elapsed_ms&quot;: round(elapsed_ms, 2),

<strong>关键设计：rationale 字段。</strong> 每次查询都返回它选择了哪个模式以及为什么。这不是可选项——在调试和优化路由策略时，这是最重要的信息。你不知道有多少次，我通过看 rationale 发现了路由错误，然后调整规则顺序修复了问题。



<strong>detail=True 标志：</strong> 默认返回精简结果（node_id, label, kind, score）。当 detail=True 时，查询后自动从数据库加载完整节点数据（weight, data, tags）。这个设计避免了不必要的数据传输——大多数调用者只需要 ID 和标签，不需要完整的节点 payload。



<strong>kwargs 透传：** 高级用户可以通过 kwargs 传递特定参数给底层方法，比如 decay=0.3 控制扩散激活的衰减率，或 max_iterations=3 控制 DRIFT 的迭代次数。这使得 query() 不会成为功能瓶颈——它路由，但不限制。






<h2>启发式 vs 机器学习：为什么五条规则就够了</h2>



一个常见的问题：<strong>为什么不用 ML 分类器来路由？</strong>



理论上，可以训练一个模型，输入是问题文本，输出是最优检索方法。但在实践中：


  1. <strong>标注数据不足。</strong> 你需要成千上万的 (question, best_method) 对。在生产中，你不知道"正确"的检索方法是什么——你只知道用户满不满意结果。
    2. 

  2. <strong>延迟成本。</strong> ML 推理需要 50-200ms。而五条 Python 字符串匹配在 0.1ms 内完成。对于每次查询都要调用的路由器，这个差距是致命的。
    3. 

  3. <strong>可解释性。</strong> 当路由出错时，你需要知道为什么。启发式规则是白盒的——rationale 字段直接告诉你哪条规则被触发。ML 模型是个黑盒。
    4. 

  4. <strong>零样本冷启动。</strong> 你的记忆系统第一天上线，一个节点都没有。ML 模型需要训练数据，启发式规则不需要。
    5. 



<strong>但最重要的原因是：五条规则已经覆盖了 95% 的场景。</strong> 剩下的 5% 可以通过 mode='drift' 手动覆盖。当简单方法有效时，不要用复杂方法。

<h2>可操作建议</h2>



如果你在自己的 Agent 记忆系统中实现自适应路由，这些建议可能有用：


  1. <strong>从启发式开始，不要从 ML 开始。</strong> 五条 if 语句覆盖了大部分场景。等到启发式不够用了（通常不会），再考虑 ML。
    2. 

  2. <strong>返回 rationale。</strong> 每次路由都附带理由。这不仅方便调试，还能让用户知道系统"在想什么"——增加信任感。
    3. 

  3. <strong>允许手动覆盖。</strong> mode='drift' 强制指定模式。高级用户（或上层 agent）有时比路由器更懂需求。
    4. 

  4. <strong>规则顺序就是优先级。</strong> 把最具体的规则放前面，最通用的放后面。特别是：复杂检查在关系检查之前，否则 "how is X connected" 会被误路由。
    5. 

  5. <strong>测试边界情况。</strong> 空图、单节点图、超长查询、非英文查询——这些是路由器崩溃的常见原因。每种情况都要有测试。
    6. 

  6. <strong>kwargs 透传。** 不要让路由器成为功能瓶颈。底层方法可能需要调整参数，通过 kwargs 传递。
    7. 






<h2>结语：好的抽象消灭决策</h2>



GraphRAG 和 LightRAG 论文提出了自适应路由的概念，但它们没有告诉你的是：<strong>实现它只需要五条 if 语句。</strong>



这不是因为问题简单，而是因为好的抽象不需要复杂。query()` 函数的核心价值不是它有多精妙的算法——而是它让调用者不再需要思考 "我该用哪个检索方法"。五种策略、一个入口、零决策成本。
