当 Agent 记忆学会「联想」：subgraph 与 shortestPath 的设计哲学

2026-05-24 · AI Agent 知识图谱 记忆系统

最近在给 agent-memory-graph 加了两个新 API：subgraph() 和 shortestPath()。这两个函数解决的不是「存储」问题，而是「联想」问题——Agent 有了记忆，但怎么从一条记忆出发，找到所有相关上下文？怎么知道两条看似无关的记忆之间，其实暗藏联系？

问题：记忆有了，但「联想」是硬编码的

之前 MemoryGraph 只有 neighbors()，它做一件事：从某个节点出发，BFS 返回相邻节点列表。听起来够用，但实际使用中有三个致命缺陷：

  1. 扁平列表丢掉了结构。 neighbors() 返回 [Node, Node, Node]，你看不出谁连着谁，也拿不到边的关系类型。对于 Agent 注入上下文来说，这是一堆散沙。
  2. 没有边界控制。 你只想看「2 跳以内的相关记忆」，但 neighbors() 的 depth 参数只控制 BFS 深度，不控制节点数量。一个高度连接的中心节点可能拉出几百个邻居。
  3. 无法回答「A 和 B 怎么关联的」。 这是最核心的问题。Agent 知道记忆 A 和记忆 B 都存在，但不知道它们之间的路径。就像人脑的「突然想起来」——两个概念通过一系列中间关联被串联起来。

subgraph()：给 Agent 一个「联想窗口」

设计目标很明确：从一个起点出发，提取一个自包含的子图（nodes + edges），可以直接序列化扔进 Agent 的上下文窗口。

def subgraph(self, start_id, *, depth=2, limit=50, layer=None):
    """BFS from start node, returns {root, nodes, edges, depth}."""
    visited = {start_id}
    frontier = [start_id]
    all_nodes = []
    all_edges = []

    for d in range(depth):
        next_frontier = []
        for nid in frontier:
            # 双向遍历：source→target 和 target→source
            edge_rows = self.conn.execute(
                "SELECT * FROM edges WHERE source=? OR target=?",
                (nid, nid)
            ).fetchall()
            for e in edge_rows:
                other = e["target"] if e["source"] == nid else e["source"]
                if other not in visited and len(all_nodes) < limit:
                    visited.add(other)
                    node = self._row_to_node(other)
                    all_nodes.append(node)
                    all_edges.append(Edge(
                        e["source"], e["target"],
                        e["relation"], e["weight"]
                    ))
                    next_frontier.append(other)
        frontier = next_frontier

    return {
        "root": start_id,
        "nodes": all_nodes,
        "edges": all_edges,
        "depth": depth
    }

几个设计决策值得说：

  • 双向遍历。 知识图谱中边是有方向的（A caused B），但联想是双向的。你想到 B 的时候，也可能需要回溯到 A。所以 subgraph 沿着 source→target 和 target→source 两个方向扩展。
  • limit 参数。 这是给 Agent 用的关键约束。LLM 上下文窗口有限，你需要说「最多给我 50 个节点」而不是无限制地扩展。这个参数在生产中直接控制 token 预算。
  • 返回完整边信息。 不只是节点列表，而是 {nodes, edges}。这让 Agent 能看到关系的类型（caused、related_to、belongs_to），而不仅仅是一堆概念。

shortestPath()：回答「它们怎么连着的」

如果说 subgraph 是「以点带面」，那 shortestPath 就是「顺藤摸瓜」。它回答的问题是：从记忆 A 到记忆 B，最短的关系链是什么？

# 使用示例
graph.add("Docker 部署失败", kind="event")
graph.add("磁盘写满", kind="event")
graph.add("日志没有 rotation", kind="fact")
graph.add("运维规范缺失", kind="concept")

graph.link("Docker 部署失败", "磁盘写满", "caused_by")
graph.link("磁盘写满", "日志没有 rotation", "root_cause")
graph.link("日志没有 rotation", "运维规范缺失", "systemic_issue")

result = graph.shortest_path("Docker 部署失败", "运维规范缺失")
# → ["Docker 部署失败", "磁盘写满", "日志没有 rotation", "运维规范缺失"]
# 距离 3，经过 caused_by → root_cause → systemic_issue

BFS 在无权图上天然给出最短路径，时间复杂度 O(V+E)，对于 SQLite 后端的知识图谱完全够用。关键是返回的不仅是节点列表，还包括每条边的 relation 和 weight——这让 Agent 能做 类型感知推理：「部署失败」通过 caused_by 连到「磁盘写满」，再通过 root_cause 连到「日志没有 rotation」——整条因果链一目了然。

Autoresearch 方法论的实际运作

这篇文章本身就是一个 autoresearch 循环的产物。过去 24 小时的实验记录：

  • Loop A: 基线 98/98 测试通过，确定 starting point
  • Loop B: 加 export_json/import_json，5 个新测试，35/35 通过
  • Loop C: 加 subgraph/shortestPath，13 个新测试，100/100 通过

每个 loop 的模式相同：识别缺陷 → 设计最小方案 → 实现 → 测试全部通过 → 记录到 experiments.tsv → commit。没有 design doc，没有 sprint planning，没有 code review——就是一个持续紧凑的 observe-design-implement-verify 循环。

Autoresearch 的核心洞察：当你把「探索想法」变成一个可重复的循环时，创新不再是灵感爆发，而是工程产出。每个循环都可以是失败的（revert），但成功的循环会稳步积累。

可操作的建议

如果你在构建 Agent 记忆系统，有三条可以直接拿走的经验：

  1. 先有图遍历，再做向量检索。 很多团队一上来就搞 embedding + cosine similarity，但图遍历（BFS/DFS）能回答向量检索回答不了的问题：「A 和 B 之间有什么关系？」 图结构 + 向量检索才是完整方案。
  2. 用 limit 参数控制 token 预算。 Agent 的上下文窗口是稀缺资源。你的记忆系统 API 必须支持「给我最多 N 个结果」这种硬约束，而不是返回一大堆让 Agent 自己裁剪。
  3. experiments.tsv 比文档更有用。 记录每个实验的 timestamp、commit、metric、before/after、decision。一周后回头看，你的决策轨迹一目了然。这不是文档，这是思维的外化。
