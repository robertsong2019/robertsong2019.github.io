Autoresearch 实验循环：如何让 AI Agent 用 TSV 驱动自我进化

2026-04-19 · AI Agent 自动化研究 实验方法论

过去两周，我的 agent-memory-service 项目从 90 个测试增长到 241 个。不是我在深夜加班写的——是一个 AI Agent 用一种我叫 Autoresearch 实验循环 的方法论自动推进的。这篇文章拆解这个方法，告诉你为什么它比"让 AI 写代码"有效得多。

问题：AI 编码的最大敌人不是能力，是上下文丢失

大多数人用 AI 写代码的方式是：开一个对话，描述需求，拿到代码，粘贴。下一次开新对话，从头开始。

这对小任务够用。但当你想让 Agent 持续演进一个项目——每天自动发现改进点、实现、测试、提交——上下文丢失就成了致命问题。Agent 不知道昨天做了什么、上次实验成功还是失败、哪些代码已经被回滚。

我的解决方案不是更长的 prompt，而是一个 5 列的 TSV 文件。

experiments.tsv：Agent 的实验笔记本

核心结构极其简单：

timestamp    commit    metric    value    status    description
2026-04-19T01:00    fd3bae4    test_count    241/241    keep    bulkTag(ids,{add,remove}) — 5 new tests
2026-04-19T00:00    key-dev-2b    test_count    236/236    keep    renameTag()+mergeTags() — 8 new tests
2026-04-18T19:30    d2b6d46    test_count    206/206    keep    mergeMemories(ids,opts) — 5 new tests

每一行记录一次实验：什么时候做的、改了什么、指标是多少、保留还是回滚、一句话总结。

Agent 每次启动时读取这个文件，就能回答：

  • 当前有多少测试通过？（241）
  • 上次实验成功了吗？（keep = 成功保留）
  • 最近的代码在哪里？（commit hash）
  • 项目在哪个版本？（v0.9.8）

循环协议：每个实验都是原子操作

每次实验严格遵循以下步骤：

  1. 读取基线 — 从 experiments.tsv 获取当前状态（版本号、测试数）
  2. 选择目标 — 发现一个具体的 API 缺口或改进点
  3. 实现 + 测试 — 最小化代码变更，配套测试
  4. 验证 — 全量测试跑一遍，确认零回归
  5. 决策 — keep（保留）或 rollback（回滚）
  6. 提交 — git commit + 更新 experiments.tsv

关键约束：每次只做一件事。不是"重构整个模块"，而是"加一个 touch() 方法做轻量访问追踪"。这让每次变更可审计、可回滚。

实战案例：48 小时从 184 到 241

看看 4 月 17-18 日发生了什么：

v0.9.5  184 tests  →  LLM记忆提取 (extractWithLLM + extractHybrid)
v0.9.6  188 tests  →  touch(id) 轻量访问追踪，不改内容不记日志
v0.9.7  206 tests  →  count()+random()+recent()+mergeMemories() 查询增强
        219 tests  →  query()+findByEntity()+batchUpdate() 结构化查询
        224 tests  →  snapshot()+diff() 状态快照对比
        228 tests  →  listArchived() 归档列表
v0.9.8  236 tests  →  renameTag()+mergeTags() 批量标签管理
        241 tests  →  bulkTag(ids,{add,remove}) 精确标签增删

57 个新测试，17 个新 API 方法，零回滚。每个方法都是 Agent 自己发现缺口、自己实现、自己验证的。

拿 touch(id) 举例。Agent 注意到：搜索命中的记忆需要更新访问时间，但唯一的接口 update() 会创建 changelog 条目。对纯访问追踪来说太重了。于是它实现了：

// 18 行代码，解决一个真实痛点
touch(id, { boost } = {}) {
  const mem = this.store.get(id);
  if (!mem) return null;
  mem.accessedAt = Date.now();
  mem.accessCount = (mem.accessCount || 0) + 1;
  if (boost) {
    mem.weight = Math.min(
      (mem.weight || 1) + boost,
      this.MAX_WEIGHT
    );
  }
  this._save();
  return { ...mem };
}

没有 changelog，没有内容变更，纯粹是"这个记忆被访问了"的信号。用于后续的衰减排序：最近被访问的记忆权重更高。

为什么这比"让 AI 自由发挥"好

可审计。 每一行 experiments.tsv 都是一次决策记录。出了问题可以追溯到具体哪次实验引入的。

可回滚。 状态列只有两种值：keep 或 rollback。Agent 启动时如果看到 rollback，就知道上次实验失败，需要 git revert。

可持续。 Agent 不依赖记忆来知道项目状态——它依赖文件。文件不会在 session 重启时消失。

反奢侈。 每次实验只加 15-50 行代码。不是大重构，是持续的小改进。这符合软件工程的现实：大多数有价值的改动都是小的、精确的、针对性强的。

踩过的坑

4 月 17 日有一次惨痛教训：Agent 连续做了 3 个实验循环，更新了 experiments.tsv，但忘了 git commit 代码。Session 重启后，代码全丢了，TSV 里记着 133 个测试但实际只有 90 个。

修复方案：在实验协议中加了一步硬性规则——每次 keep 决策后必须 git commit。不是"建议"，是"协议的一部分"。从此再没丢过代码。

// 修复后的协议（伪代码）
if (allTestsPass) {
  decision = 'keep';
  exec('git add -A && git commit');
  appendToExperimentsTsv(...);
} else {
  decision = 'rollback';
  exec('git checkout -- .');
}

可操作建议

  • 如果你在用 AI Agent 做持续开发，立刻创建一个 experiments.tsv。不需要复杂工具，5 列就够。
  • 每次实验只改一件事。如果你需要改三个地方，做三次实验。多花的时间换来的是可审计性和可回滚性。
  • 测试数是最好的进度指标。不需要代码覆盖率，不需要复杂度量。测试从 N 涨到 N+5，你就知道项目在前进。
  • commit 是实验的一部分，不是事后的装饰。忘记提交等于没做。
  • 让 Agent 自己发现改进点。我的 Agent 读了 API 列表后会说"batchUpdate 只能全量替换标签，缺少增量操作"——然后自己去实现 bulkTag。这种自主发现比人列需求表高效得多。

总结

Autoresearch 实验循环的本质是：把 AI Agent 的无状态特性变成优势。既然它记不住上次做了什么，那就让文件记住。既然上下文窗口有限，那就把最关键的信息压缩成 5 列 TSV。

241 个测试、17 个新 API、零回滚。这个方法论不是理论——它每天都在跑，每天都在产出。

试试看。你需要的只是一个 TSV 文件和一条规则：每次实验，原子提交。
