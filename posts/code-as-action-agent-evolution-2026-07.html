代码即动作：从 JSON 到程序式编排的 Agent 进化

2026-07-11 · 罗嵩 · Code as Action AI Agent GPT-5.6 Programmatic Tool Calling
当你的 Agent 还在一个一个地请求工具时，GPT-5.6 已经开始写程序来编排工具链了。这不是渐进改进——这是行动空间的范式跳跃。从 {"tool": "search", "args": {"q": "..."}} 到 results = [search(q) for q in queries]，看似只多了一层循环，实则是从"原子操作"到"程序化编排"的质变。

一、JSON Tool Calling 的三道天花板

2023 年 OpenAI 推出 Function Calling 时，整个行业都认为这就是 Agent 调用工具的标准答案。三年后回头看，它只是起步。JSON 格式的工具调用有三层根本限制：

  • 不可组合：每次只能调一个工具。想"搜索→过滤→排序"？三轮请求，三次上下文窗口往返。
  • 不可迭代：没有 if、没有 for、没有 while。条件分支需要模型反复"思考"再请求，延迟倍增。
  • 不携带状态：中间结果必须序列化为文本回传给模型，模型再在下一轮解析。100KB 的搜索结果变成了 100KB 的 prompt token。

这些问题在实践中意味着什么？看一个真实的例子：

JSON 方式：搜索+过滤+汇总需要 3 轮对话

# Round 1: 搜索
LLM → {"tool": "search", "args": {"query": "Rust web frameworks"}}
← [{"name": "Actix", "stars": 12000}, {"name": "Axum", "stars": 18000}, ...]

# Round 2: 过滤（模型要把结果塞回 prompt 再请求）
LLM → {"tool": "filter", "args": {"data": [...], "min_stars": 5000}}
← [{"name": "Actix", "stars": 12000}, {"name": "Axum", "stars": 18000}]

# Round 3: 排序
LLM → {"tool": "sort", "args": {"data": [...], "key": "stars"}}
← [{"name": "Axum", "stars": 18000}, {"name": "Actix", "stars": 12000}]

三轮 API 调用，三轮延迟，三轮 token 计费。而这三轮做的事情，一行 Python 就能搞定。

代码方式：一轮搞定

# Code Action：一轮 API 调用
results = search("Rust web frameworks")
filtered = [r for r in results if r["stars"] >= 5000]
sorted_results = sorted(filtered, key=lambda x: x["stars"], reverse=True)
return sorted_results

一轮对话，模型直接写出完整的处理逻辑，在沙箱中执行，返回最终结果。这就是 Code as Action 的核心洞察。

二、进化时间线：四个阶段

2022.10 ReAct — 文本推理 + 文本动作 2023.02 Toolformer — LLM 自主学习调用 API 2023.05 Voyager — 代码技能库 + 终身学习 2023.05 LATM — LLM 作为工具制造者 2023.10 Eureka — 代码生成超越人类专家 2024.02 CodeAct — 代码作为统一行动空间 (ICML) 2024.05 SWE-Agent — Agent-Computer Interface 2025.01 SmolAgents — 极简代码 Agent 框架 2026.07 GPT-5.6 — 程序式编排 + 多 Agent 并行

从文本动作到程序式编排，每一步都在扩大 Agent 的行动空间。但要理解为什么代码胜出，需要拆解每个阶段的关键突破。

三、关键转折：Voyager 和 CodeAct

Voyager（2023.05）：技能库的诞生

Voyager 是第一个让我确信"代码即动作"是对的方向的系统。它在 Minecraft 中实现了终身学习——不断探索、发现新技能，并把成功的策略保存为可复用的代码函数。

# Voyager 的技能库概念（简化）
class SkillLibrary:
    def add_skill(self, description: str, code: str):
        """成功的行为保存为技能"""
        self.skills.append({
            "description": description,
            "code": code,
            "embedding": embed(description)
        })

    def retrieve(self, task: str, top_k=3):
        """新任务可以检索和复用已有技能"""
        return vector_search(task, self.skills, top_k)

# 第一次遇到树：学会了砍树
library.add_skill("chop_wood", "def chop(): ...")
# 第二次需要做木板：不需要从头学
skill = library.retrieve("make planks")
# skill 就是之前保存的代码

这意味着 Agent 的能力是递增的——今天解决过的问题，明天变成一个函数调用。这不是记忆，这是编程式的知识积累。

CodeAct（2024.02, ICML）：17 个模型的消融实验

CodeAct 论文做了一件非常漂亮的事：它在 17 个不同模型上系统对比了 JSON 行动和代码行动。结果压倒性——代码行动在复杂任务上成功率高出最多 20%。

为什么？因为代码天然解决了 JSON 的三个限制：

限制JSON代码
组合性每次只能一个工具任意嵌套、链式调用
迭代性无控制流if/for/while + 异常处理
状态携带中间结果需序列化回传变量直接在沙箱内存活

四、Agent-Computer Interface：隐藏的 6 倍乘数

SWE-Agent 的发现可能是 2024 年最被低估的洞察。

核心发现：仅仅改变 Agent 与计算机交互的界面设计，就能让 pass@1 从 2% 跳到 12.5%。同样的模型，同样的任务，6 倍效率差异。

这就像给人类工程师从记事本换到 IDE——智能水平没变，但工具界面完全不同。SWE-Agent 设计了专门的 ACI（Agent-Computer Interface）：

  • 搜索-查看-编辑三步循环，而不是自由浏览
  • 每次编辑都显示 diff，而不是整个文件
  • 限制上下文窗口中可见的文件数量
  • 提供专门的 lint/test/run 命令而非通用 shell
# SWE-Agent 的 ACI 设计原则（Python 伪代码）
class AgentComputerInterface:
    """为 Agent 专门设计的交互界面"""

    def search(self, pattern: str) -> list[str]:
        """返回匹配的文件:行号，不是全文"""
        return grep(pattern)

    def view(self, file: str, lines: tuple[int, int]) -> str:
        """只看指定行范围，不是整个文件"""
        return read_lines(file, lines)

    def edit(self, file: str, old: str, new: str) -> str:
        """字符串替换 + diff 预览"""
        return replace_and_show_diff(file, old, new)

    def run(self, cmd: str) -> str:
        """受限命令集：lint, test, run"""
        return safe_exec(cmd)

# Agent 要自己搞清楚怎么搜索、怎么编辑、怎么测试
# 结果：大量 token 浪费在"操作工具"而非"解决问题"

GPT-5.6 的"inspect and refine the rendered result"本质上就是高级 ACI——模型不只生成代码，还能看到渲染结果并自动修正。这不是模型的进步，是界面的进步。

五、2026 年的新维度：推理时并行

GPT-5.6 的 Multi-Agent Ultra 模式展示了一个全新方向：不是让一个模型想更久，而是让多个 Agent 并行工作。

配置Agent 数BrowseComp 准确率延迟
标准模式132%~30s
Ultra 模式4（默认）51%~15s
Ultra 模式最大1658%~8s

4 个 Agent 并行不仅更快，而且更准。这暗示了一个深刻的转变：

复杂任务的瓶颈不是推理深度，而是探索广度。一个 Agent 想再久，也可能困在一条错误的路径上。四个 Agent 并行探索不同路径，找到正确答案的概率显著提高。

结合程序式编排，GPT-5.6 的 Agent 可以在 V8 沙箱中编写 JavaScript 来协调工具链，同时多个 Agent 各自探索子问题，最终由一个协调器汇总结果。这是从"工具使用者"到"分布式程序"的质变。

六、安全模型的重构

⚠️ 核心安全问题：当模型从"调用预定义函数"变成"编写任意代码"，安全模型完全改变了。JSON 调用只需验证参数，代码执行需要沙箱、权限隔离、输出过滤。

当前的安全实践分三层：

// 第一层：沙箱隔离（GPT-5.6 方案）
const sandbox = createV8Isolate({
    memoryLimit: '128m',
    cpuLimit: '5s',
    filesystem: false,      // 无文件系统访问
    network: false,         // 无网络
    childProcess: false,    // 无子进程
});

// 第二层：白名单模块
const allowedModules = ['lodash', 'mathjs'];
const require = (name: string) => {
    if (!allowedModules.includes(name)) throw new Error('Blocked');
    return loadModule(name);
};

// 第三层：输出验证
const result = sandbox.run(userCode);
if (typeof result !== 'object' || result === null) {
    throw new Error('Must return object');
}
if (JSON.stringify(result).length > 1_000_000) {
    throw new Error('Output too large');
}

但大多数开源 Agent 框架还在用 exec() 直接运行模型生成的代码。这是巨大的安全债务。SmolAgents、AutoGen 的早期版本、以及无数个人项目都在把模型输出直接喂给 Python exec()——这在演示中看起来很酷，在生产中是灾难。

可落地的安全方案

如果你正在构建 Code Action Agent，以下是我推荐的渐进式安全策略：

  1. 第一阶段（1-2 周）：Python eval() + 白名单内建函数。禁止 __import__、open、exec、eval（递归）。只允许数学运算和列表操作。
  2. 第二阶段（1-2 月）：迁移到 firejail 或 Docker 容器，限制文件系统和网络访问。
  3. 第三阶段（3-6 月）：使用专业沙箱（E2B、Modal、Fly Machines），按请求分配隔离环境。
# 第一阶段：Python 白名单 eval（最小可行安全）
import ast

SAFE_NODES = {
    ast.Expression, ast.BinOp, ast.UnaryOp, ast.Compare,
    ast.Constant, ast.Name, ast.Load, ast.List, ast.Tuple,
    ast.Dict, ast.ListComp, ast.DictComp, ast.Call,
    ast.Attribute, ast.Subscript, ast.For, ast.If,
    ast.Assign, ast.AugAssign, ast.Return, ast.FunctionDef,
}

SAFE_BUILTINS = {
    'len': len, 'range': range, 'enumerate': enumerate,
    'sorted': sorted, 'filter': filter, 'map': map,
    'sum': sum, 'min': min, 'max': max, 'abs': abs,
    'round': round, 'zip': zip, 'any': any, 'all': all,
    'str': str, 'int': int, 'float': float, 'bool': bool,
    'list': list, 'dict': dict, 'set': set, 'tuple': tuple,
}

def safe_eval(code: str, context: dict) -> any:
    """安全执行模型生成的代码"""
    tree = ast.parse(code)
    for node in ast.walk(tree):
        if type(node) not in SAFE_NODES:
            raise SecurityError(f"Blocked: {type(node).__name__}")
    return eval(compile(tree, '', 'exec'),
                {'__builtins__': SAFE_BUILTINS, **context})

七、技能库：从工具使用者到工具制造者

LATM（LLM as Tool Maker）和 Voyager 代表了我认为最重要的转变：Agent 不只是调用预定义工具，而是创造新工具。

在我的 agent-memory-graph 项目中，这个概念体现为"代码技能的自动积累"：

# Agent 第一次处理"找到所有孤立节点"
def find_orphan_nodes(graph):
    """手动实现的策略"""
    result = []
    for node in graph.nodes:
        if graph.degree(node) == 0:
            result.append(node)
    return result

# Agent 自动保存为技能
skill_library.add(
    name="find_orphan_nodes",
    description="找出图中没有连接的孤立节点",
    code=find_orphan_nodes,
    tags=["graph", "connectivity"]
)

# 下次遇到类似任务"找到无引用的记忆"
# Agent 不需要从头实现，直接检索并复用
skill = skill_library.retrieve("find unlinked memories")
# skill.code 就是之前保存的函数，稍作修改即可使用

这意味着 Agent 的能力增长是复利式的：今天解决的问题变成明天的工具，明天可以用今天的工具组合出更复杂的能力。

八、实战建议：如何升级你的 Agent

短期（1-2 周）

  • 把一个高频工具调用链从 JSON 迁移到代码格式，用 Python eval() + 白名单开始
  • 对比迁移前后的延迟、token 消耗、成功率
  • 关注三个指标：API 轮次（应该下降 60%+）、总 token（应该下降 40%+）、任务成功率（应该提升 10%+）

中期（1-2 月）

  • 实现技能库机制：成功完成的任务自动保存为可复用函数
  • 设计 ACI：不要给 Agent 一个原始 shell，为它设计专门的搜索-查看-编辑界面
  • 迁移到 firejail 或 Docker 沙箱

长期（3-6 月）

  • 探索多 Agent 并行：复杂任务分解为子任务，多个 Agent 并行处理
  • 构建技能市场：不同 Agent 共享和交易技能库
  • 实现"假设推理"：用世界模型模拟操作后果，选择最优方案

核心观点

1. 代码是天然的统一行动空间。JSON 是代码的子集——任何 JSON 调用都可以写成代码，但代码能做的事 JSON 做不了。当你发现"行动空间"可以用图灵完备的语言描述时，回到 JSON 就感觉像被绑住了手脚。

2. 界面设计是隐藏的乘数。SWE-Agent 证明，同样的模型在好的 ACI 下效率可以提升 6 倍。在你追逐更强的模型之前，先检查你的 Agent 界面设计是否在拖后腿。

3. 探索广度比推理深度更重要。GPT-5.6 的多 Agent 并行证明了这一点。与其让一个模型想 60 秒，不如让 4 个模型各自想 15 秒。

4. 安全不是可选项。如果你的 Agent 还在用 exec() 运行模型输出，你现在就在积累技术债。在 Code as Action 的世界里，沙箱是基础设施，不是优化项。
