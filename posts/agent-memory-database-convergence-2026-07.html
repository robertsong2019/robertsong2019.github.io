纯向量检索已死：Agent 记忆数据库的三引擎融合

2026-07-16 · 罗嵩 · Agent Memory Knowledge Graph Vector Search Mem0 Zep
你的 Agent 记得用户上周说要换工作吗？记得他半年前提过的过敏药物吗？如果你的答案是"我们用了向量数据库"——你可能在解决一个比你想的难得多的问题，用了一个比你想的弱得多的工具。

这篇文章讲的是 2024-2026 年间，AI Agent 记忆系统经历的一次安静但深刻的架构革命。这不是又一篇"RAG 入门"——这是一份关于记忆基础设施正在如何被重新设计的深度分析。

一、为什么"把所有东西丢进向量库"不好使了？

先说一个让很多人不舒服的事实：RAG 是为文档设计的，不是为对话设计的。

传统 RAG 的流程很优雅：文档 → 切块 → 嵌入 → 向量数据库 → 查询时 Top-K → 拼到 prompt 里。对于"查询公司请假政策"这种场景，它很好用。

但 Agent 的记忆需求完全不同：

  • 事实会变："我用 Postgres" → 三个月后变成 "我们迁移到了 CockroachDB"。向量库里两条记录都在，哪条是当前的？
  • 关系很重要："张三是我老板"不仅是一个事实，它还隐含了"张三批准我的休假"、"张三的优先级高于我"。向量检索看不到关系。
  • 时间维度："上次他说喜欢 A" 和 "他现在喜欢 A" 是完全不同的信息。向量嵌入没有时间戳的概念。
  • 来源要可追溯：当 Agent 基于一条记忆做决策时，你需要知道这条记忆是从哪次对话中提取的。出了问题，你得能溯源。

所以问题不是"向量检索不好"，而是单一信号——无论是什么信号——都不够。

二、三引擎融合：SQL + Vector + Graph

2025-2026 年最值得关注的架构创新，是存储层的垂直分离。Mem0 给出了最清晰的实现：

存储引擎存什么解决什么查询
SQL 数据库结构化事实 + 元数据精确查询、过滤、排序
向量数据库语义嵌入"跟这个概念相似的记忆有哪些"
图存储实体关系"跟这个人/项目相关的所有事实"

这不是"三个库做三件事"——而是同一条记忆同时存在于三个存储中，每个存储服务于不同的检索信号。当你查询时，三路结果通过排序融合（如 RRF）给出最终答案。

类比一下：就像人脑有工作记忆（前额叶）、情景记忆（海马体）、语义记忆（新皮层）——不同脑区负责不同模式的信息处理，但它们是协同工作的。

三、时间是记忆的第一公民

这是我认为被严重低估的一个创新。

Zep/Graphiti 的核心贡献不是"用了知识图谱"——GraphRAG 也用了。它真正的突破是双时序知识图谱（Bi-temporal Knowledge Graph）：

  • 每条事实有 valid_from 和 valid_to 两个时间戳
  • 当事实改变时，旧事实被 废止（invalidate），而非删除
  • 查询可以指定 "as-of" 语义："截至 2026 年 3 月，用户的偏好是什么？"
💡 为什么这比听起来重要？

想象一个客服 Agent。用户 1 月说"我对青霉素过敏"，3 月说"医生确认了可以用药"。如果没有时序管理，Agent 可能同时检索到两条矛盾的信息，不知道哪条是最新的。有了时序图谱，Agent 知道 3 月的信息废止了 1 月的——矛盾自动消解。

这比图谱结构本身更重要。图谱是骨架，时序是语义。一个没有时序的知识图谱，就像一个没有时间轴的历史书——所有事件都堆在一起，你分不清因果。

四、四信号混合检索：新的工程标准

基于 Mem0 和 Zep 的实践，行业已经收敛出四个正交的检索信号：

信号技术擅长失败场景
语义向量相似度"跟我概念上相关的东西"精确名称、代码、ID
关键词BM25名称、标识符、精确匹配同义词、概念变体
实体图遍历关系查询、多跳推理独立的事实查询
时序时间窗口过滤"最近""之前""什么时候"非时间性查询

关键工程问题：怎么融合四路结果？

标准答案是 Reciprocal Rank Fusion (RRF)：

# RRF 融合公式——简单但极其有效
def rrf_fusion(rankings, k=60):
    scores = {}
    for ranking in rankings:
        for rank, doc_id in enumerate(ranking):
            scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
    return sorted(scores, key=scores.get, reverse=True)

# 四路检索 → 各自排序 → RRF 融合
final_results = rrf_fusion([
    semantic_search(query, top_k=50),
    bm25_search(query, top_k=50),
    entity_graph_search(query, top_k=50),
    temporal_filter_search(query, top_k=50),
])

RRF 之美在于它不需要校准分数——只看排名。语义相似度 0.87 和 BM25 分数 15.3 没有可比性，但排名 #1 和排名 #3 是可比的。这使得融合极其鲁棒。

五、Mem0 vs Zep：两大阵营的设计哲学

这是 2026 年 Agent 记忆领域最重要的技术选型决策。让我用一个比喻来解释它们的区别：

Mem0 像一个高效的档案管理员——它听你说话，提取关键信息，分类归档。查询时，它快速翻到最相关的几页。简单、高效、适合"记住用户偏好"级别的需求。

Zep 像一个侦探——它不仅记录你说的话，还追踪你说这些话的时间、你后来改了什么主意、各条信息之间有什么隐藏的联系。查询时，它可以做时间线重建和多跳推理。复杂、强大、适合需要深度推理的场景。

核心架构向量 + 实体图 + SQL时序知识图谱
记忆更新追加式（新事实不会自动废止旧事实）自动废止（新事实自动标记旧事实无效）
时间推理有限（时间元数据过滤）完整（bi-temporal，支持 as-of 查询）

还有第三个选择：Letta（原 MemGPT）。它的设计哲学完全不同——不是"存什么"，而是"怎么管"。它把 LLM 的 context window 当作 RAM，把外部存储当作硬盘，用类似操作系统的页面置换算法自动管理上下文。适合超长对话和需要"反思"能力的 Agent。

六、来源追踪：从可选功能到安全必需品

2026 年的 Agent 安全年，记忆投毒攻击已经成为真实威胁。ShadowMerge 论文展示了 93.8% 成功率攻破 Mem0 图记忆系统——攻击者不需要任何特殊权限，只需要通过正常对话就能把恶意信息注入 Agent 的知识图谱。

在这种背景下，Zep 的 Episodes 设计——每条派生事实都追溯到原始输入——从审计需求升级为安全防线：

// 每条记忆的结构
{
    fact: "User prefers Postgres",
    confidence: 0.95,
    source: {
        episode_id: "ep_2026_03_15_002",
        raw_message: "我们项目决定用 Postgres",
        timestamp: "2026-03-15T10:30:00Z",
        speaker: "user"
    },
    valid_from: "2026-03-15",
    valid_to: null  // 仍然有效
}

当一条高影响力记忆（影响 Agent 决策路径的记忆）的来源可疑时——比如 confidence 低、source 不匹配、或来自非正常对话时段——安全系统可以自动拦截。

⚠️ 实践建议

如果你在设计生产 Agent 的记忆系统，从第一天就加上 provenance 追踪。事后补的成本是指数级的——你需要为每条已有记忆回溯来源，而很多来源信息可能已经丢失。

七、MCP 正在统一记忆访问协议

最后说一个趋势：记忆正在从"每个 Agent 一个私有库"变成"跨 Agent 共享基础设施"。

Zep 已经发布了 MCP Server，Mem0 也在跟进。这意味着什么？你的 Claude Desktop、Cursor 编辑器、自研 Agent 可以共享同一个用户记忆图谱——你在 Cursor 里告诉它"我用 Tailwind CSS"，你的客服 Agent 自动也知道这个信息。

这跟数据库的历史如出一辙：

  1. 第一阶段：每个应用内嵌自己的存储（文件、SQLite）
  2. 第二阶段：独立的数据库服务（MySQL、Postgres）
  3. 第三阶段：标准化的查询协议（SQL、ODBC、JDBC）

Agent 记忆正在进入第三阶段。MCP 就是记忆层的 SQL。

你今天可以做什么

不需要从零构建一个 Zep。以下是三个可以立刻执行的改进：

1. 给向量检索加上 BM25（如果你还没做）

这是性价比最高的改进。80 行代码实现 BM25 + RRF 融合，对精确匹配场景的提升是立竿见影的。大多数向量数据库（Qdrant、Weaviate、Milvus）已经内置了混合检索支持，只需要开启。

2. 给记忆加时间戳和 as-of 查询

不需要完整的时序知识图谱。在现有记忆表里加 valid_from 和 valid_to 列，更新记忆时设置旧记忆的 valid_to 而非删除。查询时支持时间过滤。50 行代码的改动，解决 80% 的时序问题。

3. 记录记忆来源

在记忆写入时额外存储 source_message_id、source_type（对话/文档/系统推断）、confidence。这不解决当前问题，但当你遇到记忆投毒或错误推理时，你会感谢过去的自己。

记忆系统是 Agent 的地基。地基不牢，上面的推理、规划、工具调用都是空中楼阁。2026 年的地基标准已经从"有个向量库"升级到了"三引擎融合 + 时序管理 + 来源追溯"。这不是过度工程，这是生产环境的最低要求。
