健康评分驱动自愈：AI Agent 记忆系统的六维体检

2026-05-04 · AI Agent 自愈系统 记忆管理

最近在做 agent-memory-service 的自动维护时，遇到了一个典型问题：你有 compactBM25Index()，也有 compactEmbedCache()，还有 purge() 和 compactChangelog()，但这些工具都是被动调用的。没人告诉你什么时候该用哪个。

结果是：索引膨胀了、缓存过时了，系统照常运行，只是越来越慢，越来越不准。像一颗无声的蛀牙。

问题根源：维护操作与诊断脱节

传统的维护模式是定时的：

// 每100次写入后自动清理
if (writeCount % 100 === 0) {
  await compact();
}

这有两个致命问题：

  1. 盲目执行——即使数据很健康也在压缩，浪费资源
  2. 遗漏场景——BM25 索引和 embedding 缓存的膨胀速率完全不同，不能用同一个周期

解决方案：healthScore 驱动 autoMaintain

核心思路：先体检，再治疗。healthScore() 返回六个维度的评分，autoMaintain() 只修复不及格的维度。

六维健康评分

healthScore() {
  const details = {
    utilization: this._memUtilization(),   // 记忆容量使用率
    freshness:   this._memFreshness(),     // 最近更新距今
    changelog:   this._changelogHealth(),  // 变更日志膨胀度
    search:      this._searchHealth(),     // 搜索索引完整性
    bm25:        this._bm25Health(),       // BM25 索引孤立条目
    embedCache:  this._embedCacheHealth(), // Embedding 缓存孤立条目
  };
  // 每个维度 0-100，综合取加权平均
  const overall = weightedAverage(details);
  return { score: overall, details, recommendations };
}

关键在于 bm25 和 embedCache 这两个维度。它们的计算逻辑是：拿索引/缓存中存储的所有 ID，与实际记忆中的有效 ID 做差集。差集就是孤立条目——对应记忆已被删除，但索引还留着。

_bm25Health() {
  const liveIds = new Set(this.memories.keys());
  const indexedIds = this.bm25Index.docIds();
  const orphans = indexedIds.filter(id => !liveIds.has(id));
  return 100 - (orphans.length / indexedIds.length) * 100;
}

autoMaintain 只修复不及格项

async autoMaintain(tasks = null) {
  const health = this.healthScore();
  const defaults = [
    'purge', 'compactChangelog',
    'compactBM25', 'compactEmbedCache',
    'decay', 'reindex'
  ];
  const toRun = tasks || defaults;

  for (const task of toRun) {
    switch (task) {
      case 'compactBM25':
        if (health.details.bm25 < 90) await this.compactBM25Index();
        break;
      case 'compactEmbedCache':
        if (health.details.embedCache < 90) await this.compactEmbedCache();
        break;
      // ... 其他任务同理
    }
  }
}

注意阈值是 90 分——允许 10% 的「不完美」。这是有意的：过度清理比少量孤立更有害，因为压缩操作本身有成本。

这个模式的独到之处

大多数开发者处理缓存/索引清理时，用的是时间驱动（定时任务）或事件驱动（写后清理）。这两种方式的问题是：它们不关心系统的实际状态。

健康评分驱动（health-driven）是第三条路：

  • 时间驱动：每 N 秒跑一次 → 盲目，浪费资源
  • 事件驱动：每次写后清理 → 过度敏感，高频场景性能差
  • 健康驱动：先体检，只治有病的 → 精准，高效

这和人去医院一个道理。你不是每天做全身体检（时间驱动），也不是每咳嗽一次就做 CT（事件驱动）。你定期查关键指标，异常了才深入检查和治疗。

实际效果

在 agent-memory-service 中实施后：

  • autoMaintain() 大部分调用只执行 purge + changelog 压缩，跳过 BM25 和 embed cache（因为它们通常很健康）
  • 当大量删除记忆后，下次 autoMaintain 自动检测到 BM25 和 embedCache 评分下降，触发压缩
  • 312 个测试全部通过，包括边界场景：空索引、全孤立、零孤立

可操作建议

如果你的系统有任何形式的缓存或索引（哪个系统没有呢？），考虑这个模式：

  1. 为每个存储维度定义健康指标——不是「上次清理时间」，而是「当前数据质量」。孤立率、命中率、膨胀率，选一个能度量的。
  2. 设阈值而不是定时器——让维护操作成为「条件触发」而非「周期执行」。
  3. 先诊断后治疗——healthScore() 必须先于 autoMaintain() 存在。如果你不知道系统健不健康，你就不该动手修。
  4. 允许不完美——90 分阈值比 100 分更实用。追求零孤立会带来频繁压缩的成本。
好的自愈系统不是「什么都能修」，而是「知道什么时候该修、什么时候别瞎折腾」。
