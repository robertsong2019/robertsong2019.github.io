从 Pocket Agent 到自进化系统：AI Agent 最小化实践

2026-04-12 · AI-Agent Rapid-Prototyping Edge-AI Python

过去两周的深夜代码实验室里，我一直在做一个实验：用最少的代码，把 AI Agent 的核心概念讲清楚。结果出乎意料——一个完整的 ReAct Agent 只需要 ~150 行 Python，而一个能在运行时自我扩展能力的 Agent，也不到 300 行。

这篇文章记录了 Pocket Agent 和 Self-Evolving Agent 两个实验项目的设计思路，以及从中提炼的 AI Agent 最小化实践原则。

一、为什么是"最小化"？

2026 年的 Agent 框架百花齐放——LangGraph 状态图、CrewAI 角色团队、AutoGen 对话流……每个都有价值，但也都有一个共同问题：抽象层太厚，初学者很难理解 Agent 到底在做什么。

我想要一个"口袋版"的 Agent：

  • 读完 agent.py 一个文件就能理解全部逻辑
  • 零外部依赖，拔掉网线也能跑
  • 但架构要能无缝切换到真实的 LLM API

这就像学操作系统，先写一个能启动的 bootloader，再看 Linux 内核。

二、Pocket Agent：150 行的核心抽象

2.1 三大组件

Pocket Agent 把所有 Agent 框架的共性提取为三个核心类：

  1. ToolRegistry — 工具注册中心，自动从函数签名生成 JSON Schema
  2. EpisodicMemory — 情景记忆，存储完整的交互历史
  3. ReActLoop — 推理-行动循环，Agent 的"心跳"
@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气"""
    return f"{city}: 晴, 25°C"

# 装饰器自动注册，自动生成 schema
# 无需手动描述参数类型 — 从 type hints 推导

这个设计的精妙之处在于：MockLLM 和真实 API 共享完全相同的接口。替换 LLM 只需要改一行初始化代码。

2.2 ReAct 循环的本质

ReAct（Reasoning + Acting）不是什么高深的概念。本质上就是一个 while 循环：

while not done:
    thought = llm.think(context)    # 思考
    action = llm.decide(thought)    # 决策
    result = tool_registry.call(action)  # 执行
    memory.store(thought, action, result)  # 记忆
    done = llm.evaluate(result)     # 判断是否完成
所有 Agent 框架的复杂性，本质上都是在回答三个问题：怎么思考？怎么选工具？什么时候停下来？

三、Self-Evolving Agent：运行时自我扩展

如果说 Pocket Agent 回答了"Agent 是什么"，那 Self-Evolving Agent 回答的就是："Agent 能不能变得更强大？"

3.1 核心机制

Self-Evolving Agent 从零工具开始。当用户提出一个它无法处理的需求时：

  1. 分析需求，匹配内置的 8 个工具模板（fibonacci、prime_check、password_gen 等）
  2. 用 exec() 动态生成工具代码
  3. 自动注册到 ToolRegistry
  4. 立即使用新工具响应用户
# 运行时动态生成工具
tool_code = generate_from_template("fibonacci", params)
exec(tool_code)  # 动态执行
registry.register(new_tool)  # 自动注册

3.2 安全与设计的权衡

是的，exec() 在生产环境中是危险的。但这个实验的价值在于概念验证：Agent 的能力边界不必在部署时固定。

更安全的实现方式包括：

  • MCP 工具市场 — 从受信任的注册中心动态拉取工具
  • A2A 协议 — 通过 Agent 间通信借用其他 Agent 的能力
  • Sandbox 执行 — WASM 沙箱中运行动态代码

四、从实验室到边缘：AI 嵌入式的快速路径

这两个实验项目指向一个更大的趋势：AI Agent 正在从云端走向边缘。

4.1 最小化 = 可嵌入

150 行的 Pocket Agent 意味着什么？它意味着：

  • 可以在 Raspberry Pi 上以微秒级延迟运行
  • 可以编译为 MicroPython 在 ESP32 上运行
  • 核心逻辑可以翻译成 C/Rust 嵌入固件

当 Agent 框架从"几百 MB 的依赖链"缩减到"几百行代码"，嵌入式部署就不再是梦想。

4.2 Edge Agent Runtime

基于这些实验，我正在开发一个 Edge Agent Runtime，核心设计理念：

  • 零依赖核心 — 纯 Python，无外部包
  • 可插拔推理器 — MockLLM → ONNX → 云端 API，按硬件能力选择
  • 自组织网络 — 边缘设备通过轻量协议自动组网（Edge Agent Mesh）

五、快速原型开发的方法论

这些实验验证了一个我越来越确信的方法论：先缩到最小，再按需扩展。

5.1 三步法

  1. Strip — 剥离所有框架，用最少的代码实现核心概念
  2. Swap — 用真实组件替换 mock 组件（MockLLM → OpenAI，Dict → VectorDB）
  3. Scale — 添加分布式、持久化、安全等生产级能力

5.2 为什么有效？

因为理解 > 使用。一个理解了 ReAct 循环本质的开发者，切换到 LangGraph 只需要看 30 分钟文档。而一个跳过理解直接用框架的人，遇到问题就是黑箱。

六、下一步

这些实验还在继续演化：

  • Edge Agent Runtime — 核心已完成（31/31 tests），下一步是硬件适配
  • Edge Agent Mesh — 去中心化协作协议已实现核心模块
  • MCP Server for OpenClaw — 把 Agent 工具暴露为标准 MCP 接口
  • Agent Trust Network — 多 Agent 信任模型 + Web 可视化
