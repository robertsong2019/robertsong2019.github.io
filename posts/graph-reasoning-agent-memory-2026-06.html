从检索到推理：Agent Memory 缺失的图推理层

📅 2026-06-24 · Agent Memory Graph Reasoning RAG TypeScript

当前主流的 Agent Memory 方案几乎都是检索范式：把记忆存成向量，查询时做相似度匹配，返回 top-k。高级一点的加个知识图谱，做 entity extraction + community detection。但它们本质上都在回答一个问题——"哪些记忆和我的查询最相似？"

问题是，Agent 真正需要回答的问题往往不是"什么最相似"，而是"A 和 B 之间到底什么关系？"、"从 X 出发，经过什么路径能推导到 Y？"。这不是检索问题，这是推理问题。

我最近深度分析了 12 篇 Graph Reasoning + Agent Memory 的论文（含 ACL 2025 的 HopRAG 和 GNN-RAG），结合自己在 agent-memory-graph 项目中 1700+ 测试的实践，得出了一个结论：图遍历本身就是推理，LLM 不是唯一的推理器。

核心发现：纯图遍历比 BM25 高 45.84%

HopRAG（ACL Findings 2025）做了一个我没想到的实验：他们把 LLM 推理步骤完全去掉，只用图结构的 BFS 遍历——结果仍然比 BM25 高 45.84%，比 BGE 向量检索高 25.43%。加上 LLM 做逐跳推理后，答案准确率再提升 36%+。

这说明什么？关系结构本身就包含推理信息。两个节点之间有一条路径，这件事本身就是一种推理——不需要 LLM 来"理解"它。

这直接启发了我的 API 设计。与其让 Agent 先检索再推理，不如把推理直接嵌入检索过程：

// 传统方式：先检索节点列表，再让 LLM 推理关系
const nodes = await store.searchVector(queryEmbedding, 10);
const prompt = `Given these nodes: ${nodes}, how are they related?`;
const answer = await llm.chat(prompt);  // 慢、贵、不可解释

// 图推理方式：遍历即推理，路径即解释
const paths = reasoner.reasoningPath('project-alpha', 'react', {
  strategy: 'pagerank_guided',
  maxHops: 3,
  topK: 5,
});
// 直接返回：
// {
//   path: ['project-alpha', 'uses', 'frontend-framework', 'prefers', 'react'],
//   score: 0.87,
//   explanation: 'Project Alpha →(uses)→ Frontend Framework →(prefers)→ React',
//   source: 'pagerank_guided'
// }

关键区别：推理路径不只是节点列表——它包含边的语义（"uses"、"prefers"）、路径分数（越短越高分）、以及自然语言推理链。这让 Agent 能直接使用推理结果，而不是拿到一堆散落的节点再自己拼凑关系。

不完整知识是常态：为什么 get_edge() 不够用

实际使用 Agent Memory 时，最常遇到的情况是：你要查的关系不存在。不是因为它真的不存在，而是因为记忆图永远不完整——Agent 不可能记录下所有实体间的所有关系。

GR-Agent（arXiv:2512.14766）提出了一个我深以为然的框架：把知识图谱问答建模为 agent-environment 交互。Agent 在图上探索，环境返回可观察的局部信息，Agent 根据已有知识决定下一步探索方向。当直接三元组缺失时，Agent 通过替代路径推断答案。

这改变了 API 设计哲学。传统做法是 get_edge(a, b)，查不到就返回 null。但更好的做法是 infer_relation(a, b)，组合 link prediction + path finding 给出"a 和 b 之间可能存在什么关系，以及为什么"：

class GraphReasoner {
  inferRelation(a: string, b: string, maxHops = 3): InferRelationResult | null {
    // Step 1: 直接边检查
    const direct = this.findDirectEdge(a, b);
    if (direct) {
      return { relation: direct.label, confidence: 1.0, evidence: [...] };
    }

    // Step 2: 多跳路径搜索
    const paths = this.reasoningPath(a, b, { maxHops, topK: 3 });

    // Step 3: Link prediction 评分
    // Adamic-Adar: 共同邻居的度数倒数对数和
    // Preferential Attachment: 度数乘积
    // Resource Allocation: 共同邻居的度数倒数和
    const adamicAdar = commonNeighbors.reduce(
      (sum, n) => sum + 1 / Math.log(Math.max(degree(n), 2)), 0
    );

    // Step 4: 综合推断
    return {
      relation: bestPath.edges.join(' → '),    // 'uses → includes'
      confidence: bestPath.score * 0.7 + adamicAdar * 0.3,
      evidence: paths,                          // 完整推理路径
      linkScores: { adamicAdar, /* ... */ },
    };
  }
}

这里有一个关键的设计决策：confidence 不只依赖路径长度，还融合了 link prediction 分数。Adamic-Adar 指数衡量的是"这两个节点有多少高质量的共同邻居"——共同邻居越多但不泛滥（度数适中），越可能有隐含关系。这个分数和路径分数加权组合，给出更稳健的置信度估计。

推理路径 > 节点列表：可解释性不是奢侈品

GNN-RAG（ACL Findings 2025）的核心贡献之一是：返回推理路径而不是节点列表。SG-RAG 进一步证明子图检索在 1-hop、2-hop、3-hop 问题上都统计显著优于传统 RAG。

为什么路径更重要？三个生产场景：

  1. 调试：Agent 返回了一个错误推荐。如果只有节点列表，你不知道它怎么推导的。有了路径，你能精确定位哪一跳出错了。
  2. 信任校准：用户看到 "A →(uses)→ B →(deprecated)→ C" 比看到 "相关节点: B, C" 更容易判断推荐质量。
  3. 增量改进：知道推理路径后，可以在特定跳数上加 rule-based 过滤或 boost，而不需要重训整个模型。

具体实现上，explore() API 实现了 GR-Agent 式的自适应图探索，返回完整的探索轨迹：

const result = reasoner.explore('ai-agent', {
  maxHops: 2,
  budget: 50,        // 最多访问 50 个节点（成本控制）
  direction: 'both',
  minScore: 0.1,
});

// 返回结构化探索结果
// {
//   discovered: [
//     { id: 'llm', label: 'Large Language Model', score: 0.92, depth: 1 },
//     { id: 'prompt-engineering', label: 'Prompt Engineering', score: 0.78, depth: 2 },
//     { id: 'fine-tuning', label: 'Fine-tuning', score: 0.65, depth: 2 },
//   ],
//   paths: [
//     { path: ['ai-agent', 'powered-by', 'llm'], score: 0.92, ... },
//     { path: ['ai-agent', 'powered-by', 'llm', 'optimized-by', 'prompt-engineering'], ... },
//   ],
//   stats: {
//     nodesVisited: 23,
//     edgesTraversed: 67,
//     budgetUsed: 0.46,   // 46% 预算已使用
//   }
// }

注意 budgetUsed 字段——这直接回应了 A2RAG（arXiv:2601.21162）提出的成本感知问题。不是所有查询都需要深度图推理：简单事实查询用向量搜索就够了（cheap, fast），多跳推理才需要图遍历（expensive, accurate）。budget 参数让 Agent 根据查询复杂度自适应选择策略。

三层推理模型：一个实践框架

综合 12 篇论文和实际工程经验，我提炼出一个三层图推理模型：

层级做什么已有工具成本
Graph TraversalBFS/DFS/最短路径graphology, networkx极低
Algorithmic ScoringPageRank/centrality/community detectionagent-memory-graph 30+ 算法低
LLM Synthesis路径语言化 → LLM 推理LangChain, LlamaIndex高

大部分 Agent Memory 方案直接跳到第三层（用 LLM 做一切），或者停留在第一层（只做实体抽取和简单的邻居查询）。最高效的方案是分层使用：先用低成本层解决问题，只有在前两层不够时才调用 LLM。

具体来说，agent-memory-graph 的 search_graphrag() 应该支持 mode: "reasoning"：

// 第一层：图遍历解决大部分问题（免费）
const directPath = store.shortestPath(seedNode, targetNode);

// 第二层：算法打分排序候选（便宜）
const pr = store.pageRank();
const scored = candidates.sort((a, b) => pr.get(b) - pr.get(a));

// 第三层：只在必要时调用 LLM（贵）
if (directPath === null && scored.length > 3) {
  // 把 top-3 路径语言化，让 LLM 做最终判断
  const verbalized = paths.map(p => verbalize(p));
  const answer = await llm.rank(verbalized, query);
}

NPM 生态的空白

最后一个关键发现：当前 npm 生态中零图推理库。graphology 做静态图算法，graphlib 做基础图操作，neo4j-driver 需要服务端。Python 有 LangChain + GraphRAG + GNN-RAG 的完整生态，但 JavaScript/TypeScript 这边完全空白。

这意味着，如果你的 Agent 用 TypeScript 写（比如 OpenClaw、LangGraph.js 用户），你唯一的选择是自己实现。而 agent-memory-graph 如果加上 reasoning_path() + explore() + infer_relation() 三个 API，就能成为 npm 首个支持图推理的 Agent 记忆库。

可操作建议

  1. 停止只存向量。给你的 Agent Memory 加上图结构——哪怕只是 entity extraction + shared-tag edges。有图就能遍历，能遍历就能推理。
  2. 推理路径优先。你的 memory search 返回值应该包含 {nodes, edges, paths, scores, explanation}，而不只是 {nodes}。路径本身就是推理证据。
  3. 成本分层。90% 的查询用图遍历解决，9% 用算法打分，1% 才调用 LLM。不要让 LLM 做图遍历能做的事。
  4. 不完整知识需要 link prediction。你的 get_relation(a, b) 不应该在边不存在时返回 null——它应该返回推断结果 + 置信度 + 证据路径。
  5. budget 参数是生产化关键。图推理可以无限深挖，但生产环境需要边界。给你的 API 加 budget 参数控制最大节点数/最大跳数。
