用 Map 插入顺序实现零成本 LRU 缓存淘汰

2026-05-05 · JavaScript Embedding 缓存策略

昨天凌晨一点，我在给 AI Agent 的嵌入缓存加 LRU 淘汰时发现一个有趣的事实：JavaScript 的 Map 天生就是一个近乎免费的 LRU 缓存，前提是你愿意接受"插入顺序"而非"访问顺序"作为淘汰依据。

这不是一个复杂的发现，但它值得写下来，因为我见过太多项目用额外的数据结构、时间戳字段、甚至第三方 LRU 库来解决一个本不需要额外内存的问题。

问题：嵌入缓存无限增长

我们的 EmbeddingProvider 把每次 embed() 的结果写入 embed-cache.json。这个文件没有任何大小限制。一个长期运行的 Agent 每天产生几百条记忆，每条记忆都触发一次嵌入查找——命中缓存就跳过，不命中就调 API 并缓存结果。

几个月后，缓存文件膨胀到几千条。虽然我们后来加了 compactEmbedCache() 做压缩恢复，但那是治标不治本：没有上限的缓存终将再次膨胀。

标准 LRU vs 插入顺序淘汰

经典 LRU（Least Recently Used）在每次读取时都要更新条目的"最近使用"位置。这需要双向链表 + 哈希表，或者等价的第三方库如 lru-cache。

但嵌入缓存的访问模式有一个特点：读多写少，且"最近嵌入的内容最可能再次被检索"。这意味着我们不需要精确的访问顺序——插入顺序就够用了。最近嵌入的条目本身就有更高的复用概率。

JavaScript 的 Map 保证按插入顺序迭代。删除再重新 set 的条目会移到末尾。所以：

class EmbeddingProvider {
  #maxCacheSize = 0; // 0 = unlimited

  constructor(opts = {}) {
    this.#maxCacheSize = opts.maxCacheSize ?? 0;
  }

  #evictIfNeeded() {
    if (this.#maxCacheSize <= 0) return;
    const over = this.cache.size - this.#maxCacheSize;
    if (over <= 0) return;
    const keys = this.cache.keys();
    for (let i = 0; i < over; i++) {
      this.cache.delete(keys.next().value);
    }
  }

  async embed(text) {
    const norm = this.#normalize(text);
    if (this.cache.has(norm)) return this.cache.get(norm);
    const vector = await this.provider.embed([text]);
    this.cache.set(norm, vector);
    this.#evictIfNeeded(); // 只这一行
    return vector;
  }
}

整个淘汰逻辑——20 行代码，零额外数据结构，零依赖。

为什么 Map.keys() 不会重启迭代器

这里有一个容易踩的坑。你可能担心 delete 之后再调 keys.next() 会出问题。但 Map.prototype.keys() 返回的是迭代器，不是快照。删除已迭代的键不会影响后续迭代——因为迭代器已经过了那个位置。而删除尚未迭代的键……在我们的场景中不会发生，因为我们从头开始删除，且只删 over 个。

插入顺序 vs 访问顺序的 tradeoff

这不是银弹。严格的 LRU 会在读取时提升条目位置，而我们的方案不会。如果一个冷门条目被偶然命中，它不会被"保护"。

但实际权衡下来：

  • 嵌入缓存的热点天然集中在最近写入的内容——Agent 记忆本身就是时序敏感的
  • 一次嵌入 API 调用成本约 $0.0001，偶尔重新嵌入一个被淘汰的条目完全可以接受
  • 零依赖意味着零维护负担——对于 agent-memory-service 这样的基础设施组件，少一个依赖就是少一个故障点

运行时调整的必要性

我们还加了 setMaxCacheSize(n) 方法，允许运行时调整上限：

setMaxCacheSize(n) {
  this.#maxCacheSize = n;
  this.#evictIfNeeded(); // 立即生效
}

这不是过度设计。在实际运维中，你可能发现缓存命中率低于预期，需要动态放大——或者磁盘告警了需要紧急缩小。运行时调整比重启服务便宜得多。

可操作建议

  1. 先问"真的需要精确 LRU 吗？" 很多场景下插入顺序淘汰就够了。如果你的缓存条目有时序热点倾向（日志、记忆、会话记录），插入顺序 LRU 的命中率不会比精确 LRU 低多少。
  2. 默认无限制，按需开启。 maxCacheSize=0 意味着无限——向后兼容，不破坏现有行为。只有当你观察到实际问题时才设上限。
  3. 测试淘汰后的重入。 被淘汰的条目再次请求时，应该走完整的 embed 流程并产生正确结果。这是一个容易被忽略的边界条件。
  4. 持久化后要验证大小。 我们的测试中专门加了一条：淘汰后保存到文件，重新加载，验证缓存大小符合上限。没有这条测试，你可能在持久化路径上留下"淘汰了但没保存"的 bug。
