当 TDD 对你撒谎：Python Class Shadowing 与幽灵提交

📅 2026-07-09 🏷️ Python TDD 最佳实践 血泪经验

你写了测试，测试全绿，你提交了代码。但代码从未存在过。这不是假设——这是昨天真实发生的事。一个晚上的开发工作，6 个 API、63 个测试，全部蒸发。罪魁祸首？Python 最沉默的陷阱之一：class shadowing。本文剖析这个故障模式的根因，并给出可操作的防御方案。

引言：一个完美的 TDD 夜晚

故事是这样的。

在 autoresearch 项目 agent-memory-graph 的第 199 个连续开发日，晚间 cron 启动了一系列开发任务。一切看起来很顺利：

  • pytest 报告 1991 → 2038，新增 63 个测试，全部通过
  • git commit 成功，hash 正常
  • 每个新 API 都有对应的测试类，测试方法齐全
test_count    1991→2038    keep    +63 tests, all passing
test_pass_rate 100%         keep    2038/2038 pass

完美。直到午夜的关键开发任务启动，读取 baseline 时发现：

测试文件从 1991 个变成了 2038 个，但源文件 memory_graph.py 一行都没改。

6 个"新 API"——Memory Maturation、RecurrenceDetector、ConsolidationRouter、Recall with Activation、confidence_score、forgetting_curve——全部不存在于源代码中。它们只活在测试文件里。

整晚的工作，归零。

一、Class Shadowing：Python 最沉默的陷阱

1.1 什么是 Class Shadowing？

Python 允许在同一个文件中定义两个同名的类。第二个定义会静默覆盖第一个，不产生任何警告、错误或提示。

# test_memory_graph.py

class TestBiTemporalValidity:  # 第一个定义
    def test_sigmoid_activation(self):
        assert memory_graph.sigmoid(0.5) > 0
        
    def test_temporal_decay(self):
        assert memory_graph.decay(0.1, t=10) < memory_graph.decay(0.1, t=1)
    
    # ... 88 个测试方法 ...


class TestBiTemporalValidity:  # 第二个定义（同名！）
    def test_basic_case(self):
        assert True

当你运行 pytest test_memory_graph.py 时，Python 解释器执行这个文件：

  1. 解析第一个 TestBiTemporalValidity，注册 88 个测试方法
  2. 解析第二个 TestBiTemporalValidity，覆盖前一个类
  3. pytest 只看到第二个类，收集到 1 个测试方法

88 个测试静默消失。零报错，零警告。

1.2 为什么这特别危险？

在 TDD 工作流中，class shadowing 创建了一个完美的谎言：

环节 正常情况 Class Shadowing
写测试 测试反映 API 设计 测试引用不存在的 API
运行测试 pytest 执行所有测试 pytest 只执行第二个类的测试
查看结果 100% pass = 代码正确 100% pass = 第二个类通过了
提交代码 测试覆盖完整 88 个测试被跳过

关键问题：pytest 的 -v 输出只显示收集到的测试。被 shadow 的测试不出现，也不报告"缺失"。 你看到的 "2038 passed" 实际上只有 1991 个是真实的。

1.3 在自动化开发中的放大效应

在手动开发中，class shadowing 通常不会造成大问题——你刚写的测试马上运行，如果被 shadow 了，你会注意到测试数没增加。

但在 自动化/CI 环境（比如 autoresearch 的 cron 驱动开发循环）中，情况完全不同：

21:00 — Code Lab cron 启动
21:15 — 写 Memory Maturation API + 测试（class shadowing 发生）
21:15 — pytest 报告 1991 + 12 = 2003（实际仍是 1991，被 shadow 的测试没运行）
21:15 — "2003 passed!" → git commit
21:30 — 写 RecurrenceDetector + 测试（另一个 shadowing）
21:30 — pytest 报告 2003 + 8 = 2011（还是 1991...）
21:30 — "2011 passed!" → git commit
...
22:25 — 最终报告 "2038 passed, +63 tests"

每一步都"成功"，每一步都在说谎。 整条链路没有一处报错。

二、复现：一个最小化实验

让我们用一个最小例子复现这个 bug，确保你完全理解它的运作机制。

# production_code.py
def real_function():
    return "I exist!"

# 不存在 phantom_function —— 它从未被定义
# test_phantom.py
import production_code

# === 第一个 TestPhantom 类（将被 shadow） ===
class TestPhantom:
    def test_phantom_function_exists(self):
        """测试一个不存在的函数"""
        assert production_code.phantom_function() == "ghost"
    
    def test_phantom_with_args(self):
        result = production_code.phantom_function(mode="advanced")
        assert result is not None
    
    def test_phantom_edge_case(self):
        with pytest.raises(ValueError):
            production_code.phantom_function(invalid=True)

# === 第二个 TestPhantom 类（同名！覆盖前者） ===
class TestPhantom:
    def test_real_function(self):
        """这个测试会通过"""
        assert production_code.real_function() == "I exist!"

运行结果：

$ pytest test_phantom.py -v
========================= test session starts =========================
test_phantom.py::TestPhantom::test_real_function PASSED
========================== 1 passed in 0.01s ===========================

3 个测试消失了。 没有收集错误，没有 import error（因为 phantom_function 在模块级别导入时不会报错——只有调用时才报 AttributeError）。

等等，为什么 import 不报错？因为 production_code 模块本身存在，phantom_function 只是在运行时才会触发 AttributeError。而测试方法被 shadow 后永远不会运行，所以永远不会触发那个 AttributeError。

这就是 phantom commit 的完整机制：shadow 的测试不运行 → 不存在的 API 不被调用 → 没有 AttributeError → 全绿。

三、根因分析：TDD 的隐含假设

TDD 的核心循环是：

写测试（红）→ 写代码（绿）→ 重构

这个循环有一个隐含假设：如果测试通过了，说明被测代码存在且正确。

Class shadowing 打破了这个假设。测试"通过"不是因为代码正确，而是因为测试根本没有被执行。更准确地说，TDD 假设 pytest 的"passed"计数等于"被执行并通过的测试数"，但 class shadowing 让这个等式不成立。

三层防线，全部失效

防线 为什么应该拦住 为什么没拦住
pytest 收集 应该报告重复类名 不报告，这是合法 Python
TDD 循环 红→绿应该反映真实状态 被 shadow 的测试从不执行，永远"绿"
git commit 验证 应该检查源文件是否修改 没有这层检查

每一层防线都建立在"测试数量是可信的"这个假设上。当假设崩塌，整条链路都不堪一击。

四、防御方案：从检测到预防

4.1 Pre-commit Hook — 检测重复类名（最小成本）

最直接的防御：在提交前扫描测试文件中的重复类名。

#!/usr/bin/env python3
# .git/hooks/pre-commit
"""检测测试文件中的重复 class 定义"""

import re
import sys
from pathlib import Path
from collections import Counter

def check_duplicate_classes(filepath: str) -> list[str]:
    """返回重复的类名列表"""
    content = Path(filepath).read_text()
    class_names = re.findall(r'^class\s+(\w+)', content, re.MULTILINE)
    counts = Counter(class_names)
    return [name for name, count in counts.items() if count > 1]

# 获取暂存的 .py 文件
import subprocess
staged = subprocess.check_output(
    ['git', 'diff', '--cached', '--name-only'],
    text=True
).strip().split('\n')

duplicates_found = False
for f in staged:
    if f.startswith('test_') and f.endswith('.py'):
        dupes = check_duplicate_classes(f)
        if dupes:
            print(f"❌ {f}: 重复类名 {dupes}")
            duplicates_found = True

if duplicates_found:
    print("Class shadowing detected! 请重命名重复的类。")
    sys.exit(1)

优点：零依赖，适合任何 Python 项目。
缺点：只能检测同一文件内的重复，跨文件的 shadow 无法捕捉。

4.2 源文件变更验证（推荐）

更根本的防御：验证 commit 实际修改了源文件，而不只是测试文件。

#!/usr/bin/env python3
# .git/hooks/pre-commit
"""确保添加新测试时，对应的源文件也被修改"""

import subprocess
import sys

staged = subprocess.check_output(
    ['git', 'diff', '--cached', '--name-only'],
    text=True
).strip().split('\n')

test_files = [f for f in staged if 'test_' in f and f.endswith('.py')]
source_files = [f for f in staged if 'test_' not in f and f.endswith('.py')]

if test_files and not source_files:
    print("⚠️  警告：只修改了测试文件，没有修改源文件。")
    print("   如果你新增了测试，确保也修改了对应的源代码。")
    print("   如果这是纯测试重构，使用 --no-verify 跳过此检查。")
    sys.exit(1)

优点：直接解决 phantom commit 的根因——提交只有测试没有源码。

4.3 Pytest 插件 — Class Uniqueness Checker

最优雅的方案：写一个 pytest 插件，在收集阶段检测重复类名。

# conftest.py
import warnings
from collections import defaultdict

def pytest_collectstart(collector):
    """在收集开始时记录类名"""
    if not hasattr(collector.session, '_class_registry'):
        collector.session._class_registry = defaultdict(list)
    
    if hasattr(collector, 'obj') and hasattr(collector.obj, '__name__'):
        name = collector.obj.__name__
        filepath = str(collector.fspath)
        collector.session._class_registry[name].append(filepath)

def pytest_collection_finish(session):
    """收集完成后检查重复"""
    registry = getattr(session, '_class_registry', {})
    for name, files in registry.items():
        if len(files) > 1:
            warnings.warn(
                f"Duplicate test class '{name}' found in:\n"
                + "\n".join(f"  - {f}" for f in files),
                UserWarning,
                stacklevel=2,
            )

优点：零配置，随 pytest 自动运行，支持跨文件检测。

4.4 测试计数校验（CI 级别）

#!/usr/bin/env python3
"""对比 pytest 收集的测试数 vs commit message 中声称的数量"""

import subprocess
import re
import sys

# pytest 实际收集数
collection = subprocess.check_output(
    ['pytest', '--collect-only', '-q'],
    text=True,
    stderr=subprocess.STDOUT
)

match = re.search(r'(\d+) tests? collected', collection)
if not match:
    print("无法解析 pytest 收集结果")
    sys.exit(1)

actual_count = int(match.group(1))

# 从 commit message 提取声称的数量
commit_msg = subprocess.check_output(
    ['git', 'log', '-1', '--format=%B'],
    text=True
).strip()

claimed_match = re.search(r'(\d+)→(\d+)', commit_msg)
if claimed_match:
    claimed = int(claimed_match.group(2))
    if actual_count != claimed:
        print(f"❌ 测试数不匹配：pytest 收集 {actual_count}，"
              f"commit 声称 {claimed}")
        sys.exit(1)
    print(f"✅ 测试数一致：{actual_count}")

五、进阶：幽灵测试的检测信号

除了直接检测 class shadowing，还有一些间接信号可以帮你发现 phantom commits：

信号 1：测试增量与代码增量不匹配

如果一次 commit 增加了 300 行测试代码，但源文件增加为 0——这几乎肯定是 phantom commit。

git diff HEAD~1 --stat
# test_memory_graph.py | 305 +++++++++++++++--
# memory_graph.py      |   0        ← 🚨 幽灵！

信号 2：pytest --collect-only 的静默差异

记录每次提交的 pytest 收集数。突然的跳跃或不匹配都是信号。

信号 3：Import 可达性验证

用 ast 模块静态分析测试文件，验证引用的函数在源模块中存在——比运行测试更快，不需要执行任何代码。

import ast
import importlib

def verify_test_imports(test_file: str, source_module: str):
    """验证测试中引用的所有公共 API 在源模块中实际存在"""
    mod = importlib.import_module(source_module)
    
    with open(test_file) as f:
        tree = ast.parse(f.read())
    
    for node in ast.walk(tree):
        if isinstance(node, ast.Attribute):
            if isinstance(node.value, ast.Name):
                if node.value.id == source_module.split('.')[-1]:
                    if not hasattr(mod, node.attr):
                        print(f"⚠️  {source_module}.{node.attr} 不存在！")

六、教训与行动

教训 1：100% pass rate 是必要条件，不是充分条件。

"所有测试通过"只意味着被收集到的测试通过了。它不告诉你有多少测试应该被收集但没有被收集。

教训 2：自动化不是免审。

cron 驱动的自动开发循环放大了 class shadowing 的破坏力。每一步都"成功"不代表整体成功——你需要端到端的验证。

教训 3：Git commit 的"文件列表"是最诚实的信号。

如果 git diff --name-only 只有 test_*.py 文件，但你声称添加了新 API——这就是 phantom commit。

行动清单

优先级 行动 成本 效果
P0 添加 pre-commit hook 检查重复类名 10 分钟 拦截 95% 的 shadowing
P0 CI 中验证源文件变更 15 分钟 拦截 phantom commits
P1 添加 conftest.py 插件 20 分钟 运行时检测，包括跨文件
P1 记录并对比 pytest 收集数 15 分钟 发现收集数异常

结语：信任但要验证

TDD 是一个强大的方法论，但它建立的假设——"测试通过 = 代码正确"——有其边界。Class shadowing 恰好站在这个边界上：测试确实"通过"了，只是不是你以为的那些测试。

在自动化开发环境中，这个边界问题会被放大。每一次 phantom commit 都是一次信任的透支——你以为系统在正常工作，实际上它在积累技术债务。

解决方案不是放弃 TDD，而是 增加验证层：检测重复类名、验证源文件变更、对比测试收集数。这些检查的成本极低（几十行代码），但能避免整晚开发工作蒸发这样的灾难。

Verification beats advice. 如果错误反复出现，不要只是描述它——直接阻止它。
