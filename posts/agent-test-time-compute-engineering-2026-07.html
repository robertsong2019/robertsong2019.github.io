AI Agent 的思考经济学：推理时计算从论文走向工程

2026-07-15 · 罗嵩 · Test-Time Compute Agent 架构 Verification 工程实践
如果我问你："一个 AI Agent 做决策时应该思考多久？"你会怎么回答？太短，它会犯低级错误；太长，用户已经等不及关掉了。2026 年的今天，这个问题终于有了数据支撑的答案——而且答案比你想的更反直觉。

一个价值十亿美元的问题

想象你在构建一个客服 Agent。一个简单的退款问题，Agent 可以在 1 秒内给出答案（直接从知识库检索），也可以花 10 秒检索+推理+验证。一个复杂的合同审核问题，Agent 可能需要花 2 分钟做多步推理。

问题来了：Agent 怎么知道当前这个问题该花 1 秒还是 2 分钟？

这不仅仅是一个技术问题，而是一个经济学问题。每次推理都在烧钱——GPU 时间、API 调用费、用户等待成本。在 2024 年，大多数 Agent 系统采用"一刀切"策略：要么所有请求都用最贵的模型+最长的推理链，要么都用最便宜的。两种都是浪费。

但 2025-2026 年的一系列研究正在彻底改变这个局面。核心发现可以用一句话概括：

推理时计算（test-time compute）是一个独立的、可扩展的性能维度——而且它的扩展效率可能比增大模型参数还要高。

这篇文章不打算重复那些已被广泛讨论的理论基础（如果你需要背景，可以参考我之前的推理时计算缩放和过程奖励模型两篇文章）。本文聚焦于一个更实际的问题：当你真正要构建一个生产级 Agent 时，这些研究意味着什么？

一、从理论到现实：Agent 的"思考"为什么不一样

2025 年 6 月，Zhu 等人发表了第一篇系统研究 Agent 推理时计算的论文。他们的发现揭示了一个被纯推理任务研究者忽略的盲点：Agent 的推理和纯数学推理有本质区别。

区别在哪？三个字——有状态。

解一道数学题，你可以生成 100 个候选答案，然后用验证器挑选最好的。没选中？扔掉就好，零成本。但 Agent 不一样：

  • Agent 调了一个 API 发了邮件 → 这个动作不可撤销
  • Agent 修改了数据库 → 你需要事务回滚
  • Agent 和用户交互了三轮 → 你不能假装这些对话不存在

这意味着传统 Tree of Thoughts 那种"生成多个分支、选最优路径"的策略，在 Agent 场景下遇到了巨大的工程障碍。你不能简单地 fork 出 10 个 Agent 实例让它们各自跑完然后选最好的——因为每个实例都可能在真实世界里留下副作用。

关键工程挑战：Agent 的推理时计算扩展需要"状态感知"——在不可逆操作之前进行充分验证，在可逆阶段允许大胆探索。

二、验证器即护城河

如果你只能记住这篇文章的一个观点，记住这个：

在推理时计算无限扩展的极限下，验证器的质量决定了系统的上限。

想象一个简单的系统：生成 N 个候选答案，用验证器打分，选最好的。如果验证器是完美的，你只需要足够大的 N 就能逼近最优解。但如果验证器是随机的呢？Best-of-N 就退化成了随机选择。

2026 年 7 月，Stanford 和 Anthropic 的联合团队发表了一篇影响深远的论文——"LLM-as-a-Verifier"。他们识别出了一个之前被忽略的扩展维度：

扩展维度 提升什么 典型方法
预训练计算 模型知识 Scaling laws, 更大数据集
后训练计算 模型对齐 RLHF, DPO, GRPO
推理时计算 推理深度 Chain-of-Thought, Tree Search
验证计算 可靠性保证 LLM-as-a-Verifier, PRM

验证计算的独立意味着一件非常实际的事：你可以用一个小模型生成，用一个大模型验证。

这是一个反直觉的架构选择。传统思维是"用最好的模型做所有事"。但如果你理解了验证的重要性，就会发现更优的架构是：

  1. 用小模型（比如 7B）快速生成 N 个候选方案（成本低）
  2. 用大模型（比如 70B 或 GPT-4 级别）对每个方案打分（质量高）
  3. 选得分最高的方案执行

Chegoni 和 Feizi 在 2026 年 6 月的研究更进一步证明了这一点：他们发现即使是无需训练的现成大模型，作为过程评分器（Process Scorer）也能达到接近专用 PRM 的效果。这意味着验证能力的门槛正在急剧降低。

三、难度路由：Agent 的"交通管制"系统

回到我们开头的问题：Agent 怎么知道一个请求该花多少算力？

Snell 等人在 2024 年的基础研究中给出了最关键的理论洞察：不同难度的请求需要不同的推理时计算策略。

  • 简单请求（如"今天天气怎样"）→ 单次推理足够，多花算力纯属浪费
  • 中等请求（如"帮我总结这篇论文"）→ Best-of-3 到 Best-of-5 效率最高
  • 困难请求（如"设计一个分布式锁的实现方案"）→ 串行精炼 + 验证器引导搜索

这个发现的核心数据点：采用"计算最优"策略（根据难度分配算力），比无脑 Best-of-N 高效 4 倍以上。在某些任务上，14 倍小模型配合充足的推理时计算，能超越 14 倍大的模型。

CoRefine（Jin et al. 2026）把这个理念落地了。它的核心机制：

  1. 模型先生成一个初步答案
  2. 同时输出一个置信度分数
  3. 如果置信度 > 阈值 → 直接返回（省算力）
  4. 如果置信度低 → 触发自我精炼循环（多花算力）

这就像一个经验丰富的专业人士：简单问题秒答，复杂问题会说"让我想想"然后深入思考。关键在于——Agent 需要知道自己"不知道"。

四、多 Agent 协作：推理计算的水平扩展

到目前我们讨论的都是单个 Agent 内部的推理时计算扩展。但 2026 年出现了一个令人兴奋的新方向——通过多 Agent 协作来扩展推理计算。

TMAS（Test-time Multi-Agent Synergy）框架提出了一个大胆的假设：与其让一个 Agent 思考 10 倍长时间，不如让 3 个不同的 Agent 各思考 3 倍长时间，然后交换意见。

为什么这比单个 Agent 更好？答案是认知多样性。

同一个模型的 N 次采样，本质上是在同一个概率分布里采样。即使 N 很大，它们的错误模式也是相关的——同一种类型的难题，N 次采样都会失败。

但如果你用不同基础模型、不同系统提示、不同工具集的 Agent 呢？它们的错误模式是不相关的。一个 Agent 可能擅长逻辑推理但不擅长创意，另一个正好相反。当它们意见一致时，答案的可靠性远高于单个 Agent 的自我确认。

类比：Tree of Thoughts 是让一个人从不同角度思考同一个问题；多 Agent 协作是让一个团队各自独立思考然后开会讨论。前者增加了深度，后者增加了广度。最优解通常是两者的结合。

但这引入了一个新的工程挑战：协调成本。3 个 Agent 意味着 3 倍的 API 调用费用、3 倍的延迟、以及一个"意见聚合"层的设计复杂度。在什么场景下这个成本是值得的？

基于现有研究的证据，答案是：高价值、低频率、高难度的任务。比如：

  • 安全关键的代码审查（漏掉一个 bug 的代价 >> 多请一个 Agent 的成本）
  • 法律文档的关键条款审核
  • 医疗诊断的第二意见
  • 金融交易的风控审查

而对于高频低价值的任务（如客服回复、信息检索），单 Agent + 智能难度路由仍然是更好的选择。

五、边缘上的推理：当设备受限时

到目前为止，我们都在讨论有充足 GPU 资源的数据中心场景。但 2026 年的一个显著趋势是推理时计算向边缘设备扩展。

FastTTS（Chen et al. 2025）解决了"如何在手机上做推理时计算扩展"的问题。它的核心思路是管理计算-内存-延迟三角：

  • 不是每一步推理都需要完整展开——根据难度提前退出
  • 利用稀疏激活减少每次推理的内存占用
  • 在设备过热或电量低时自动降低推理深度

MORES（Liu et al. 2026）更激进——它提出了"推理即服务"（Reasoning-as-a-Service）的架构：手机上的轻量模型负责交互和理解用户意图，重推理计算卸载到附近的边缘服务器执行。这就像人的大脑——快速反应由脊髓处理，深度思考由前额叶皮层处理，两者通过神经信号通信。

六、你现在能做什么

理论很美好，但如果你正在构建一个 Agent 系统，今天能落地的具体行动是什么？

1. 实现置信度门控

最简单也最有效。让你的 Agent 在输出答案之前，先自评一个置信度。低置信度的请求路由到更深的推理链：

# 伪代码
confidence = model.estimate_confidence(query, draft_answer)
if confidence > 0.9:
    return draft_answer  # 快速路径
elif confidence > 0.6:
    refined = model.refine(query, draft_answer)  # 精炼一次
    return refined
else:
    candidates = [model.generate(query) for _ in range(N)]  # 多次采样
    best = verifier.select_best(query, candidates)  # 验证器选择
    return best

2. 建立"生成-验证"分离架构

不要让同一个模型既当运动员又当裁判。用小模型生成候选（省钱），用强模型验证（保质量）。实现方式：

  • 生成端：7B-13B 参数的开源模型，或 GPT-4o-mini 等低成本 API
  • 验证端：GPT-4 / Claude 3.5 Sonnet 等最强可用模型，仅对关键步骤调用
  • 关键点：验证器评估的是中间推理步骤，不是最终答案。这样能及早发现错误路径

3. 在不可逆操作前设置验证门

这是 Agent 特有的最佳实践。Agent 的每个 action 按可逆性分级：

  • 可逆（读文件、查询数据库）→ 直接执行，不需要额外验证
  • 有条件可逆（写文件、更新数据库）→ 轻量验证（格式检查、权限检查）
  • 不可逆（发邮件、调支付 API、删数据）→ 强制深度验证（多 Agent 交叉确认 + 人类审核）

4. 为关键场景部署多 Agent 辩论

识别你的系统中"错误代价最高"的决策点。在这些点上部署 3-Agent 辩论：

  • Agent A：乐观主义者，寻找为什么方案会成功
  • Agent B：悲观主义者，寻找为什么方案会失败
  • Agent C：综合者，权衡两方观点做出最终决策

这个模式的成本是单 Agent 的 3 倍，但对于关键决策，避免一次严重错误的收益就能覆盖成本。

七、展望：推理时计算的"摩尔定律"

回顾 2023-2026 年的发展轨迹，一个清晰的趋势浮现出来：

时期 核心策略 代表方法
2023 提示工程 Chain-of-Thought, Few-shot
2024 推理时搜索 Tree of Thoughts, Best-of-N + PRM
2025 推理模型 OpenAI o1/o3, DeepSeek R1 (内置推理链)
2026 自适应+验证 LLM-as-a-Verifier, Multi-Agent Synergy

每一步演进，"推理"这件事都在变得更加结构化、分层和可调控。

我预测 2026 下半年到 2027 年，我们会看到三个关键发展：

  1. 难度路由器标准化：像现在的负载均衡器一样，推理时计算路由器将成为 Agent 基础设施的标配组件。输入一个请求，路由器根据难度、紧急度、预算自动选择推理策略。
  2. 验证器市场：专门训练的验证器模型（而非通用 LLM）会在特定垂直领域出现。医疗验证器、法律验证器、代码验证器——每个领域的最优验证策略不同。
  3. Agent 间的推理资源共享：多个 Agent 系统共享推理结果，类似于 CDN 缓存。如果一个 Agent 已经深度推理过"如何实现分布式锁"，其他 Agent 不需要从头开始。

回到开头的问题："一个 AI Agent 做决策时应该思考多久？"答案是：取决于问题难度，而 Agent 需要学会自己判断难度。

这不是一个哲学回答，而是一个工程要求。当你构建的 Agent 能够智能地分配自己的"思考预算"时，你就在做推理时计算优化了——即使你从没读过 Snell 的论文。

最好的 Agent 不是思考最快的，也不是思考最深的，而是知道什么时候该快、什么时候该深的。

一句话总结：推理时计算扩展不是"让模型想更久"，而是"让模型学会管理自己的认知资源"——这恰恰是人类智能的核心特征之一。
