第 03 课  ·  C vs Rust  ·  形式化验证的根本差异

C 能被证明，
Rust 是不是
一样？ — 答案是「能」，但「证明什么」完全不同

你已经看到 C 怎么靠 AutoCorres 翻译成数学。Rust 也能这么干—— 但关键洞察是：Rust 的类型系统已经替你证明了 C 要辛苦证明的很多东西。 所以同样写一行代码，证明的「起点」完全不同。

看直接答案 三大差异
01 / 直接答案
TL;DR

能证明。 但 Rust 不需要「翻译管道」—— 它的类型系统已经是部分数学， 所以证明工具往往直接嵌在 Rust 里。

C 离数学远（指针、副作用、UB），所以需要 C parser + AutoCorres 把它一路翻译成 monad。 Rust 的仿射类型（affine type）和生命周期本身就是逻辑概念—— borrow checker 在编译期就证明了内存安全。 你不用「翻译」，你在 Rust 里直接写证明。

这意味着同一个 strlen，在两种语言里要证明的工作量天差地别。 下面这个对比表是整份文档的核心——仔细看。

02 / 三大根本差异

不是「能/不能」，
是「起点不同」。

i

免费证明

Rust 编译器免费送了大量 C 必须辛苦证明的性质： 内存安全、没有数据竞争、没有空指针解引用。这些在 C 里要写几百行证明。

C: 写规范 + 200 行证明
Rust: 编译期自动 ✓ (borrow ck)
ii

无需翻译

C 要 C parser + AutoCorres 翻译成 monad，因为 C 离数学太远。 Rust 本身就是部分数学—— 你在 Rust 语法里写证明，不用切到外部证明助手。

C: → Isabelle → SIMPL → ...
Rust: 直接 #[verus::spec] 注解
iii

聚焦功能

因为内存安全已「免费」，Rust 验证把精力集中在 功能正确性—— 程序算出的结果对不对。这恰恰是 C 和 Rust 都难的部分。

C: 80% 内存安全 + 20% 功能
Rust: 5% 内存 + 95% 功能

04 / 同一个 strlen，两种证明

对比一下证明脚本。

下面是同一个 strlen，左边是 C + AutoCorres，右边是 Rust + Verus。 注意 Rust 版本更短、更内嵌——你不用切换语言，证明就在 Rust 代码里。

my_strlen · C + AutoCorres ~200 行证明 
// 1. C 源代码
unsigned my_strlen(char *s) {
    unsigned i = 0;
    while (*s) { i++; s++; }
    return i;
}

(* 2. AutoCorres 抽象后 *)
my_strlen :: char ptr ⇒ natMonad
my_strlen s = do {
    ... (* monad 形式 *)
}

(* 3. 证明内存安全 + 功能 *)
lemma my_strlen_correct:
  shows "Γ ⊨⇩t {valid s}
       my_strlen s
       {r. r = len s}"
proof ...
  (* 写循环不变式 *)
  (* 证明内存安全 *)
  (* 证明功能正确 *)
qed
(* ↑ 总共约 200 行 *)
my_strlen · Rust + Verus ~30 行证明 
// 规范和实现都在 Rust 里
#[verus::spec]
fn spec_strlen(s: Seq<u8>) -> nat
    recommends {
    // 数学定义：找第一个 0
    s.position(|c| *c == 0).unwrap_or(s.len())
}

#[verus::verifier]
fn my_strlen(s: &[u8]) -> usize
    ensures |r: usize|
        r == spec_strlen(s@)
{
    let mut i = 0;
    while i < s.len() && s[i] != 0 {
        i += 1;
    } invariant i <= s.len()
        && /* 已扫部分无 0 */ true;
    i
}
// ↑ 内存安全免费；
//   只证功能正确（~30 行）

关键区别：左边要写内存安全证明（~150 行）+ 功能证明（~50 行）； 右边内存安全由 borrow checker 免费保证， 只需写功能证明（~30 行）。同样的程序，工作量 1/6。

05 / 工具栈对比

两个生态，完全不同的工具。

因为证明范式不同，Rust 有自己的一套验证工具，跟 C 的 Isabelle 生态是平行的。 下面是最重要的几个。

C / 系统软件验证

AutoCorres + C parser
seL4 团队 · Isabelle/HOL
C → SIMPL → 抽象 monad。seL4 同款。 翻译管道式，证明在 Isabelle 里写。
VST
Princeton · Coq + 分离逻辑
用分离逻辑验证 C 代码。CompCert（形式化验证的 C 编译器）同生态。
Frama-C / WP
CEA · ACSL 规范 + SMT
工业界用得多。ACSL 注解 + WP 插件 + SMT solver 自动证明。

Rust 验证

Verus
Microsoft Research · CMU · VMware
最重要。直接在 Rust 语法里写证明，用 SMT solver 自动验证。 目标：完整功能正确性。
Flux
AWS · CMU
轻量级 refinement type。AWS s2n-quic（QUIC 实现）在用。 检查整数范围、向量长度等不变式。
Creusot
LRI · Paris-Saclay
基于 Why3 + Pearlite。把 Rust 翻译成 Why3，再证明。 介于 Verus 和传统翻译管道之间。
RefinedRust
MPI-SWS · Iris / Coq
基于 Iris 分离逻辑。类型系统层做 refinement。 RustBelt 的延续（证明 Rust 自己的安全性）。
06 / 必须说清楚的陷阱
⚠ THE CATCH

但 Rust 有 unsafe。

Rust 的「免费证明」只在 safe Rust 里成立。一旦你写 unsafe， 你就手动绕过了 borrow checker——回到了和 C 一样的境地： 必须自己证明内存安全、无别名、无 UB。

这就是为什么 Verus、Flux 这些工具特别强调能验证 unsafe 代码—— 因为内核、嵌入式、FFI 这些场景不可避免要用 unsafe。 真实工程里：safe Rust 的部分免费安全，unsafe Rust 的部分要辛苦证明。

结论：Rust 不是「不用证明」，而是「把证明的边界从全部代码缩小到 unsafe 边界」。 这仍然是巨大的胜利——但不是万能的。

07 / 总结

回到你的问题：Rust 是不是一样？

能证明，但起点不同。 C 从「全凭自己」开始（指针、UB、别名都要证）； Rust 从「内存安全已免费」开始（只证功能）。

工具不同。 C 靠 AutoCorres 这种「翻译管道 + 外部证明助手」； Rust 靠 Verus 这种「在 Rust 里直接写证明 + SMT 自动验证」。

真实价值： 验证同样一段系统代码，Rust 的工程量约为 C 的 1/3 到 1/6。 这就是为什么 Microsoft、AWS 都在投 Rust 验证—— 它让形式化验证第一次有可能进入工业日常。
