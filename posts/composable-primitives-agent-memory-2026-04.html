可组合原语：Agent Memory Service 的 API 设计哲学

2026-04-29 · 技术随笔 · 基于 Agent Memory Service (AMS) 开发实践

过去一个月，我用 autoresearch 方法论持续迭代 Agent Memory Service (AMS)——一个给 AI Agent 用的本地记忆引擎。从最初 90 个测试到今天的 540 个，API 从 5 个方法膨胀到 30+。但真正有意思的不是数量增长，而是一个反复出现的设计模式：可组合原语（Composable Primitives）。

什么是可组合原语

Unix 哲学说"做好一件事"。可组合原语是这个思想在 API 层面的延伸：每个方法只做一件事，但几件事可以串起来变成强大的工作流。

举个例子。AMS 最近加了三个方法，看似独立，其实形成了一条完整的流水线：

findDuplicatePairs() → memoryMerge() → bulkMerge()
  • findDuplicatePairs() — 检测哪些记忆是重复的
  • memoryMerge(id1, id2, opts) — 合并两条记忆，支持多种冲突策略
  • bulkMerge(pairs, opts) — 批量合并，自动跳过已删除的

单独看，每个方法都很简单。memoryMerge 不过 60 行代码，bulkMerge 只有 20 行——它本质上就是个循环。但组合起来，你就有了一个完整的去重管线。

为什么不让 bulkMerge 直接包含检测逻辑？

这是一个关键的设计决策。你完全可以写一个 autoDeduplicate() 一把梭——检测、合并、删除全包。很多库确实这么做。但这样的问题是：

  1. 无法调试 — 出了问题你不知道是检测错了还是合并错了
  2. 无法复用 — 也许你想手动挑选合并哪些对？
  3. 无法测试 — 你得为整个流程写集成测试，而不是为每步写单元测试
  4. 无法中断 — 用户无法在检测和合并之间插入手动审核

拆成原语之后，每一步都可独立验证、独立使用。10 个测试覆盖 memoryMerge 的策略组合，5 个测试覆盖 bulkMerge 的错误处理，而不是 15 个测试试图覆盖两者的笛卡尔积。

代码示例：memoryMerge 的策略模式

这是一个有趣的微型设计。合并两条记忆时，内容和标签的冲突解决策略不同：

// memoryMerge 的核心逻辑（简化版）
function memoryMerge(id1, id2, opts = {}) {
  const primary = store.get(id1);
  const secondary = store.get(id2);
  if (!primary || !secondary || id1 === id2) return null;

  // 内容策略：concat | keep-longer | keep-newer | manual
  const strategies = {
    concat: () => primary.content + '\n' + secondary.content,
    'keep-longer': () => primary.content.length >= secondary.content.length
      ? primary.content : secondary.content,
    'keep-newer': () => primary.updatedAt >= secondary.updatedAt
      ? primary.content : secondary.content,
    manual: () => opts.content,
  };

  const contentStrategy = opts.contentStrategy || 'concat';
  primary.content = strategies[contentStrategy]();

  // 标签策略：union（默认）| primary | secondary
  const tagStrategy = opts.tagStrategy || 'union';
  if (tagStrategy === 'union') {
    primary.tags = [...new Set([...primary.tags, ...secondary.tags])];
  }

  // 链接策略：rewire（迁移到 primary）| drop
  if (opts.linkStrategy !== 'drop') {
    rewireLinks(secondary.id, primary.id);
  }

  store.delete(secondary.id);
  return primary;
}

注意这里没有用 class、没有用策略模式的继承体系——就是一个简单的对象映射。够用就行，不过度设计。这是 autoresearch 方法论的核心精神：先让它工作，先测好它，需要时再抽象。

分支系统的原语三角

另一个例子是 AMS 的分支系统。记忆不是静态的——agent 会迭代、修正、分支。三个原语构成了完整的分支工作流：

contentBranch()（创建）→ searchByBranch()（查询）→ memoryMerge()（合并） 
// searchByBranch: BFS 遍历记忆分支树
function searchByBranch(rootId, opts = {}) {
  const depth = opts.depth ?? Infinity;
  const includeSelf = opts.includeSelf ?? false;
  const branches = [];
  const queue = [{ id: rootId, depth: 0 }];

  while (queue.length > 0) {
    const { id, depth: d } = queue.shift();
    if (d > depth) continue;

    const mem = store.get(id);
    if (!mem) continue;

    if (includeSelf || d > 0) {
      branches.push({ memory: mem, depth: d });
    }

    // 沿 derived_from 链接向下探索
    const children = linkStore.getOutgoing(id, 'derived_from');
    for (const child of children) {
      queue.push({ id: child.targetId, depth: d + 1 });
    }
  }

  return { branches };
}

又是 ~20 行代码。但和 contentBranch、memoryMerge 组合后，你拥有了完整的记忆版本管理：创建分支、探索分支树、合并回主线。这不是 Git，但设计思路一样——每个命令做好一件事。

可操作建议

1. 用"三角验证"检查 API 设计

每加一个新功能，问自己：它能和现有的哪两个方法组成三角？如果答案是"不能"，它可能是孤岛功能，需要重新考虑。

2. 测试覆盖率是新 API 的信号

如果一个方法需要超过 20 个测试才能覆盖所有路径，它可能做了太多事。AMS 的经验：每个原语 5-10 个测试是健康的甜区。memoryMerge 的 10 个测试恰好覆盖了 4 种内容策略 × 标签策略 × 链接策略的关键组合。

3. 批量操作是原语的消费者，不是原语本身

bulkMerge 不应该知道怎么合并——它只知道怎么调用 memoryMerge。这种分层让批量操作天然具有错误隔离：一个失败不影响其他的。代码只有 20 行，但它消耗了 memoryMerge 的全部能力。

// bulkMerge: 简单得惊人
function bulkMerge(pairs, opts = {}) {
  const results = [], errors = [];
  for (const [primary, secondary] of pairs) {
    try {
      const result = memoryMerge(primary, secondary, opts);
      if (result) results.push(result);
      else errors.push({ pair: [primary, secondary], reason: 'null_result' });
    } catch (e) {
      errors.push({ pair: [primary, secondary], reason: e.message });
    }
  }
  return { results, errors, merged: results.length, failed: errors.length };
}

这不是给 CI 看的，是给未来的自己看的。一个月后回头看，你能清晰地看到 API 的演化路径——哪些被保留了，哪些被回滚了。540 个测试、23 天零回滚率，这个信号比任何代码审查都有说服力。

总结

可组合原语不是新概念——Unix 管道、React 组件、Lisp 函数都是这个思想。但在 AI Agent 基础设施的语境下，它有一个额外的好处：LLM 更容易理解小的、职责单一的 API。

一个 agent 面对两个 20 行的方法，比面对一个 100 行的"万能方法"，能做出更好的调用决策。这可能是 agent-native API 设计最重要的原则——不只是给人读的，也是给 LLM 读的。

设计 API 就像设计乐高积木：每块只做一种形状，但拼在一起能建城堡。
