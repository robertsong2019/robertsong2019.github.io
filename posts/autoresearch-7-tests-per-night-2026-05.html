每晚7个测试：Autoresearch方法论的实战复盘

2026-05-14 · AI Agent 自动化编程 测试驱动

过去48小时，我的AI Agent在两个项目上自动完成了4次迭代。每次迭代：约40行代码，7个测试，零回滚。测试数从 278 涨到 292，从 132 涨到 139。没有一行代码是人工审查后打回的。

这不是偶然。这是一个叫 autoresearch 的方法论在起作用。今天我想拆解它为什么有效。

核心约束：每次只动一件事

Autoresearch 的第一条规则是：每次迭代只解决一个明确的、可测试的问题。不是"改进性能"，而是"检测循环依赖"。不是"加搜索功能"，而是"支持按标签+时间+前缀组合过滤"。

这个约束看起来简单，但它是整个系统的地基。因为如果你每次只做一件事，你就：

  • 知道该写什么测试
  • 知道什么时候做完了
  • 知道出了问题该看哪里

实战案例：依赖图验证

以昨晚的一次迭代为例。项目是 nano-agent（一个PRD驱动的自主编程Agent），用户故事（story）之间有依赖关系：

stories:
  - id: "auth"
    dependencies: []
  - id: "login-ui"
    dependencies: ["auth"]
  - id: "dashboard"
    dependencies: ["login-ui", "auth"]

问题：如果有人写了循环依赖（A依赖B，B依赖A），或者依赖了一个不存在的story，get_next_story() 会无限循环或者静默卡住。没有任何机制在运行前捕获这些错误。

解决方案：写一个 validate_dependencies() 方法，用 DFS 遍历依赖图：

def validate_dependencies(self) -> dict:
    """DFS-based dependency graph validation."""
    stories = self._get_active_stories()
    ids = {s["id"] for s in stories}
    circular, missing, self_deps = [], [], []

    def dfs(sid, visited, stack):
        if sid in stack:
            cycle = stack[stack.index(sid):] + [sid]
            circular.append(cycle)
            return
        if sid in visited:
            return
        visited.add(sid)
        stack.append(sid)
        for dep in stories_dict[sid].get("dependencies", []):
            dfs(dep, visited, stack)
        stack.pop()

    for s in stories:
        for dep in s.get("dependencies", []):
            if dep not in ids:
                missing.append((s["id"], dep))
        if s["id"] in s.get("dependencies", []):
            self_deps.append(s["id"])

    return {
        "valid": not circular and not missing and not self_deps,
        "circular_deps": circular,
        "missing_deps": missing,
        "self_deps": self_deps,
    }

~45行代码。7个测试覆盖：干净图、合法链、缺失依赖、自依赖、2节点环、3节点环、混合问题。从285到292个测试，全部通过，提交 cbd7772。

为什么是7个测试？

观察多次迭代，7个测试几乎成了一个自然常数。原因很简单：

  1. 正常路径（1-2个）：验证基本功能工作
  2. 边界条件（2-3个）：空输入、极限值、零状态
  3. 错误路径（2-3个）：各种失败场景

少于5个测试通常说明你漏了边界。多于10个说明你可能在一次迭代里做了太多事。

递增式构建的复利效应

真正有意思的是这些迭代之间的衔接。看看 agent-context-store 项目的时间线：

  • Day 1 Loop A: 搜索按时间范围 + 批量删除 → 69 tests
  • Day 1 Loop B: 组合搜索（tags + prefix + age + length） → 76 tests
  • Day 2 Loop A: 命名空间隔离 → 125 tests
  • Day 2 Loop B: 操作审计日志（changelog） → 139 tests

每次迭代都在前一次的基础上构建。组合搜索利用了已有的标签索引。审计日志挂载在已有的 put/delete 方法上。没有一次是推倒重来。

这就是"7个测试"策略的真正威力：不是单次迭代的质量，而是 60天累积400+测试且零回滚 的可持续性。每天进步一点，从不倒退。

三个可操作建议

1. 把"做完"定义为"测试通过"，而不是"代码写完"。

如果你不能为你的改动写测试，要么你的改动太模糊，要么你的架构不支持可测试性。两种情况都值得停下来想想。

2. 每次 commit 只做一件事，但把这件事做完。

"做完"的意思是：代码 + 测试 + 文档（如果需要）。半成品堆积是技术债的根源。我们的 agent 在每次迭代结束时自动生成结构化报告（基线、目标、实现、结果、决策），这比任何 code review checklist 都有效。

3. 用实验日志替代"感觉"。

我们维护一个 experiments.tsv，记录每次迭代的 commit、测试数变化、保留/回滚决策。两行示例：

2026-05-13T01:00	214e16b	test_count	285/285	keep	plan_batch(): dry-run batch planning. +7 tests (278→285).
2026-05-14T01:00	cbd7772	test_count	292/292	keep	validate_dependencies(): DFS cycle detection. +7 tests (285→292).

这不是事后补的文档，是 agent 自动写的。当你能回溯每一天的决策和结果，你就不再需要依赖记忆或直觉来判断"这段代码好不好"。

反直觉的结论

大多数人认为AI自主编程的关键是让AI更聪明。我的经验恰恰相反：关键是给AI一个足够小的操作单元，让错误无处藏身。

7个测试，40行代码，一个明确的问题。这个约束看起来是在限制AI，实际上是在释放它——因为当每次迭代都可靠时，你可以让AI一直迭代下去。

30天零回滚不是目标。是方法论的自然结果。
