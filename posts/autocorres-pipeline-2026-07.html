01 / 翻译管道

C 代码不是数学，
但可以被翻译成数学。

AutoCorres 不是直接读 C 代码做证明——中间隔着四层抽象。 每一层都把上一层变得更接近 Isabelle 能处理的形式。下面这个例子贯穿全程： 一个计算字符串长度的 C 函数（简化版）。

关键认知：每一层都比上一层更简单——不是语义不同， 而是细节更少。AutoCorres 的核心贡献是第 2→3 层的自动抽象， 它自动证明「我的抽象没改变程序含义」，所以你只在第 3 层做证明，结论自动传递回第 1 层。

02 / 四层代码对比

同一个 strlen，
四种形态。

点下面的 tab，看 my_strlen 这个函数在每一层长什么样。 注意细节怎么一层层被剥离——直到第 4 层只剩纯数学。

my_strlen.c  ·  人类可读的 C 源代码
// 计算以 null 结尾的字符串长度
unsigned my_strlen(char *s) {
    unsigned i = 0;
    while (*s != '\0') {
        i++;
        s++;
    }
    return i;
}

LAYER 01 · SOURCE

C 源代码

这是工程师写的代码。包含 指针（*s）、 可变状态（i++, s++）、 循环、内存读取。

这些特性让 C 高效，但无法直接数学推理—— 数学没有「指针」「副作用」「内存」这些概念。

问题：C 代码有指针别名、未定义行为、整数溢出。 要证明它正确，必须先把这些细节翻译成数学能处理的对象。
my_strlen_C.txt  ·  C parser 自动产出（示意）
(* C parser 把每个语句翻译成一个 state monad *)
my_strlen' :: string_mem ⇒ string_mem × nat
my_strlen' = do {
    s ← get "s";              (* 读指针 s 的当前值 *)
    i ← return 0;             (* 初始化 i *)
    while (λ(s,i). heap_get s ≠ 0)
          (λ(s,i). do {
              s ← return (s + 1);   (* s++ *)
              i ← return (i + 1);   (* i++ *)
              return (s, i)
          });
    return i
}

LAYER 02 · SIMPL

C parser 产出

C-to-Isabelle Parser（Michael Norrish 写的） 把 C 代码机械地翻译成 Isabelle 里的 state monad。

每个语句变成 do { ... } 块。指针变成对「堆」(heap_get) 的显式访问。副作用变成 monad 的串联。

这一层忠实但难用：它保留了 C 的所有细节—— 每次访问内存都要写堆的状态。证明一个简单性质可能要写几百行。 这就是 AutoCorres 要解决的问题。
my_strlen_AC.thy  ·  AutoCorres 抽象后
(* AutoCorres 把 state monad 抽象成 lifting monad *)
my_strlen :: "char ptr ⇒ natMonad"
my_strlen s = do {
    (* 指针操作被提升为高层函数 *)
    gs ← gets (λσ. heap_C σ);
    (* 循环用 whileLoop 表示，带测度函数 *)
    (i, _) ← whileLoop
        (λ(i, p). is_c_null p = False)
        (λ(i, p). return (i + 1, p +Ptr 1))
        (0, s);
    return i
}

LAYER 03 · AUTOCORRES

AutoCorres 抽象

AutoCorres（David Greenaway 写的）把 SIMPL 自动抽象成更接近数学的形式：

• 指针类型显式化 (char ptr)
• 内存访问封装成 gets/sets
• 循环带测度函数（用于终止性证明）
• 整数操作提升为数学的 nat

核心魔法：AutoCorres 自动证明「这个抽象等价于原始 C 语义」。 你在这层做证明，结论自动传递回 C 代码。这就是它叫 Auto 的原因。
my_strlen_proof.thy  ·  证明目标
(* 用 Hoare 三元组写规范 *)
lemma my_strlen_correct:
  shows "∀ σ s.
      valid_string σ s ⟶
      <Γ ⊨⇩
        {σ}
          my_strlen s
        {r σ'. r = actual_len s ∧
               preserves_mem σ σ'}"
proof (induct s)
    (* 这一步要写 50-200 行证明 *)
    (* 包括：循环不变式、终止性、内存安全 *)
...
qed

(* 翻译成人话：                              *)
(*   如果输入是指向合法字符串的指针，          *)
(*   那么 my_strlen 返回的值 = 字符串实际长度，*)
(*   并且不破坏其他内存。                     *)

LAYER 04 · PROOF

证明目标

到这一层，C 函数变成了一个 Hoare 三元组：

{前置条件} 程序 {后置条件}

你要证的是：在合法输入下（valid_string）， 函数返回值等于实际长度，且不破坏内存。

这就是 seL4 证明的形态。 每个内核函数都对应这么一个 lemma。seL4 有约 10,000 行 C 代码， 对应几千个这样的证明，加起来 200,000 行证明脚本。

读完这四层你应该感受到：从 C 到证明，每一层都在剥离无关细节。 Layer 2 忠实地保留了所有 C 的复杂性；Layer 3 把它抽象得可证； Layer 4 只剩下纯逻辑。AutoCorres 是这条管道的核心加速器。

03 / seL4 完整精化链

AutoCorres 只是其中一环。

seL4 的完整验证不止 C→证明。它走的是一条精化（refinement）链： 从最抽象的「设计规范」一层层下降到 C 代码。每一层都证明「我是上一层的正确实现」。 AutoCorres 负责链的最底层。

04 / 真实难点

为什么 seL4 要20 人年？

即使有 AutoCorres 加速，验证真实 C 代码仍然非常难。下面是 4 个主要难点—— 它们解释了「1 行 C 代码 ↔ 多少行证明」的真实比例。

▣ 难点 01

指针别名（Aliasing）

两个指针可能指向同一块内存。改了 *p，*q 也变了—— 引擎必须追踪所有可能的别名关系。AutoCorres 用分离逻辑处理， 但证明复杂度爆炸式增长。

▣ 难点 02

整数溢出

C 的整数会溢出。i + 1 在 i = UINT_MAX 时回绕到 0。 你必须证明在整个执行过程中不会溢出，否则推理无效。 每个算术操作都附带这个负担。

▣ 难点 03

循环不变式

每个 while 循环都需要手写循环不变式——一个在循环每一步都成立的性质。 不变式找得不好，证明就卡住。这是人工智慧（不是 AI）最难自动化的部分。

▣ 难点 04

未定义行为

C 标准留了大量「未定义行为」——读未初始化内存、空指针解引用、数组越界。 AutoCorres 必须证明这些都不会发生，否则 C 语义不明确，无法推理。

▶ 这就是 AI 的价值所在

这 4 个难点目前都靠人海战术——seL4 团队 20 人干几年。 但它们本质上都是模式化的：指针别名有固定处理套路、溢出检查有标准不变式、 循环不变式可以从规格合成。

如果 AI 能自动生成 80% 的这部分证明，把 1:20 的比例降到 1:5， 形式化验证就从「国家级项目」变成「日常工程实践」。 AlphaProof 已证明 AI 能做非平凡数学推断——这条路是通的，只是时间问题。
