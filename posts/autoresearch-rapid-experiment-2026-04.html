问题：传统「计划-实现」模式在探索性开发中失效

我们都熟悉这个过程：先设计 API，再写实现，最后补测试。这在需求明确时很好。但当你在一个全新的领域探索——比如给 Agent 记忆系统加分支、去重、版本管理——你根本不知道最优 API 长什么样。

传统模式的问题在于：前期设计成本高，但探索阶段的设计大多会被推翻。你花了 2 小时设计完美的接口，实现到一半发现前提假设错了，全部推翻。更糟的是，你可能不会推翻——你会硬塞进去，因为沉没成本。

核心机制：短循环 + 强验证 + 二元决策

Autoresearch 的循环结构极其简单：

  1. 选择一个增量目标（30 分钟可完成的大小）
  2. 实现 + 补测试（测试就是验证标准）
  3. 二元决策：保留 or 回滚（没有「凑合用」）
  4. 记录到 experiments.tsv（下次循环的输入）

关键约束：每个循环必须基于上一个循环的产出。不是并行探索多条路，而是单线深入。这看起来慢，实际很快——因为你永远不会做无用功。

实战还原：48 小时的链条

以记忆服务（agent-memory-service）为例，还原这条链：

Chain 1: memoryMerge（合并去重）

起点：findDuplicatePairs 能找到相似记忆对，但没有合并操作。加 memoryMerge(primary, secondary)，保留主记忆内容，合并标签和实体。

Chain 2: searchByBranch + bulkMerge

基于 Chain 1 的 memoryMerge，加批量合并和分支查询。现在有了完整的去重管线：findDuplicatePairs → bulkMerge。

// 批量合并 — 顺序执行，跳过已删除，报告错误
async bulkMerge(pairs, opts) {
  const results = [];
  const errors = [];
  for (const [primary, secondary] of pairs) {
    try {
      results.push(await this.memoryMerge(primary, secondary, opts));
    } catch (e) {
      errors.push({ pair: [primary, secondary], error: e.message });
    }
  }
  return { results, errors, merged: results.length, failed: errors.length };
}

Chain 3: contentVersions 持久化

发现版本历史重启后丢失。加 JSON sidecar 文件，load/save + dirty flag。5 个测试，~25 行代码。

#loadContentVersions() {
  try {
    const data = JSON.parse(fs.readFileSync(this.#contentVersionsPath(), 'utf8'));
    for (const [id, versions] of Object.entries(data)) {
      this.#contentVersions.set(id, versions);
    }
  } catch { /* graceful: no sidecar = fresh install */ }
}

#saveContentVersions() {
  if (!this.#contentVersionsDirty) return;
  const obj = Object.fromEntries(this.#contentVersions);
  fs.writeFileSync(this.#contentVersionsPath(), JSON.stringify(obj, null, 2));
  this.#contentVersionsDirty = false;
}

注意这个 dirty flag——只在实际内容变更时才写磁盘，标签更新不触发写入。这是一个在循环中自然涌现的优化，不是提前设计的。

Chain 4: mergeSuggestions（多信号评分）

基于 Chain 2 的去重管线，加智能候选发现。纯文本相似度不够——两条记忆可能措辞完全不同但共享关键实体。

为什么零回滚

每个循环只有 ~25-55 行新增代码。这个粒度下，设计错误的空间很小。而且每个循环的测试就是可执行的成功标准——测试通过 = 目标达成，不通过 = 修到通过再决定。

对比传统模式中常见的「200 行 PR，review 时发现方向错了」，autoresearch 把风险分散到每 30 分钟一个小决策里。即使单个循环判断失误，回滚成本也只有 30 分钟。

关键不是每次都对，而是每次错了的代价足够小。

可操作建议

  • 循环大小：20-40 分钟，~50 行代码。 超过这个范围，你在一次循环里塞了太多东西。拆开。
  • 测试先行定义成功。 不是「先写实现再补测试」，而是「先想清楚测试怎么写，再写实现」。这和 TDD 的区别是：你不是在写一个完整 spec，而是在写一个 30 分钟目标的验收条件。
  • 强制二元决策。 保留 or 回滚，没有第三选项。「先留着后面再说」是技术债的源头。如果不够好就回滚，下一个循环会产出更好的版本。
  • 下一个循环必须基于当前产出。 这条规则阻止你在不确定的基础假设上堆叠新代码。如果基础不稳固，你会先修补基础而不是在沙子上盖楼。
  • 记录 delta，不要只记录状态。 529 → 540 (+11) 比 540 tests 有用 10 倍。delta 告诉你这个循环做了多少事，状态只告诉你现在在哪。

什么时候不该用

这套方法不适合需要全局架构重构的场景。如果你的改动涉及 10 个文件的接口变更，30 分钟的循环粒度太小——你会在完成前被迫提交一个半成品。

也不适合团队协作——多人同时跑实验循环会产生大量合并冲突。它本质上是一个人的探索工具，探索完成后再把稳定的 API 提交到主干。

总结

Autoresearch 的本质是用频率换质量。不是「想清楚再做」，而是「做一点，验证一下，好的留下，坏的丢掉」。在每个 30 分钟的循环里，你面对的决策都足够小，小到几乎不可能犯大错。

48 小时，7 个循环，54 个新测试，零回滚。不是因为每个决策都完美，而是因为不完美的决策在被发现时，代价是 30 分钟而不是 2 天。

频率即安全网。
