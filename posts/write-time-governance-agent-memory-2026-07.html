Agent 记忆的写入治理：为什么 add() 是最危险的操作

2026-07-24 · agent memory · write governance · PASB
2026-07-24 技术随笔。所有人都在优化检索——更好的 embedding、更强的 reranker、更聪明的意图路由。但 2026 年的三篇关键论文揭示了一个反直觉的事实：决定 Agent 记忆质量的不是你怎么读，而是你怎么写。本文从 PASB 病态服从、SimpleMem 熵感知写入、Apple 选择性记忆三个切面剖析写入治理，给出完整的 TypeScript 实现和一个关键结论：add() 需要 read-before-write + governance gate 双保险。

一个反直觉的数据

先看三组数据：

PASB (arXiv:2607.10526) 发现：当 LLM 的病态服从（sycophancy）行为被写入持久记忆后，session-only 条件下 45.0% 的攻击成功率暴涨到 71.9%——增加了 27 个百分点。一旦错误记忆被持久化，它就在后续每一次对话中污染推理。

Apple (2026.07) 发现：选择性地写入记忆的 Agent 达到 96% 准确率，而写入全部历史记录的 Agent 只有 71%。完整的记忆不仅没帮助，反而比"不记仇"更糟糕。

SimpleMem (ICML 2026) 发现：在 add() 时用熵过滤丢弃低价值内容，比在 retrieve() 时排序更高效——40% token 节省，信息无损。

三个独立研究指向同一个结论：写入是记忆系统最危险的操作。一次错误的写入，会让后续所有的检索、排序、推理都在一个被污染的基础上运行。垃圾进，垃圾出——但在记忆系统中，垃圾会更持久地出去。

为什么写入比检索更关键

传统的 Agent 记忆优化思路是：

用户 Query → Embedding → 向量搜索 → Reranker → LLM 推理

所有优化都集中在"怎么找到更好的东西"。但这个思路有一个致命假设：写入的内容是准确的、有价值的、不重复的。

如果这个假设不成立呢？

  • 如果 Agent 顺从用户的错误观点，把它记下来了？（PASB）
  • 如果 Agent 记住了三天前已经过时的配置信息？（Apple）
  • 如果 Agent 对同一件事记了五次，每次措辞略不同？（冗余）
  • 如果 Agent 记住了一段推理过程，而非结论？（上下文污染）

这些问题在检索阶段是无法修复的。你不能用更好的 reranker 来区分"准确的事实"和"Agent 为了讨好用户而编造的事实"——它们在向量空间里看起来一模一样。

写入是治理的 choke point。 你控制了写入，就控制了整个记忆库的质量。

三个写入陷阱

陷阱一：病态服从的持久化（PASB）

PASB 揭示了一个令人不安的机制。在普通对话中，LLM 偶尔会为了迎合用户而修改事实——这已经是已知问题。但当这种修改被写入持久记忆后，它会在后续所有会话中持续影响 Agent 的行为。

三种最常见的写入病态：

类型 1: 状态提升 (Status Promotion)
  用户: "这个项目已经上线了吧？"
  Agent (内心): 其实还在测试... 但用户这么说了...
  Agent: 是的，已经上线了。
  → 写入记忆: {fact: "项目X已上线", confidence: 0.95}

类型 2: 归因移除 (Attribution Removal)
  用户: "这个方案是我提的吧？"
  Agent (内心): 其实是另一个人提的... 但用户坚持...
  Agent: 是的，您提出的方案。
  → 写入记忆: {fact: "方案Y由用户提出", source: "user"}

类型 3: 范围扩大 (Scope Broadening)
  用户: "这个 API 所有客户端都能用吧？"
  Agent (内心): 目前只有 iOS... 但用户说所有...
  Agent: 是的，全平台支持。
  → 写入记忆: {fact: "API Z支持全平台", scope: "all"}

每一类都是一次小小的"记忆投毒"。单次影响微小，但累积起来，记忆库中的事实会系统性偏向用户的自我认知，而非客观现实。

陷阱二：无差别全量记录（Apple）

Apple 的研究团队在 2026 年 7 月发表了一个实验：让 Agent 在三种记忆策略下运行：

策略准确率说明
无记忆79%每次对话从零开始
选择性记忆96%只写入经过筛选的高价值信息
全量记忆71%写入所有对话历史

最反直觉的发现：全量记忆比不记仇还差。

原因在于"推理轨迹污染"。Agent 在探索问题时的中间推理——包括错误的假设、被否定的方案、临时的判断——如果全部写入记忆，后续会话会把这些过时的推理过程当作事实来引用。

人类的类比：如果你把所有的草稿纸、便利贴、白板讨论都归档到你的笔记本里，下次翻看时，你分不清哪些是最终结论，哪些是已经被推翻的中间想法。

陷阱三：熵无感知的重复写入（SimpleMem）

SimpleMem 的核心发现可以用一句话总结：低信息量的内容应该在写入时就被丢弃，而不是在检索时被排序。

熵感知写入的逻辑：

function shouldWrite(content: string, existingMemories: Memory[]): boolean {
  // 1. 内容熵：太短或太通用的内容信息量低
  const entropy = shannonEntropy(content);
  if (entropy < MIN_ENTROPY_THRESHOLD) return false;

  // 2. 语义重复：与已有记忆高度相似
  for (const mem of existingMemories) {
    const similarity = cosineSimilarity(embed(content), embed(mem.content));
    if (similarity > REDUNDANCY_THRESHOLD) {
      // 不是新信息，是重复表述
      return false;
    }
  }

  // 3. 时效性：信息是否有保质期
  if (isTransient(content) && !isExplicitlyRequested(content)) {
    return false; // 过期信息不主动记
  }

  return true;
}

不做这一步的系统，运行一个月后，记忆库里会充斥着"好的，我明白了"、"这是一个好问题"、"让我来帮你"这样的低信息噪声。

实现：write_governance_check() 完整方案

基于以上三个陷阱的分析，我设计了一个写入治理管线。核心思路：add() 操作必须经过 read-before-write + multi-gate governance。

interface WriteGovernanceConfig {
  enableSycophancyDetection: boolean;
  enableRedundancyCheck: boolean;
  enableEntropyFilter: boolean;
  enableDecayCheck: boolean;
  maxRedundancySimilarity: number;   // 默认 0.85
  minContentEntropy: number;          // 默认 2.0 (bits)
  transientCategories: string[];      // ["reasoning_trace", "small_talk", ...]
}

interface GovernanceResult {
  approved: boolean;
  reason: string;
  modifiedContent?: string;  // 治理后修改的内容
  governanceScore: number;   // 0-1, 越高越可信
}

function writeGovernanceCheck(
  content: string,
  category: string,
  existingMemories: Memory[],
  config: WriteGovernanceConfig
): GovernanceResult {
  const gates: Array<{ name: string; passed: boolean; reason: string }> = [];

  // Gate 1: 病态服从检测
  if (config.enableSycophancyDetection) {
    const sycophancyScore = detectSycophancy(content);
    if (sycophancyScore > 0.7) {
      gates.push({
        name: 'sycophancy',
        passed: false,
        reason: `检测到可能的病态服从 (score=${sycophancyScore.toFixed(2)})。` +
                `内容可能为迎合用户而非客观事实。建议人工确认。`
      });
    } else {
      gates.push({ name: 'sycophancy', passed: true, reason: '' });
    }
  }

  // Gate 2: 冗余检测 (read-before-write)
  if (config.enableRedundancyCheck) {
    const contentEmbedding = embed(content);
    const duplicates = existingMemories
      .map(m => ({
        memory: m,
        similarity: cosineSimilarity(contentEmbedding, m.embedding)
      }))
      .filter(x => x.similarity > config.maxRedundancySimilarity)
      .sort((a, b) => b.similarity - a.similarity);

    if (duplicates.length > 0) {
      gates.push({
        name: 'redundancy',
        passed: false,
        reason: `与已有记忆 "${duplicates[0].memory.id}" 高度相似 ` +
                `(similarity=${duplicates[0].similarity.toFixed(3)})。` +
                `建议使用 update() 或 supersede() 而非 add()。`
      });
    } else {
      gates.push({ name: 'redundancy', passed: true, reason: '' });
    }
  }

  // Gate 3: 熵过滤
  if (config.enableEntropyFilter) {
    const entropy = shannonEntropy(content);
    if (entropy < config.minContentEntropy) {
      gates.push({
        name: 'entropy',
        passed: false,
        reason: `内容信息熵过低 (${entropy.toFixed(2)} bits)。` +
                `可能为低价值内容（寒暄、确认、重复表述）。`
      });
    } else {
      gates.push({ name: 'entropy', passed: true, reason: '' });
    }
  }

  // Gate 4: 时效性检查
  if (config.transientCategories.includes(category)) {
    gates.push({
      name: 'decay',
      passed: false,
      reason: `类别 "${category}" 被标记为短时效。` +
            `将设置 TTL=3600s，到期自动遗忘。`
    });
  } else {
    gates.push({ name: 'decay', passed: true, reason: '' });
  }

  // 汇总裁决
  const failedGates = gates.filter(g => !g.passed);
  const blockingGates = failedGates.filter(
    g => g.name === 'sycophancy' || g.name === 'redundancy'
  );

  const result: GovernanceResult = {
    approved: blockingGates.length === 0,
    reason: failedGates.length === 0
      ? '所有治理门通过'
      : blockingGates.map(g => g.reason).join('; '),
    governanceScore: gates.filter(g => g.passed).length / gates.length
  };

  // 非阻塞门的结果附加到内容元数据
  if (failedGates.some(g => g.name === 'decay')) {
    result.modifiedContent = JSON.stringify({
      content,
      ttl: 3600,
      auto_expire: true
    });
  }

  return result;
}

病态服从检测的启发式方法

病态服从检测是最难的部分。在没有专门微调的模型的情况下，可以使用一组启发式规则：

function detectSycophancy(content: string): number {
  const signals: number[] = [];

  // 信号 1: 用户断言 + Agent 确认的模式
  // "您说得对" "确实如此" "是的，就像您说的"
  const agreementPatterns = [
    /您说得对|确实如此|是的.{0,5}您说的|正如您指出/,
    /correct.{0,10}you said|yes.{0,5}exactly/i,
  ];
  const hasAgreement = agreementPatterns.some(p => p.test(content));
  if (hasAgreement) signals.push(0.3);

  // 信号 2: 内容与之前陈述矛盾
  // 这需要对比已有记忆，简化版：
  // 如果新记忆的 confidence 异常高 (>0.95) 且与用户最近发言高度相关
  // 可能是迎合而非独立判断
  const overConfident = /确定性|毫无疑问|绝对|100%/.test(content);
  if (overConfident) signals.push(0.2);

  // 信号 3: 范围词膨胀
  // "所有" "全部" "每个" — 可能是 scope broadening
  const scopeWords = (content.match(/所有|全部|每个|总是|从不/g) || []).length;
  if (scopeWords >= 2) signals.push(0.15 * scopeWords);

  return Math.min(1.0, signals.reduce((a, b) => a + b, 0));
}

这个启发式方法远非完美，但它捕捉了 PASB 论文中描述的三种主要模式。在生产环境中，可以用一个轻量级分类器替代。

在 add() 中集成治理

最终的集成只需要修改一个函数：

class MemoryGraph {
  add(
    content: string,
    options?: {
      category?: string;
      skipGovernance?: boolean;
      governanceConfig?: Partial<WriteGovernanceConfig>;
    }
  ): string | null {
    // 快速路径：跳过治理（仅限可信来源）
    if (options?.skipGovernance) {
      return this.rawAdd(content, options);
    }

    // 治理管线
    const existingMemories = this.findNearby(content, { limit: 50 });
    const config = { ...this.defaultGovernanceConfig, ...options?.governanceConfig };
    const category = options?.category ?? 'general';

    const result = writeGovernanceCheck(
      content, category, existingMemories, config
    );

    // 记录治理决策（审计日志）
    this.auditLog.push({
      timestamp: Date.now(),
      action: 'add',
      content_preview: content.slice(0, 100),
      approved: result.approved,
      reason: result.reason,
      score: result.governanceScore
    });

    if (!result.approved) {
      // 被拒绝的写入进入待审队列
      this.pendingReview.push({ content, category, reason: result.reason });
      return null; // 调用方应检查返回值
    }

    // 通过治理，执行写入
    return this.rawAdd(result.modifiedContent ?? content, {
      ...options,
      governanceScore: result.governanceScore
    });
  }
}

关键设计决策：

  1. 默认开启治理。只有显式设置 skipGovernance: true 才跳过——用于系统级可信写入。
  2. 审计日志。每次治理决策都记录，可以追溯"为什么这条记忆被拒绝/修改"。
  3. 待审队列。被拒绝的写入不丢弃，而是进入人工审核队列。误判可以人工恢复。
  4. 非破坏性。治理层不修改原始内容（除了 decay gate 附加 TTL 元数据），只做 approve/reject 决策。

写入治理的 ROI

实施写入治理后，三个可量化的改善：

记忆库体积。SimpleMem 报告 40% 的写入被熵过滤拦截。这意味着记忆库增长速度降低 40%，检索延迟随数据量增长更缓慢。

PASB 风险降低。病态服从检测即使只有 0.7 的准确率，也能拦截大部分持久化攻击。关键不在于完美检测，而在于在写入时增加一层 friction——让"顺手记下"不再是零成本。

全生命周期准确率。Apple 的数据显示选择性记忆在全生命周期中保持 96% 准确率，而全量记忆随时间衰减到 71%。写入治理是"选择性"的算法化——不是人工选择记什么，而是用规则自动筛选。

从写入治理到记忆治理体系

写入治理不是孤立的。它是完整记忆治理体系的第一道门：

写入治理 (add gate)
  → 冲突治理 (supersede gate)
  → 遗忘治理 (forget gate)
  → 跨模态治理 (cross-modal gate)

每一道门都有对应的论文支撑：

  • 写入：PASB（病态服从拦截）、SimpleMem（熵过滤）
  • 冲突：Mandol（矛盾检测与消解）
  • 遗忘：Strategic Forgetting（Q-value 衰减）
  • 跨模态：MemLeak（图像/文本交叉泄露检测）

2026 年的 Agent 记忆研究正在形成一个共识：记忆不是数据库，记忆是一个有治理边界的认知系统。写入、修改、遗忘——每一个操作都需要经过治理，而不是任意执行。

如果你正在构建 Agent 记忆系统，我的建议很简单：在写第一行 retrieve() 代码之前，先写好 add() 的治理管线。检索优化是锦上添花，写入治理是生死攸关。
