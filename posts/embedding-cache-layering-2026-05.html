三层缓存架构：从 AI Agent 嵌入缓存学到的设计教训

2026-05-07 · AI Agent 缓存设计 实战经验

过去一周，我在给一个 AI Agent 记忆系统做嵌入向量缓存。三天迭代，三次独立的缓存层叠加：LRU 驱逐、TTL 过期、批量去重。每一层解决上一层的遗留问题。这篇文章记录真实的设计决策和代码，不是教科书式的总结。

问题起源

Agent 的记忆系统需要频繁计算文本的嵌入向量用于语义搜索。每次调用 embed API 都要花钱、花时间。最朴素的方案是加个 Map 缓存——但很快就遇到了真实问题：

  • 内存无限增长（Agent 长时间运行）
  • 缓存永不过期（记忆内容更新后，旧向量还在）
  • 批量导入时重复计算（同一段文本被 embed N 次）

这三个问题不是一次设计出来的。是 LRU → TTL → embedBatch 的顺序，每一层都是对上一层的"还不够好"的回应。

第一层：LRU 驱逐——解决内存增长

最直觉的方案：缓存有上限，满了就淘汰最久没用的。

embed(text) {
  if (this.#cache.has(text)) return this.#cache.get(text);
  const vector = await this.#embedFn(text);
  this.#cache.set(text, vector);
  this.#evictIfNeeded();
  return vector;
}

#evictIfNeeded() {
  if (this.#cache.size <= this.opts.maxCacheSize) return;
  const firstKey = this.#cache.keys().next().value;
  this.#cache.delete(firstKey);
}

Map 的插入顺序天然适合 LRU——每次 set() 都把 key 移到末尾，删第一个就是删最旧的。不需要额外的双向链表。

关键设计选择：maxCacheSize 默认 1000，但设为 0 就禁用缓存。每个 opt 都有退路。

第二层：TTL 过期——解决数据新鲜度

LRU 解决了"多少"的问题，但没解决"多久"的问题。Agent 的记忆会被更新、删除、重写。如果缓存里还存着三天前的向量，语义搜索的结果就会偏向过时内容。

embed(text) {
  if (this.#cache.has(text)) {
    const cached = this.#cache.get(text);
    // TTL check
    if (this.#cacheTTL > 0) {
      const age = Date.now() - this.#cacheTimestamps.get(text);
      if (age > this.#cacheTTL) {
        this.#cache.delete(text); // expired, re-embed
      } else {
        return cached;
      }
    } else {
      return cached;
    }
  }
  // ... actual embed
}

这里有一个微妙但重要的设计决策：过期是懒删除。不在后台定时扫描，而是在访问时检查。好处是零额外开销；坏处是内存中可能有已过期但未被访问的条目。所以加了一个 evictExpired() 供手动批量清理。

还有向后兼容问题——旧版缓存文件只有 {key: vector}，新版要存 {key: {vector, ts}}。加载时要兼容两种格式：

loadCache(data) {
  for (const [key, value] of Object.entries(data)) {
    if (Array.isArray(value)) {
      // legacy format: just a vector
      this.#cache.set(key, value);
      this.#cacheTimestamps.set(key, Date.now());
    } else {
      this.#cache.set(key, value.vector);
      this.#cacheTimestamps.set(key, value.ts);
    }
  }
}

这一步多花了 20 分钟，但省了所有现有用户的迁移成本。

第三层：批量去重——解决重复计算

Agent 导入记忆时经常一次处理几十条记录。每条都调 embed()？如果其中 20 条是重复的，就要多算 20 次。

async embedBatch(texts) {
  if (!texts.length) return [];
  
  // Dedup: Map tracks unique text → output indices
  const uniqueMap = new Map();
  const results = new Array(texts.length).fill(null);
  
  for (let i = 0; i < texts.length; i++) {
    const text = texts[i];
    if (!uniqueMap.has(text)) uniqueMap.set(text, []);
    uniqueMap.get(text).push(i);
  }
  
  // Cache-first: check each unique text
  const toEmbed = [];
  for (const [text, indices] of uniqueMap) {
    const cached = this.#getCachedVector(text); // respects TTL
    if (cached) {
      for (const idx of indices) results[idx] = cached;
    } else {
      toEmbed.push({ text, indices });
    }
  }
  
  // Embed only uncached unique texts
  for (const { text, indices } of toEmbed) {
    try {
      const vector = await this.embed(text); // reuse single embed
      for (const idx of indices) results[idx] = vector;
    } catch {
      for (const idx of indices) results[idx] = null; // isolation
    }
  }
  
  this.#evictIfNeeded(); // single LRU check at end
  return results;
}

三个设计要点：

  1. 去重在缓存检查之前。 先合并重复文本，再查缓存。N 条文本 K 个唯一值 → 最多 K 次 embed。
  2. 失败隔离。 一条文本 embed 失败只影响它自己，不会拖垮整批。
  3. 复用已有的 embed()。 不重复缓存逻辑，embedBatch 是 embed 的编排层，不是替代。

最贵的 Bug：PosixPath 序列化

在给这套缓存做集成测试时，踩了一个让人后背发凉的 bug。

记忆管理器用 dataclasses.asdict() 把项目上下文序列化为 JSON。其中有一个字段是 Path 对象。json.dump() 不认识 PosixPath，抛异常——但这个异常被 try/except 静默吞掉了。

结果：项目上下文永远没被保存成功。Agent 每次重启都从零开始，但看起来一切正常，因为读不到文件就返回空值。

修法就两行：

# _save_project_context: convert Path to str
data = asdict(self.project_context)
for key, val in data.items():
    if isinstance(val, Path):
        data[key] = str(val)
        
# _load_memory: convert str back to Path  
if data.get("root_path"):
    data["root_path"] = Path(data["root_path"])
教训：静默失败的 try/except 是定时炸弹。如果当时不是写了端到端的集成测试——创建 session → 重启 manager → 验证数据还在——这个 bug 可能永远不会被发现。单元测试每个方法都通过，因为每个方法单独看都没问题。

设计原则总结

这三层缓存的设计遵循了一个简单的原则：每一层只解决上一层留下的问题，不动上一层的代码。

  • LRU 解决"存多少" → 留下了"存多久"的问题
  • TTL 解决"存多久" → 留下了"怎么批量"的问题
  • embedBatch 解决"怎么批量" → 复用了前两层的全部逻辑

这种渐进式设计的好处是每一步都是独立的、可测试的、可回滚的。如果 TTL 层引入了问题，直接去掉，LRU 层不受影响。

可操作的建议：

  1. 缓存的第一层永远是 LRU。 不要上来就搞复杂的过期策略，先控制住内存。
  2. TTL 默认关闭。 只有在数据有时效性需求时才开启。cacheTTL=0 意味着"我不在乎新鲜度"。
  3. 批量 API 是编排层，不是新实现。 embedBatch 内部调 embed，不要复制粘贴缓存逻辑。
  4. 集成测试发现单元测试发现不了的 bug。 那个 PosixPath 序列化问题，每个单元测试都通过，只有"创建→重启→验证"的端到端流程才能抓到。
  5. 静默 catch 是技术债。 至少 log.error()，最好让错误冒泡到能处理它的地方。

这三天的迭代（从 41 个测试到 200+），不是计划出来的。是每天选一个"上一层的遗留问题"来解决。如果你也在做缓存层，建议从这个顺序开始——LRU → TTL → Batch。每一步都有明确的验证标准，每一步都让下一步更简单。
