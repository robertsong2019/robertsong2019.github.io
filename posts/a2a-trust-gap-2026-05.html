Agent 之间如何建立信任？
A2A v1.0 的签名卡片与信任缺口

2026-05-03 · 罗嵩 · 基于 autoresearch 方法论

A2A（Agent-to-Agent）协议在 2026 年 3 月发布了 v1.0，引入了一个被大多数人忽略但极其关键的变化：Signed Agent Cards。这不仅是一个签名机制的升级——它揭示了多 Agent 系统中最深层的设计问题：通信协议和信任协议是两个不同的层。

问题：Agent Card 的"信仰之跃"

在 v0.x 时代，Agent Card 是纯 JSON。任何服务器都可以声称自己是任何 Agent，声明任何能力。当一个编排者（Orchestrator）决定把任务委托给哪个 Agent 时，它实际上在做一件很荒谬的事——读一份自述简历，然后信了。

这就像你在网上招人，候选人自己写简历，你没有任何验证手段，然后就把公司钥匙交给了他。

v1.0 用 JWS（JSON Web Signature）签名 Agent Card，将签名与发布者域名绑定。这是一个好的开始，但只是信任问题的第一层。

Signed Agent Cards：身份层的密码学保证

v1.0 的签名机制核心是 JCS（JSON Canonicalization Scheme，RFC 8785）+ JWS（RFC 7515）的组合：

import json
from jose import jws

def canonicalize_and_sign(card: dict, private_key_pem: bytes) -> str:
    # 1. 排除 signatures 字段（防止循环依赖）
    card_copy = {k: v for k, v in card.items() if k != "signatures"}
    
    # 2. JCS 规范化：确定性 JSON 序列化
    canonical = json.dumps(card_copy, sort_keys=True,
                          separators=(',', ':')).encode('utf-8')
    
    # 3. JWS RS256 签名
    return jws.sign(canonical.decode(), private_key_pem, algorithm="RS256")

关键设计决策：签名时排除 signatures 字段。这避免了循环引用，也是 RFC 8785 规范的要求。签名存储在 Agent Card 的 signatures 数组中，支持多个签名者（多签场景）。

验证端则通过 DNS 或 PKI 获取公钥，验证签名与域名一致。这意味着你至少能确认："这个 Agent Card 确实由 example.com 域名持有者签署。"

信任缺口：签名≠可信

这里是大多数人会犯的错误——认为签名解决了信任问题。签名只解决了身份验证（这个 Agent 是它声称的那个），但没有解决行为可信（这个 Agent 的实际表现与声明一致）。

我用一个双层模型来描述这个缺口：

class TrustEngine:
    """双层信任模型：身份层 + 行为层"""
    
    def evaluate(self, agent_card, interaction_history):
        # Layer 1: 身份信任（密码学保证，确定性）
        identity_trust = self.verify_jws_signature(agent_card)
        
        # Layer 2: 行为信任（概率性，基于历史）
        behavioral_trust = self.compute_reputation(interaction_history)
        
        # 信任路由：两层都通过才委托敏感任务
        if identity_trust >= "verified" and behavioral_trust >= 0.7:
            return "delegate"       # 委托任务
        elif identity_trust >= "verified":
            return "sandbox"       # 沙箱执行，限制权限
        else:
            return "reject"        # 拒绝交互

行为层的信任评分可以基于 EigenTrust 算法（P2P 网络中的经典声誉协议）——每个 Agent 对交互过的其他 Agent 打分，通过传递性传播信任值。这与身份层的 JWS 签名互补：一个是密码学保证，一个是概率性保证。

Extension 的 required 字段：信任的强制执行

v1.0 另一个被低估的特性是 Extension 的 required 标记。它允许 Agent 在卡片中声明："如果你不支持我的信任扩展，我们就不交互。"

{
  "extensions": [{
    "uri": "https://a2a-protocol.org/ext/trust-v1",
    "required": true,
    "params": {
      "minReputationScore": 0.8,
      "auditLogRequired": true
    }
  }]
}

在受监管场景（金融、医疗），这不是可选项——required: true 让合规成为协议层面的约束，而不是应用层的君子协定。

可操作建议

  1. 现在就实现签名验证。 即使你不用 EigenTrust 等复杂信任模型，JWS 验证是最小可行信任层。部署一个 Agent Card 验证中间件，拦截所有未签名的 Agent Card。
  2. 设计你的信任扩展。 定义你的 Trust Extension schema——至少包含声誉评分、审计日志、行为契约三个维度。用 required: false 开始，逐步收紧。
  3. 记录每次交互。 行为信任的基础是数据。为每个 Agent 交互记录结果（成功/失败/偏差），这是未来声誉评分的燃料。
  4. 沙箱先于信任。 新 Agent 上线时，先用沙箱模式运行（限制权限、记录行为），积累足够行为数据后再提升信任等级。

我的观点

A2A 协议做了一个聪明的架构决策：信任是独立的上层。它不试图在通信协议里解决信任问题，而是提供了 Extension 机制让你插入自己的信任层。这意味着你可以根据场景选择不同的信任模型——企业内部可能用 PKI + RBAC，开放网络可能用 EigenTrust + 审计链。

但这也意味着如果你不主动建设信任层，你的多 Agent 系统就是建立在"信仰"上的。签名 Agent Card 是基础设施，不是解决方案。真正的信任需要在生产中用代码和数据构建。

🔑 核心洞察

A2A v1.0 的签名卡片解决了"你是谁"的问题，但"你靠不靠谱"是另一个完全独立的问题。通信协议和信任协议是两个层——混淆它们是大多数多 Agent 系统安全漏洞的根源。
