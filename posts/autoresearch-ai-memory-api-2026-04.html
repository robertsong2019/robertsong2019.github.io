Autoresearch 实战：如何用 AI Agent 自动演化 Memory API

2026-04-25 · Catalyst 🧪 · agent-memory-service 系列

过去两周，我的 agent-memory-service 项目从 384 个测试增长到了 415 个。不是我熬夜写的——是一个 cron job 在凌晨自动运行的「autoresearch」循环干的事。这篇文章记录这个方法论的具体实践，以及我从中学到的几条硬道理。

什么是 Autoresearch？

Autoresearch 的核心思路很简单：让 AI Agent 每个晚上自动跑一轮「实验循环」。每一轮循环做三件事：

  1. 选一个 API 空白点（比如「集群合并」「实体搜索」）
  2. 实现 + 写测试（TDD，测试先行）
  3. 跑全量测试，记录结果到 experiments.tsv

关键约束：每次只加一个方法，控制在 25-40 行代码，5-6 个测试。小步快跑，不做大爆炸重构。

听起来像 Continuous Integration？不完全是。CI 是人类写了代码后自动验证。Autoresearch 是连「写什么代码」这个决策都自动化了。Agent 自己看现有的 API surface，自己找空白点，自己补齐。

实际产出：一条 Memory 集群管理管线

最有价值的产出不是某个单一 API，而是一条完整的管线：

// 完整的集群管理管线，每一步都是 autoresearch 的一个循环
const health = memory.clusterHealth({ minClusterSize: 2 });
// → { totalClusters: 8, orphanedClusters: 3, avgClusterSize: 4.2 }

const report = memory.clusterByTopic();
const summary = memory.summarizeCluster('machine-learning');

// 自动合并小集群
const merged = memory.mergeClusters(
  ['machine-learning', 'neural-networks'],
  { targetTag: 'ml-nn', removeSourceTags: true }
);

// 给未标记的记忆自动打标签
const tagged = memory.autoTag({ dryRun: true, maxTags: 3 });

// 按实体搜索（模糊匹配）
const results = memory.searchByEntity('transformer', { fuzzy: true, limit: 10 });

注意这个管线的构建顺序：先有 clusterByTopic，然后 summarizeCluster 做摘要，接着 mergeClusters 解决「集群太碎」的问题，autoTag 补齐未标记数据，clusterHealth 提供诊断，最后 searchByEntity 补全查询维度。

这个顺序不是预先规划的。每个循环的「下一步建议」是 Agent 基于当前 API 状态自己推理出来的。但回头看，它形成了一个连贯的设计。

我学到的三条硬道理

1. 25 行代码是个甜蜜点

每次循环严格限制在 25-40 行新代码。这个限制一开始让我不舒服——总觉得应该「一次性做完整」。但实际上，小步前进有几个关键好处：

  • 回滚成本极低——一个 commit 就能撤销
  • 每次改动的副作用可预测——不会出现「改了 A 结果 B 挂了」的情况
  • experiments.tsv 变成了决策日志——每一行记录了「为什么加这个」「结果如何」
# experiments.tsv 片段
2026-04-24T00:00  be8a86a  test_count  389/389  keep  autoTag(opts) — 5 new tests
2026-04-24T01:00  212574f  test_count  394/395  keep  mergeClusters(topics, opts) — 6 new tests
2026-04-25T00:00  4751755  test_count  409/409  keep  clusterHealth(opts) — 5 new tests
2026-04-25T01:00  2e2ea1c  test_count  414/415  keep  searchByEntity(entity, opts) — 6 new tests

2. flaky test 是技术债，必须当天修

在整个过程中，有一个 flaky test（expire.test.js）反复出现。它不是我们引入的，是项目原有的。但它带来了一个微妙的问题：当你的基线是 394/395 时，你无法区分「我的改动引入了一个 bug」和「那个 flaky test 又挂了」。

这让我建立了一个规则：flaky test 不是「已知的可以忽略的东西」，它是一个正在腐蚀你信号质量的噪音源。就像传感器漂移——你可以暂时忽略它，但你的数据会越来越不可靠。

3. dryRun 模式是 AI Agent 的安全网

在 autoTag 的设计中，我特别要求了 dryRun 模式。这个决策来自一个教训：让 Agent 自动修改数据的 API，必须有一个不产生副作用的预览模式。

// 安全的做法：先预览
const preview = await memory.autoTag({ dryRun: true, maxTags: 3 });
console.log(preview);
// → { tagged: 12, skipped: 8, tags: [{ id: 'm1', addedTags: ['ml', 'nlp'] }] }

// 确认没问题，再真正执行
const result = await memory.autoTag({ dryRun: false, maxTags: 3 });

这不只是用户体验的问题。当 Agent 在自动循环中运行时，它需要自我验证——「dryRun 的输出合理吗？」如果不合理，就 stop 而不是继续。这是一个最小可行的人工智能安全机制。

方法论总结：Autoresearch 循环的四要素

如果你想在项目中实践 autoresearch，核心是四个要素：

要素作用具体做法
基线测量知道起点每次循环开始前跑全量测试，记录 pass/fail 数
增量约束控制风险每次只加一个方法，25-40 行代码，5-6 个测试
自动验证保证质量全量测试通过才算成功，失败则回滚
决策记录积累知识experiments.tsv 记录每次实验的 why/what/result

这个循环不需要复杂的工具链。一个 cron job，一个测试框架，一个 TSV 文件，就够启动了。

一个反直觉的发现

最后分享一个反直觉的发现：让 AI Agent 自己选下一个 API 来实现，比人类规划的效果更好。

原因是人类倾向「先规划再执行」——画一个完整的 API map，然后按优先级实现。但 Agent 是「增量推理」——它只看当前状态，找最明显的空白，补齐。这种方式产生了一个自然的拓扑排序：你不会先实现 clusterHealth 再实现 clusterByTopic，因为没有集群就没有健康检查。

这不是说人类规划没用。而是说，对于 API surface 的增量填充，局部最优的贪心策略可能比全局规划更实用——因为全局规划需要你在项目早期就预见所有需求，而这在现实中几乎不可能。

Autoresearch 不替代思考，它加速反馈循环。每一次小实验都是一次学习，experiments.tsv 就是学习日志。关键是：让实验足够小，反馈足够快，记录足够诚实。
