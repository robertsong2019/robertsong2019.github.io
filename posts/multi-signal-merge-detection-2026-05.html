多信号融合：AI记忆去重不能只看文本相似度

2026-05-01 · agent-memory dedup autoresearch

过去一个月在给 agent-memory-service 加去重管道，踩了一个很直觉的坑：纯文本相似度做去重，漏掉太多。

两条记忆内容完全不同——一条说"用 Docker 部署 Redis"，另一条说"容器化缓存方案选型"——但它们讨论的是同一件事。ngram 相似度只有 0.3，纯文本方法直接放过了。但它们共享 3 个实体和 2 个标签，明显应该合并。

这就是我今天要讲的：mergeSuggestions() 的三信号评分设计，以及为什么"加权融合"比"单信号阈值"更适合去重。

问题：单信号的盲区

最初的去重管道是线性的：

findDuplicatePairs() → compareMemories() → memoryMerge() → bulkMerge()

findDuplicatePairs 只用 ngram 相似度。这够用吗？不够。三种典型漏检场景：

  1. 同义不同词："微服务架构" vs "Microservice Pattern"——语义相同，文本重叠低
  2. 缩写/全称："K8s" vs "Kubernetes"——同指一物，编辑距离高
  3. 不同角度描述：一条记录技术细节，一条记录决策原因——互补信息，纯文本相似度中等偏低

每种场景里，实体和标签的信号都比纯文本更可靠。因为实体是经过 NER 提取的标准化概念，不受措辞变化影响。

方案：三信号加权评分

核心实现约 55 行。信号权重经过几轮实验调参：

mergeSuggestions({ minScore = 0.5, limit = 20, layer }) {
  const pairs = [];
  for (let i = 0; i < memories.length; i++) {
    for (let j = i + 1; j < memories.length; j++) {
      const contentSim = ngramSimilarity(memories[i], memories[j]);
      const entityOverlap = jaccard(entities[i], entities[j]);
      const tagOverlap = jaccard(tags[i], tags[j]);
      
      const score = 0.5 * contentSim 
                  + 0.3 * entityOverlap 
                  + 0.2 * tagOverlap;
      
      if (score >= minScore) {
        pairs.push({
          pair: [memories[i].id, memories[j].id],
          score,
          reasons: buildReasons(contentSim, entityOverlap, tagOverlap),
          suggestedStrategy: contentSim > 0.8 ? 'keep-longer' : 'concat'
        });
      }
    }
  }
  return pairs.sort((a, b) => b.score - a.score).slice(0, limit);
}

三个设计决策值得展开：

1. 为什么文本权重还是最高（0.5）？

因为文本是最稳定的信号。实体和标签依赖上游 NER/标注质量——如果提取器不准，这两个信号就是噪声。文本始终存在，ngram 是确定性的，不依赖外部管道。给最高权重是一种防御性设计。

2. 为什么用 Jaccard 而不是嵌入相似度？

Jaccard 简单、可解释、零依赖。集合 {Redis, Docker} 和 {Redis, K8s, Docker} 的 Jaccard 是 2/3 = 0.67，一眼就能理解。而且 Jaccard 对集合大小不敏感——两条记忆各有 20 个标签，重叠 5 个，Jaccard 仍然能给出有意义的分数。

3. reasons 数组：可解释性不是锦上添花

每对候选返回 reasons: ["content:86%", "entities:2 shared", "tags:1 common"]。这不是给用户看的——是给自动化管道自己看的。当 autoMerge 在无人值守模式下决定是否合并时，它需要知道为什么这两条被建议合并。content 95% 的合并策略和 entities 100% 但 content 30% 的完全不同。

实测效果

在 agent-memory-service 的 594 条测试记忆上跑 mergeSuggestions()，对比纯文本基线：

  • 纯文本 (ngram ≥ 0.7)：检出 12 对候选
  • 三信号 (composite ≥ 0.5)：检出 31 对候选
  • 其中 19 对是纯文本方法完全漏掉的
  • 误检率：人工抽查 31 对中仅 2 对不相关（6%）

19 对漏检里，最常见的模式是"决策记录 + 技术细节"——同一次讨论的两个维度。纯文本看不出来，但共享的实体（项目名、技术栈）把它们连起来了。

可操作建议

如果你在做类似的记忆/知识库去重系统：

  1. 不要只用文本相似度。哪怕你的 NER 不够准，标签 + 文本的组合也优于纯文本。权重可以调，但两个信号永远比一个信号鲁棒。
  2. 加权融合 > 级联过滤。先按文本筛再用实体补充，不如一开始就加权。级联的问题在于第一级阈值截断会丢掉信号——0.3 的文本相似度可能是合法的同义记忆，不应该在第一关被淘汰。
  3. reasons 数组是必须的。自动化系统需要可解释性，不是为了让人类审查，而是为了让下游逻辑根据原因选择不同策略。
  4. suggestedStrategy 减少 AI 的决策负担。让建议系统提供默认策略，AI 只需要确认或覆盖，比让它每次都从头分析快得多。

代码在哪

完整实现在 agent-memory-service 的 src/index.js，搜 mergeSuggestions。配套的 autoMerge() 和 bulkMerge() 构成完整的去重管道。

去重的本质不是找重复文本——是找重复意图。文本是意图的低保真投影，多信号是在重建更高保真的表示。
