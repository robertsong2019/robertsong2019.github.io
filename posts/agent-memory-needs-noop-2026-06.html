Agent 记忆系统最需要的操作是 NOOP

2026-06-15 · 罗嵩

所有主流 Agent 记忆系统——Mem0、Letta、Zep、以及我自己的 Agent Memory Service——在处理新信息时都面临同一个问题：默认操作是 ADD。用户说了一句话？存下来。工具返回了结果？存下来。日积月累，记忆库膨胀到几千条，检索精度直线下降，重要信息被噪声淹没。

2026 年 ACL 接收的 Memory-R1 论文用强化学习训练了一个记忆管理器，核心发现出乎意料：在四个操作（ADD / UPDATE / DELETE / NOOP）中，NOOP 是被触发最多的，也是最关键的。大部分对话轮次根本不需要修改记忆。

这篇文章分享我在 agent-memory-graph 中实现 Memory-R1 风格操作评分的具体方案——不需要 RL 训练，用启发式就能捕获 80% 的价值。

为什么默认 ADD 是灾难

先看一个真实场景。Agent 在一次会话中处理以下信息：

# 对话轮次 1
用户: "我用的是 Python 3.12"
→ ADD: "用户使用 Python 3.12"

# 对话轮次 2
用户: "对，Python 3.12，配 venv"
→ ADD: "用户使用 Python 3.12 配 venv"  # 冗余！

# 对话轮次 3
用户: "帮我看看这个 bug"
→ ADD: "用户请求查看 bug"  # 噪声！

# 对话轮次 4
用户: "其实我升级到 3.13 了"
→ ADD: "用户升级到 Python 3.13"  # 矛盾但没删旧的！

4 轮对话产生了 4 条记忆，其中 3 条是有问题的：第 2 条冗余，第 3 条是噪声，第 4 条与第 1 条矛盾。典型的启发式系统（"有信息就存"）在这里全军覆没。

Memory-R1 用 RL 训练后发现，最优策略的分布大致是：

操作触发比例说明
NOOP~60-70%大部分信息不值得记忆
ADD~15-20%真正新颖的信息
UPDATE~10-15%修正或补充已有记忆
DELETE<5%仅限明确矛盾/过期

注意 NOOP 的压倒性比例。这不是说 Agent 不需要记忆，而是说大部分信息不值得主动记忆——它们要么是冗余的、要么是短暂的、要么是无关的。

启发式操作评分：不训练也能用

Memory-R1 用 PPO+GRPO 训练，效果很好但成本很高。对于自建记忆系统，我设计了一套基于内容相似度的启发式评分方案，核心思想是：用新颖度（novelty）区分 ADD 和 NOOP，用相似度（similarity）区分 ADD 和 UPDATE。

核心算法

def score_memory_ops(content, existing_keys=None, noop_bias=0.15):
    """对新信息评分 4 种操作。返回按分数降序排列的操作列表。"""
    existing_keys = existing_keys or []

    # 1. 在已有记忆中找最相似的节点
    best_match, best_sim = None, 0.0
    if existing_keys:
        for kid in existing_keys:
            node = get_node(kid)
            sim = content_similarity(content, node.label)
            if sim > best_sim:
                best_sim, best_match = sim, kid
    else:
        # 搜索近似结果
        results = search_unified(content, limit=5)
        for r in results:
            sim = content_similarity(content, r.label)
            if sim > best_sim:
                best_sim, best_match = sim, r.id

    # 2. novelty = 1 - best_similarity
    novelty = 1.0 - best_sim

    # 3. 四操作评分
    scores = [
        {"op": "ADD",
         "score": novelty * (1.0 - noop_bias),
         "reason": f"novelty={{novelty:.2f}"},

        {"op": "UPDATE",
         "score": best_sim * (1.0 - noop_bias) if best_sim > 0.3 else 0.0,
         "target": best_match},

        {"op": "DELETE", "score": 0.0,
         "reason": "仅限矛盾/过期，需手动确认"},

        # NOOP: 基础偏置 + 高重叠奖励
        {"op": "NOOP",
         "score": min(noop_bias + best_sim * 0.3, 1.0)},
    ]
    return sorted(scores, key=lambda x: x["score"], reverse=True)

关键设计决策：

  • noop_bias=0.15 — 给 NOOP 一个基础偏置，让它不会在所有情况下得 0 分。Memory-R1 训练后的 NOOP 触发率约 60-70%，0.15 的偏置让启发式版本接近这个分布。
  • novelty = 1 - similarity — 如果新信息和已有记忆高度相似（sim=0.9），novelty 只有 0.1，ADD 分数极低。这自动抑制了冗余写入。
  • UPDATE 阈值 0.3 — 相似度低于 0.3 时不考虑 UPDATE，因为差异太大意味着是独立信息。
  • DELETE 永远是 0 分 — 删除操作的风险不可逆，必须人工确认。这是安全边界。

content_similarity 的实现

相似度计算用 trigram Jaccard 系数——足够轻量，不需要 embedding：

def _content_similarity(a: str, b: str) -> float:
    """Trigram Jaccard 系数。O(n) 复杂度，无外部依赖。"""
    def trigrams(s):
        s = s.lower().strip()
        return {s[i:i+3] for i in range(len(s) - 2)} if len(s) >= 3 else {s}

    ta, tb = trigrams(a), trigrams(b)
    if not ta or not tb:
        return 0.0
    return len(ta & tb) / len(ta | tb)

为什么不用 embedding？两个原因：（1）embedding 计算需要模型调用，延迟 50-200ms，而 trigram 只需 <0.1ms；（2）对于短文本（Agent 记忆的典型场景），trigram Jaccard 的效果出奇地好，因为精确字符重叠本身就是强信号。

端到端：从决策到执行

有了评分，端到端执行就很直接了：

def execute_memory_op(content, kind="fact", threshold=0.5, noop_bias=0.15):
    decision = decide_memory_op(content, threshold, noop_bias)
    op = decision["op"]

    if op == "ADD":
        node = add(content, kind)
        return {"op": "ADD", "result": "created",
                "node_id": node.id}

    if op == "UPDATE" and decision.get("target_key"):
        old = get_node(decision["target_key"])
        new_label = f"{old.label} + {content[:50]}"
        update_node(decision["target_key"], label=new_label)
        return {"op": "UPDATE", "result": "merged",
                "old": old.label, "new": new_label}

    return {"op": "NOOP", "result": "no_action",
            "reason": decision["reason"]}

关键细节：threshold=0.5 是 ADD 的最低门槛。如果 ADD 分数不到 0.5，即使它是四个操作中得分最高的，也会降级为 NOOP。这个安全网防止"低置信 ADD"污染记忆库。

UPDATE 操作不是覆盖，而是 merge——把新内容追加到已有标签后面。这保留了历史信息，同时记录了演变过程。真正的冲突消解留给更高层的逻辑（或未来的 RL 训练）。

记忆审计与有界遗忘

操作评分解决的是"新信息要不要存"的问题。但已有记忆也需要定期维护。MemoryArena（ICLR 2026）定义了 Agent 记忆的四项核心能力，其中 selective forgetting 是现有系统最薄弱的。

我实现了两个补充工具：

1. memory_audit — 全局健康评分

def memory_audit(max_nodes=500, staleness_days=30):
    """全局记忆审计: 健康评分 + 冗余分析 + 过期检测。"""
    total = stats()["nodes"]
    stale = count_nodes(accessed_before=now - staleness_days * 86400)
    avg_weight = avg(node.weight for node in all_nodes)
    redundant = count_redundant_pairs(threshold=0.7)

    health = 100 * (
        0.3 * (1 - stale / total) +        # 新鲜度
        0.3 * (1 - redundant / total) +     # 无冗余
        0.2 * min(avg_weight, 1.0) +        # 重要性
        0.2 * min(total / 100, 1.0)         # 规模合理性
    )
    return {health_score, total_nodes, stale_nodes,
            redundant_pairs, avg_importance, suggestions}

健康评分 0-100，四个维度加权。低于 60 分时系统会给出具体建议（"删除 N 个过期节点"、"合并 M 组冗余记忆"）。

2. fifa_forget — 有界遗忘

Find-and-Forget（FiFA）策略来自 MemoryArena 的 selective forgetting 评测：

def fifa_forget(budget=50, min_importance=0.1):
    """删除 budget 个最低权重 + 最陈旧的节点。"""
    # 按 weight × 时间衰减 排序，取最后 budget 个
    candidates = query("""
        SELECT id, label, weight, accessed
        FROM nodes
        WHERE weight < ? AND accessed < ?
        ORDER BY weight * freshness_score ASC
        LIMIT ?
    """, (min_importance, cutoff, budget))

    for node in candidates:
        remove_node(node.id)

    return {"forgotten": len(candidates),
            "remaining": stats()["nodes"]}

min_importance=0.1 是安全阈值——权重高于 0.1 的节点永远不会被自动遗忘。这防止了"清除所有记忆"的灾难性场景。

从反馈中学习：在线阈值调整

启发式方案的固有缺陷是阈值是固定的。AgeMem 论文的一个关键贡献是让模型从反馈中在线调整策略。我用了一个简化版本：

def memory_feedback(corrections):
    """从用户纠正数据调整 noop_bias 和 threshold。"""
    false_adds = sum(1 for c in corrections if c["predicted"] == "ADD" and c["actual"] == "NOOP")
    missed_adds = sum(1 for c in corrections if c["predicted"] == "NOOP" and c["actual"] == "ADD")

    # 误 ADD 多 → 提高 bias（更保守）
    # 遗漏 ADD 多 → 降低 bias（更积极）
    delta = 0.02 * (false_adds - missed_adds) / max(len(corrections), 1)
    new_bias = max(0.05, min(0.6, current_noop_bias + delta))

    return {"old_bias": current_noop_bias,
            "new_bias": new_bias,
            "adjustment": delta}

每次调整幅度被限制在 ±0.02 以内，bias 被限制在 [0.05, 0.6] 区间。这保证了调整是渐进的，不会因为少量反馈就剧烈震荡。这个设计直接借鉴了 AgeMem 的 step-wise 策略更新思想，但不需要梯度计算。

独到观点

💡 核心洞察

Agent 记忆系统的问题不是"记不住"，而是"忘不掉"。 大部分系统过度存储，导致检索质量随时间线性下降。解决方案不是更大的上下文窗口或更强的 embedding，而是在写入端就做好过滤——而 NOOP 是这个过滤器的核心。

几个不常见的观点：

  1. NOOP 不是"不做事"，而是一个主动的决策。 选择不记忆本身就是一种智能——人类大脑每秒丢弃 99% 的感官输入，这不是缺陷而是特性。Agent 记忆系统应该把 NOOP 作为默认操作，ADD 作为需要理由的例外。
  2. trigram Jaccard 在 Agent 记忆场景下够用了。 很多文章推荐 embedding 做语义去重，但 Agent 记忆的典型长度是 20-100 字符的短文本，这个长度范围内 trigram 的精确匹配能力反而比 embedding 的语义泛化更可靠。省下的 50-200ms 延迟可以花在更有价值的地方。
  3. DELETE 应该永远是手动的。 Memory-R1 敢于让 RL 自动 DELETE，但它有明确的奖励信号（QA 准确率）。在生产系统中，误删的代价远大于冗余存储。把 DELETE 限制为"标记为候选删除，人工确认后执行"是更安全的选择。
  4. 记忆质量比记忆数量重要 10 倍。 我的 memory_audit 实现中，冗余率和过期率各占健康评分的 30%。一个有 200 条高质量记忆的 Agent，在检索精度上会显著优于有 2000 条未过滤记忆的 Agent——这不只是性能问题，更是信噪比问题。

可操作建议

如果你在构建 Agent 记忆系统，以下是具体的行动项：

  1. 把默认操作从 ADD 改为 NOOP。 在写入前先做相似度检查，novelty < 0.5 的信息直接跳过。这一个改动可以把记忆库增长率降低 60-70%。
  2. 实现 merge 而非 overwrite。 UPDATE 操作应该追加信息而不是替换。这样既保留了历史轨迹，又避免了信息丢失。
  3. 定期运行 memory_audit。 每天或每 100 次写入后做一次健康检查。冗余率超过 20% 或过期率超过 30% 时触发清理。
  4. 收集反馈数据。 记录用户的"这个不对"和"你怎么不记得"反馈，用来调整 noop_bias。即使每天只有 2-3 条反馈，一个月积累下来就能显著改善策略。
  5. 给 DELETE 加确认门。 永远不要自动删除。最多做到"候选删除"——标记后等待人工确认或超时自动取消。

下一步：从启发式到学习式

当前的启发式方案捕获了 Memory-R1 约 80% 的价值，剩下的 20% 需要 RL 训练。但 80% 已经足够让记忆库从"垃圾场"变成"图书馆"。

我的实现路径是：先用启发式跑起来，收集足够的操作日志（每次 ADD/UPDATE/NOOP 决策 + 最终用户是否满意），然后用这些日志做蒸馏训练——不需要完整的 PPO 循环，简单的行为克隆就能超越启发式天花板。

Memory-R1 论文的一个重要启示是：精确匹配奖励就足以教会复杂的记忆整合行为。这意味着即使没有 RL 基础设施，只要有一个"用户最终是否满意"的信号，就能逐步从启发式迁移到学习式。

Agent 记忆系统的未来不是更大的上下文窗口或更强的 embedding 模型，而是更聪明的写入策略。而最聪明的写入策略，大部分时候就是——什么都不做。
