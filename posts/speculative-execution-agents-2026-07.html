Agent 推测执行：从 CPU 流水线偷回 3-5 倍延迟

2026-07-21 · agent architecture · speculative execution
Agent 的 ReAct 循环是串行的——思考、调用工具、等待、再思考。但 Agent 的工具调用模式高度可预测：93.8% 的下一步调用可以从历史轨迹中推断。借鉴 CPU 的推测执行架构，我们可以在不牺牲准确率的前提下，把 Agent 延迟降低 40-70%。本文给出 TypeScript 实现和三层加速架构。

问题：Agent 的串行税

你写过一个 5 步的 Agent 任务吗？每一步都是"LLM 推理 → 工具调用 → 等待结果 → 再推理"。假设每步 3 秒，5 步就是 15 秒。用户盯着 loading 图标等 15 秒，在 2026 年是不可接受的用户体验。

但真正的问题不是单步慢——而是步骤之间是串行的，即使它们不需要串行。

看看这个典型的 Agent 执行序列：

用户："帮我查一下最近三天的天气和新闻，然后总结"
Step 1: [LLM 推理 800ms] → 决定调用 get_weather
Step 2: [get_weather 1200ms] → 返回天气数据
Step 3: [LLM 推理 800ms] → 决定调用 get_news
Step 4: [get_news 1500ms] → 返回新闻数据
Step 5: [LLM 推理 1200ms] → 生成总结
总计：5500ms

问题在哪？Step 3 和 Step 1 之间没有真正的数据依赖——"查新闻"不需要等"查天气"的结果。如果并行发起两个调用：

Step 1: [LLM 推理 800ms] → 同时调用 get_weather + get_news
Step 2: [max(1200, 1500) = 1500ms] → 两个调用并行完成
Step 3: [LLM 推理 1200ms] → 生成总结
总计：3500ms（省 36%）

这是最简单的优化——并行工具调用。但很多 Agent 框架（包括 LangChain 的早期版本）都不做这件事。

更激进的问题：如果我们能在 Step 1 的 LLM 还在推理时，就预测到 Step 2 要调用 get_weather，然后提前执行呢？如果预测对了，省下整个等待时间。如果错了，回退重来。这就是推测执行。

CPU 给我们的三个启示

推测执行不是新概念。1990 年代的 CPU 就面临同样的问题：分支预测失败代价高昂，但不预测的代价更高（流水线停顿）。CPU 设计师的解法影响了三十年后的 AI Agent 架构：

启示 1：无损优化范式

CPU 推测执行的核心承诺是：只能加速，不会变慢。预测正确，省一个时钟周期；预测错误，丢弃结果，回到正确路径。

Agent 推测执行继承了同样的承诺。SpecEyes（ECCV 2026）的实验显示：推测执行不仅把多模态 Agent 加速 3.35 倍，准确率反而上升了 6.7%——因为并行探索的分支提供了更丰富的上下文。

这跟传统的 Agent 加速方法（模型量化、知识蒸馏、缓存）完全不同。那些方法都有精度-速度的 tradeoff。推测执行没有。

启示 2：分支可预测性远超预期

PASTE（arXiv:2603.18897, 2026）在生产环境中测得：Agent 工具调用的下一步预测命中率达到 93.8%，其中 top-1 准确率 27.8%。

这意味着什么？Agent 的行为模式远比想象中规律。"查用户资料"之后几乎总是"检查权限"。"搜索文档"之后几乎总是"读取匹配项"。这些模式可以被挖掘和利用。

启示 3：多层优化是乘法关系

CPU 的优化是分层的：分支预测 + 乱序执行 + 缓存 + 流水线。Agent 也一样：

层技术加速比实现难度
L1并行工具调用2-3.7x⭐ 简单
L2推测执行1.5-3x⭐⭐ 中等
L3Token 级优化3-6x⭐⭐⭐ 复杂
总计三层叠加5-10x—

一个 30 秒的任务，理论上可以压缩到 3-6 秒。

第一层：并行工具调用（今天就能做）

这是 ROI 最高的优化，实现只需要一个依赖图分析。核心思路：当 LLM 返回多个工具调用时，分析它们之间的数据依赖，然后并行执行无依赖的调用。

interface ToolCall {
  id: string;
  name: string;
  args: Record<string, unknown>;
  /** 此调用依赖的前序调用 ID 列表 */
  dependsOn?: string[];
}

async function parallelExecute(
  calls: ToolCall[],
  executor: (call: ToolCall) => Promise<unknown>
): Promise<Map<string, unknown>> {
  const results = new Map<string, unknown>();
  const pending = new Set(calls.map(c => c.id));

  while (pending.size > 0) {
    // 找出所有依赖已完成的调用（无依赖或依赖已就绪）
    const ready = calls.filter(c =>
      pending.has(c.id) &&
      (c.dependsOn ?? []).every(dep => results.has(dep))
    );

    if (ready.length === 0) {
      throw new Error("Deadlock: circular dependency in tool calls");
    }

    // 并行执行所有就绪的调用
    const execResults = await Promise.all(
      ready.map(async call => {
        const result = await executor(call);
        return [call.id, result] as const;
      })
    );

    for (const [id, result] of execResults) {
      results.set(id, result);
      pending.delete(id);
    }
  }

  return results;
}

使用方式：

// LLM 返回三个调用，其中只有 summarize 依赖前两个
const calls: ToolCall[] = [
  { id: "1", name: "get_weather", args: { city: "上海" } },
  { id: "2", name: "get_news", args: { topic: "科技" } },
  { id: "3", name: "summarize", args: {}, dependsOn: ["1", "2"] },
];

const results = await parallelExecute(calls, async (call) => {
  return tools[call.name](call.args);
});
// get_weather 和 get_news 同时执行，summarize 等它们都完成

效果：3 个独立调用从 3.6 秒 → 1.2 秒（取最慢一个的延迟）。

OpenAI 的 parallel_tool_calls 参数和 Anthropic 的编排层已经在做这件事。但如果你用的是自建 Agent 框架，大概率还没实现。

第二层：模式感知推测执行

这是 PASTE 的核心贡献。它不等待 LLM 推理完成，而是从历史执行轨迹中挖掘工具调用模式，提前执行下一步。

模式挖掘

interface CallPattern {
  /** 触发上下文：前 N 步的工具调用序列 */
  context: string[];
  /** 预测的下一步工具 */
  predictedTool: string;
  /** 参数推导函数（从前序结果提取参数） */
  argExtractor?: (prevResults: unknown[]) => Record<string, unknown>;
  /** 历史命中率 */
  hitRate: number;
  /** 推测安全等级 */
  safety: "readonly" | "dry-run" | "forbidden";
}

class PatternMiner {
  private patterns: CallPattern[] = [];

  /** 从执行日志中挖掘 n-gram 模式 */
  train(traces: string[][]): void {
    const ngramCounts = new Map<string, number>();
    const nextToolCounts = new Map<string, Map<string, number>>();

    for (const trace of traces) {
      for (let i = 0; i < trace.length - 1; i++) {
        const key = trace[i];
        const next = trace[i + 1];

        ngramCounts.set(key, (ngramCounts.get(key) ?? 0) + 1);
        if (!nextToolCounts.has(key)) nextToolCounts.set(key, new Map());
        nextToolCounts.get(key)!.set(next,
          (nextToolCounts.get(key)!.get(next) ?? 0) + 1);
      }
    }

    // 转换为模式规则，只保留命中率 >30% 的
    for (const [context, counts] of nextToolCounts) {
      const total = ngramCounts.get(context) ?? 0;
      for (const [predictedTool, count] of counts) {
        const hitRate = count / total;
        if (hitRate > 0.3) {
          this.patterns.push({
            context: [context],
            predictedTool,
            hitRate,
            safety: this.classifySafety(predictedTool),
          });
        }
      }
    }
  }

  /** 预测下一步工具调用 */
  predict(recentCalls: string[]): CallPattern | null {
    const last = recentCalls[recentCalls.length - 1];
    const candidates = this.patterns
      .filter(p => p.context[p.context.length - 1] === last)
      .sort((a, b) => b.hitRate - a.hitRate);
    return candidates[0] ?? null;
  }

  private classifySafety(tool: string): CallPattern["safety"] {
    const READONLY = ["search", "query", "get", "fetch", "list", "read"];
    const FORBIDDEN = ["send", "write", "delete", "transfer", "publish"];
    if (READONLY.some(r => tool.toLowerCase().includes(r))) return "readonly";
    if (FORBIDDEN.some(f => tool.toLowerCase().includes(f))) return "forbidden";
    return "dry-run";
  }
}

推测调度器

class SpeculativeExecutor {
  constructor(
    private miner: PatternMiner,
    private tools: Record<string, (args: any) => Promise<any>>,
  ) {}

  async runStep(
    llmDecision: Promise<ToolCall>,
    recentHistory: string[],
  ): Promise<{ result: unknown; speculative: boolean }> {
    // 1. 在 LLM 推理的同时，预测可能的下一步
    const prediction = this.miner.predict(recentHistory);

    if (prediction && prediction.safety === "readonly") {
      // 2. 对只读操作，立即推测执行
      const speculativePromise = this.tools[prediction.predictedTool](
        prediction.argExtractor
          ? prediction.argExtractor([])
          : {}
      ).catch(() => null); // 静默失败，不影响主流程

      // 3. 等 LLM 的正式决策
      const actualCall = await llmDecision;

      // 4. 命中检查
      if (actualCall.name === prediction.predictedTool) {
        const result = await speculativePromise;
        return { result, speculative: true }; // 命中！省下整个工具调用等待
      }
      // 未命中，丢弃推测结果，执行正式调用
      const result = await this.tools[actualCall.name](actualCall.args);
      return { result, speculative: false };
    }

    // 非只读操作，走正常路径
    const actualCall = await llmDecision;
    const result = await this.tools[actualCall.name](actualCall.args);
    return { result, speculative: false };
  }
}

关键设计决策：

  1. 只对只读操作推测——搜索、查询、读取。写入、发送、删除永远走正式路径。
  2. 静默失败——推测执行失败不影响主流程，最坏情况是浪费一点算力。
  3. 安全分级——readonly 直接推测，dry-run 可先模拟，forbidden 禁止推测。

PASTE 用这个架构在生产环境中拦截了 602 次潜在副作用操作（20000+ 次推测中），零输出偏差。

第三层：Token 级优化（进阶）

SimpleTool（2026）发现：工具调用的 JSON 结构本身就有大量冗余。函数名、参数名、嵌套结构——这些 token 可以被压缩。

// 原始调用（~180 tokens）
const verboseCall = {
  tool: "search_documents",
  arguments: {
    query: "agent memory architecture",
    limit: 5,
    filter: { type: "research_paper", year: 2026 }
  }
};

// 压缩编码（~30 tokens，6x 压缩）
const compactCall = ["sd", "agent memory architecture", 5, "rp", 2026];
// tool[0]=search_documents, filter[3]=research_paper

这需要工具端配合实现编解码器，但效果显著——SimpleTool 报告 3-6x 的单步加速。

我的观点：Agent 框架的"性能债"

大多数 Agent 框架（2024-2025 年创建的那些）都有严重的性能债。它们在设计时只考虑了功能正确性，完全没有考虑执行效率。这就像 1990 年代写了一个功能正确但没有流水线的 CPU——能跑，但慢得离谱。

具体来说：

  1. 串行工具调用是 Anti-pattern——2026 年了，如果一个 Agent 框架不支持并行工具调用，那它不应该被用于生产环境。这是 10 行代码的改动，但能省 60% 的延迟。
  2. 推测执行应该是标配——不是可选优化。Agent 行为的可预测性（93.8% 命中率）已经足够高，不做推测就是浪费。实现成本是一个中间件层，不需要修改 Agent 核心逻辑。
  3. 安全门控不能事后补——推测执行的第一天就要设计安全策略系统。PASTE 的三级分类（完全可推测 / 干运行 / 禁止）是好的起点。不要等到出了事故再加。
  4. 可观测性是前提——你需要追踪推测命中率、回退率、节省的延迟。如果命中率低于 50%，要么是你的 Agent 行为太不可预测，要么是你的模式挖掘算法太粗糙。两种情况都需要知道。

五步行动清单

优先级行动预期收益工作量
P0审计现有 Agent 的工具调用，找出可并行的调用对2-3x 延迟降低1 天
P0实现并行工具执行（上面的 parallelExecute）即时生效1-2 天

结语：从"能跑"到"跑得快"

2026 年的 Agent 领域有一个奇怪的矛盾：模型推理速度越来越快（GPT-5 的首 token 延迟已经降到 200ms 以内），但 Agent 任务的端到端延迟几乎没有改善。原因很简单——瓶颈不在模型推理，而在工具调用的串行等待。

CPU 在 1990 年代就解决了类似的问题。我们在 Agent 领域不需要重新发明轮子——只需要把三个已知的优化层叠加起来。并行调用是地基，推测执行是加速器，Token 优化是锦上添花。

最有意思的是：这些优化都是无损的。你不需要牺牲任何准确率来获得加速。你只需要承认一个事实——Agent 的行为比你想象的更可预测，而可预测性就是免费的性能。
