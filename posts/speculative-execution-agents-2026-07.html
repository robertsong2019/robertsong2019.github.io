Agent 推测执行：让 AI 学会 CPU 的"抢跑"哲学

2026-07-20 · 罗嵩 · Speculative Execution AI Agent 性能优化 并行计算
你正在用 AI Agent 处理一个复杂任务。Agent 想了一下，决定调用搜索 API——等了 300 毫秒。拿到结果，又想了一下，决定读取一个文件——又等了 200 毫秒。再想，再调……15 步下来，用户等了快一分钟。但如果你仔细看 Agent 的行为模式，会发现一个令人沮丧的事实：它下一步要做什么，往往在当前步骤就已经决定了——只是它还在"认真思考"而已。那我们为什么不提前帮它把事做了？

这正是 2025-2026 年 AI Agent 领域最激动人心的工程突破：推测执行（Speculative Execution）。这个概念借自已经用了 30 年的 CPU 设计——处理器不会傻等分支判断的结果，它会预测哪条路更可能，提前跑起来。猜对了，白赚时间；猜错了，回退重来。现在，这套思路终于来到了 AI Agent 的世界。

一、问题：Agent 的"排队综合征"

要理解为什么推测执行如此重要，先得看清当前 Agent 的致命弱点。

今天绝大多数 AI Agent 都遵循 ReAct 模式——一个"思考-行动-观察"的循环：

用户请求 ↓ ┌──→ [LLM 推理] ──→ [调用工具] ──→ [等待结果] ──┐ │ │ └────────────────────────────────────────────────┘ ↓ 最终回答

这个循环的问题在于：完全串行。每一步必须等上一步完成才能开始。工具调用（API 请求、数据库查询、文件读取）占总时间的 35-61%。当任务需要 5-15 步时，延迟线性累积。

一个具体的例子：你让 Agent "总结我的账户信息"。它需要：查余额（300ms）→ 查交易记录（300ms）→ 查通知设置（300ms）→ 综合分析（1s）。总计 1.9 秒。但前三个调用完全独立——如果并行执行，只需 300ms 而非 900ms，总时间砍到 1.3 秒。

更极端的场景：两个 SOTA Agent 下一盘棋，因为每步都是串行 API 调用链，可以花好几个小时。不是模型推理慢，而是串行等待把时间吃光了。

二、三层加速：从简单到激进

2026 年的 Agent 加速技术已经形成了清晰的三层架构，每层解决不同维度的延迟：

┌─────────────────────────────────────────┐ │ 用户请求 / Agent 任务 │ ├─────────────────────────────────────────┤ │ Layer 3: Token 级优化 (SimpleTool) │ │ 函数名+参数并行生成，JSON 压缩 4-6x │ │ 效果：3-6x intra-step 加速 │ ├─────────────────────────────────────────┤ │ Layer 2: 推测执行 (PASTE/SpecEyes) │ │ 预测下一步+预执行，认知门控+回滚 │ │ 效果：1.5-3x inter-step 加速 │ ├─────────────────────────────────────────┤ │ Layer 1: 并行工具调用 (LLMCompiler) │ │ 依赖图分析，独立任务并发 │ │ 效果：最高 3.7x 并行加速 │ ├─────────────────────────────────────────┤ │ 基础层: 工具后端 / API / 数据库 │ └─────────────────────────────────────────┘

这三层是乘法关系。一个 30 秒的任务，经过三层优化后可以压缩到 5-8 秒。对语音 Agent 和机器人控制来说，这意味着从"概念验证"到"生产可用"的跨越。

第一层：并行工具调用——最低悬的果实

最简单也最有效的优化。当多个工具调用互相独立时，同时发起。

UC Berkeley 的 LLMCompiler（ICML 2024）将这个思路形式化。它借鉴传统编译器的三阶段设计：

  1. Planner：分解任务，分析依赖关系，构建 DAG（有向无环图）
  2. Task Fetching Unit：为每个独立任务准备输入参数
  3. Executor：并发执行所有不互相依赖的任务

结果：3.7x 延迟加速，6x 成本节省（更少的 LLM 推理轮次），甚至准确率提升约 9%（因为减少了多轮交互中的错误累积）。

OpenAI 的 GPT-4/4o 和 Anthropic 的 Claude 已经原生支持并行函数调用。但对更复杂的依赖分析，需要像 LLMCompiler 这样的专门系统。

第二层：推测执行——CPU 的智慧

并行调用优化的是同一步内的延迟。推测执行更进一步，优化步与步之间的延迟。

核心思路用一句话概括：用小模型猜大模型下一步会做什么，提前把那件事办了。

这直接来自 CPU 设计的灵感。你的处理器每天都在做这件事：

CPU 流水线: 指令 A: cmp r1, r2 → 跳转到 ??? ↓ (不等结果，猜 "大于") 指令 B (speculative): mov r3, [r4] ↓ 判断结果: 确实 "大于" → ✓ 提交 B 的结果 判断结果: 不是 "大于" → ✗ 丢弃 B 的结果，回到正确分支 Agent 推测执行: LLM 推理中... "我需要查用户权限" ↓ (不等 LLM 输出完毕，小模型猜下一步) 推测执行: fetch_user_permissions(user_id) ↓ LLM 输出完毕: 确实要查权限 → ✓ 推测结果直接可用 LLM 输出完毕: 要查别的 → ✗ 丢弃推测结果

MIT 和 Cornell 的 Speculative Actions 框架（2025年10月）是这个领域的开创性工作。它用更小更快的"草稿模型"（draft model）预测大模型的下一步行动，在游戏、电商、搜索环境中实现了 55% 的预测准确率和最高 20% 的端到端延迟降低。

关键特性：无损（lossless）。猜对了省时间，猜错了回退到串行路径。准确率不会降低，只会变快。

为什么"无损"是范式转变？
传统的 Agent 加速方法——模型量化、知识蒸馏、KV-Cache 压缩——都有精度-速度 tradeoff。你需要回答"牺牲多少精度换多少速度"。

推测执行完全不同。优化空间从"牺牲什么换什么"变成了"花多少算力换多少速度"。这是一个没有下界的优化——你只需要找到预测准确率足够高的 draft model。

第三层：Token 级优化——压榨最后一份冗余

即使推测执行和并行调用到位，每次 LLM 生成函数调用时还有大量浪费。函数调用的 JSON 输出——{"function": "search", "arguments": {"query": "..."}}——包含大量低熵的冗余 token：花括号、参数名、引号。

SimpleTool（2026年3月）的方案：引入特殊 token 压缩这些序列 4-6 倍，同时并行解码函数名和参数。因为知道你在调 search_database 对推断查询字符串几乎没有帮助——两者因果关系很弱。

结果：3-6x 端到端加速。在消费级 GPU 上，4B 参数模型实现 61.2ms P50 延迟——达到 16Hz 控制频率，满足实时机器人控制需求。

三、PASTE：当推测执行走进生产环境

如果说 Speculative Actions 证明了概念，PASTE（Pattern-Aware Speculative Tool Execution）则把它推向了生产级。这篇来自上海交大和微软研究院的工作（2026年3月）有几个关键突破：

洞察：Agent 行为是有模式的

PASTE 的核心发现是：Agent 的工具调用序列远比想象中规律。

  • 查了用户资料？86% 的概率下一步查权限
  • 读了文件？73% 的概率接下来要编辑
  • 搜索了关键词？68% 的概率要打开某个结果

PASTE 从执行轨迹中挖掘这些模式，构建"模式元组"（pattern tuples），每个元组包含：上下文、预测工具、参数推导函数、经验成功率。

数据说话

指标结果
平均任务完成时间降低43.5%
工具执行吞吐量提升1.8x
P95 延迟降低48.6%
P99 延迟降低61.9%
预测总体命中率93.8%
Top-1 精确预测率27.8%

93.8% 的命中率意味着：在近 94% 的情况下，PASTE 预测的"可能调用列表"中包含了正确的工具。27.8% 的 top-1 准确率更令人印象深刻——超过四分之一的场景中，它精确猜中了下一步调用哪个工具、用什么参数。

安全性：不是所有工具都能"抢跑"

推测执行对只读操作（搜索、查询、读取）是安全的——预执行一个搜索不会产生任何副作用。但对写入操作（发邮件、数据库写、金融交易），"抢跑"可能造成不可逆后果。

PASTE 的策略系统将工具分为三档：

  • Fully speculatable：GET 请求、搜索、文件读取——随便推测
  • Dry-run eligible：可以先模拟（如数据库事务的 BEGIN-ROLLBACK）
  • Speculation-prohibited：写入、删除、转账——绝对禁止推测

在测试中，PASTE 在 20000+ 次推测执行中检测并拦截了 602 次潜在副作用操作，最终输出与基线零偏差。这是"无损"承诺的工程验证。

部署极轻

PASTE 以中间件形式部署，无需修改 Agent 逻辑。资源开销极小：

  • CPU：0.02 core-seconds/秒延迟降低
  • 内存：2.6 MB
  • 网络：0.9 MB/秒延迟降低

轻到可以跑在 sidecar container 里。TypeScript 8000 行 + Python 4000 行，已集成 Gemini-CLI、Qwen-DeepResearch 等真实系统。

四、SpecEyes：让多模态 Agent 也能"抢跑"

SpecEyes（ECCV 2026）把推测执行带入了多模态领域。对多模态 Agent（如视觉问答系统），串行瓶颈更严重——每次调用视觉工具都要处理高分辨率图像，单步延迟可达数秒。

SpecEyes 的创新是"认知门控"（Cognitive Gate）：

  1. 用一个轻量级无工具小模型（如 0.5B VLM）快速"扫一眼"图像和问题
  2. 测量"答案可分性"（answer separability）——这个模型对自己的答案有多确定？
  3. 如果确定——直接返回，跳过昂贵的工具链
  4. 如果不确定——交给大模型做完整工具调用

关键结果：准确率不降反升 6.7%。这说明很多中间工具调用本来就是冗余的——小模型直接给出的答案和走完完整工具链的结果一样好，甚至更好（因为避免了多步累积错误）。

加速幅度：1.1x 到 3.35x，取决于任务复杂度。

深层洞察：SpecEyes 的结果暗示了一个更反直觉的结论——Agent 的很多"思考"步骤不是必要的推理，而是"习惯性确认"。就像人不需要每次开门都检查门把手的设计一样，Agent 也不需要每次都调用完整的工具链来验证它已经知道的事情。

五、这些技术如何叠加？

来看一个实际的例子。一个 Agent 需要完成一个涉及 6 次工具调用的任务，其中 3 次是独立的：

优化阶段等价步骤预计时间
基线串行6 串行调用 + 6 次 LLM 推理~30 秒
+ 并行调用4 等效步骤（独立对合并）~20 秒
+ 推测执行2-3 次推测命中省去完整往返~12 秒
+ Token 优化每次推理 3-6x 更快~5-8 秒

从 30 秒到 5-8 秒，约 4-6 倍加速。对一个语音 Agent 来说，这意味着从"用户等待时听 30 秒沉默"到"对话级别的自然响应"。

六、实践的 tradeoff

技术这么好，为什么不是所有 Agent 都在用？因为真正的工程决策总有代价：

计算成本

推测执行用算力换延迟。每次错误预测都是浪费的 CPU 周期、API 调用费用。PASTE 的开销被证明很小（"轻量 sidecar"），但在大规模部署中，即使每请求很小的浪费也会累积。关键问题是：延迟降低带来的用户体验提升，是否值额外的算力成本？

依赖分析的复杂性

并行调用需要精确的依赖分析。两个调用看起来独立，但可能通过隐式共享状态产生竞争条件。大多数框架选择保守策略——只并行化模型明确声明独立的调用。更激进的系统（如 LLMCompiler）做自动依赖图分析，但这本身就是计算开销。

副作用的不可逆性

对只读操作推测执行是安全的。但真实 Agent 经常需要写操作。一个 Agent 如果推测执行了"发送邮件"然后发现预测错了——你不能"撤销"一封已发的邮件。PASTE 的策略系统是必要的，但也限制了适用范围。

冷启动问题

PASTE 依赖历史调用模式来预测未来。一个新部署的 Agent 没有历史数据，预测准确率很低。需要积累足够的执行轨迹后，推测执行才能发挥作用。

七、更大胆的思考：这指向什么？

Agent 行为的可预测性

PASTE 的 93.8% 命中率揭示了一个深刻的事实：Agent 行为远比我们以为的可预测。如果你能 94% 地猜中 Agent 下一步要调用什么工具，那 Agent 的"智能"到底体现在哪里？

答案可能是：Agent 的智能主要体现在处理异常和边界情况。90%+ 的常规调用可以被视为"模式匹配后的模板执行"，真正需要推理的是那些落在模式之外的情况。推测执行本质上是在自动识别并优化"不需要智能"的部分。

推测执行 vs 世界模型

从更高层面看，推测执行是世界模型的工程化实现。Agent 在"脑中模拟未来"（小模型预测大模型行为），然后根据模拟结果决定行动。这与机器人学中的 Model Predictive Control（MPC）异曲同工。

未来，我们可能会看到推测执行和世界模型的融合——不再只是预测"下一步调什么工具"，而是预测"如果执行这个操作，世界状态会如何变化"，据此决定是否提前执行。

"中间步骤冗余"的启示

SpecEyes 准确率反升 6.7% 的发现，对 Agent 设计有直接启发：很多中间步骤不仅浪费时间，还可能有害。每多一次工具调用，就多一次引入错误的机会。如果能跳过不必要的步骤，Agent 既更快又更准确。

八、你可以立刻做什么

如果你在开发 AI Agent，以下是按优先级排列的实践建议：

Level 1：本周可做——并行化审计

  1. 审查你的 Agent 的工具调用日志
  2. 标出哪些调用对是真正独立的（无数据依赖、无共享状态）
  3. 对独立调用对使用 Promise.all（JS）或 asyncio.gather（Python）并行执行
  4. 这通常能带来 2-3x 加速，几乎零风险

Level 2：两周可做——模式分析

  1. 收集 100+ 次完整 Agent 执行轨迹
  2. 对工具调用序列做 n-gram 分析，计算 top-1 和 top-3 预测准确率
  3. 如果 top-3 准确率 > 70%，推测执行值得投入
  4. 识别你的 Agent 的"标志性模式"——每个 Agent 都有自己的习惯

Level 3：一个月可做——推测执行原型

  1. 参考 PASTE 架构，构建中间件层
  2. 从只读操作开始（搜索、查询、读取）
  3. 实现简单的"工具调用预测器"（n-gram 或小模型）
  4. 添加命中率监控作为关键可观测指标
  5. 逐步扩大推测范围，持续监控输出偏差

Level 4：长期方向——多模型协作

  • 部署 0.5B 小模型作为"草稿预测器"
  • 与 Agent 记忆系统集成——从历史中学习调用模式
  • 探索多步推测（预测 2-3 步之后的行为）
  • 与 Agent 记忆系统联动——历史模式从记忆中检索

结语：并行思维的时代

AI Agent 领域有一种迷思：只要模型足够强，一切问题都能解决。但现实是，即使 GPT-5 级别的模型也逃不开物理定律——API 调用有延迟，工具执行需要时间，串行循环有上限。

推测执行告诉我们：让 Agent 更快的杠杆，往往不在模型本身，而在系统架构。就像 CPU 靠流水线和分支预测实现了千倍加速（而非靠单个晶体管更快），Agent 也需要从"串行思维"转向"并行思维"。

从 LLMCompiler 的并行调用，到 PASTE 的模式感知推测，再到 SimpleTool 的 token 级压缩——这些工作的共同信息是：Agent 系统中到处都是冗余和浪费，只要你愿意用工程的眼光去审视。

2026 年的 Agent 推测执行，就像 1990 年代 CPU 的超标量设计——一个看似简单的好主意，最终改变了整个系统的架构范式。
