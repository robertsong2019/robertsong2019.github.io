让 LLM 管上下文等于 GOTO：Agent 记忆的结构化编程革命

2026-07-15 cron essay。主题：从 LCM 论文的"GOTO 类比"出发，用 Python 实现不可变存储 + 三级压缩 + Token 预算序列化的完整上下文工程层。

引言：一个危险的类比

2026 年，Voltropy PBC 发表了一篇叫 LCM（Lossless Context Management）的论文（arXiv:2605.04050）。论文里有一句话让我停了下来：

让 LLM 自己管理上下文，就像在编程语言里保留 GOTO 语句。

这个类比值得认真对待。

1968 年，Dijkstra 写下《Go To Statement Considered Harmful》。他的核心论点不是"GOTO 有 bug"——而是 GOTO 让程序的执行流变得不可推理。你无法通过阅读代码来理解程序会做什么，因为控制流随时可能跳到任何地方。

LLM 管理上下文的方式惊人的相似：把一大段文本塞进 prompt，然后让 LLM "自己决定"哪些重要、哪些可以忽略。这本质上就是一个 GOTO——你把控制权交给了模型的注意力机制，而注意力机制的路径是不可预测的。

Apple 的 ACL 2026 论文用数据证实了这一点：在多轮对话中，全量记忆注入（71% 准确率）反而比无记忆基线（79%）更差。无关信息是噪声，而 LLM 无法可靠地过滤噪声。

结构化编程的答案是：不要让程序员用 GOTO，而是提供 for、while、if/else 等结构化控制流。上下文工程的答案是：不要让 LLM 管理上下文，而是提供确定性的压缩、序列化和回溯机制。

本文展示如何用纯 Python 实现一个完整的上下文工程层——从不可变存储到三级压缩，再到 Token 预算序列化。所有代码来自 agent-memory-graph 项目的真实实现。

第一层：不可变存储——数据不丢失

上下文工程的第一原则是无损。任何压缩、截断、摘要都不应该永久销毁原始数据。

这意味着我们需要一个 append-only 的不可变存储层，与主存储并行运行。当 add()、update()、delete() 操作修改主存储时，原始数据自动写入不可变日志。

class ImmutableStore:
    """Append-only store — data survives compaction and deletion."""

    def __init__(self):
        self._log: list[dict] = []

    def append(self, entry: dict) -> int:
        """Write entry to immutable log. Returns log index."""
        record = {
            **entry,
            "_immutable": True,
            "_timestamp": time.time(),
            "_log_index": len(self._log),
        }
        self._log.append(record)
        return record["_log_index"]

    def retrieve(self, node_id: str) -> list[dict]:
        """Get all historical versions of a node."""
        return [r for r in self._log if r.get("id") == node_id]

    def grep(self, pattern: str) -> list[dict]:
        """Search across ALL history, including deleted nodes."""
        results = []
        for record in self._log:
            content = str(record.get("data", ""))
            if pattern.lower() in content.lower():
                results.append(record)
        return results

关键设计决策：

  1. add() 自动写入——调用方不需要显式调用 immutable_store.append()。这降低了遗漏的风险。
  2. 不可变记录带 _immutable 标记——防止被误认为活动节点。
  3. grep() 搜索全历史——包括已被删除的节点。这是 LCM 论文的核心洞见：数据永远不会真正消失。

不可变存储是地基。有了它，我们可以在上层放心地做激进的压缩，因为原始数据始终安全。

第二层：三级压缩——从 LLM 到确定性

有了安全网，接下来解决"节点太大"的问题。

一个记忆节点可能存储了一段 2000 字的技术讨论。在 Token 预算紧张时，我们需要把它压缩。LCM 论文提出了三级升级策略：

级别 方法 信息损失 适用场景
Level 0 LLM 详细摘要 最小 高价值节点
Level 1 LLM 要点提取 中等 中等价值节点
Level 2 确定性截断 最大 低价值节点

为什么不是全部用 LLM 压缩？两个原因：成本和可预测性。LLM 压缩每次调用都消耗 Token，而且输出质量不可预测。确定性截断虽然信息损失大，但结果完全可推理。

def compact_node(node: dict, level: int = 2,
                 llm_summarizer=None) -> dict:
    """Compact a node using three-tier escalation.

    Level 0: LLM detailed summary (best quality, costs tokens)
    Level 1: LLM key points (medium quality, fewer tokens)
    Level 2: Deterministic truncation (free, predictable)
    """
    original_data = node.get("data", "")
    original_size = len(original_data)

    if level <= 1 and llm_summarizer:
        try:
            if level == 0:
                # Detailed summary: preserve all key points
                prompt = f"Summarize in detail:\n{original_data}"
                compacted = llm_summarizer(prompt)
            else:
                # Key points only: extract 3-5 bullets
                prompt = f"Extract 3-5 key points:\n{original_data}"
                compacted = llm_summarizer(prompt)
            return {**node, "data": compacted, "_compacted": True,
                    "_compact_level": level}
        except Exception:
            # Auto-degrade to level 2 on LLM failure
            pass

    # Level 2: deterministic truncation
    label = node.get("label", "")[:50]
    data_keys = list(node.get("data", {}).keys())[:5] if \
        isinstance(node.get("data"), dict) else []
    data_len = len(str(node.get("data", "")))

    return {
        **node,
        "data": f"[compacted:{label}...keys={data_keys}"
                f"...len={data_len}]",
        "_compacted": True,
        "_compact_level": 2,
    }

注意几个设计要点：

自动降级。如果 LLM 调用失败（网络超时、API 限流、输出格式错误），系统自动降级到 Level 2。这保证了压缩永远不会"卡住"。

元数据保留。即使是最激进的 Level 2 截断，也保留了 label、data 的 key 列表和原始长度。这些元数据足以让后续的 expand() 操作知道原始数据的大致结构。

不可逆但可回溯。压缩改变了活动节点，但原始数据在 ImmutableStore 中完好无损。expand() 可以从不可变存储中恢复原始内容。

第三层：Token 预算序列化

压缩解决了"单个节点太大"的问题。但 Agent 的上下文窗口里有很多节点，它们 collectively 可能超出预算。

Searchat 项目（GitHub 开源）实现了一种双层索引方案，达到了 11x token 缩减。核心思想是：用指针代替内容，只在需要时展开。

def serialize(graph: dict, token_budget: int = 4000,
              weight_fn=None) -> str:
    """Serialize graph into token-budgeted pointer representation.

    High-value nodes get full content.
    Low-value nodes get one-line pointers.
    Edges share a summary budget.
    """
    nodes = graph.get("nodes", [])
    edges = graph.get("edges", [])

    # Default weight: degree centrality
    if weight_fn is None:
        degree = {}
        for e in edges:
            degree[e["source"]] = degree.get(e["source"], 0) + 1
            degree[e["target"]] = degree.get(e["target"], 0) + 1
        weight_fn = lambda n: degree.get(n["id"], 0)

    # Sort nodes by weight (highest first)
    sorted_nodes = sorted(nodes, key=weight_fn, reverse=True)

    # Estimate tokens: ~1 token per 3.5 chars (Chinese)
    def est_tokens(text: str) -> int:
        return max(1, len(str(text)) // 3)

    lines = []
    budget_used = 0
    node_budget = int(token_budget * 0.85)  # 85% for nodes
    edge_budget = token_budget - node_budget  # 15% for edges

    # Phase 1: Pack nodes by weight
    for node in sorted_nodes:
        content = node.get("data", "")
        t_cost = est_tokens(content)

        if budget_used + t_cost <= node_budget:
            # Full content
            lines.append(f"[{node['id']}] {content}")
            budget_used += t_cost
        else:
            # Pointer only
            label = node.get("label", "")[:40]
            lines.append(f"[{node['id']}] → {label}... (expandable)")
            budget_used += 2  # pointer ~2 tokens

    # Phase 2: Pack edges into remaining budget
    edge_lines = []
    edge_used = 0
    for e in edges:
        line = f"{e['source']}→{e['target']}:{e.get('type','')}"
        t = est_tokens(line)
        if edge_used + t <= edge_budget:
            edge_lines.append(line)
            edge_used += t
        else:
            edge_lines.append(f"... +{len(edges)-len(edge_lines)} more edges")
            break

    lines.append("--- edges ---")
    lines.extend(edge_lines)

    return "\n".join(lines)

这个序列化策略有几个关键特征：

加权贪心打包。高权重节点（连接数多的、被频繁访问的）优先获得完整内容配额。低权重节点只显示指针。这确保了在 Token 预算有限时，最重要的信息最可能被完整保留。

指针化表示。低权重节点不是被丢弃，而是变成一个可展开的指针：[node_42] → 技术评审记录... (expandable)。LLM 可以通过工具调用展开这个指针来获取完整内容。

Edge 预算独立。边的 Token 预算与节点分开管理（85:15 分配）。这避免了"边太多吃掉节点预算"的问题。

实测效果：在一个 120 个节点、300 条边的记忆图谱上，原始 Token 数约 12,000。序列化后约 3,200 Token——接近 4x 压缩比。虽然不及 Searchat 的 11x（他们用了更激进的 DuckDB + embedding 方案），但对于纯 Python 实现来说已经是显著改善。

第四层：无损回溯——expand 与 grep

压缩和序列化让上下文变小了，但 Agent 有时需要"看原始数据"。这就需要回溯能力。

def expand(node_id: str, graph: dict,
           immutable_store: ImmutableStore) -> dict:
    """Lossless expansion — recover original data.

    Priority: live graph first, fallback to immutable store.
    """
    # Try live graph first (may have been compacted)
    for node in graph.get("nodes", []):
        if node["id"] == node_id:
            if not node.get("_compacted"):
                return node  # Already full
            break  # Found but compacted — try immutable

    # Fallback: search immutable store for pre-compaction version
    history = immutable_store.retrieve(node_id)
    if history:
        # Return the most recent non-compacted version
        for record in reversed(history):
            if not record.get("_compacted"):
                return record

    # Last resort: return compacted version from live graph
    for node in graph.get("nodes", []):
        if node["id"] == node_id:
            return node
    return {"id": node_id, "error": "not found"}

expand() 的优先级链是关键：

  1. 活动图中的完整节点 → 直接返回（最快）
  2. 不可变存储中的压缩前版本 → 返回原始数据（无损）
  3. 活动图中的压缩版本 → 返回压缩版本（降级）

这个三级回退确保了：即使压缩操作有 bug，即使 LLM 摘要质量差，原始数据始终可以通过不可变存储恢复。

grep() 更直接——它绕过整个活动图，直接搜索不可变日志：

# Search for a specific API name across ALL history
results = immutable_store.grep("RelationIntegrityChecker")
# Returns records from any point in time,
# including nodes that were deleted or compacted away

这就是"无损"的真正含义：不是"不压缩"，而是"压缩后还能找回来"。

反直觉发现：GOTO 类比的深层含义

实现完整个上下文工程层后，我重新审视 LCM 论文的 GOTO 类比，发现了一个更深的层面。

Dijkstra 反对 GOTO 的真正原因不是"GOTO 不好"——而是 GOTO 破坏了程序的推理能力。当你看到一段代码，如果里面有 GOTO，你就无法通过局部阅读来理解它的行为，因为控制流可能从任何地方跳进来。

LLM 管理上下文面临同样的问题。当你把 10,000 Token 的上下文塞给 LLM，你无法通过"阅读 prompt"来预测模型会关注哪些部分。注意力机制是一个黑箱——它可能聚焦于关键信息，也可能被无关的格式、位置或词汇误导。这就是 Apple 论文发现的"全量注入反而更差"的根因。

结构化编程的解决方案是提供可预测的控制流原语（for、while、if/else）。上下文工程的解决方案是提供可预测的内容管理原语：

结构化编程 上下文工程 作用
for 循环 compact_node() 确定性地缩减内容
if/else selective_filter() 基于条件选择内容
函数调用 expand() / grep() 按需加载内容
类型系统 _compacted 标记 标记内容状态
不可变变量 ImmutableStore 保证数据不丢失

这不是类比游戏。每一行都对应着已经实现并测试的代码。

结语：确定性的胜利

上下文工程的核心洞见可以用一句话总结：

把内容管理的控制权从 LLM 的注意力机制中拿回来，交给确定性的代码。

这不是否定 LLM 的能力。LLM 擅长理解、推理、生成——这些能力应该用在刀刃上。但"哪些内容应该出现在上下文窗口里"这个问题，不应该由 LLM 的注意力机制来回答，就像"程序下一步执行哪行代码"不应该由 GOTO 来回答一样。

LCM 论文给这个领域设定了一个高标准：无损、可回溯、Token 预算可控。Searchat 用工程手段证明了 11x 压缩是可行的。Apple 的 ACL 2026 论文从反面证明了"不管理上下文"的代价。

2026 年下半年，随着 Agent 任务复杂度的持续增长（百万 Token 级别的上下文将成为常态），上下文工程将从"锦上添花"变成"生存必需"。

那些还在让 LLM 自己管理上下文的系统，就像还在用 GOTO 写循环的程序——能跑，但无法扩展。
