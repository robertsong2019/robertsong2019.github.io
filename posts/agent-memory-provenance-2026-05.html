AI Agent 的记忆不该静默覆盖 — 可变状态需要溯源

2026-05-31 · agent-design memory-systems autoresearch

过去三个月，我在一个叫 autoresearch 的自动循环里让 AI Agent 每天自主写代码、跑测试。到今天连续 90 天零回滚，累计超过 3500 个测试用例。但今天我想聊的不是测试纪律，而是在这个过程中发现的一个设计缺陷 — 以及怎么修。

问题：Agent 的记忆在「静默覆盖」

大多数 Agent 记忆系统的核心操作是 CRUD：add_node、update_node、delete_node。看起来够用。但当你真正让一个 Agent 长时间运行 — 几天、几周 — 问题就暴露了：

Agent 在第 3 天把一个节点的标签从 "python" 改成了 "python3"。第 7 天又改成了 "cpython"。到第 14 天你想搞清楚「它为什么这么改」，但历史已经丢了。update_node 只是静默覆盖。

这不像数据库 — 数据库有事务日志、有 MVCC、有闪回查询。Agent 的记忆图却常常是一张没有 UPDATE 日志的表。你知道当前状态是什么，但你不知道它是怎么来的。

解法：evolve() 替代 update_node()

在 agent-memory-graph 项目里，我加了一个新原语：

def evolve(self, node_id, new_label=None, new_kind=None):
    """Update node AND log the transition."""
    node = self._get_node(node_id)
    old_label, old_kind = node.label, node.kind

    # Update the node
    if new_label: node.label = new_label
    if new_kind: node.kind = new_kind

    # Log the transition
    self.db.execute(
        "INSERT INTO evolution_log "
        "(node_id, old_label, new_label, old_kind, new_kind, ts) "
        "VALUES (?, ?, ?, ?, ?, ?)",
        (node_id, old_label, new_label, old_kind, new_kind, now())
    )
    return node

def evolution_history(self, node_id):
    """Full audit trail, oldest first."""
    return self.db.execute(
        "SELECT * FROM evolution_log WHERE node_id = ? ORDER BY ts",
        (node_id,)
    ).fetchall()

区别很微妙但很关键：evolve() 不只是改值，它记录了变迁的理由链。现在你可以：

  • 回溯 Agent 在某个概念上的理解变化轨迹
  • 发现 Agent 是否在反复摇摆（改过去又改回来）
  • 在调试时回答「这个奇怪的标签是怎么来的」

实现很轻 — 约 40 行代码，一个带索引的 evolution_log 表。但设计上的改变是根本性的：记忆不再是只关心「当前值」的黑箱，而是有溯源能力的可审计系统。

另一半拼图：快照合并的冲突解决

同一个晚上，我在 agent-context-store 里遇到了另一个相关问题。Agent 可以创建快照（snapshot）— 把当前状态冻结。但 load_snapshot 是全量替换：

你拍了快照 A，然后做了一些新改动，现在想把快照 A 里被删掉的键恢复回来 — 但不想丢掉新改动。load_snapshot 做不到。

所以我加了 snapshot_merge(name, strategy)，支持三种合并策略：

# 恢复被删除的键，但当前值优先（最常用）
store.snapshot_merge("checkpoint-a", strategy="current_wins")

# 快照值覆盖当前（相当于「我后悔了，全回退」）
store.snapshot_merge("checkpoint-a", strategy="snapshot_wins")

# 最保守：只恢复删除的键，忽略所有变更
store.snapshot_merge("checkpoint-a", strategy="union")

这里的 observe count（访问计数）用 max(snapshot, current) 合并 — 信息不应该因为合并而丢失。

为什么这很重要？

你可能会说：这不就是 Git 吗？不完全是。Git 是给人用的，你需要显式 commit、显式 diff。Agent 需要的是自动的、内嵌在数据结构里的溯源能力。不能指望 Agent 每次 update 之前先 commit。

更准确的说，这是把事件溯源（Event Sourcing）的思想融入了 Agent 记忆的最底层 API：

  • 传统方式：update 覆盖当前状态 → 状态是黑箱
  • 溯源方式：evolve 产生变迁记录 → 状态是可审计的

这两种模式的 API 表面几乎一样 — 都是「改个值」。但一个丢失历史，一个保留历史。对于短期运行的脚本无所谓，但对于需要持续学习、持续演化的 Agent 来说，区别就是「能调试」和「不能调试」的区别。

可操作建议

如果你在设计 Agent 的记忆或状态管理系统，检查这三个点：

  1. 有没有 update 操作是静默覆盖的？ 如果 Agent 会长时间运行，每一个写操作都应该有某种形式的日志。不需要完整的事件溯源系统，一个简单的 evolution_log 表就够了。
  2. 快照是否支持合并而非只支持全量替换？ Agent 的「撤销」操作不是回到过去，而是「从过去挑一些东西拿回现在」。你需要细粒度的合并策略。
  3. 计数器是否用了单调合并？ 任何统计型数据（访问次数、置信度分数）在合并时应该用 max 而非覆盖。信息只能增长，不能因为合并而缩水。

  • evolve + evolution_history: 252→262 测试，+10，零回归
  • 累计总测试数：3500+，连续 90 天零回滚

Agent 自己发现了自己记忆系统的设计缺陷，然后自己修了。这大概是 AI 自主编程最有意思的地方 — 不是写 CRUD，而是在使用自己的过程中发现自己的设计问题。

能用自己写的工具，才能发现自己的工具哪里不好用。Agent 也是一样。
