Agent = Model + Harness：一个正在形成的世界观

2026-06-30 · AI Agent Harness Engineering 架构思考

当 Claude Code、Codex、OpenClaw、Hermes、Cursor 这些产品同台竞争时，一个自然的问题浮现：它们都在用同样的前沿模型（Claude/GPT/Gemini/GLM），那差异到底在哪？

答案指向一个词：Harness（挽具/脚手架）。Agent 的本质就是模型外面裹的那一层工程——但这个"外面一层"正在变得越来越厚、越来越复杂，甚至比模型本身更值得认真对待。

Model + Harness = Agent

一、什么是 Harness？

Harness 原意是"马具/挽具"——套在马身上、把马的力量转化为可用功的那套皮带和缰绳。在 AI Agent 语境下，它指包裹在 LLM 外层的所有工程层，把"能聊天"的模型变成"能干活"的 Agent。

┌──────────────────────────────────────────────┐ │ Agent (用户看到) │ │ │ │ ┌────────────────────────────────────────┐ │ │ │ Harness 层 │ │ │ │ │ │ │ │ · Agent Loop (感知→推理→行动→观察) │ │ │ │ · Tool Use (工具定义/调用/结果解析) │ │ │ │ · Memory (短期/长期/工作记忆) │ │ │ │ · Planning (任务拆解/规划/反思) │ │ │ │ · Permissions (权限模型/安全护栏) │ │ │ │ · Sub-agents (子Agent编排) │ │ │ │ · Context Engineering (上下文窗口管理) │ │ │ │ · Evals (质量评估/自检) │ │ │ │ · Multi-channel (IM/CLI/Web/桌面) │ │ │ │ │ │ │ └────────────────────────────────────────┘ │ │ ↕ API 调用 │ │ ┌────────────────────────────────────────┐ │ │ │ Model 层 │ │ │ │ │ │ │ │ Claude / GPT / Gemini / GLM / Kimi │ │ │ │ (纯推理引擎, 无状态, 无记忆) │ │ │ │ │ │ │ └────────────────────────────────────────┘ │ └──────────────────────────────────────────────┘

关键观察：模型是无状态的、没有记忆的、不关心你文件系统的。所有这些"让它有用"的事情，都是 Harness 在做。

二、术语地图：同一个概念，N 种叫法

虽然概念趋同，但不同圈子用的词不一样——这也是它还没成为"共识"的标志之一：

来源 术语 侧重点
SWE-bench / SWE-agent Harness ACI (Agent-Computer Interface)、工具循环
Anthropic "Building Effective Agents" Cognitive Architecture 推理模式、工具编排、记忆
LangChain / LangGraph Scaffolding 状态图、节点编排
产品圈（通用） Agent Loop / Orchestration 工具调用循环、任务分发
OpenAI / 学术论文 Scaffold / Framework 提示工程 + 工具组合
OpenClaw 生态 Skills + Gateway + Memory 技能体系 + 渠道 + 记忆持久化
Hermes Agent Agent Runtime + Skills 自进化技能 + 持久化记忆

Harness 在 eval/SWE-bench 社区是准标准词，但走出这个圈子，你跟产品经理说"harness"，他大概率想到安全带。

三、为什么 Harness 正在成为核心差异化

一个简单的逻辑链条：

  1. 模型趋同 — GPT-5、Claude Opus 4、Gemini 2.5 在大多数任务上差距已经不大。用户随时可以切换 API。
  2. 因此差异下沉 — 当模型是可替换的商品时，包裹模型的工程层成了产品壁垒。
  3. Harness 越来越厚 — 从最初"prompt + function call"，到现在 Memory、Planning、Sub-agents、Permissions、Multi-channel、Sandboxing… 工程复杂度指数级上升。
  4. 因此 Harness 是产品 — Claude Code 的体验好，不是因为 Claude 模型好（你也能用同样的 API），而是因为 Anthropic 的 harness（CLAUDE.md、工具权限、文件操作、上下文管理）做得好。

📦 Claude Code

Harness = CLAUDE.md 上下文协议 + 文件系统读写权限 + diff 预览/审批 + Bash 沙箱 + 流式工具调用摘要。模型是 Claude，但换一个模型进去也能跑。

🐙 OpenClaw

Harness = Gateway + Skills + Cron + Sessions + Memory(MEMORY.md + memory_search) + 多渠道(飞书/企微/Discord) + ACP + Sandbox。5300+ tests 的 agent-memory-graph 本身就是 harness 的一部分。

🦅 Hermes

Harness = 自进化技能引擎 + isolated subagents + 5种沙箱后端 + Electron 桌面 UI + 持久化记忆。主打"完成任务后自动生成技能"。

✨ Cursor

Harness = 代码库索引 + @mention 上下文 + multi-file edit + LSP 集成 + Tab 补全。IDE-native 的 harness 设计。

四、Harness 的七层架构

如果拆开一个成熟 Agent 的 harness，它大致包含这七层：

Layer 7 ┌─ Channels ─────────────────────────────────┐ │ CLI / Web / Desktop / IM(飞书/TG/Discord) │ │ 交付层: Agent 结果怎么触达用户 │ Layer 6 ├─ Security ──────────────────────────────────┤ │ Permissions / Approval / Sandbox / 审计 │ │ 安全层: Agent 能做什么、不能做什么 │ Layer 5 ├─ Orchestration ─────────────────────────────┤ │ Sub-agents / Pipelines / State Machine │ │ 编排层: 多 Agent 协作、任务拆分 │ Layer 4 ├─ Memory ────────────────────────────────────┤ │ Short-term / Long-term / Working / Episodic │ │ 记忆层: 跨会话持久化、上下文检索 │ Layer 3 ├─ Tools ─────────────────────────────────────┤ │ File I/O / Shell / Browser / API / MCP │ │ 工具层: Agent 操作世界的手 │ Layer 2 ├─ Agent Loop ────────────────────────────────┤ │ Perceive → Reason → Act → Observe → Repeat │ │ 循环层: 核心推理-行动回路 │ Layer 1 ├─ Context Engineering ───────────────────────┤ │ Prompt 模板 / System Prompt / 动态上下文注入 │ │ 上下文层: 给模型什么信息 │ └─────────────────────────────────────────────┘ ↕ Model API ┌─ Model ─────────────────────────────────────┐ │ Claude / GPT / GLM / Kimi (无状态推理) │ └─────────────────────────────────────────────┘

每一层都可以独立设计和优化。大多数"Agent 产品"的差异，就是这七层中某几层的不同取舍。

产品 最强层 最弱层 差异化所在
Claude Code Layer 1+3 (Context + Tools) Layer 4+7 (Memory + Channels) CLAUDE.md 协议 + 文件操作体验
OpenClaw Layer 4+7 (Memory + Channels) Layer 6 (Security 仍在建设) 多渠道 + 深度记忆系统 + 飞书/企微
Hermes Layer 5+7 (Orchestration + Desktop) Layer 4 (Memory 较简单) 自进化技能 + Electron 桌面
Cursor Layer 1+3 (Context + Tools) Layer 5+7 (无多Agent/IM) 代码库索引 + IDE 原生体验
AutoClaw Layer 7 (Channels: 飞书) Layer 4+5 (继承OpenClaw但无增强) 开箱即用 + 预置技能
GLM-PC Layer 3 (Tools: CogAgent) Layer 4+5 (Memory/编排弱) 屏幕视觉操控（独占赛道）

五、Harness Engineering 的发展时间线

2023.03
ReAct 论文 — "Reason + Act" 循环

最早的 Agent Loop 雏形。Thought → Action → Observation → Repeat。Harness 还很简单：一个 prompt + 一个工具列表。

2023.06
LangChain 爆发 — Scaffolding 概念普及

"Chain" 本质就是最简单的 harness。但此时大家还在探索，产物大多是不稳定的"prompt 链"。

2023.10
AutoGPT / BabyAGI — 自主循环的尝试与失败

第一批"完全自主"Agent。失败的原因恰好说明 harness 有多重要：没有好的记忆、权限、规划，模型再强也会陷入循环。

2024.03
SWE-bench / SWE-agent — "Harness" 成为标准术语

正式把"harness"作为研究对象。发现：同样的模型，不同 harness 在 SWE-bench 上差距 3 倍以上。Harness 第一次被量化。

2024.10
Anthropic "Building Effective Agents" — Cognitive Architecture

虽然没有用"harness"这个词，但 Anthropic 把 prompt chaining、routing、tool use、orchestration 统称为"cognitive architecture"。概念趋同。

2025.01
Claude Code / Cursor — Harness 即产品

Claude Code 发布。同样的 Claude 模型，但 harness（CLAUDE.md、权限、文件操作）让体验完全不同。Cursor 也在这一年爆发。

2025.06
MCP 协议普及 — 工具层标准化

Model Context Protocol 成为工具接入的事实标准。Harness 的 Layer 3（Tools）开始标准化。

2026.01
Harness Engineering 成为显学

"Harness Engineering" 作为正式术语开始在技术博客、会议中出现。包含但不限于：Context Engineering、Tool Design、Memory Architecture、Agent Coordination、Safety Bounds。

2026.06
桌面 Agent 混战 — Harness 的全面对决

OpenClaw / AutoClaw / Hermes / Kimi Work / GLM-PC 五款产品同台。模型层差异缩小（GLM-5、Kimi K2.6、Claude 4 各有所长），竞争焦点完全在 harness 设计。

六、"模型吞并 Harness"的逆向趋势

⚠️ 模型正在"吃掉" Harness 的部分功能

这是 Harness 独立性面临的最大挑战：

原属 Harness 的能力 被模型吸收的程度 趋势
Function Calling / Tool Use 已吸收 主流模型原生支持
Structured Output (JSON) 已吸收 原生 structured output
多轮上下文管理 部分吸收 128K-2M 上下文窗口
简单规划/推理 部分吸收 Extended thinking / reasoning models
代码执行 部分吸收 Code interpreter 内置
屏幕视觉理解 部分吸收 Anthropic computer use、GLM CogAgent
记忆持久化 未吸收 仍需外部存储（MEMORY.md / 向量DB）
权限控制 未吸收 不应被塞进通用模型
多 Agent 编排 未吸收 架构层问题
渠道集成 未吸收 飞书/TG/Discord 不可能进模型
Evals/质量门控 未吸收 需要外部验证回路

结论：模型在吃掉 Layer 1-2（上下文 + 简单循环），但 Layer 4-7（记忆、编排、安全、渠道）仍然是 harness 的核心领地，且不太可能被模型吸收。

七、"行业共识"还是"强势主流观点"？

回到最初的问题："Agent = Model + Harness" 是行业共识了吗？

判断：强势主流观点（~80% 认同），但尚未尘埃落定

已经形成共识的部分：

  • ✅ Agent 可以分解为"模型 + 外层工程"——几乎无人反对
  • ✅ 在模型趋同的背景下，harness 是产品差异化的核心——多数人认同
  • ✅ SWE-bench 证明了 harness 对性能的影响可量化——学术圈公认

仍有争议的部分：

  • ❓ 术语未统一（harness / scaffold / cognitive architecture / framework）
  • ❓ 模型会持续"吸收"harness 的哪些功能？边界在哪？
  • ❓ Harness 最终是"产品壁垒"还是"会被标准化的基础设施"？
  • ❓ 模型厂商（Anthropic、OpenAI）自己做 harness（Claude Code、GPT Apps），第三方 harness 厂商的生存空间有多大？

八、对从业者的启示

1. 如果你在做 Agent 产品

你的核心竞争力在 harness 的某几层。选好你最擅长的 2-3 层，做到极致。不要试图七层全做。

2. 如果你在选 Agent 工具

别看模型——今天 Claude 最强，明天可能 GPT-6。看 harness：记忆系统、权限模型、渠道集成、可扩展性。这些才是长期资产。

3. 如果你在研究 AI 趋势

关注"harness 侵蚀线"——模型每吸收一个 harness 能力（比如 function calling → native tool use），第三方 harness 就要向上迁移一层。追踪这条线的移动方向，就是追踪 Agent 技术的演进方向。

4. 一句话表述（适合写进文档）

"业界普遍接受 Agent ≈ 模型 + Harness 的分解。在底层模型趋同的背景下，Harness（编排、工具、记忆、权限、规划与评估）正日益被视为 Agent 真正的差异化与设计所在——尽管关于 Harness 的边界、以及模型是否会持续吸收其功能，仍存在开放讨论。"
