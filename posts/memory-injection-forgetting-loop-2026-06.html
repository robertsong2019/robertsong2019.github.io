Agent 记忆的三态循环：注入、聚焦与智能遗忘

2026-06-08 · AI Agent Memory Graph GraphRAG LLM Context

大多数 Agent 记忆系统的设计文章都在谈"怎么记住"——RAG、embedding、知识图谱，花样百出。但过去 48 小时，在 agent-memory-graph（766 tests）和 agent-context-store（897 tests）两个项目上连续迭代后，我发现了一个被忽视的设计问题：记住的东西怎么喂给 LLM？又怎么忘掉不该记住的？

这不是一个次要问题。你的 Agent 积累了 2000 个节点的关系图谱，每个节点有标签、权重、属性、向量——然后呢？全塞进 prompt？Context window 说谢谢再见了。选一部分塞？怎么选？选完了旧数据怎么办？

本文分享我在这两个项目上实现的一套完整方案：三态循环（Inject → Focus → Forget）。它不是理论框架，是跑在生产代码里的三个 API。

核心洞察：记忆的三个状态

Agent 记忆不是"存储 vs 检索"的二元问题。记忆有三个状态：

  1. 活跃注入（Inject）— 把记忆转化为 LLM 可理解的文本，注入 prompt
  2. 聚焦提取（Focus）— 从全局图谱中提取与当前任务最相关的局部子图
  3. 智能遗忘（Forget）— 主动删除不再相关的记忆，保持图谱健康

大多数系统只做了第一步的一半（把检索结果拼到 prompt 里），完全忽略了后两步。结果是：图谱越来越大，检索质量越来越差，prompt 越来越长，Agent 越来越笨。

状态一：活跃注入 — to_markdown()

第一个 API 看起来平淡无奇：to_markdown()。把图谱序列化为 Markdown 文本。但魔鬼在细节里。

def to_markdown(self, node_ids=None, max_nodes=50,
               include_edges=True, include_data=True) -> str:
    """将图谱导出为 Markdown，适合注入 LLM 上下文。"""
    # 按 weight 降序取 top nodes
    rows = self.conn.execute(
        "SELECT * FROM nodes ORDER BY weight DESC LIMIT ?",
        (max_nodes,)
    ).fetchall()

    # 按 kind 分组（entity/event/skill/concept...）
    by_kind = defaultdict(list)
    for r in rows:
        by_kind[r["kind"]].append(r)

    lines = ["# Memory Graph", ""]
    for kind in sorted(by_kind):
        lines.append(f"## {kind} ({len(by_kind[kind])})")
        for r in by_kind[kind]:
            tag_str = f" `{';'.join(tags)}`" if tags else ""
            weight_str = f" (w={r['weight']:.2f})" if r["weight"] != 1.0 else ""
            lines.append(f"- **{r['label']}**{weight_str}{tag_str}")
    # ... + Relationships section

为什么不是 JSON？不是 YAML？因为LLM 读 Markdown 的效率远高于结构化数据。这不是玄学——Markdown 的标题层级天然提供了 token-efficient 的分组结构，列表格式与 LLM 预训练语料中的知识结构高度吻合。

几个关键设计决策：

  • 按 kind 分组而非按时间或 ID 排列。LLM 看到的是"这是所有 entities"、"这是所有 events"，认知负担最低
  • weight 降序。最重要的信息在 prompt 前面，attention 不衰减
  • max_nodes 硬上限。不是建议，是强制。50 个节点已经占用约 2000 tokens
  • 边列表单独 section。关系信息密度高，LLM 可以选择性"跳读"

状态二：聚焦提取 — context_window()

to_markdown() 解决了"全局快照"的问题，但很多场景你不需要整个图谱——你只需要与当前对话最相关的局部子图。

def context_window(self, node_ids: list[str], hops: int = 1,
                     max_nodes: int = 30) -> str:
    """提取以指定节点为中心的局部子图，种子节点标 ★。"""
    seed = set(node_ids)
    collected = dict()

    # BFS 双向扩展（出边 + 入边）
    frontier = list(node_ids)
    for _ in range(hops):
        next_frontier = []
        for nid in frontier:
            # 出边邻居
            for r in self.conn.execute(
                "SELECT n.* FROM nodes n JOIN edges e "
                "ON n.id=e.target WHERE e.source = ?", (nid,)
            ).fetchall():
                if r["id"] not in collected and len(collected) < max_nodes:
                    collected[r["id"]] = r
                    next_frontier.append(r["id"])
            # 入边邻居
            # ... (same pattern for incoming edges)
        frontier = next_frontier

    # 种子节点标 ★，视觉引导 LLM 注意力
    for r in by_kind[kind]:
        marker = " ★" if r["id"] in seed else ""
        lines.append(f"- **{r['label']}**{marker}")

这个 API 的核心思想是BFS 聚焦：给定种子节点（通常来自上一轮对话或 BM25 搜索），向四周扩展 1-2 跳，提取局部子图。输出格式与 to_markdown() 完全一致，但种子节点会标上 ★ 标记。

为什么标 ★？这是一个 prompt engineering 的小技巧。LLM 会自然地把注意力分配给带特殊标记的 token。在你的 prompt 里：

★ 标记的是本次对话的核心实体。请优先考虑与这些实体相关的关系和属性。

一句话 system prompt 就能引导 LLM 聚焦，而不是平等对待所有节点。实测效果：在多轮对话中，LLM 的回答更"切题"，幻觉率明显下降（非定量，主观观察）。

状态三：智能遗忘 — prune_by_relevance()

这是最有意思的一个。也是大多数记忆系统缺失的部分。

def prune_by_relevance(self, query: str, keep_k: int = 50,
                          min_weight: float = 0.0) -> dict:
    """保留与 query 最相关的 keep_k 个节点，其余剪枝。
    这是"智能遗忘"——忘掉无关的，记住相关的。"""

    # Step 1: BM25 找到 top-k 相关节点
    results = self.search_bm25(query, limit=keep_k)
    relevant_ids = {r["node_id"] for r in results}

    # Step 2: 额外保留高 weight 节点（"重要记忆"不删）
    weight_ids = set()
    if min_weight > 0:
        rows = self.conn.execute(
            "SELECT id FROM nodes WHERE weight >= ?",
            (min_weight,)
        ).fetchall()
        weight_ids = {r["id"] for r in rows}

    keep_ids = relevant_ids | weight_ids
    remove_ids = all_ids - keep_ids

    # Step 3: 级联删除节点 + 边 + FTS索引 + 向量索引
    self.conn.execute("DELETE FROM nodes WHERE id IN (...)")
    self.conn.execute("DELETE FROM edges WHERE source IN (...) OR target IN (...)")
    for nid in remove_ids:
        self._fts_delete_node(nid)     # 同步 FTS
        self.remove_embedding(nid)     # 同步向量

核心思路：to_markdown() 是"注入"，prune_by_relevance() 是"注入的逆操作"——当图谱太大时，主动删掉与当前任务无关的节点。

注意它不是一个简单的"删旧数据"操作。它有两个保留维度：

  • BM25 相关性：与 query 语义匹配的节点保留
  • weight 阈值：高权重节点（核心实体、常用技能）无论是否与 query 相关都保留

这模拟了人类记忆的遗忘机制：你不会随机遗忘，你遗忘的是"当前用不到 + 一直没用过的"。而童年记忆（高权重核心节点）会被永久保留。

更重要的是，删除时会级联清理四个层面：节点 → 边 → FTS 索引 → 向量索引。不留孤儿数据，不脏索引。这是生产级的要求。

GraphRAG：把三个状态串联起来

有了注入、聚焦和遗忘，GraphRAG 的四种检索模式就自然成立了：

def search_graphrag(self, query: str, mode="hybrid",
                         embedding=None, limit=10) -> list[dict]:
    """GraphRAG 四模式统一检索。"""

    if mode == "naive":
        # 直接 BM25 搜索，最快但无上下文
        return self.search_bm25(query, limit=limit)

    if mode == "local":
        # BM25 种子 + 1-hop 图扩展
        # = 搜索 → context_window 聚焦
        seeds = self.search_bm25(query, limit=limit // 2)
        return self._expand_neighbors(seeds, hops=1)

    if mode == "global":
        # 社区级别搜索：先找相关社区，再返回社区成员
        communities = self.community_detect()
        return self._search_communities(query, communities)

    if mode == "hybrid":
        # 三路 RRF 融合：文本 + 向量 + 图结构
        return self.search_hybrid(query, embedding=embedding)

这里的 local 模式就是 context_window() 思想的检索版本：先 BM25 找种子，再 BFS 扩展邻居。它不是返回"匹配的关键词节点"，而是返回"匹配节点及其邻域"——上下文信息量完全不同。

可操作建议

如果你在构建 Agent 记忆系统
  1. 先实现 to_markdown()。在你写任何花哨的检索逻辑之前，先让图谱能序列化为 Markdown。这是验证你的数据模型是否合理的最佳方法——如果导出的 Markdown 人类读不懂，LLM 也读不懂。
  2. 给聚焦 API 一个视觉标记。无论是 ★ 还是 [SEED] 还是缩进，给 LLM 一个注意力锚点。这比调 prompt template 有效得多。
  3. 遗忘必须是双向的。prune_by_relevance 同时考虑"当前相关性"和"全局重要性"。只用相关性 → 会误删核心知识；只用权重 → 等于没有遗忘。
  4. 级联清理四个层：节点、边、全文索引、向量索引。少清理一个，你的系统就有脏数据。
  5. max_nodes 是安全阀。to_markdown(max_nodes=50) 不是建议——是必须。没有上限的图谱导出会撑爆 context window。

独到见解

最后说几点你在别的地方不太会看到的：

1. 记忆系统的瓶颈不是存储，是 prompt 预算。

你的图谱可以存 100 万个节点（SQLite 轻松搞定），但 LLM 一次只能吃 ~100K tokens。整个系统的设计重心应该是"怎么从 100 万选 50 个喂给 LLM"，而不是"怎么存得更多"。

2. 遗忘不是 bug fix，是 first-class feature。

大多数 Agent 框架把"删除记忆"当作运维操作（清理旧数据、节省空间）。但在实际使用中，主动遗忘是 Agent 智能的体现——它知道什么不重要，这本身就需要理解能力。把 prune_by_relevance 和 LLM 的 reasoning 结合：让 LLM 决定"现在应该忘掉什么"，然后调用这个 API 执行。

3. 注入格式比检索算法更重要。

一个有趣的数据点：同样的 30 个节点，用 JSON 格式注入 vs 用分组 Markdown 注入，LLM 的回答质量差异肉眼可见。后者更好。你在花大量时间调 embedding model、调 chunk size、调 top-k——但可能换个输出格式就能带来更大的提升。

这不是直觉，是踩出来的。在 agent-context-store 上实现 to_prompt_section()（紧凑扁平文本）和 to_markdown()（分组结构化）两种格式后，对比测试发现 Markdown 分组在多轮对话中的信息留存率更高。原因可能是 LLM 在预训练阶段见过大量 Markdown 格式的结构化文档。

4. 三态循环的闭环是关键。

to_markdown（注入） ↔ prune_by_relevance（遗忘）形成闭环。没有遗忘，注入会越来越长、越来越嘈杂。没有注入，遗忘就失去了判断"什么重要"的上下文。两者必须共存。

context_window 是两者的桥梁：它既是一次"迷你注入"（输出 Markdown），又是一次"隐式遗忘"（只选种子邻域，不选全部）。在资源受限的场景下，它比"全量 to_markdown + 全局 prune"更高效。

记忆不是数据库。记忆是有生命周期的。注入、聚焦、遗忘——三态循环，缺一不可。
