Agent 信任传播引擎：
从扁平评分到图结构传播

2026-06-05 · 罗嵩 · 基于 autoresearch 方法论

过去两周，我的 a2a-trust-prototype 经历了从 29 个测试到 59 个测试的跨越。但比数字更重要的是一个结构性问题：当 Agent A 委托 Agent B，Agent B 又调用 Agent C 时，整条链路的信任度到底怎么算？

这不是学术讨论。在真正的多 Agent 系统中，委托链无处不在——编排器调用工具 Agent，工具 Agent 调用外部 API，API 又触发下游 Agent。如果你只有扁平的 per-agent 信任分数，你在每一跳都在做一个盲目的赌博。

这篇文章记录了我从"扁平信任"到"图信任传播"的完整思考路径，包含 4 个算法实现和 22 个测试验证的全部代码。

扁平信任的致命缺陷

先看现有系统是怎么做的。大部分 Agent 框架——包括我自己的 TrustEngine V1——用的是一个简单模型：

// 扁平信任模型 (TrustEngine V1)
trustScore[agentId][skill] = computeScore(interactions)

// 委托决策：只看目标 Agent 的个人分数
if (trustScore[targetAgent][skill] > threshold) {
    await delegate(targetAgent, task)
}

这里有一个隐藏假设：Agent 的可信度是独立的，与调用者无关。但现实完全不是这样。同一个 Agent，对高信任调用者可能表现优秀，对陌生调用者可能敷衍了事。更重要的是，当委托链变长时，扁平模型完全忽略了链路的累积风险。

具体来说，假设一条 5 跳的委托链，每跳的可靠性是 0.9：

// 扁平模型：只检查最终目标的分数
finalTarget.score // → 0.9 ✅ 看起来没问题

// 实际链路可靠性：几何平均
0.9 ** 5 // → 0.59 ❌ 实际只有 59%

这就是 Microsoft AgentGraph 团队在 autogen 讨论 #7476 中提出的核心观点："多 Agent 系统的信任强度取决于最薄弱的组件"。

四种信任传播算法

我设计了一个 TrustGraph 引擎，实现了四种互补的信任传播算法。每种解决不同的问题，组合起来覆盖完整的需求矩阵。

算法 1：组合信任（Compositional Trust）

解决最直接的问题：委托链的累积信任度。用几何平均作为聚合函数——这意味着一个薄弱环节就能拉低整条链，这恰好是我们想要的安全特性。

/**
 * 计算委托链的信任度
 * @param {string[]} chain - 有序的 Agent ID 列表: [A, B, C, D]
 * @returns {trust: number, valid: boolean, bottleneck: string}
 */
computeChainTrust(chain) {
    if (chain.length > this.maxDepth) {
        return { trust: 0, valid: false, bottleneck: 'max_depth_exceeded' }
    }

    let product = 1
    let minTrust = Infinity
    let bottleneck = ''

    for (let i = 0; i < chain.length - 1; i++) {
        const edge = this.getEdge(chain[i], chain[i + 1])
        if (!edge || edge.polarity < 0) {
            return { trust: 0, valid: false, bottleneck: chain[i + 1] }
        }
        const effective = edge.weight * (this.getNode(chain[i + 1]).score / 100)
        product *= effective
        if (effective < minTrust) {
            minTrust = effective
            bottleneck = chain[i + 1]
        }
    }

    // 几何平均：惩罚长链和弱链接
    return {
        trust: Math.pow(product, 1 / (chain.length - 1)) * 100,
        valid: true,
        bottleneck
    }
}

关键设计决策：

  • 几何平均而非算术平均——几何平均对极小值更敏感，一个 0.3 的薄弱环节会把整条链拉到 0.05 附近，而算术平均可能还能维持在 0.7。在安全场景中，我们宁愿误报也不愿漏报。
  • 最大深度硬约束——超过 5 跳直接返回 0。不是因为计算不了，而是因为在实际 Agent 系统中，超过 5 跳的委托链几乎一定意味着架构设计有问题。
  • 瓶颈节点报告——返回 bottleneck 字段，告诉你链条上最薄弱的环节是谁。这对调试和链路优化至关重要。

算法 2：主观传递信任（Subjective Transitive Trust）

这是整个系统中我认为最有价值、也最容易被误解的算法。

EigenTrust（后面会讲）产生一个全局信任向量——每个节点有一个统一的信任分数。这在 P2P 文件共享中有效，因为文件的质量对所有人是一样的。但 Agent 系统更像社交网络：同一个 Agent 对不同调用者的表现可能完全不同。

主观信任的核心公式：从节点 i 的视角看节点 j：

// Subjective trust: observer 对 target 的综合评价
// = 直接交互 + 间接推荐（通过中间人传递）

T(i→j) = α × direct(i,j) + (1-α) × Σ_k [ T(i→k) × T(k→j) ] / Σ_k T(i→k)

// 其中 k 是 i 信任的中间人，α 是直接经验的权重

用 BFS 实现时，从观察者出发逐层扩展，每跳乘以衰减因子：

subjectiveTrust(observer, target) {
    const queue = [{ id: observer, accumulatedTrust: 1.0, depth: 0 }]
    let totalTrust = 0, totalWeight = 0

    while (queue.length > 0) {
        const { id, accumulatedTrust, depth } = queue.shift()
        if (depth >= this.maxDepth) continue

        for (const { id: neighborId, weight } of this.getTrustedNeighbors(id)) {
            const hopTrust = accumulatedTrust * weight * this.decayFactor

            if (neighborId === target) {
                totalTrust += hopTrust * (this.getNode(target).score / 100) * 100
                totalWeight += hopTrust
            }

            queue.push({ id: neighborId, accumulatedTrust: hopTrust, depth: depth + 1 })
        }
    }

    // 直接信任和间接信任的加权融合
    const directEdge = this.getEdge(observer, target)
    if (directEdge?.polarity > 0) {
        return directTrust * 0.6 + transitive * 0.4  // 直接经验优先
    }
    return totalWeight > 0 ? totalTrust / totalWeight : 50 // 无路径返回中性
}

💡 关键洞察

主观信任天然具备 Sybil 抗性：攻击者创建再多假节点也无法提升自己的信任分数，因为信任只能从别人那里获得，不能自封。这比 EigenTrust 的预信任锚定策略更优雅。

算法 3：EigenTrust 全局收敛

虽然主观信任更适合 Agent 系统，但全局信任仍有价值——它提供了一个"宏观信任地图"，适合做系统级监控和异常检测。

EigenTrust 是 PageRank 的信任版本。核心迭代公式：

// t^(k+1) = (1-α) C^T t^(k) + α p
// C = 归一化信任矩阵, p = 预信任向量, α = teleport 概率

computeGlobalTrust(preTrusted = []) {
    // 1. 构建归一化信任矩阵 C
    // 2. 初始化预信任向量 p（锚定已知可信节点）
    // 3. 幂迭代直到收敛

    for (let iter = 0; iter < 100; iter++) {
        const tNew = new Array(n).fill(0)
        let maxDelta = 0

        for (let j = 0; j < n; j++) {
            let sum = 0
            for (let i = 0; i < n; i++) {
                sum += C[i][j] * t[i]  // 转置矩阵 × 向量
            }
            tNew[j] = (1 - this.teleport) * sum + this.teleport * p[j]
            maxDelta = Math.max(maxDelta, Math.abs(tNew[j] - t[j]))
        }

        t = tNew
        if (maxDelta < this.convergenceThreshold) break  // 收敛
    }

    // 归一化到 [0..100]
    return normalize(t)
}

预信任节点（preTrusted）是 Sybil 防御的关键。少量经过验证的 "root agents" 作为信任锚点，恶意节点即使建立大量假身份，也无法通过迭代获得高全局信任——因为假身份之间互相投票的权重远低于真实锚点的权重。

算法 4：信任感知子图提取

这是连接信任引擎和记忆系统的桥梁。Schneider 在 2026 年的 memory poisoning 研究中指出：每条记忆在写入时应该标记来源和信任分数，检索时信任度高的记忆优先填充上下文窗口。

// 从观察者视角提取信任子图
// 用于 agent-memory-graph 的信任感知检索
trustSubgraph(observer, threshold = 60) {
    const trustedNodes = []

    for (const [id] of this.nodes) {
        if (id === observer) continue
        const trust = this.subjectiveTrust(observer, id)
        if (trust >= threshold) {
            trustedNodes.push(id)
        }
    }

    // 只保留高信任节点之间的边
    const trustedEdges = [...this.edges]
        .filter(([key, edge]) => {
            const [from, to] = key.split('→')
            return edge.polarity > 0
                && trustedNodes.includes(to)
                && (from === observer || trustedNodes.includes(from))
        })

    return { nodes: [observer, ...trustedNodes], edges: trustedEdges }
}

集成路径很清晰：

// agent-memory-graph 查询时过滤低信任来源
const subgraph = trustGraph.trustSubgraph(callerAgent, 60)
const memories = memoryGraph.query({
    filter: node => subgraph.nodes.includes(node.agentId),
    orderBy: node => relevance(node) * trustWeight(node.trustScore)

SIGMA 带来的启示：负信任 ≠ 低信任

实现过程中，SIGMA 论文（arXiv:2605.19418）给了我最深的认知冲击。它指出了一个大多数系统都存在的盲区：不信任不等于负信任。

状态分数模型实际含义传播行为
主动冲突 (polarity = -1)score ≈ 20"我和你对着干"应主动抑制传播
缺乏信息 (polarity = 0)score ≈ 50"我不认识你"不参与传播
积极信任 (polarity = +1)score ≈ 80"我们合作愉快"主动促进传播

在传统的纯分数模型中，"两个 Agent 互相不信任"（polarity=-1, score=20）和"两个 Agent 互不相识"（polarity=0, score=50）看起来是两个不同的分数。但当我们做信任传播时，这两者的行为应该截然不同：

  • 负信任应该主动减去邻居的信任——"我不信任 X，所以如果你信任 X，我对你的信任也要打折"
  • 中性信任应该不参与传播——"我不认识 X，这不影响我对你的看法"

SIGMA 的符号消息传递公式把这一点变成了数学：

h_i^(l+1) = σ( Σ_{j∈N+(i)} w_ij · h_j^(l)  // 正邻居：聚合
                - Σ_{j∈N-(i)} w_ij · h_j^(l) ) // 负邻居：减去

正邻居的特征被聚合（加），负邻居的特征被减去（减）。中性节点不参与。这比纯分数模型的"低信任 = 传播弱"精确得多。

三明治架构：信任的三个层次

综合这些研究，我现在认为生产级 Agent 系统的信任架构应该是三层"三明治"结构：

层回答的问题技术手段本文算法
L1 密码学层"你是谁？"ES256 签名、JCS 规范化已有（a2a-trust-prototype）
L2 图传播层"别人怎么看你？"EigenTrust、主观信任、组合信任TrustGraph（本文）
L3 行为层"你实际表现如何？"交互成功率、时间衰减、技能细分已有（TrustEngine V1）

这三层不是替代关系，而是互补：

  • L1 防止身份伪造——你无法冒充别人
  • L2 防止信任操控——你无法通过 Sybil 攻击提升声誉
  • L3 防止信任过期——昨天的可靠不代表今天的可靠

⚠️ 实践建议

不要试图用单一算法解决所有信任问题。我的实验数据表明：subjectiveTrust 应该是主要 API（80% 的调用），computeGlobalTrust 作为辅助监控（15%），computeChainTrust 用于委托验证（5%）。过度依赖全局信任会导致"Agent 的社会阶层固化"——新 Agent 永远无法获得初始信任。

可操作的建议

如果你的 Agent 系统需要信任管理，这里是从 0 到 1 的路径：

  1. 先实现扁平信任——per-agent + per-skill 的分数模型。这只需要一个 Map 就够。不要一上来就建图。
  2. 加入极性属性——区分"主动冲突"和"缺乏信息"。这是从 SIGMA 学到的最重要的教训：在 edge 上增加 polarity: +1 | -1 | 0 字段。
  3. 实现组合信任——当你有了委托链（比如编排器 → Agent → 工具），用几何平均计算链路信任度。设置最大深度硬约束。
  4. 引入主观信任——当 Agent 数量超过 ~20 个时，全局信任开始不够用。此时实现 BFS 的主观信任传播。
  5. 图信任 + 记忆——最后一步，将信任子图与记忆检索结合。这一步的价值最大，但也需要前面的基础。

这个演进路径的每一步都能独立产生价值，不需要一步到位。
