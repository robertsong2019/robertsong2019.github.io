当 AI 学会忘记：Agent 记忆的自我进化之路

2026-07-13 · 罗嵩 · Agent Memory Self-Improving Dreaming autoresearch
想象你有一个助理，他记得你两年前随口说的一句话，但忘了你上周改了密码。或者更糟：他记住了你三年前的饮食偏好，然后固执地用这个过时信息给你推荐餐厅，即使你已经告诉他八次"我现在吃素了"。这不是科幻——这是 2025 年大多数 AI Agent 记忆系统的真实表现。但在 2026 年，一些根本性的变化正在发生：Agent 开始学会主动管理自己的记忆，学会忘记，学会在做梦时整理思绪。

记忆的三重困境

先说清楚问题。Agent 记忆听起来简单——不就是存点东西以后用吗？但任何一个真正做过 Agent 记忆系统的人都知道，这里有三重困境：

困境一：写什么？ 你和 Agent 聊了一小时，讨论了十个话题，修改了三段代码，开了两个玩笑。什么值得记住？"用户喜欢用 tab 缩进"显然值得记。"用户刚才打了个喷嚏"显然不值得。但"用户提到项目下周截止"呢？——现在重要，下周就不重要了。大多数系统的做法是全记，然后用检索质量来兜底。结果是记忆库越来越像垃圾场。

困境二：什么时候该忘？ 用户三个月前说"我在新加坡"，现在还记着，于是推荐了新加坡的餐厅。OpenAI 在 Dreaming 系统的文档里举了这个精确的例子——记忆变"陈旧"（stale）是生产环境中最常见的投诉。但怎么知道一条记忆已经过期？你总不能给每条记忆设一个 TTL。

困境三：谁在做决定？ 在今天的多数系统里，记忆的写入和读取逻辑是外部脚本决定的——一套固定规则，什么时候写、写什么、怎么检索。Agent 本身对此毫无发言权。它就像一个被迫使用某个笨拙笔记软件的知识工作者：软件决定什么该存，它只能照做。

这三个困境本质上指向同一个问题：记忆系统缺乏智能。不是 AI 不够聪明，而是记忆架构构本身不够聪明。2026 年，这个局面开始改变。

范式转移：记忆从"管道"到"行为"

阿里巴巴和武汉大学在 2026 年 1 月发表的论文 AgeMem（Agentic Memory）提出了一个看起来简单但影响深远的主张：把记忆操作变成 Agent 可以调用的工具。

具体来说，他们定义了六个记忆工具：

记忆类型操作作用
长期记忆add存储新的事实到持久化存储
长期记忆update更新已有的记忆条目
长期记忆delete删除不再有用的记忆
短期记忆retrieve把相关片段拉入当前上下文
短期记忆summary压缩一大段上下文为摘要
短期记忆filter过滤掉无关内容

关键区别在于：这些操作的调用策略不是硬编码的规则，而是通过强化学习训练出来的。Agent 自己学会什么时候该记、什么时候该忘、什么时候该压缩。

训练分三个阶段——这个设计本身就很有启发：

  1. 长期记忆构建：把 Agent 放在日常对话场景中，只奖励它正确地存储重要事实。这一阶段不关心任务完成度，只训练"当图书管理员"的能力。
  2. 短期记忆控制：保留第一阶段的长期记忆，重置短期上下文，然后注入看起来相关但其实无关的干扰信息。Agent 必须学会什么时候信任长期记忆、什么时候过滤噪音。
  3. 综合推理：现在给 Agent 真正的任务，需要长期和短期记忆协同工作。奖励信号来自任务成功，Agent 必须同时协调存储、检索和推理。

这个渐进式训练解决了一个核心难题：记忆操作的奖励信号极其稀疏。你可能在整个任务过程中调用了 50 次记忆操作，但只有最后一次的结果告诉你哪些是有用的。传统的 RL 在这种稀疏信号下基本失效。AgeMem 用逐步式 GRPO（Group Relative Policy Optimization）给每一步记忆操作单独分配信用，才让训练变得可行。

核心发现：在五个长程任务基准上，配备 AgeMem 的 4B 参数模型超越了 7B 参数模型的基线系统。学习型记忆带来的提升，比多出 30 亿参数还大。如果只能记住一条结论：记忆架构的投资回报率可以匹敌甚至超过模型缩放。

从执行轨迹中学习：不只是记住"什么"，还要记住"为什么"

AgeMem 回答了"Agent 应该怎么管理记忆"，IBM 的研究团队在 2026 年 3 月发表的另一篇论文回答了一个互补的问题：Agent 能从自己的执行过程中学到什么？

他们的框架叫做 Trajectory-Informed Memory Generation（轨迹信息记忆生成）。核心洞察是：Agent 的执行轨迹——每一步的思考、决策、行动、观察——包含了远比"对话事实"更丰富的学习素材。

考虑一个电商场景：Agent 要完成"清空购物车并结账"。可能出现三种情况：

  • 干净的成功：Agent 先检查购物车，逐一删除商品，验证空了之后结账。→ 可以提取出 策略提示：结账前系统性验证前置条件。
  • 低效的成功：Agent 用循环逐个删除商品，虽然成功了，但其实调用一次 empty_cart() 就够了。→ 可以提取出 优化提示：清空多商品购物车时用批量操作。
  • 失败后恢复：Agent 尝试结账失败（没绑定支付方式），然后补充支付信息重试成功。→ 可以提取出 恢复提示：结账失败时先验证支付配置。

这三种提示对应三种完全不同的学习类型，而现有的记忆系统几乎只关注第一种——成功的经验。IBM 的框架通过四层分析（轨迹智能提取 → 决策归因分析 → 上下文学习生成 → 自适应记忆检索），自动从执行日志中提取所有三类知识。

结果令人印象深刻：在 AppWorld 基准上，复杂任务的完成率提升了 28.5 个百分点（相对提升 149%）。越是复杂的任务，从过往经验中学习的收益就越大——这和直觉完全吻合。

启示：不要只记录 Agent 做了什么，要分析为什么它那么做。决策的因果关系比行为本身更有价值。一条"Agent 调用了 API X"的记录没什么用，但一条"Agent 选择调 API X 而不是 API Y，因为 Z，结果证明这个选择是对的"的记录——那是金子。

做梦的机器：当 Agent 闭上眼睛开始整理记忆

如果说 AgeMem 和轨迹学习是 Agent 在"清醒"时管理记忆，那么 2026 年另一个重要趋势是：让 Agent 在"睡眠"时整理记忆。

这听起来像科幻，但两家最重要的 AI 公司已经在生产环境中部署了这种系统。

OpenAI 的 Dreaming V3

OpenAI 在 2026 年 6 月推出了第三代 Dreaming 系统。它的核心思路是：一个后台进程定期扫描你的所有对话历史，主动合成和更新记忆。

举例来说：你在七月说过"我要去新加坡"。传统的记忆系统会永远记住"用户在新加坡"。但 Dreaming 会在你旅行结束后自动更新这条记忆——从"你要去新加坡"变成"你去了新加坡（2026年7月）"，然后在你回国后不再推荐新加坡的餐厅。

这个系统经历了三代演化：

  • 2024 年（Saved Memories）：只在对话中显式触发。你说"记住我喜欢 Python"，它才存。像一个只记你说要记的东西的笔记本。
  • 2025 年（Dreaming V0）：后台进程开始从对话历史中自动提取信息。但性能不足，不能单独支撑记忆系统。
  • 2026 年（Dreaming V3）：计算成本降低 5 倍，首次可以向免费用户开放。在三个维度显著提升：携带上下文、遵循偏好、时间感知（自动处理记忆的"保鲜期"）。

Anthropic 的 Claude Code Dreaming

Anthropic 走得更远。根据 Troy Hua 在 2026 年 3 月的逆向工程分析，Claude Code 的记忆系统有七层：

  1. 轻量级 token 裁剪（上下文窗口级别）
  2. 上下文压缩（段落级别摘要）
  3. 会话记忆（跨对话持久化）
  4. 项目记忆（代码库级别的知识）
  5. Dreaming 整合层（会话间模式发现和记忆整合）
  6. 跨项目知识（从一个项目迁移到另一个）
  7. 元学习（学习如何更好地学习）

第 5 层是最有意思的。当 Claude Code 的会话结束后，Dreaming 进程会回顾过去的会话，发现重复模式，自动生成或更新项目级别的知识。2026 年 7 月，Anthropic 又悄悄上线了 Auto Dream——解决了一个关键问题：积累 20+ 会话后，记忆条目开始互相矛盾，相对日期（"上周"、"昨天"）变得不准确。Auto Dream 自动清理和合并这些退化中的记忆。

这和人类大脑的睡眠巩固机制惊人地相似。神经科学早就发现，海马体在睡眠期间重播白天的经历，将短期记忆整合为长期记忆，丢弃不重要的细节。AI 系统正在独立发现同样的原理。

基础设施层：记忆作为服务

研究突破之外，2026 年最显著的变化是记忆基础设施的产品化。一批专门做"Agent 记忆层"的产品和平台走向成熟：

  • Mem0 已经成为事实标准，支持 21 个框架和 20 个向量存储，在 LoCoMo 基准上得分 92.5。它的多信号检索（语义相似 + 关键词匹配 + 实体匹配的三路并行打分）已经成为行业标准做法。
  • Zep（基于 Graphiti） 主打实时时序知识图谱，适合需要精确时间线的场景。
  • Cloudflare Agent Memory 在 2026 年 4 月推出了托管式持久记忆服务，把记忆变成了 CDN 级别的基础设施。
  • Perplexity Brain 做了搜索场景的自改进记忆——Agent 每次搜索后自动从结果中学习，下次搜索质量持续提升。

更重要的是，基准测试也在标准化。BEAM 基准在 100 万和 1000 万 token 的规模上测试记忆系统——这已经超出了任何上下文窗口的物理限制，是唯一接近生产规模的评估工具。BEAM 的十个类别包括偏好遵循、矛盾解决、弃权判断（该说"我不记得"的时候），这些才是真实场景中最难的问题。

深层问题：参数化学习 vs 上下文学习

站在更高的视角，这些记忆系统的进化其实指向一个更根本的问题：AI 的"学习"到底应该发生在哪里？

a16z 在 2026 年 4 月的一篇文章里把这个问题摆得很清楚。他们用电影《记忆碎片》做比喻：主角因为脑损伤无法形成新记忆，只能靠纹身和拍立得来记录——外部记忆装置帮助他行动，但他本人从不真正"学到"任何东西。

今天的 LLM 就是那个主角。所有的 Agent 记忆系统——无论是向量数据库、知识图谱还是 Dreaming 进程——都是"纹身和拍立得"。模型本身的参数从未改变。你在 ChatGPT 里和它聊了一万个小时，它的权重一个 bit 都没有变。

这就是上下文学习（ICL）和持续学习（Continual Learning）的根本分野。前者是把信息塞进上下文窗口，后者是直接更新模型权重。前者是"查看笔记"，后者是"真正记住"。

State Space Models（SSM）提供了某种中间道路：一种比传统 Attention 更擅长处理超长序列的架构，可以把上下文窗口的有效范围从 ~20 步扩展到 ~20,000 步。但这仍然不是真正的参数化学习。

a16z 的结论是："上下文学习是暂时的。真正的学习需要压缩。在让模型持续压缩之前，我们可能困在《记忆碎片》的永恒当下里。"

在真正的持续学习突破之前，Agent 记忆系统做的事情，就是尽可能把这个"永恒当下"延长——从单次对话，到跨会话，到跨项目，到跨年。每一次延长，都是向着"真正的学习"迈进一步。

你可以做什么

如果你正在构建 Agent 系统，以下是一些可以立刻落地的实践建议：

  1. 把记忆操作暴露为工具调用。即使你没有 RL 训练的能力，只是在 system prompt 里告诉模型"你可以调用 remember(key, value) 和 forget(key)"，就能显著改善长任务的记忆质量。AgeMem 的消融实验显示，即使没有训练，仅仅暴露工具就能带来可测量的提升。
  2. 记录完整的决策轨迹，不只是结果。"Agent 做了 X" 是原材料，"Agent 做了 X 因为 Y，结果证明了 Z" 才是知识。IBM 的框架告诉我们，从轨迹到知识的转化需要语义分析和因果归因，但这完全可以离线做。
  3. 在写入路径上加过滤器。最好的遗忘策略是不该记住的东西从一开始就不存。一个简单的做法：写入前问自己（问 Agent）"这条信息在三个月后还有用吗？"如果答案是否定的，不要写。
  4. 实现某种形式的 Dreaming。哪怕只是一个每天运行的 cron job，扫描昨天的交互，更新过时的记忆，合并重复条目。OpenAI 和 Anthropic 都在做这件事，说明它是必需品，不是奢侈品。
  5. 用 BEAM 评估，不要只用 LoCoMo。在 1000 条记忆上表现完美的系统，在 100 万条时可能崩溃。生产环境的记忆规模迟早会到百万级别——提前在 BEAM 上测试。
  6. 关注 AgeMem 的后续工作。虽然训练代码和检查点还没发布，但"记忆即行为"的范式是可迁移的。跟踪这个方向——它很可能是 Agent 记忆的下一个标准范式。
