问题：AI 编程的"空转"陷阱

大多数 AI 辅助开发停在这里：你提需求，它写代码，你 review。这是 被动模式。Agent 等待指令，不会自己发现问题、提出假设、验证方案。

但真正的工程研究不是这样运作的。一个好的工程师会：

  1. 看到系统的缺口（"我们没有时间范围查询"）
  2. 设计最小方案（"加一个 search_by_time_range()"）
  3. 实现并测试
  4. 记录结果，决定保留或丢弃

问题是：怎么让 Agent 做同样的事？

核心机制：Key Development 循环

Autoresearch 的核心是一个三循环结构，每天自动触发：

Loop A: 分析项目 → 找到最大缺口
Loop B: 实现 + 测试 (agent-context-store 等)
Loop C: 实现 + 测试 (better-ralph-core 等)

每个循环产出一份结构化的实验报告：

### 🎯 Target
问题：没有时间范围查询，只能用相对时间
方案：search_by_time_range(start, end, field="created_at")

### 📊 Results
92 → 97 tests (+5, all passing)
Committed: 4d974af

### ✅ Decision: RETAIN
理由：填补真实 API 缺口，零依赖，最小实现

关键设计决策：每个实验必须独立可评估。不是"改进了整体代码质量"，而是"+5 tests, 4d974af, keep"。

experiments.tsv：不可篡改的实验账本

所有实验记录在一个 TSV 文件里，格式统一：

timestamp    commit    metric    value    status    description
2026-05-11   4d974af   tests     97/97    keep      search_dups + move: dedup via trigram + atomic rename

这不是日志，是账本。每一行都是一个可审计的决策：keep 还是 revert。30 天后回看，你能精确知道每天做了什么、为什么保留、测试覆盖如何变化。

观点：大多数 AI 编程工具的问题不是生成代码的能力，而是缺乏结构化的决策记录。没有 experiments.tsv，你只是在和 Agent 聊天。有了它，你在做工程研究。

代码示例：一个实验的最小实现

以 search_dups 为例，看看 autoresearch 的"最小实现"原则：

def search_dups(self, content: str, min_similarity: float = 0.5) -> list[tuple[str, float]]:
    """Find entries with similar content using trigram overlap."""
    def trigrams(s: str) -> set[str]:
        return {s[i:i+3] for i in range(len(s) - 2)} if len(s) >= 3 else {s}
    
    query_tri = trigrams(content.lower())
    results = []
    for key, entry in self._store.items():
        if entry.get("_expired"):
            continue
        entry_tri = trigrams(entry["content"].lower())
        if not query_tri or not entry_tri:
            continue
        score = len(query_tri & entry_tri) / len(query_tri | entry_tri)
        if score >= min_similarity:
            results.append((key, score))
    
    return sorted(results, key=lambda x: x[1], reverse=True)

15 行。零依赖。复用了已有的 trigram 方法。3 个测试覆盖核心场景。这就是 autoresearch 的品味：解决问题的最小代码，而不是展示技巧的最大代码。

批量执行：run_batch 的设计哲学

同时期的 better-ralph-core 项目（一个 PRD 驱动的自主编码器）也通过 autoresearch 演化。最有意思的添加是 run_batch：

def run_batch(self, max_iterations=10, max_consecutive_failures=3):
    """Execute up to N iterations with early-stop on systemic failure."""
    consecutive_failures = 0
    for i in range(max_iterations):
        if self.is_complete():
            break
        result = self.execute_iteration()
        if not result.success:
            consecutive_failures += 1
            if consecutive_failures >= max_consecutive_failures:
                break  # Stop: systemic failure, not individual story issue
        else:
            consecutive_failures = 0  # Reset on success

设计洞察：max_consecutive_failures 区分了"单个 story 有问题"（可以跳过继续）和"系统有问题"（应该停止）。这不是过度工程——它是从实际运行中提炼的防护措施。

深层洞察：为什么这能工作？

Autoresearch 不是"让 AI 自己写代码"——那是 vibe coding。区别在于三个约束：

1. 结构化的缺口发现。 Agent 不是随机写功能，它分析现有 API 的使用模式，找到真正缺失的部分。每个实验的 "Problem" 部分是它自己写的——它需要解释为什么这个功能值得添加。

2. 不可协商的测试覆盖。 每次添加必须附带测试，且全部通过。experiments.tsv 记录的是测试数而不是代码行数，因为测试数是质量的代理指标。

3. 保留/丢弃的显式决策。 每个实验以 keep 或 revert 结尾，附带理由。这阻止了功能膨胀——如果 Agent 不能说服自己一个功能值得保留，它就不会保留。

我的观点：AI Agent 的价值不在于它能写多少代码，而在于它做决策的质量。Autoresearch 的本质是一个决策框架，代码只是副产品。

可操作建议

  • 从 experiments.tsv 开始。 不需要复杂的工具，一个 TSV 文件就够了。每行：时间、提交哈希、指标、值、状态、描述。
  • 强制"Problem → Solution → Result → Decision"格式。 这四段式让每个实验可评估。缺少任何一段，就不是一个完整的实验。
  • 用测试数做北极星指标。 代码行数可以膨胀，测试数不能。它是唯一不会说谎的质量指标。
  • 每天最多 3 个实验。 速度不是目的，可审计性才是。超过 3 个实验意味着每个都不够深入。
  • 让 Agent 自己写 Problem。 如果 Agent 不能清晰描述"为什么需要这个功能"，这个功能大概率不需要。
