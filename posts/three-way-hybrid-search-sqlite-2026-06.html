三路混合搜索：在单个 SQLite 里融合 BM25 + Vector + Graph

2026-06-07 · 罗嵩 · 混合检索 SQLite Agent Memory 图算法

大多数 Agent 记忆库的搜索只做两件事：关键词匹配（BM25）和语义相似（Vector Embedding）。这很好，但漏掉了一个关键维度——文档之间的结构关系。上周在 agent-memory-graph 项目中，我把图算法作为第三条检索通道加进来，用 Reciprocal Rank Fusion (RRF) 做三路融合。结果有些出乎意料：一些完全不含搜索词的文档，通过图连接被准确召回，排名提升了 2-3 位。

这篇文章拆解这个架构：为什么需要三路、怎么在单个 SQLite 里实现、以及几个踩过的坑。

一、为什么两路不够

先看一个具体场景。Agent 记忆库里有这些节点：

  • 节点 A：「如何配置 LangGraph Supervisor」
  • 节点 B：「LangGraph 架构概述」（与 A 有边连接）
  • 节点 C：「PageRank 实现笔记」（与 A 无连接，但与 D 连接）
  • 节点 D：「图算法入门」（与 C 有边连接）

当用户查询「Supervisor 模式」时：

检索方式能找到什么漏掉什么
BM25节点 A（含 "Supervisor"）B、C、D 都不含这个词
Vector节点 A + 可能有语义关联的 BC 和 D 语义距离太远
Graph从 A 出发能到达 B不评估内容相关性

BM25 和 Vector 都是文档级独立性评分——它们不知道文档之间是否有关系。图信号利用的是文档间的结构性关联，这是纯内容分析永远无法捕获的维度。

但反过来，图信号也不能独立工作——高连接度的节点不一定相关。三路的价值在于交叉验证：当三种信号一致时置信度高；当它们分歧时，说明存在值得探索的隐含关联。

二、架构：一个 SQLite，三个引擎

传统三路融合需要三个系统：Elasticsearch（BM25）、Pinecone/Qdrant（Vector）、Neo4j（Graph）。我的方案是全部塞进一个 SQLite：

通道SQLite 技术信号类型擅长捕获
BM25 (Sparse)FTS5 内建扩展词频 + IDF精确关键词、代码标识符、错误码
Vector (Dense)sqlite-vec (vec0)语义嵌入相似度概念匹配、跨语言、同义改写
Graph (Relational)Recursive CTE / 应用层 BFS结构重要性 + 邻域权威节点、知识枢纽、隐含关联

这带来的好处不仅是省了三个服务的运维：

  1. ACID 事务——三路检索看到一致的数据快照，不会出现 FTS5 刚写入但向量还没索引的窗口期
  2. 零网络延迟——全在进程内，查询延迟 < 1ms
  3. 单文件部署——整个 Agent 记忆是一个 .db 文件，拷贝即迁移

三、核心代码：三路 RRF 融合

融合用经典的 Reciprocal Rank Fusion，但扩展为三路加权：

const RRF_K = 60; // 标准 RRF 常数

function rrf(rankings, weights = [1.0, 1.0, 0.5]) {
  const scores = new Map();
  for (let i = 0; i < rankings.length; i++) {
    const ranking = rankings[i];
    const w = weights[i];
    for (const [rowid, rank] of ranking) {
      const contribution = w * (1 / (RRF_K + rank));
      scores.set(rowid, (scores.get(rowid) || 0) + contribution);
    }
  }
  return [...scores.entries()]
    .sort((a, b) => b[1] - a[1]);
}

为什么图权重是 0.5 而不是 1.0？因为图信号的作用是「打破平局」和「发现隐含关联」，而不是替代内容相关性。如果图权重过高，会把高连接但低相关的节点过度提升——这叫「枢纽污染」。实测 0.3-0.5 是最佳区间。

三路查询的实现

function searchHybrid(queryText, queryVec) {
  // 通道 1: BM25（注意 FTS5 的隐含 AND 陷阱！）
  // FTS5 MATCH 'graph algorithms' 等价于 graph AND algorithms
  // Agent 记忆场景下几乎不可能所有词同时出现，要改成 OR
  const ftsQuery = queryText.split(/\s+/).join(' OR ');
  const bm25Rows = db.prepare(`
    SELECT rowid, bm25(memories_fts) AS rank
    FROM memories_fts WHERE memories_fts MATCH ?
    ORDER BY rank LIMIT 20
  `).all(ftsQuery);
  const bm25Ranking = new Map(bm25Rows.map((r, i) => [r.rowid, i + 1]));

  // 通道 2: Vector KNN
  const vecRows = db.prepare(`
    SELECT rowid, distance FROM memories_vec
    WHERE embedding MATCH ?
    ORDER BY distance LIMIT 20
  `).all(new Float32Array(queryVec));
  // ⚠️ BigInt rowid 坑！vec0 返回 BigInt，Map 查找会失败
  const vecRanking = new Map(vecRows.map((r, i) => [Number(r.rowid), i + 1]));

  // 通道 3: 图邻域增强
  // 取向量 Top-3 结果，沿图边扩展找邻居
  const graphRanking = new Map();
  for (const r of vecRows.slice(0, 3)) {
    const neighbors = db.prepare(`
      SELECT target AS node FROM memory_edges WHERE source = ?
      UNION
      SELECT source AS node FROM memory_edges WHERE target = ?
    `).all(Number(r.rowid), Number(r.rowid));
    neighbors.forEach((n, i) => {
      if (!graphRanking.has(n.node)) graphRanking.set(n.node, i + 1);
    });
  }

  // 加权 RRF 融合
  return rrf([bm25Ranking, vecRanking, graphRanking], [1.0, 1.0, 0.5])
    .slice(0, 10);
}

四、三个踩过的坑

坑 1：FTS5 默认 AND 是个隐形杀手

SQLite FTS5 的 MATCH 'graph algorithms memory' 等价于 graph AND algorithms AND memory。在 Agent 记忆场景下，三个词同时出现在同一文档中的概率极低，导致召回率骤降。

解法：查询预处理，把空格替换成 OR ：

// 错误：AND 语义导致空结果
db.prepare('SELECT ... WHERE memories_fts MATCH ?').get('graph algorithms memory');

// 正确：OR 语义提高召回
db.prepare('SELECT ... WHERE memories_fts MATCH ?').get('graph OR algorithms OR memory');

这个坑在单元测试中不容易发现——测试数据通常很小，AND 也能命中。但在真实 Agent 记忆（几百到几千条，每条几十字）中，AND 几乎杀死了所有查询。

坑 2：sqlite-vec 的 BigInt rowid

在 better-sqlite3 中，vec0 表的 rowid 返回 BigInt 而非 number。JavaScript 的 BigInt !== number，导致 Map<number, ...> 查找失败，排名数据莫名其妙丢失。

// 错误：BigInt key 在 Map 中无法匹配 number key
const vecRanking = new Map(vecRows.map((r, i) => [r.rowid, i + 1]));
vecRanking.get(1); // undefined！因为 key 是 1n 不是 1

// 正确：统一转换为 number
const vecRanking = new Map(vecRows.map((r, i) => [Number(r.rowid), i + 1]));
vecRanking.get(1); // ✅ 正常工作

这个问题在 Node.js 生态中几乎是 sqlite-vec 独有的，文档里也没明确说明。我在集成时花了两个小时排查「为什么有些结果消失了」。

坑 3：PageRank 在小图上退化

PageRank 是最经典的图权威性算法，但在节点数 < 50 时，所有节点的分数接近 1/N，区分度极低。这意味着在 Agent 早期记忆（节点少）时，PageRank 信号基本是噪声。

自适应策略——根据图规模动态选择算法：

节点规模推荐算法区分度
< 50度中心性 (Degree Centrality)好
50 - 500HITS (hub/authority)好
> 500PageRank优秀
> 5000k-core decomposition好（找核心簇）
function selectGraphStrategy(nodeCount) {
  if (nodeCount < 50) return degreeCentrality;
  if (nodeCount < 500) hits;
  return pageRank;
}

这个不需要在 V1 实现，但架构设计要预留接口。Agent 记忆库的特点是持续增长——今天 50 个节点，下个月可能 5000 个，算法切换应该自动完成。

用 8 个节点、15 条边的小图做对照实验，查询「graph algorithms memory」：

关键观察：「PageRank Implementation」节点完全不包含「graph algorithms memory」中的任何词，纯 BM25 根本不会返回它。但它通过向量相似度（与「Graph Algos」语义接近）+ 图连接（是「Graph Algos」的邻居）被三路融合成功召回，排名第 4。

这就是三路融合的核心价值——发现你不知道自己知道的东西。

六、性能优化：级联搜索

不是每次查询都需要跑三路。按成本级联：

function cascadeSearch(query, queryVec, topK = 10) {
  // Step 1: BM25（最便宜，~0.1ms/1K docs）
  const bm25Results = bm25Search(query, topK);
  if (bm25Results.length >= topK && bm25Results[0][1] > CONFIDENCE_THRESHOLD) {
    return bm25Results; // 80% 的查询到此为止
  }

  // Step 2: 加入 Vector（中等成本，~1ms/10K vectors）
  const hybridResults = rrf([bm25Results, vectorSearch(queryVec, topK)]);
  if (hybridResults.length >= topK) {
    return hybridResults;
  }

  // Step 3: 加入 Graph 扩展（最贵，邻域遍历）
  return rrf([bm25Results, vectorSearch(queryVec, topK), graphExpand(hybridResults.slice(0, 3))]);
}

实测约 80% 的查询只需 BM25 就能得到好结果。Vector 在涉及同义词/概念改写时被触发。Graph 在需要发现「意外相关」内容时才上场。贵的通道只在便宜的不够用时才跑。

没有任何竞品同时提供三种检索 + 图算法 + 单文件 SQLite。这不是说别人做不到，而是说大多数 Agent 框架的注意力还在「怎么接 LLM」上，记忆检索还停留在关键词或纯向量阶段。图增强检索是一个被忽略的蓝海。

八、可操作建议

  1. 如果你在做 Agent 记忆——不要只做 Vector。先加 BM25（FTS5 零成本），效果立竿见影。如果你的记忆有结构关系（A 依赖 B、A 引用 B），加图邻域增强只需要多一张 edges 表。
  2. 如果你已有 BM25 + Vector——加图的 ROI 最高。不需要图数据库，一张 edges 表 + Recursive CTE 就能做邻域扩展。代码量 < 50 行。
  3. 如果你刚开始——从 SQLite 开始，FTS5 + sqlite-vec + edges 表，三个 CREATE TABLE 就搞定。等真正需要再考虑独立服务。
  4. 权重调优——BM25 和 Vector 各给 1.0，Graph 给 0.3-0.5。图权重超过 0.5 会导致枢纽节点污染结果。这个数字在 200-500 节点规模下实测稳定。
  5. 记得修 FTS5 的 AND 陷阱——query.split(/\s+/).join(' OR ')，一行代码避免 80% 的零结果查询。

结语

三路融合的核心理念很简单：不同检索方式看到的世界不同。BM25 看到词，Vector 看到语义，Graph 看到关系。让它们一起投票，比任何单一方式都更接近「这个 Agent 真正需要什么」的答案。

这套系统已经在 agent-memory-graph 中跑了 665 个测试，150+ 个 API。一个 .db 文件就是完整的 Agent 记忆——可搜索、可迁移、可审计。没有 Pinecone 的账单，没有 Neo4j 的运维，只有一个 SQLite 文件和三行 CREATE TABLE。

有时候，最好的架构不是选择最好的工具，而是让一个工具发挥最大的潜力。
