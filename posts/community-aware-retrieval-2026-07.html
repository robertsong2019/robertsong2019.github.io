当 Agent 的记忆开始「聚类思考」：社区感知检索的工程实践

2026-07-05 · 罗嵩 · 基于 autoresearch 方法论

Agent 的记忆图越大，检索就越像在嘈杂的人群中找人喊话。社区检测让记忆自组织成语义部落，检索从"全局广播"变成"精准投递"。

引言：扁平检索的天花板

过去两周，我在 agent-memory-graph 项目中陆续实现了 Q-value scoring、bi-temporal validity、strategic forgetting 等记忆管理能力。每个功能都在解决同一个问题的不同侧面：如何让 Agent 在记忆增长后依然保持高质量的检索和推理？

但这些功能有一个共同的局限——它们都在一个扁平的记忆图上操作。所有的记忆节点处于同一个层级，检索时遍历整个图。当记忆从 100 个节点增长到 10,000 个节点时，问题开始显现：

  • BM25 全局搜索返回的 top-10 结果可能来自完全无关的领域
  • 图遍历（如 Personalized PageRank）在密集图上扩散过快，丢失局部结构
  • Agent 无法区分"这是一组关于用户偏好的记忆"和"这是一组关于项目架构的记忆"

人类不是这样回忆的。当你想到"上次和团队讨论数据库选型的场景"，你的大脑会自然聚焦到那个语境下的相关信息——团队成员的意见、技术对比的笔记、最终的决策——而不会同时浮现你昨天看的菜谱。

这种能力在图论中有一个精确的数学对应：社区结构。

一、从 GraphRAG 到 Agent Memory

GraphRAG 的核心创新

2024 年微软提出的 GraphRAG 给出了一个优雅的方案：在构建知识图谱后，运行社区检测算法（如 Leiden），将图划分为层次化的社区，然后为每个社区生成摘要。检索时，根据问题范围选择在社区摘要层或原始节点层搜索。

GraphRAG 的 Global mode 在宏观问题上显著优于传统 RAG——因为传统 RAG 根本无法回答需要综合全库信息的问题。

但 GraphRAG 是为文档问答设计的。Agent 的记忆系统有几个关键差异：

维度GraphRAG（文档问答）Agent Memory
数据来源静态文档库动态增量写入
查询模式问答（一次检索）多步推理（反复检索）
时间维度无Bi-temporal
更新频率构建后很少更新每次对话都可能写入
社区结构离线计算，层次固定需要增量更新

这意味着 Agent Memory 不能直接照搬 GraphRAG 的方案。我们需要一个轻量、增量友好、单库即可运行的社区检测方案。

选择 LPA 的理由

在比较了 Leiden、Louvain、Label Propagation Algorithm (LPA) 三种主流算法后，我选择了 LPA：

Leiden/Louvain 是模块度优化算法，社区质量最高，但计算成本高，增量更新困难，实现复杂（~500+ 行）。

LPA 是基于标签传播的分布式算法：每个节点初始为自己的社区，迭代地采纳邻居中最常见的标签，收敛快（通常 5-10 次迭代），近线性时间复杂度，天然增量友好。实现不到 80 行。

LPA 的社区质量略逊于 Leiden，但对于 Agent Memory 的场景——社区检测主要服务于检索优化，不需要完美的分区——这个 trade-off 是合理的。

二、实现：从零到社区感知检索

2.1 构建邻接表

社区检测的第一步是理解图的拓扑结构。记忆图是有向图，但社区检测需要无向图——因为"社区"是双向的语义聚类。

def _build_adjacency(self) -> dict[str, set[str]]:
    """Build undirected adjacency map from directed edges.
    Excludes quarantined nodes."""
    active_ids = {
        row["id"] for row in self.conn.execute(
            "SELECT id FROM nodes WHERE quarantined=0 OR quarantined IS NULL"
        ).fetchall()
    }
    adj: dict[str, set[str]] = defaultdict(set)
    for row in self.conn.execute("SELECT source, target FROM edges").fetchall():
        s, t = row["source"], row["target"]
        if s in active_ids and t in active_ids:
            adj[s].add(t)
            adj[t].add(s)  # 无向化
    for nid in active_ids:
        adj.setdefault(nid, set())
    return dict(adj)

一个关键细节：隔离节点（quarantined）被排除在外。这些是被 conflict_resolve() 标记的、被新事实推翻的旧记忆。它们不应该参与社区形成——就像被否决的提案不应该影响委员会的分组。

2.2 标签传播算法核心

LPA 的核心逻辑出奇地简洁：

def detect_communities(self, max_iterations=20, resolution=1.0) -> dict:
    adj = self._build_adjacency()
    node_ids = sorted(adj.keys())
    
    # 初始化：每个节点是自己所在的社区
    labels = {nid: i for i, nid in enumerate(node_ids)}
    
    iterations = 0
    changed = True
    while changed and iterations < max_iterations:
        changed = False
        iterations += 1
        for nid in node_ids:
            neighbors = adj[nid]
            if not neighbors:
                continue
            
            # 投票：邻居的标签哪个出现最多？
            votes = defaultdict(float)
            for nb in neighbors:
                votes[labels[nb]] += resolution
            
            # 确定性 tie-break：相同票数选小编号
            best_label = max(votes, key=lambda lb: (votes[lb], -lb))
            if best_label != labels[nid]:
                labels[nid] = best_label
                changed = True
    
    # 重新编号为连续整数
    unique = sorted(set(labels.values()))
    remap = {old: new for new, old in enumerate(unique)}
    # ...

整个算法就是：反复让每个节点采纳邻居中最流行的标签，直到收敛。

这看起来太简单了——但它确实工作。直觉上的解释是：密集连接的节点群会迅速就一个标签达成共识，而稀疏连接的不同群体之间的标签传播会被"稀释"。

resolution 参数 是一个有趣的扩展。标准 LPA 中每个邻居的投票权重相同。但在 Agent Memory 中，同类型的节点（比如两个 fact 类型的记忆）之间的投票权重更高：

for nb in neighbors:
    weight = resolution
    if kind_map.get(nid) == kind_map.get(nb):
        weight *= resolution  # 同类 boost
    votes[labels[nb]] += weight

这让社区形成有了语义偏好：关于"数据库"的事实更倾向于聚在一起，而不是和"用户偏好"的临时指令混在一组。

2.3 模块度评估：社区质量有多好？

检测出社区后，需要评估分区质量。模块度（Modularity Q）是标准指标：

@staticmethod
def _modularity(adj, labels, n) -> float:
    """Q = (1/2m) * Σ_ij [A_ij - k_i*k_j/(2m)] * δ(c_i, c_j)"""
    if n == 0:
        return 0.0
    degree = {nid: len(nbrs) for nid, nbrs in adj.items()}
    two_m = sum(degree.values())  # 2m
    if two_m == 0:
        return 0.0
    
    # 同社区的边数（每条边被计算两次）
    intra = 0.0
    for nid, nbrs in adj.items():
        for nb in nbrs:
            if labels.get(nid) == labels.get(nb):
                intra += 1.0
    
    q = (intra / two_m) - sum(
        (degree[nid] / two_m) ** 2 for nid in degree
    )
    return q

Q 值范围在 [-0.5, 1.0]：

  • Q ≈ 0：社区结构和随机图没有区别（社区检测无效）
  • Q > 0.3：存在明显的社区结构
  • Q > 0.7：非常强的社区结构（实际中罕见）

Q 的公式有一个优雅的概率解读：(intra / two_m) 是同社区边的实际比例，Σ(k_i/2m)² 是随机图下的期望比例。Q 衡量的是实际超过了期望多少。

三、社区感知检索：从"全局搜索"到"局部探索"

社区检测的最终目的不是画好看的分区图，而是改变检索策略。

传统 BM25 检索:
  Query → 全库 BM25 排序 → Top-K 结果

社区感知检索 (search_community):
  Query → 全库 BM25 找 seed → 定位 seed 所属社区 
        → 在社区内（+ 邻接社区）BM25 → Top-K 结果

这个"先定位再深入"的策略有一个关键优势：同社区的相关记忆会被优先返回，即使它们的关键词匹配分数不是最高。

def search_community(self, query: str, limit: int = 10) -> list[Node]:
    """Community-aware retrieval: find the best community first,

    # Step 1: 全局 BM25 找 seed
    seed_results = self.search_bm25(query, limit=1)
    if not seed_results:
        return self.recall(query, limit=limit)

    cid = cache["node_community"].get(seed_id)

    member_ids = set(cache["communities"].get(cid, []))

为什么扩展到邻接社区？ 因为社区边界不是硬性的。一个关于"PostgreSQL 性能调优"的记忆可能属于"数据库"社区，但通过一条 related_to 的边连接到"DevOps"社区。限制在单一社区内会丢失这些跨域关联。

四、桥节点：记忆图中的"结构洞"

社会学家 Ronald Burt 在 1992 年提出了结构洞（Structural Holes）理论：在社会网络中，跨越不同社群的人拥有独特的信息优势——他们能将不同群体的知识组合起来，产生创新。

在 Agent 的记忆图中，这对应着桥节点（Bridge Nodes）：那些连接不同社区的节点。它们是跨域推理的关键。

考虑这个场景：Agent 的记忆图中有两个社区：

  • C0（用户偏好）：用户喜欢 Python、注重代码质量
  • C1（项目状态）：项目用 TypeScript、需要做 API 测试

桥节点是："用户写过一篇 pytest vs vitest 的对比文章"——它同时属于用户偏好和项目状态两个社区。

当用户问"我们该用什么做 API 测试？"时，Agent 通过桥节点可以推理出：

"项目用 TypeScript（C1），但用户偏好 Python（C0）且有测试框架对比经验（桥节点）。考虑到 pytest 在 API 测试上的成熟度，建议用 pytest + requests 做 E2E，vitest 做单元测试。"

没有桥节点，Agent 可能只看到 TypeScript 就直接推荐 vitest，错过了用户的 Python 偏好。

五、超图缩减：当社区图变成新的记忆层

当记忆图有 10,000 个节点时，一个自然的想法是：把每个社区压缩成一个超节点，构建社区级别的超图。

def community_graph(self) -> dict:
    """Build a reduced graph where each community is a supernode."""
    supernodes = []
    for s in self.community_stats():
        supernodes.append({
            "community_id": s["community_id"],
            "size": s["size"],
            "dominant_kind": max(s["kinds"].items(), key=lambda x: x[1])[0],
            "avg_weight": s["avg_weight"],
            "density": s["density"],
        })
    
    # 统计社区间的边数
    inter = defaultdict(int)
    for row in self.conn.execute("SELECT source, target FROM edges").fetchall():
        src_c = node_community.get(row["source"])
        tgt_c = node_community.get(row["target"])
        if src_c != tgt_c:
            key = (min(src_c, tgt_c), max(src_c, tgt_c))
            inter[key] += 1
    
    superedges = [
        {"source": s, "target": t, "edges": cnt}
        for (s, t), cnt in sorted(inter.items())
    ]
    return {"supernodes": supernodes, "superedges": superedges}

10,000 个节点的原图 → 30 个超节点的社区图。在超图上运行图算法快几个数量级。

六、工程实践中的取舍

6.1 缓存策略

社区检测不是免费的。在 1000+ 节点的图上，LPA 大约需要 50-200ms。方案是惰性缓存：

def community_of(self, node_id: str) -> Optional[int]:
    cache = getattr(self, "_community_cache", None)
    if cache is None:
        self.detect_communities()  # 首次调用时计算
        cache = getattr(self, "_community_cache", None)
    return cache["node_community"].get(node_id)

更理想的方案是增量更新：新增节点加入其邻居的社区，只有当社区结构显著变化时才全量重算。但这需要实现增量 LPA，留作后续迭代。

6.2 LPA 的随机性问题

LPA 有一个已知问题：节点更新顺序会影响最终分区。我的处理方式是确定性排序加确定性 tie-break。确保相同输入永远产生相同输出，代价是失去了并行可扩展性。对于 Agent Memory 的单库场景，这是正确的 trade-off。

6.3 小图的退化

当记忆图只有 3-5 个节点时，LPA 会把每个节点分成独立的社区。我在测试中用 LPA-tolerant 断言处理了这种情况——更重要的是验证社区感知检索不报错，而不是追求精确的社区数量。

七、社区检测在 Agent 记忆链路中的位置

把社区检测放回完整能力链路中：

写入 → 冲突检测 → 战略性遗忘 → 社区检测 → 社区感知检索
  ↑                                              ↓
  └──────── Q-value 反馈 ←──────────────────────┘
环节维度解决什么
冲突检测时间新旧事实矛盾时隔离旧事实
战略性遗忘价值淘汰过期/低价值记忆

这五个维度共同构成了一个自组织、自维护的记忆系统。

八、与前沿研究的对话

DF-Leiden（ICML 2026）：动态流式 Leiden 算法，在 10⁶ 节点图上比标准 Leiden 快 105×。LPA 是更轻量的替代，但 DF-Leiden 的增量思路值得借鉴。

GraphRAG 层次摘要：我们的 community_profile() 是一个轻量版摘要——不调用 LLM 生成自然语言摘要，而是提取结构化特征。对于 token 预算有限的 Agent 场景，比 LLM 摘要更经济。

MemoryArena 的 Agency Gap：社区感知检索直接瞄准了 recall 和 agency 之间的鸿沟。通过语义聚类，让返回的记忆在同一个语境中，降低了 Agent 跨语境推理的负担。

结语

Agent 的记忆系统正在从"存得更多"走向"想得更深"。社区检测让记忆不再是散落的点，而是自组织的网络。

当 Agent 检索"PostgreSQL"时，它不再只是在 10,000 个节点中做关键词匹配，而是先定位到"项目架构"社区，在那个 82 个节点的语义簇中做精准搜索。当 Agent 需要跨域推理时，桥节点提供了结构洞上的桥梁——就像一个既懂技术又懂用户偏好的同事，能在对的时间把对的信息连起来。

社区感知检索不只是一个搜索优化技巧。它是 Agent 记忆从数据库走向知识图谱的标志——从"存什么"到"怎么组织什么"的范式转变。

而这一切的算法核心——LPA——只有不到 30 行有效代码。有时候，最有价值的工程不是最复杂的，而是在正确的位置做了正确的 trade-off。

本文基于 agent-memory-graph 项目 Cycle 176-178 的工程实践（+39 tests, 1599 passed, 零回滚率 174 天），以及对 GraphRAG、DF-Leiden、MemoryArena 等前沿研究的调研。
