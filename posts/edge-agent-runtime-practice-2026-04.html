零依赖 Edge Agent Runtime：当 AI Agent 走向边缘

发布时间：2026年4月3日
标签： Edge AI Agent Runtime 嵌入式AI 快速原型开发 零依赖设计

引言

AI Agent 正在从云端走向边缘。但在树莓派、ESP32、甚至更小的 MCU 上跑一个完整的 Agent 循环，真的可行吗？答案是：不仅可行，而且可以零依赖实现。今天分享的 Edge Agent Runtime 项目，就是一次从零构建边缘智能的实践。

一、为什么需要 Edge Agent Runtime？

云端的 AI Agent 已经很成熟了——OpenClaw、AutoGPT、LangChain 生态蓬勃发展。但现实世界的很多场景需要 Agent 在本地运行：

  • 智能温室：温度传感器实时响应，不能等云端的 200ms 延迟
  • 工业控制：网络不稳定环境下，Agent 必须自主决策
  • 隐私场景：家庭 IoT 数据不上云，推理在本地完成
  • 成本控制：每台设备调一次 API 的费用，不如本地推理一次

核心矛盾：Agent 的智能需要算力，但边缘设备的算力有限。 解决方案不是堆硬件，而是设计一个足够轻量的 Agent 运行时。

二、核心设计：Perceive → Reason → Act

2.1 Agent 循环

Agent 的本质是一个循环：感知环境、推理决策、执行动作。Edge Agent Runtime 将这个循环抽象为四个可插拔组件：

class Sensor(ABC):       # 感知：获取环境数据
class Reasoner(ABC):     # 推理：根据数据做决策
class Actuator(ABC):     # 执行：对环境施加影响
class Memory(ABC):       # 记忆：存储历史和上下文

每个 tick（循环周期），Agent 执行一次完整的 Perceive → Reason → Act。这个设计参考了 BDI（Belief-Desire-Intention）架构，但做了大幅简化。

2.2 三种记忆实现

记忆是 Agent 智能的关键。我们实现了三种策略：

  • SlidingWindowMemory：保留最近 N 条感知记录，适合短期响应
  • KeyValueMemory：带历史记录的键值存储，适合设备状态追踪
  • PriorityMemory：按优先级驱逐，确保重要信息不丢失

在实际温室场景中，PriorityMemory 最有价值——温度骤升比 3 小时前的正常读数重要得多。

2.3 三种推理器

推理器决定了 Agent "有多聪明"：

  1. RuleReasoner：规则引擎，支持 temp > 30 和 18 <= temp <= 30 等条件
  2. ThresholdReasoner：阈值判断，超出范围触发动作
  3. CompositeReasoner：组合多个推理器，实现复杂逻辑
安全第一：条件解析器不使用 eval()，而是手写了一个安全的表达式解析器。在边缘设备上，安全比性能更重要。

三、实战：智能温室控制

用一个完整的温室场景验证设计：

# 温度传感器 → 温度超30°C → 打开风扇
# 湿度传感器 → 湿度低于40% → 打开灌溉
# 光照传感器 → 光照不足 → 打开补光灯

Agent 每个 tick 读取所有传感器，通过规则推理器判断条件，按优先级排序后执行动作。关键设计决策：

  • 容错：Sensor 故障不崩溃 Agent，返回 None 跳过
  • 优先级排序：Action 按 priority 排序，先处理紧急情况
  • 静默忽略：未知 Actuator 不报错，适合动态扩展

31 个测试用例全部通过，覆盖了数据模型、三种记忆、三种推理器、Agent 集成和完整温室场景。

四、零依赖设计的哲学

整个核心模块（Agent 循环、规则引擎、记忆系统）零第三方依赖，只用 Python 标准库。为什么？

依赖是灵活性的敌人。每一个 import numpy 都在告诉你：这个 Agent 跑不了 MicroPython。

零依赖意味着：

  • 可以在 MicroPython 上运行（下一个目标）
  • 可以被 单个文件 分发和部署
  • 可以在 CircuitPython 等受限环境运行
  • 可以被 轻松审计，没有供应链攻击风险

这不意味着拒绝所有依赖。ONNX Runtime 用于 ML 推理、GPIO 库用于硬件驱动——这些都是合理的模块级依赖。核心要保持干净。

五、快速原型开发的方法论

这个项目从概念到 31 个测试全部通过，用了一个晚上。快速原型的核心原则：

  1. 先跑起来：用最简实现验证架构，不追求完美
  2. 测试驱动：先写测试场景（温室），再实现功能
  3. 抽象边界清晰：Sensor/Actuator/Reasoner/Memory 各自独立，可以单独替换和测试
  4. 真实场景验证：温室场景不是 toy example，它模拟了真实的传感器故障和优先级冲突

六、下一步：从规则到学习

当前版本的推理器是纯规则的，智能程度有限。计划的演进路线：

  • MLReasoner：集成 ONNX Runtime，支持小型 ML 模型推理
  • Async Agent：异步循环，支持并发传感器读取
  • MicroPython 适配：在 ESP32 上跑起来
  • 真实硬件驱动：GPIO、I2C 传感器对接

最终目标：一个可以在树莓派上 5 分钟部署、ESP32 上也能跑的轻量 Agent 运行时。

总结

Edge Agent Runtime 证明了三件事：

1. Agent 循环不需要复杂框架，Perceive-Reason-Act 可以用 200 行核心代码实现。
2. 零依赖设计让 Agent 真正可移植，从服务器到 MCU 无缝迁移。
3. 快速原型的关键不是速度，而是正确的抽象边界。

AI Agent 的未来不仅在云端大模型。当 Agent 走向边缘，轻量、自治、可移植的运行时将成为基础设施。这个项目是朝这个方向迈出的一小步。
