Schema-Valid ≠ Content-Correct：LLM 结构化输出的双重验证陷阱

2026-05-22 · 阅读时间约 8 分钟
#LLM #Structured Output #TypeScript #Agent

过去两个月，我在 agent-context-store 项目上跑了连续 68 天的 autoresearch 循环——每天自动选功能、实现、测试、提交，零人工干预，零回滚。370 个测试全绿。但上周末在做 Structured Output Toolkit 的深度研究时，我发现了一个让我后背发凉的盲点：

Constrained Decoding 保证 100% 的 schema-valid 输出，但没有任何机制保证 content-correct。

这不是理论问题。这是一个在生产中会咬你一口的真实陷阱。

三层结构化输出的演进

2026 年的 LLM 结构化输出已经形成了清晰的三层体系：

  1. Level 1 — Prompt Engineering（"Please respond in JSON"）。可靠性约 35%，只适合原型。
  2. Level 2 — Function Calling / Tool Use。可靠性 95-99%，当前生产环境的主流选择。
  3. Level 3 — Constrained Decoding。在 token 生成时用有限状态机（FSM）屏蔽非法 token，保证输出 100% 符合 JSON Schema。

Level 3 很诱人。OpenAI 的 response_format: { type: "json_schema" }、Gemini 的 responseSchema 都支持。100% schema-valid 听起来像是解决了所有问题。

但它只解决了格式问题，没解决语义问题。

一个具体的例子

假设你用 constrained decoding 让 LLM 做情感分析：

const sentimentSchema = {
  type: 'object',
  properties: {
    sentiment: { type: 'string', enum: ['positive', 'negative', 'neutral'] },
    confidence: { type: 'number', minimum: 0, maximum: 1 },
    topics: { type: 'array', items: { type: 'string' } },
    summary: { type: 'string' }
  },
  required: ['sentiment', 'confidence', 'topics', 'summary']
};

输入："This product is amazing! Best purchase ever."

LLM 输出：

{
  "sentiment": "positive",
  "confidence": 0.92,
  "topics": ["product quality", "customer satisfaction"],
  "summary": "The customer is very happy with the product"
}

完美。Schema-valid ✅。但想象另一种情况——输入是一段完全无关的文本，比如"明天上海天气如何"，而 LLM 返回：

{
  "sentiment": "neutral",
  "confidence": 0.5,
  "topics": ["weather"],
  "summary": "Asking about tomorrow's weather in Shanghai"
}

这也完全 schema-valid ✅。但它的"情感分析"结果毫无意义——原始文本根本不是在表达情感。Schema 约束了形状，没约束内容是否合乎任务意图。

双重验证：不可省略的第二道门

这就是我说的"双重验证"：

  1. Schema Validation（格式验证）—— JSON 结构是否符合定义？Constrained decoding 在这层做到了 100%。
  2. Business Logic Validation（语义验证）—— 内容是否在业务上有意义？这层 LLM 做不了，必须你来做。

大多数框架只帮你做了第一层。第二层你得自己写。

Validation-Aware Retry Pattern

关键不是验证失败了就重试——关键是把验证错误的具体信息反馈给 LLM，让它理解哪里错了、为什么错。

class StructuredLLMClient {
  private cache = new SchemaCache();

  async extract<T>(opts: RequestOptions): Promise<T> {
    const schema = this.cache.getOrCreate(opts.schemaKey, () => opts.schema);
    let lastErrors: string[] = [];

    for (let attempt = 0; attempt <= this.maxRetries; attempt++) {
      // 把验证错误反馈给 LLM
      let prompt = opts.prompt;
      if (lastErrors.length > 0) {
        prompt += `\n\n上一次尝试的验证错误:\n${
          lastErrors.map(e => `- ${e}`).join('\n')
        }\n请修复这些错误。`;
      }

      const raw = await this.callLLM(prompt, schema);
      const parsed = JSON.parse(raw);

      // 第一层：schema validation
      const schemaResult = validateSchema(parsed, schema);
      if (!schemaResult.ok) {
        lastErrors = schemaResult.errors;
        continue;
      }

      // 第二层：business logic validation
      const bizResult = this.businessValidate(parsed, opts.rules);
      if (!bizResult.ok) {
        lastErrors = bizResult.errors;
        continue;
      }

      return parsed as T;
    }

    throw new Error(`Failed after ${this.maxRetries + 1} attempts`);
  }

  private businessValidate(data: any, rules?: ValidationRule[]): ValidationResult {
    if (!rules) return { ok: true, errors: [] };
    const errors: string[] = [];
    for (const rule of rules) {
      const result = rule(data);
      if (!result.pass) errors.push(result.message);
    }
    return errors.length === 0 ? { ok: true, errors: [] } : { ok: false, errors };
  }
}

注意循环里的关键设计：lastErrors 不是简单的"failed"，而是具体的错误描述——"confidence 超出范围"、"topics 为空但 summary 提到了具体主题"。LLM 拿到这些信息后，修复的成功率远高于 blind retry。

Schema Cache：被大多数框架忽略的差异化组件

在 agent 系统中，同一个 schema 会被反复使用。每次请求都重新编译 schema 是浪费。但大多数框架——包括 LangChain——没有内置 schema 缓存。

class SchemaCache {
  private cache = new Map<string, { schema: SchemaDef; jsonSchema: object }>();

  getOrCreate(key: string, factory: () => SchemaDef) {
    const cached = this.cache.get(key);
    if (cached) return cached;
    const schema = factory();
    const jsonSchema = this.toJsonSchema(schema);
    this.cache.set(key, { schema, jsonSchema });
    return this.cache.get(key)!;
  }
}

一个简单的 Map 缓存。但在高频调用场景下，这意味着从 O(n) schema 编译降到 O(1) 查找。Vercel AI SDK 的 generateObject 内部做了类似的事，但暴露出来的 API 不如自己控制灵活。

Schema 设计的实用建议

基于过去几周的实战经验，几条非显而易见的建议：

  • 嵌套深度 ≤ 3 层。Constrained decoding 在深层嵌套时的 token 限制开销指数增长。大 schema 拆成多个并行小 schema 调用更划算。
  • Enum 值 ≤ 10 个，且语义不重叠。LLM 在相似 enum 值之间容易混淆。用"非常不满/不满/中立/满意/非常满意"而不是 1-10 的数字 enum。
  • 必填字段要克制。required 列表越长，LLM 编造内容填充的概率越高。能选填就选填。
  • 用 description 引导语义。每个字段加 description，不只是给人类看的——LLM 会读它来理解期望。

Provider 差异：不是所有 constrained decoding 都一样

2026 年三巨头的实现差异值得了解：

  • OpenAI：原生 FSM constrained decoding，response_format 直接传 JSON Schema。最成熟，但 schema 有复杂度限制。
  • Anthropic：没有原生 structured output。通过 tool use 模拟——把你的 schema 包装成一个"工具"让模型"调用"。需要 Zod 兜底。
  • Gemini：responseSchema 原生支持。不合规时抛 JSONSchemaValidationError。

如果你要跨 provider，Vercel AI SDK 的 generateObject 是当前最佳抽象层——自动选最优策略。但了解底层差异能帮你在 debug 时少走弯路。

从 agent-context-store 学到的设计原则

回到我的 autoresearch 实验。370 个测试、68 天零回滚的背后，有一个核心原则：

每个新功能都必须通过已有功能的测试组合。

incr() 必须和 tags、search、snapshot 兼容。merge_content() 必须触发 hooks 和 changelog。这不是额外的测试负担——这是组合性的证明。

Structured Output Toolkit 也应该遵循同样的原则。你的 retry pipeline 必须和 schema cache 组合。你的 business validator 必须和 provider adapter 组合。单独测每个组件不难，难的是组合后的正确性。

可操作的建议

  1. 永远写 business logic validator。Schema validation 是免费的（constrained decoding 替你做了），但 semantic validation 必须手写。把它当成接口契约的一部分。
  2. Retry 时带错误上下文。不要 silent retry。把具体错误信息拼进 prompt，LLM 的修复成功率从 ~40% 跳到 ~80%。
  3. 缓存 schema 编译结果。一个 Map 就够。在 agent 循环中这个优化肉眼可见。
  4. Schema 设计要克制。能选填就不必填，能平铺就不嵌套，能 5 个 enum 就不 10 个。
  5. 测试组合性，不只是单元性。确保你的 retry + cache + validation 三条路径在交叉时依然正确。

结构化输出在 2026 年已经成为标配——三巨头全部原生支持。但"能用"和"可靠"之间的距离，恰恰是 schema-valid 和 content-correct 之间的距离。别让这 20% 的缝隙咬掉你 80% 的调试时间。
