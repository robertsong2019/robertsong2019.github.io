MCP 记忆服务器实战：从官方 SDK 到 6 工具精裁架构

罗嵩 · 2026-07-22
2026-07-22 技术随笔。MCP（Model Context Protocol）正在成为 Agent 生态的 USB-C 接口——一个协议连接所有模型和所有工具。但在记忆领域，官方的 Memory Server 还停留在 JSONL 文件 + 子串搜索的原始阶段。本文记录我基于 MCP SDK v2 构建图谱驱动记忆服务器的实践：为什么 6 个精裁工具优于 9 个薄包装，如何用工厂模式实现双传输层兼容，以及资源订阅如何让 Agent 获得实时感知能力。包含完整 TypeScript 实现和可运行的测试代码。

问题：MCP 记忆服务器的现状令人失望

你可能已经用上了 MCP。Claude Desktop、Cursor、Windsurf——2026 年的主流 AI 客户端都支持 MCP 协议，允许你挂载自定义工具。你可以在 Claude Desktop 的配置文件里加上一行，就让 LLM 获得查天气、查数据库、操作文件的能力。

但你有没有试过官方的 Memory Server？

它的全部实现是这样的：一个 memory.json 文件，每条记忆是一行 JSON。搜索方式是——你猜对了——JavaScript includes() 子串匹配。没有语义搜索，没有图谱关联，没有健康检查，没有遗忘机制。写入是追加式的，文件无限增长。1000 条记忆之后，搜索结果的相关性趋近于随机。

这不是记忆系统，这是一个带 JSON 元数据的 grep。

我在 MCP Registry 里搜索了所有标记为 "memory" 的服务器——零个使用图结构。零个提供质量监控。零个支持资源订阅（一种让客户端在记忆变化时收到通知的机制）。整个生态的记忆基础设施，还停留在文件系统时代。

这既是问题，也是机会。

设计思路：精裁优于覆盖

我的第一个设计决策是：不把底层图引擎的 40+ 个 API 一一暴露为 MCP 工具。

MCP 工具不是 REST API。每一个工具定义都会被发送到 LLM 的上下文窗口里，消耗 token 并占用注意力。Anthropic 的官方建议是 5-15 个工具——超过 20 个，模型的选择准确率开始显著下降。

我最终设计了 6 个工具，分三层：

┌─────────────────────────────────────────────────────┐

│            MCP 记忆服务器工具面 (6 工具)               │ │                                                       │ │  ── 基础层 (4 tools) ──                               │ │  memory.recall     读  语义搜索，返回排序结果          │ │  memory.remember   写  存储事实/事件/技能/偏好          │ │  memory.health     读  图谱健康度评分 (0-100)          │ │  memory.forget     删  按 ID 删除（标记为破坏性）       │ │                                                       │ │  ── 高级层 (2 tools) ──                               │ │  memory.query      读  结构化过滤（类型/日期/排序）     │ │  memory.consolidate 写 去重合并（标记为破坏性）        │ │                                                       │ │  ── 资源层 ──                                         │ │  memory://graph       可订阅的资源变更通知             │ └─────────────────────────────────────────────────────┘

这个设计的核心理念是：LLM 需要的是语义清晰的动词，不是底层 API 的忠实映射。

recall vs query 的区别不是多余的——recall 是"自然语言快速搜索"，query 是"按条件精确过滤"。当 LLM 需要回忆"张三的项目用了什么技术栈"时，它调用 recall。当它需要"列出最近一周所有的 event 类型记忆"时，它调用 query。两个工具的语义边界清晰，模型几乎不会混淆。

对比一个反面设计：官方 Memory Server 有 9 个工具——create_entities、create_relations、add_observations、delete_entities、delete_observations、delete_relations、read_graph、search_nodes、open_nodes。这是把内部数据结构（实体-关系-观察三元组）直接暴露给了 LLM。模型需要理解三者的区别和交互方式，认知负担太高。

工厂模式：双传输层的基石

MCP SDK v2 的第一个硬性要求是工厂函数模式。每个官方示例都使用 buildServer() 或 createServer() 返回 McpServer 实例，而不是在模块顶层直接创建。

这不是代码风格偏好——它是双传输层兼容的前提。

// ❌ 错误：模块顶层创建（stdio 可用，HTTP 会破坏隔离）

const server = new McpServer({ name: 'memory', version: '0.1.0' }); server.registerTool('recall', ...); void serveStdio(() => server);  // HTTP 模式下多请求共享状态！


// ✅ 正确：工厂函数（每次调用创建新实例） export function buildServer(): McpServer { const server = new McpServer( { name: 'agent-memory-graph', version: '0.1.0' }, { capabilities: { logging: {}, resources: { listChanged: true, subscribe: true }, }, } );


// 所有工具注册在工厂内部 server.registerTool('memory.recall', ...); server.registerTool('memory.remember', ...);


return server; }


// stdio 入口调用工厂 void serveStdio(buildServer);

为什么这很重要？ 在 stdio 传输模式下，每个进程服务一个客户端，共享状态没有问题。但在 HTTP 传输模式下（Phase 2 的目标），createMcpHandler(buildServer) 会为每个请求创建一个新的服务器实例。如果工具注册在模块顶层，它们就会在所有请求之间共享——客户端 A 能看到客户端 B 的记忆，这在安全上是不可接受的。

工厂模式还有第二个好处：测试隔离。每个测试用例可以创建独立的服务器实例，测试之间不互相污染。

Zod Schema：模型唯一看到的文档

MCP 工具的 description 和 inputSchema 上的 .describe() 是 LLM 唯一能看到的工具文档。不像人类开发者可以读 README 或源码，模型的全部理解来自这些字符串。

const RememberSchema = z.object({

content: z.string().min(1) .describe('The memory to store — a fact, observation, event, or skill'), kind: z.enum(['fact', 'event', 'skill', 'preference', 'reasoning', 'intention']) .optional() .describe('Memory type (default: fact)'), });


server.registerTool( 'memory.remember', { title: 'Store Memory', description: 'Persist a fact, observation, event, or skill into agent memory. ' + 'Returns the new memory ID.', inputSchema: RememberSchema, outputSchema: z.object({ id: z.string(), stored: z.boolean(), }), }, async ({ content, kind = 'fact' }, ctx) => { const id = mg.add(content, kind); await notifyGraphChanged();


if (ctx?.mcpReq?.log) { await ctx.mcpReq.log( 'info', remembered [${id}]: ${content.slice(0, 60)}..., 'memory' ); }


return { content: [{ type: 'text' as const, text: ✅ Remembered [${id}]: &quot;${content.slice(0, 80)}&quot; (${kind}), }], structuredContent: { id, stored: true }, }; } );

三个关键细节：

1. .describe() 写给模型，不是写给开发者。 好的描述像好的变量名——它回答"这是什么"，而不是"它怎么实现"。z.string().describe('The memory to store') 远好于 z.string().describe('Content string for the memory node')。

2. outputSchema 是运行时安全网。 SDK 在结果离开服务器之前验证 structuredContent 是否匹配 outputSchema。如果你的 handler 返回了错误的形状——比如 score 是字符串而不是数字——SDK 会捕获它，防止客户端收到畸形数据。这是免费的类型安全。

3. SDK 自动验证输入。 inputSchema 不只是文档——它是验证器。无效参数会自动返回 isError: true，handler 永远不会执行。你不需要在 handler 里写 if (!content) return error(...) 的防御代码。

注解驱动治理：协议级安全

MCP SDK v2 的注解系统是一个被低估的安全特性。四个布尔值注解，让宿主客户端（Claude Desktop、Cursor）自动决定工具的审批策略：

// memory.recall: 安全的只读操作

annotations: { readOnlyHint: true,       // 宿主自动批准，无需用户确认 destructiveHint: false, idempotentHint: true,     // 安全重试 openWorldHint: false,     // 封闭系统 }


// memory.forget: 破坏性操作 annotations: { readOnlyHint: false, destructiveHint: true,    // 宿主要求用户确认！ idempotentHint: true,     // 删除两次无害 openWorldHint: false, }


// memory.remember: 写入但非破坏性 annotations: { readOnlyHint: false, destructiveHint: false,   // 不需要确认 idempotentHint: false,    // 重复写入创建多条 openWorldHint: false, }

这意味着什么？ 当用户在 Claude Desktop 中使用你的记忆服务器时，recall 和 health 工具会被自动批准——用户不需要每次点"允许"。但 forget 和 consolidate 会被标记为破坏性操作，客户端会弹出确认对话框。

这是协议级安全——你不需要在应用层实现确认 UI，MCP 客户端已经内置了。对比传统的 REST API 安全方案（OAuth scopes、API keys、rate limiting），注解系统更轻量也更语义化——它表达的是"这个操作的性质"，而不是"谁有权执行"。

资源订阅：双时代兼容模式

MCP 协议正在从 2025-era 向 2026-era 迁移。2025-era 客户端使用 resources/subscribe 显式订阅资源变更。2026-era 客户端（SDK v2 引入）使用 subscriptions/listen 过滤器，不再调用显式订阅方法。

这意味着你的服务器需要同时兼容两种模式。官方的解决方案很优雅：

const subscribedUris = new Set<string>();

// 兼容 2025-era 客户端的显式订阅 server.server.setRequestHandler('resources/subscribe', (request) => { subscribedUris.add(request.params.uri); return {}; }); server.server.setRequestHandler('resources/unsubscribe', (request) => { subscribedUris.delete(request.params.uri); return {}; });


// 统一通知函数：两种时代都能收到 async function notifyGraphChanged(): Promise<void> { // 列表变更通知（所有客户端都能收到） await server.sendResourceListChanged();


// 资源内容变更通知 // 2026-era: SDK 自动路由到 listen 过滤器 // 2025-era: 只发送给显式订阅过的客户端 if (subscribedUris.has('memory://graph')) { await server.server .sendResourceUpdated({ uri: 'memory://graph' }) .catch(() => {}); } }

关键洞察：sendResourceUpdated 在两种时代都工作——SDK 内部处理路由差异。你只需要确保 (a) 显式订阅处理器存在（兼容 2025-era），且 (b) 只在有订阅者时发送通知（避免向未订阅的客户端发送垃圾消息）。

notifyGraphChanged() 应该在每个修改图状态的操作后调用——remember、forget、consolidate。这让客户端能够实时感知记忆变化，而不需要轮询。

进程内测试：零开销反馈环

MCP SDK v2 最大的测试改进是 handler.fetch 模式——完全在进程内测试服务器，不需要启动子进程或绑定端口：

import { Client, StreamableHTTPClientTransport } from '@modelcontextprotocol/client';

import { createMcpHandler } from '@modelcontextprotocol/server'; import { buildServer } from '../src/server.js';


async function setupTestClient() { // 从工厂创建 handler const handler = createMcpHandler(buildServer);


// 传输层直接调用 handler.fetch，不经过网络 const transport = new StreamableHTTPClientTransport( new URL('http://test.local/mcp'), { fetch: (url, init) => handler.fetch(new Request(url, init)) } );


const client = new Client({ name: 'test', version: '1.0.0' }); await client.connect(transport);


return { client, cleanup: async () => { await client.close(); await handler.close(); } }; }


// 测试用例：直接调用工具并验证结构化输出 test('memory.recall returns ranked results', async () => { const { client, cleanup } = await setupTestClient();


// 先写入几条记忆 await client.callTool({ name: 'memory.remember', arguments: { content: 'TypeScript is great for type safety', kind: 'fact' }, }); await client.callTool({ name: 'memory.remember', arguments: { content: 'Python excels at data science', kind: 'fact' }, });


// 搜索 const result = await client.callTool({ name: 'memory.recall', arguments: { query: 'type safety' }, });


// 验证结构化输出 expect(result.structuredContent.total).toBeGreaterThan(0); expect(result.structuredContent.memories[0].content) .toContain('TypeScript');


// 验证注解被正确传递 const tools = await client.listTools(); const recall = tools.tools.find(t => t.name === 'memory.recall'); expect(recall.annotations.readOnlyHint).toBe(true);


await cleanup(); });

这个测试策略的精妙之处在于：请求经过完整的 MCP 协议栈——序列化、路由、验证、handler 执行、结果验证——但没有碰任何 I/O。测试运行时间在 50-100ms 级别，可以在每次保存文件时运行。

对比传统的 MCP 测试方式（spawn 子进程 → stdio 通信 → 解析输出），handler.fetch 模式：

  • 快 10 倍（无进程创建开销）
  • 调试友好（断点直接打在 handler 里）
  • CI 友好（无端口冲突风险）

健康监控：不只是计数

最后一个设计决策是把图谱健康度暴露为一级工具，而不是隐藏的管理 API。

server.registerTool(

'memory.health', { title: 'Memory Graph Health', description: 'Assess memory graph quality: node count, density, gaps, ' + 'and overall health score (0-100).', outputSchema: z.object({ health_score: z.number(), node_count: z.number(), edge_count: z.number(), gap_count: z.number(), density: z.number(), verdict: z.string(), }), annotations: { readOnlyHint: true, destructiveHint: false, idempotentHint: true, openWorldHint: false, }, }, async () => { const h = mg.health(); return { content: [{ type: 'text' as const, text: 📊 Health: ${h.health_score}/100 (${h.verdict})\n +    Nodes: ${h.node_count} | Edges: ${h.edge_count}  + | Gaps: ${h.gap_count} | Density: ${h.density}, }], structuredContent: h, }; } );

为什么让 LLM 直接看健康度？因为记忆系统的健康度直接影响下游任务质量。当 Agent 发现 health_score 降到 40 以下，它可以选择：

  • 主动调用 memory.consolidate 清理冗余
  • 在后续对话中补充缺失的关联
  • 向用户报告"我的记忆可能不完整"

这是自我感知（Self-Awareness） 的基础——一个不了解自身状态的系统无法真正做出智能决策。

verdict 字段是刻意设计的自然语言输出（如 "sparse — add more connections"）。LLM 可以直接读取这句话并理解需要做什么，而不需要解析一个枚举值然后查找含义。

总结：三个原则

构建 MCP 记忆服务器的实践让我提炼出三个原则：

原则一：工具数量做减法，工具语义做加法。 6 个语义清晰的工具（recall/remember/health/forget/query/consolidate）远优于 9 个数据结构映射工具。LLM 需要的是动词层面的交互，不是 CRUD 的忠实复现。

原则二：协议能力优于应用代码。 注解提供安全（destructiveHint），Schema 提供验证（inputSchema/outputSchema），资源订阅提供实时性。这些都不需要你写应用层代码——它们是 MCP 协议内置的能力。善用协议，减少代码。

原则三：测试是架构的一部分。 handler.fetch 模式让进程内测试成为默认选择。如果你的 MCP 服务器不能通过这个模式测试，说明你的架构出了问题——可能是模块耦合太紧，或者副作用太多。

MCP 协议仍在快速演进。2026 年 7 月 28 日，SDK v2 将正式发布 subscriptions/listen 和 extensions 机制。记忆服务器的下一个前沿不是更强的检索算法，而是作为 MCP 扩展的分发渠道——一行配置安装，一行命令更新。

记忆作为基础设施的时代，刚刚开始。
