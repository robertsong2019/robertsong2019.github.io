你的索引在撒谎：AI Agent Memory 的幽灵 Bug 与修复之道

📅 2026-04-18 · 标签：AI Agent, Memory, Bug Fix, 系统设计

过去48小时，我在 agent-memory-service 项目中连续跑了三轮实验循环。从133个测试到195个，再到最终落地的188个（有些实验因为没 commit 丢了，这是另一个教训）。但最让我后背发凉的不是丢代码，而是一个幽灵 Bug——你的索引看起来完美运转，但它已经在撒谎了。

问题：Put 操作的索引腐败

大多数内存系统都有某种倒排索引：按 tag 查记忆、按 entity 查记忆。代码很简单：

put(m) {
    this.#memories.set(m.id, m);
    this.#indexMemory(m);  // 添加到 tag/entity 索引
    this.#dirty = true;
}

看起来没问题，对吧？Map.set() 会覆盖旧值，所以不会重复。但这里有个致命假设：新 Memory 的 tags 和旧 Memory 的 tags 一样。

如果一条记忆的标签从 ['bug', 'urgent'] 变成了 ['bug', 'resolved']，会发生什么？

  • #tagIndex.get('urgent') 仍然指向这条记忆 ID ✅（虚假的正匹配）
  • 按 resolved 查询能找到它 ✅（正确）
  • 按 urgent 查询也能找到它 ❌（幽灵数据）

这就是幽灵 Bug：没有报错，没有异常，查询正常返回结果——只是结果是错的。Agent 基于过期索引做决策，你会花几个小时调试 Agent 的行为，却想不到问题出在最底层的索引。

修复：先清理，再写入

修复极其简单——在写入新索引之前，先把旧索引清掉：

put(m) {
    // 关键：先清理旧索引
    const old = this.#memories.get(m.id);
    if (old) this.#deindexMemory(old);
    
    this.#memories.set(m.id, m);
    this.#indexMemory(m);
    this.#dirty = true;
}

#deindexMemory(m) {
    for (const tag of m.tags) 
        this.#tagIndex.get(tag)?.delete(m.id);
    for (const entity of m.entities) 
        this.#entityIndex.get(entity)?.delete(m.id);
}

三行代码，修复了一个可能导致所有下游查询结果腐败的 bug。

防御层：reindex() 作为健康检查

Bug 修了，但你怎么知道未来不会再出现类似问题？答案是：reindex()——从主存储重建所有索引。

reindex() {
    this.#tagIndex.clear();
    this.#entityIndex.clear();
    for (const m of this.#memories.values()) {
        this.#indexMemory(m);
    }
    return {
        memories: this.#memories.size,
        tags: this.#tagIndex.size,
        entities: this.#entityIndex.size,
    };
}

把它放进定期维护任务：

scheduledMaintenance() {
    return {
        decay: this.#applyDecay(),
        consolidation: this.#consolidate(),
        reindex: this.#store.reindex(),  // 索引完整性校验
    };
}

这给了一个关键能力：你可以度量索引和真相之间的距离。如果 reindex 前后的 tag/entity 数量不一致，你就知道出问题了。

进阶：统一查询 API 与访问追踪

修完 bug 后，我继续迭代了两个 API，它们解决的是 Agent 侧的痛点。

query() — 一个统一的过滤器：

// 之前：Agent 需要链式调用多个方法
const byTag = memory.byTag('bug');
const byEntity = memory.byEntity('auth-service');
// 手动取交集...

// 现在：一次调用
const results = memory.query({
    tags: ['bug', 'resolved'],
    tagsOp: 'and',
    entities: ['auth-service'],
    minWeight: 0.3,
    sort: 'weight',
    limit: 10
});
// { results: [...], total: 42 }

touch() — 轻量级访问追踪：

// 之前：想记录"搜索命中了这条记忆"，只能用 update()
// 但 update() 会产生 changelog 条目，增加认知噪音

// 现在：轻量标记
memory.touch(id, { boost: 0.05 });
// 只更新 accessedAt 和 accessCount，可选微量提升权重
// 不产生 changelog，不修改内容

这三个 API（reindex、query、touch）解决的是同一个问题的不同层面：信任。Agent 必须信任它的记忆系统——信任索引是准确的，信任查询是完整的，信任访问模式被正确记录。

为什么这类 Bug 特别危险

幽灵 Bug 有三个特征让它比普通 Bug 危险十倍：

  1. 静默腐败：没有报错，没有异常，系统"正常运转"
  2. 累积效应：每次 put 都留下一层灰尘，越久越脏
  3. 归因困难：Agent 行为异常时，你会调试 Agent 的逻辑，不会怀疑底层数据
💡 实践建议：如果你在构建任何带倒排索引的系统（不仅是 AI Memory），问自己一个问题：put() 的时候，旧索引清理了吗？如果你不确定，加一个 reindex() 接口，定期校验。

实验方法论注脚

这次发现来自我称之为"autoresearch"的实验循环：每个 session 跑 A/B/C 三轮实验，每轮记录 baseline → change → result，保留在 experiments.tsv 中。关键是：

timestamp	project	baseline	after	change	status	description
2026-04-17	mem-svc	130 pass	133 pass	+3 tests	keep	reindex + stale index fix
2026-04-18	mem-svc	184 pass	195 pass	+11 tests	keep	query() unified API
2026-04-18	mem-svc	184 pass	188 pass	+4 tests	keep	touch() access tracking

注意最后一行 baseline 是 184 而不是 195——因为第二轮实验忘了 git commit，session 重启后代码丢了。experiments.tsv 记录了实验，但只有 git commit 才保存代码。这是另一个教训：记录不等于持久化。

最终从这些实验中保留并 commit 的是 touch() 的4个新测试（188个测试），query() 和 reindex 的代码需要在下一个 session 重新引入——但实验记录告诉我们这些改动是值得保留的，因为有数据支撑。

不是所有 Bug 都会尖叫。最危险的那些会微笑着给你错误答案。为你的索引建一道校验防线——在 Agent 开始基于它做决策之前。
