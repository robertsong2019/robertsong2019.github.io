让 LLM 输出靠谱结构化数据的三层防线

2026-06-13 · LLM Structured Output TypeScript Reliability

用 LLM 生成结构化数据（JSON）是 Agent 开发中最普遍的需求，也是最频繁的故障源。模型会偶尔吐出多余文本包裹的 JSON、字段类型偏差、语义上荒谬但 schema 上合法的结果。过去三个月，我在 structured-output-toolkit 项目中用 autoresearch 方法迭代了 14 个 cycle，测试从 12 个增长到 178 个，源码从不到 200 行增长到 1284 行。今天这篇文章不讲怎么用这个库——讲的是构建结构化输出可靠性体系的三层设计思路，以及其中非显然的工程决策。

问题拆解：错误不是平面的

很多人把 LLM 结构化输出失败当作一个单一问题处理：「给个 schema，解析失败就重试」。这忽略了错误实际上分布在三个完全不同的层级：

  1. 语法层 — 模型输出了 markdown 代码块包裹的 JSON、嵌入了注释、或者干脆不是合法 JSON
  2. Schema 层 — JSON 合法，但字段缺失、类型错误、枚举值越界
  3. 语义层 — Schema 完全合法，但内容没有意义（空字符串、数值超出业务范围、数组缺元素）

关键洞察是：每一层的修复策略不同。语法层用正则提取 + 宽松解析就能解决；Schema 层需要 Zod 这类运行时类型检查；语义层只能靠业务自定义验证器。把它们混在一起处理（比如只靠 Zod），会在语义错误面前束手无策。

第一层：Validation Sandwich — 三明治验证管线

核心思路是短路管线：语法失败就不浪费时间检查 Schema，Schema 失败就不跑语义验证。每一层失败都产出结构化错误反馈，供后续重试使用：

// 三层验证管线 — 每层短路，失败即返回
export function validateSandwich(
  raw: string,
  schema: z.ZodType,
  semanticValidator?: SemanticValidator
): ValidationReport {
  // Layer 1: 语法 — 提取并解析 JSON
  const syntax = validateSyntax(raw);
  if (!syntax.ok) {
    return shortCircuit(syntax); // 不继续
  }

  // Layer 2: Schema — Zod 运行时类型检查
  const schemaResult = validateSchema(schema, syntax.data);
  if (!schemaResult.ok) {
    return shortCircuit(schemaResult); // 不继续
  }

  // Layer 3: 语义 — 自定义业务规则
  const semantic = semanticValidator
    ? validateSemantic(schemaResult.data, semanticValidator)
    : { ok: true, data: schemaResult.data };

  return { syntax, schema: schemaResult, semantic, passed: semantic.ok };
}

语义验证器可以组合使用，这是我用得最多的模式：

// 组合多个语义检查
const validator = validators.compose(
  validators.noEmptyStrings(["title", "summary"]),
  validators.numericRange([{ key: "score", min: 0, max: 100 }]),
  validators.minItems("tags", 2)
);

const report = validateSandwich(raw, MySchema, validator);

这里有一个非显然的设计：语义验证器可以携带 transform 函数，在验证通过的同时对数据做变换（比如标准化字符串、补全默认值）。这让验证层和清洗层合二为一，避免了在管线之外再写一个后处理步骤。

第二层：Consensus Voting — 多数票共识

单次生成的可靠性有上限。ICLR 2026 的 MAKER 论文给出了一个数学上很优雅的结论：如果单个模型在每一步的准确率是 p=0.9，那么 k=6 次独立投票取多数，在 1000 步任务上的整体准确率可以达到 99.8%。这是线性冗余换指数可靠性。

但投票有一个前提条件经常被忽略：你需要一个稳定的比较函数。两个 JSON 对象在语义上相同但 key 顺序不同时，直接 JSON.stringify 会产生不同的字符串。如果不对这个问题做特殊处理，投票系统会严重低估共识率。

解决方案是 canonicalize——递归排序所有 key：

function canonicalize(value: unknown): unknown {
  if (value === null || typeof value !== "object") return value;
  if (Array.isArray(value)) return value.map(canonicalize);
  const sorted: Record = {};
  for (const key of Object.keys(value).sort()) {
    sorted[key] = canonicalize(value[key]);
  }
  return sorted;
}

function stableHash(data: unknown): string {
  return JSON.stringify(canonicalize(data));
}

有了 stableHash 之后，投票逻辑就很直接了：生成 k 个候选 → 各自验证 schema → 对通过的候选计算 hash → 取出现次数最多的：

const candidates = await Promise.allSettled(
  Array.from({ length: k }, () =>
    withTimeout(generateFn(prompt, temperature), timeoutMs)
  )
);

// 验证 + hash → 统计共识
const hashCounts = new Map();
for (const result of candidates) {
  const validated = schema.safeParse(extractAndParse(result));
  if (validated.success) {
    const hash = stableHash(validated.data);
    hashCounts.set(hash, (hashCounts.get(hash) ?? 0) + 1);
  }
}

另一个实战细节：withTimeout 不可或缺。LLM 推理延迟方差极大，没有超时保护的 Promise.allSettled 会被一个 hang 住的请求拖垮整个投票轮次。

第三层：Error Recovery — 把验证失败变成自修正循环

这是整个体系中最有价值的一层。思路是：与其在验证失败后简单重试相同的 prompt，不如把具体的错误信息打包成反馈，喂给模型让它自己修正。

// generate → validate → 如果失败：构造错误反馈 → 重试
for (let attempt = 0; attempt <= maxRetries; attempt++) {
  const temp = attempt === 0 ? 0.1 : 0.5; // 温度递增
  const raw = await generateFn(currentPrompt, temp);
  const report = validateSandwich(raw, schema, semanticValidator);

  if (report.passed) return success(report);

  // 把验证错误变成 LLM 能理解的反馈
  if (attempt < maxRetries) {
    currentPrompt = buildRetryPrompt(prompt, report);
    // ↑ 生成："你的上一次回答有以下问题：
    //   - [Schema] score: Expected number, received string
    //   - [Semantic] Field 'tags' must have at least 2 items
    //   请修正这些问题，只输出 JSON。"
  }
}

温度递增是一个来自实践的发现：第一次用低温（0.1）获取最保守的输出；如果失败，逐步升高温度（0.3→0.5），给模型更多"创造空间"来绕过导致失败的模式。

当所有重试都耗尽时，还有一个常被忽略的策略：降级到部分有效结果。很多场景下，Schema 通过但语义失败的结果比完全失败更有价值。与其 throw 一个错误，不如在候选中挑出语义错误最少的那个作为 fallback：

// 所有重试耗尽 → 选最好的部分有效结果
if (fallbacks.length > 0) {
  const best = fallbacks.reduce((best, cur) =>
    cur.report.semantic.errors.length < best.report.semantic.errors.length
      ? cur : best
  );
  return { data: best.data, usedFallback: true, validated: false };
}

在 batchRecovery 的批量场景中，这个策略更加关键——100 个 prompt 中有 2 个彻底失败不应该中断整个批次。batchRecovery 用 per-item isolation + 并发控制来保证：单个 prompt 的 RecoveryError 不会影响其他 prompt 的处理。

可操作的结论

  • 不要只靠 Schema 验证。Zod 能帮你抓住类型错误，但抓不住空字符串和数值越界。加一个语义验证层，用 compose 组合业务规则。
  • 投票之前先解决 hash 稳定性。递归排序 key 是 5 行代码的事，但不做的话投票系统形同虚设。
  • 重试时给模型具体反馈，而不是简单重发 prompt。"你的输出缺少 required field: email"比"请重试"有效得多。温度递增（0.1→0.5）能覆盖更多输出空间。
  • 设计 fallback 而不是纯抛错。在批量场景中，Schema 通过但语义部分失败的结果往往够用。让它成为 usedFallback: true 的显式降级，而不是悄悄返回错误数据。
  • 每个候选都要有独立超时。LLM 推理延迟的长尾分布会让 Promise.all 变成陷阱。

数据说话

这套三层体系在 structured-output-toolkit 中经过了 178 个测试用例的验证（从 Cycle 1 的 12 个增长到 Cycle 14 的 178 个），覆盖了语法提取边界、Schema 校验、语义组合验证、共识投票、错误恢复全链路。三个核心模块（consensus.ts 206 行、validation-sandwich.ts 274 行、error-recovery.ts 210 行）总计约 700 行 TypeScript，零运行时依赖（除了 zod）。

可靠性工程的核心不是某一个银弹，而是多层防御的纵深。每一层覆盖前一层的盲区，每一层的失败都能被下一层接住。这和网络安全里的 defense-in-depth 是同一个思路——只不过这里的攻击者不是恶意黑客，而是 LLM 那偶尔跑偏的概率分布。
