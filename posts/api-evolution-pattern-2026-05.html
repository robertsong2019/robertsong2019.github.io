一个 Context Store 的24小时进化史：从25个测试到48个

Python API设计 Agent开发 TDD
2026-05-09 · 罗嵩

过去的24小时里，我的 AI Agent 在一个叫 agent-context-store 的项目上连续做了4次自主迭代。测试从25个涨到48个，新增7个API方法，零回滚。这不是一个关于"AI多厉害"的故事——这是一个关于API设计中的隐性债务，以及如何用小手术而非大重构来消除它的故事。

起点：一个看起来够用的 Store

ContextStore 是 Agent 的短期记忆层。核心接口很简洁：put(key, content, tags)、get(key)、search(query)、delete(key)。25个测试全绿。看起来没问题。

但问题是——Agent 在实际使用中反复撞墙。

第一刀：get() 的副作用陷阱

这是最阴险的一种设计缺陷。get("key") 在返回内容的同时，会顺带删除过期的条目。这意味着：

# 你只是想检查一个条目是否存在
entry = store.get("config")
# 但如果它过期了，它已经被删了！

# 你想搜索未过期的条目
for key in store.list():
    entry = store.get(key)  # 每次get都可能删掉别的条目！
    # list()返回的key在你遍历时已经失效了

这不是 bug，是设计债。get() 承担了"读取"和"清理"两个职责。

修复：exists(key) — 纯检查，无副作用。

def exists(self, key: str) -> bool:
    entry = self._data.get(key)
    return entry is not None and not entry.expired
设计原则：检查状态的操作不应该改变状态。这不是新发现——C++ 的 const 方法、React 的纯组件都在说同一件事。但当你的 get() 里混入了副作用，这条原则就被静默违反了。

第二刀：多标签过滤的无奈

Agent 给条目打了多个标签（python、ai、production），但 list(tag="python") 只支持单标签。要找"既是 Python 又是生产环境"的条目？手动遍历。

search_by_tags(tags, match_all=False) 解决了这个问题。关键设计决策是 match_all 参数——默认 OR，可选 AND：

def search_by_tags(self, tags, match_all=False):
    results = []
    tag_set = {t.lower() for t in tags}
    for key, entry in self._data.items():
        if entry.expired:
            continue
        entry_tags = {t.lower() for t in entry.tags}
        if match_all:
            if tag_set.issubset(entry_tags):
                results.append((key, entry))
        else:
            if tag_set & entry_tags:
                results.append((key, entry))
    return sorted(results, key=lambda x: x[1].updated_at, reverse=True)
观点：默认 OR 而非 AND，因为 OR 更宽容——漏掉结果比错过结果好。这在搜索交互中是常识（Google 就是 OR 语义），但在 API 设计中经常被忽略。

第三刀：retag() — 为什么改个标签这么难？

想给已有条目加个标签？流程是：get() 内容 → delete() 条目 → put() 新标签。副作用：created_at 被重置，条目变"新"了。

retag(key, add_tags, remove_tags) 是一个15行的方法，但它解决了一个真实痛点——元数据操作不应该要求重建数据。

def retag(self, key, add_tags=None, remove_tags=None):
    entry = self._data.get(key)
    if not entry or entry.expired:
        return None
    tags = set(entry.tags)
    if add_tags:
        tags.update(add_tags)
    if remove_tags:
        remove_lower = {t.lower() for t in remove_tags}
        tags = {t for t in tags if t.lower() not in remove_lower}
    entry.tags = sorted(tags)
    entry.updated_at = datetime.now()
    self._save()
    return entry

注意大小写不敏感的删除：remove_tags=["python"] 能删掉 "Python"。这种细节在 API 文档里容易被忽略，但在使用中是高频困惑点。

第四刀：正则搜索与内容追加

最后两轮迭代加了 search_regex(pattern, search_fields) 和 append(key, text)。

search_regex 的价值在于：子串搜索无法表达模式。你要找所有版本号？search("version") 会匹配"Version 2.3.1"和"version control"，但你真正想要的是 version \d+\.\d+\.\d+。

# 精确匹配 semver 格式
results = store.search_regex(r"v?\d+\.\d+\.\d+", search_fields=["content"])

# 只在 key 里搜索
results = store.search_regex(r"^config\.", search_fields=["key"])

append 解决的是 Agent 的日志场景：Agent 在执行任务时不断追加状态更新到同一个 key。之前每次追加都要 get() + put()，风险是 created_at 丢失。现在：

store.append("task-001", "步骤3完成：数据清洗")
# created_at 不变，content 自动用换行连接

模式：手术式 API 扩展

回头看这24小时，有一个清晰的模式：

  1. 发现痛点：Agent 在使用中反复做 "get → modify → put" 的 workaround
  2. 识别根因：不是缺功能，是接口职责混乱（get 带副作用）或缺失（没有 tag 编辑）
  3. 最小手术：加一个方法，10-35行代码，不改已有代码
  4. 测试驱动：先写测试描述期望行为，再实现
  5. 验证无回归：全量测试跑过才提交

这不是大重构。每个方法都是独立的、正交的、不互相依赖的。它们可以一个一个地加，也可以一个一个地删，不会影响已有功能。

三个可操作的建议

1. 审计你的 get() 方法族。 如果 get() 有任何副作用（缓存失效、状态变更、事件触发），加一个纯查询方法。这是最容易藏 bug 的地方。

2. 元数据操作应该独立于数据操作。 改标签不应该要求重建内容。改 TTL 不应该要求重写内容。如果你发现 "改 X 要先 get 再 put"，那就是一个 API 缺口。

3. 默认宽松，选项严格。 search_by_tags 默认 OR。search_regex 默认搜所有字段。append 默认换行分隔。让简单场景一行代码搞定，复杂场景通过参数打开。这就是 "Make the common case fast" 在 API 层面的体现。

结语

48个测试，7个新方法，24小时。没有一次重写，没有一次回滚。每个方法都是在使用中"撞墙"后的精确修补。这不是炫技——这是 API 设计最朴素的原则：让调用者说人话，不要让调用者翻越你设计的障碍。

好的 API 不是设计出来的，是用出来的。
