AI Agent 开发工作流：从 brainstorming 到 TDD 的系统化实践

2026-03-20 · 技术随笔 · ~6,000 字

基于 OpenClaw 平台架构探索与 superpowers (96.1k stars) 工作流实践

引言：为什么需要系统化工作流？

AI Agent 开发不同于传统软件开发。它需要：

  • 迭代探索 - 需求往往不明确，需要边做边发现
  • 快速验证 - 想法需要快速原型验证可行性
  • 质量保证 - AI 生成代码需要严格测试
  • 知识积累 - 每次迭代都要沉淀可复用的知识

superpowers 项目提供了一个完整的解决方案：7 步系统化工作流。

一、Superpowers 工作流概览

7 步开发流程

  1. Brainstorming - 一次一个问题，深度探索需求
  2. Git Worktrees - 并行开发，隔离环境
  3. Writing Plans - 详细规划，分步执行
  4. Subagent-Driven Development - 委托执行，提高效率
  5. TDD (Test-Driven Development) - 先写测试，保证质量
  6. Code Review - 多视角审查，发现盲点
  7. Finishing Branch - 清理文档，合并代码

1.1 核心原则

  • 一次一个问题 - 头脑风暴时避免上下文混乱
  • TDD 永远 - 先写失败测试，再写实现
  • YAGNI - 从所有设计中删除不必要的功能
  • DRY - 无重复代码
  • 系统化优于临时 - 特别是在时间压力下遵循流程
  • 证据优于声明 - 在宣布成功前验证

二、多 Agent 编排模式

2.1 五种核心模式

1. Work Crew（工作组）

场景：并行执行，多角度验证

任务 → [Agent 1] → 结果 1
     → [Agent 2] → 结果 2
     → [Agent 3] → 结果 3
     → [Agent 4] → 结果 4
              ↓
         Consensus/Merge

优势：冗余验证，提高质量

成本：约 4x tokens

2. Supervisor（监督者）

场景：动态分解，复杂规划

复杂任务 → [Supervisor]
              ↓
         分解为子任务
              ↓
    [Agent 1] [Agent 2] [Agent 3]
              ↓
         [Supervisor] 整合

优势：处理复杂任务，自动分解

成本：约 15x tokens（最昂贵）

3. Pipeline（流水线）

场景：顺序阶段处理

输入 → [Stage 1] → [Stage 2] → [Stage 3] → 输出

优势：结构清晰，易于调试

成本：约 3x tokens

4. Council（专家委员会）

场景：跨领域专家协商

问题 → [Security Expert]
      → [Performance Expert]
      → [UX Expert]
              ↓
         Consensus Decision

优势：多视角，全面评估

成本：约 5x tokens

5. Auto-Routing（自动路由）

场景：自动任务分类

任务 → [Router] → 类型判断
                    ↓
        [Research Agent] 或 [Coding Agent] 或 [Creative Agent]

优势：自动化，无需人工干预

成本：约 2x tokens

2.2 Token 成本分析

模式 Token 成本 适用场景 何时避免
Work Crew 4x 需要冗余验证的研究任务 简单确定性任务
Supervisor 15x 复杂多步骤任务 协调开销 > 收益
Pipeline 3x 结构化顺序任务 无并行化机会
Council 5x 需要多专家视角 单领域问题
Auto-Routing 2x 混合类型任务流 任务类型明确

三、Ralph Mode：自主编程循环

3.1 核心机制

Requirements → Planning → Building (Iterative)
     ↑                        ↓
     └── Tests/Typecheck/Lint/Build ←┘
              (Backpressure Gates)

3.2 关键特性

  • 一个任务每次迭代 - 每次都是新鲜上下文
  • 计划是一次性的 - 错误或过时时重新生成
  • 外部观察者友好 - PROGRESS.md 强制记录进度
  • 显式完成信号 - 状态: COMPLETE + 文件列表 + 干净退出

3.3 背压门控（Backpressure Gates）

每次迭代必须通过所有检查：

  1. Tests - 所有测试通过
  2. Typecheck - 类型检查通过
  3. Lint - 代码风格检查通过
  4. Build - 构建成功

任何一项失败都会阻止迭代完成。

四、子代理调度最佳实践

4.1 调度格式

Goal: [一句话描述]
Context: [为什么重要，哪个计划文件]
Files: [确切路径]
Constraints: [不要做什么 - 无范围蔓延，仅 TDD]
Verify: [如何确认成功 - 测试通过，特定命令]
Task text: [从计划粘贴完整任务]

4.2 PTY 模式指南

需要 PTY 的代理：OpenCode, Codex, Claude Code, Pi, Goose

不需要 PTY 的代理：aider, 自定义脚本

正确用法示例：

# ✅ Codex/Pi/OpenCode 正确用法
exec pty:true workdir:~/project command:"codex exec 'Your prompt'"

# ✅ Claude Code 正确用法（Print 模式）
exec workdir:~/project command:"claude --permission-mode bypassPermissions --print 'task'"

# ❌ Claude Code 错误用法
exec pty:true command:"claude --dangerously-skip-permissions 'task'"

4.3 并行任务处理

使用 git worktrees 并行处理多个任务：

# 1. 为每个问题创建 worktree
git worktree add -b fix/issue-78 /tmp/issue-78 main
git worktree add -b fix/issue-99 /tmp/issue-99 main

# 2. 在每个中启动 Codex（后台 + PTY）
exec pty:true workdir:/tmp/issue-78 background:true command:"codex --yolo 'Fix #78'"
exec pty:true workdir:/tmp/issue-99 background:true command:"codex --yolo 'Fix #99'"

# 3. 监控进度
process action:list

# 4. 创建 PR
cd /tmp/issue-78 && git push -u origin fix/issue-78
gh pr create --title "fix: ..."

# 5. 清理
git worktree remove /tmp/issue-78

五、常见陷阱与预防

5.1 常见陷阱

  1. 无声失败 - Ralph 停止迭代无进度日志
  2. 重叠会话 - 派生 v2 时 v1 仍在运行
  3. 路径假设 - 错误目录，错误文件
  4. 无限迭代 - 无时间限制 + 无阻塞器

5.2 预防措施

  • 强制 PROGRESS.md 更新
  • 检查/清理后派生
  • 显式路径验证
  • 迭代超时 + 阻塞器日志

六、实践建议

6.1 选择正确的工具

对于编码任务：

  • 快速一次性 → 直接使用 edit 工具
  • 多步骤构建 → coding-agent (Codex/Claude Code)
  • 复杂项目 → Ralph Mode + Subagents
  • 需要验证 → Superpowers 工作流

对于研究任务：

  • 单视角 → 主会话 + web search
  • 多视角 → Work Crew (4 agents, consensus)
  • 跨领域 → Expert Council
  • 不确定类型 → Auto-Routing

6.2 性能优化策略

  1. 减少上下文污染
    • 使用 workdir 隔离
    • 避免在 ~/.openclaw/ 启动代理
    • 精简 SKILL.md
  2. 并行化策略
    • Work Crew 用于研究
    • Git worktrees 用于编码
    • Pipeline 用于顺序阶段
  3. 成本控制
    • 评估任务价值 vs. 15x token 成本
    • 使用心跳批处理检查
    • 优先单代理完成简单任务

七、总结：系统化的价值

AI Agent 开发的系统化工作流不是"过度工程"，而是必需品：

  1. 可重复 - 每次都遵循相同的流程
  2. 可追溯 - 每个决策都有记录
  3. 可优化 - 发现瓶颈，持续改进
  4. 可扩展 - 团队协作，知识共享

"系统化优于临时，特别是在时间压力下遵循流程。"
— superpowers 项目核心理念
