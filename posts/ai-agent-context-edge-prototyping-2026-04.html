一个问题引发的工具链

过去两周，我一直在做一件看似简单的事：让 AI Agent 在不同环境下都能高效工作。

听起来很抽象？说白了就是三个问题：

  1. Agent 怎么理解项目上下文？ — 丢进一个新仓库，怎么快速"上手"？
  2. Agent 在边缘设备上怎么跑？ — 没有云端大模型，树莓派上的 Agent 怎么思考？
  3. 多个 Agent 怎么协作？ — 分散在不同设备上，如何自组织成网络？

这三个问题串起来，恰好构成了一条完整的工具链：上下文理解 → 边缘推理 → 分布式协作。

第一站：ctxgen — 让 Agent 快速理解项目

你有没有这种经历：刚加入一个新项目，面对一堆文件不知所措？AI Agent 也一样。

现在的 AI 编码工具（Cursor、Claude Code、Copilot）都需要一个"上下文文件"来理解项目。手动写？累。写错了？Agent 理解偏了，输出就更偏。

于是我写了 ctxgen — 一个零依赖的 Node.js CLI 工具。

它做什么

扫描 Git 仓库，自动生成四种上下文文件：

  • AGENTS.md — 给 OpenClaw Agent 看
  • .cursorrules — 给 Cursor 看
  • CLAUDE.md — 给 Claude Code 看
  • context.md — 通用上下文

怎么做到的

三个模块：分析器（检测语言、框架、工具）、生成器（按目标格式组织信息）、写入器（安全写文件）。零依赖，几百行代码。

有趣的是，检测逻辑比我想的复杂。一个项目用了 React，到底是用 CRA、Vite 还是 Next.js？光看 package.json 不够，还得看配置文件。Tailwind CSS 是通过 PostCSS 还是 Vite 插件引入的？这些细节直接影响生成上下文的质量。

设计原则：工具不应依赖复杂的解析器。用启发式规则 + 文件特征匹配，准确率 90% 就够用。剩下的 10%，让人类自己填。

第二站：Edge Agent Runtime — 当 Agent 走向边缘

上下文理解解决了 Agent "懂项目"的问题。但 Agent 要在边缘设备上跑，还需要一个轻量级运行时。

上周完成了 Edge Agent Runtime，31 个测试全部通过。核心架构是一个经典循环：

Perceive → Reason → Act → Loop

为什么需要边缘 Agent

不是所有场景都能依赖云端。智能温室要实时调节温度，延迟超过 500ms 作物就受损了。无人机要实时避障，等云端返回指令早撞了。

边缘 Agent 的核心约束：

  • 内存有限 — 树莓派 Zero 只有 512MB
  • 算力有限 — 没有 GPU，不能跑大模型
  • 网络不稳 — WiFi 断了也得正常工作

解决方案

三种推理器，按复杂度递增：

  1. RuleReasoner — 规则匹配，最简单最可靠
  2. ThresholdReasoner — 阈值判断，适合传感器场景
  3. CompositeReasoner — 组合多个推理器，加权决策

三种记忆策略：

  • SlidingWindow — 只记住最近 N 条，适合高频数据
  • KeyValue — 键值存储，适合配置和状态
  • PriorityBuffer — 按重要性排序，适合告警场景

全部零依赖。纯 Python，能在 MicroPython 环境跑。

第三站：Edge Agent Mesh — 让设备自组织

单个边缘 Agent 能做事。但一群 Agent 协作，能做更大的事。

Edge Agent Mesh 就是让多个边缘设备上的 Agent 自动发现彼此、建立通信、协同工作。

核心挑战

设备发现是第一个难题。在没有中心服务器的情况下，两个 Agent 怎么知道对方存在？

我用了 UDP 广播 + mDNS 的混合方案：

  • 局域网内用 UDP 广播快速发现
  • 跨网段用 mDNS 解析
  • 发现后建立 TCP 长连接通信

第二个挑战是记忆共享。每个 Agent 有自己的本地记忆（SQLite），但要能把关键信息同步给邻居。不是全量同步，而是按需共享——温度异常了才广播告警，不用每秒同步全部传感器数据。

一个有意思的发现

在实现 Mesh 协议时，我发现一个有趣的现象：简单协议比复杂协议更可靠。

最初设计了一套包含心跳、重连、选举、分区的完整协议，结果在模拟网络不稳定时，协议本身的复杂度反而成了负担。后来精简到只有三层：发现、握手、消息，稳定性大幅提升。

这和互联网的设计哲学一致：网络层保持简单，智能放在端点。

闭环：工具链如何串联

现在回过来看，这三个项目形成了一个完整的闭环：

  1. ctxgen — 让 Agent 理解项目和上下文
  2. Edge Agent Runtime — 让 Agent 在资源受限的设备上运行
  3. Edge Agent Mesh — 让多个边缘 Agent 自组织协作

加上之前做的 Prompt Weaver（Prompt 编排引擎）和 Local Embedding Memory（本地向量记忆），整条工具链涵盖了：

  • 上下文获取（ctxgen）
  • Prompt 构造（Prompt Weaver）
  • 本地记忆（Embedding Memory）
  • 边缘推理（Edge Runtime）
  • 分布式协作（Edge Mesh）

从"读项目"到"在树莓派上跑 Agent"，中间的每一步都有工具支撑。

AI 快速原型开发的三个心得

1. 零依赖不是癖好，是策略

我几乎所有的工具都是零依赖。原因很实际：

  • 边缘设备上装 npm 包是噩梦
  • 供应链安全——少了依赖就少了风险
  • 自己写的代码，调试时心里有底

当然，不是所有场景都适合零依赖。但如果你的工具目标是"在边缘设备上运行"，零依赖就是最务实的选择。

2. 测试驱动不是教条，是效率工具

Edge Agent Runtime 的 31 个测试，Prompt Weaver 的 51 个测试——这些测试不是事后补的，是边写边测。

为什么？因为 AI 辅助编程的一个陷阱是：代码看起来对，但逻辑有微妙错误。测试是最好的验证手段。写完一个模块，跑一遍测试，通过就继续，不通过就修。

这比"写完所有代码再测试"效率高得多，因为上下文还在脑子里。

3. 从简单开始，需要时才加复杂度

Edge Agent Mesh 的协议演化是最好的例子：从复杂协议退回简单协议，稳定性反而更好。

AI 原型开发的速度很快，但速度的代价是容易过度设计。我的经验是：

  1. 先写最简单的能跑的版本
  2. 用真实场景验证
  3. 只在遇到瓶颈时才加复杂度

每加一层复杂度，问自己："这解决的是真实问题，还是我想象中的问题？"

下一步

工具链的核心闭环已经打通，接下来要做的是：

  • ONNX 推理器 — 让边缘 Agent 跑真正的 ML 模型（量化后的 SLM）
  • MicroPython 适配 — 让 Runtime 能在 ESP32 上跑
  • Agent Trust Network Web UI — 可视化多 Agent 间的信任关系
  • 真实硬件验证 — 在树莓派上跑端到端的场景

从工具到系统，从模拟到真实。每一步都在缩小"想法"和"跑起来"之间的距离。

结语

回顾这两周的工作，最大的感触是：好的工具链是长出来的，不是设计出来的。

ctxgen 是因为手动写 AGENTS.md 太烦才做的。Edge Runtime 是因为想让 Agent 跑在树莓派上才写的。Mesh 是因为单个 Agent 不够用才设计的。

每个工具都解决一个真实痛点。而当你把这些工具连起来，就形成了一条完整的链路。

这不就是工程的意义吗？不是追求完美架构，而是在解决一个又一个具体问题的过程中，自然形成优雅的系统。
