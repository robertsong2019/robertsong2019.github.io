58天零回滚：AI Agent 编程的方法论进化

2026-05-17 · AI Agent autoresearch 零回滚

上次我写"30天零回滚"时，有人说"有意思，看能持续多久"。现在是第58天，1327次测试提交，零回滚。方法论没有变，但理解变深了。今天不讲概念，讲两个最近24小时内实际落地的设计决策——以及它们背后的思维模式。

核心纪律：为什么零回滚不是运气

autoresearch 的每轮循环遵循一个简单规则：

选一个有明确问题定义的缺口 → 写测试先行 → 实现 → 跑全量测试 → 全过才 commit，不过就回退到 commit 前

这个规则听起来平淡，但它的力量在于约束了 AI 的自由度。LLM 最大的敌人不是能力不足，而是"顺手多改了点"。每次只解决一个明确的问题，测试定义成功的边界——这让错误变成了可检测的、可回退的。

58天里，我们的测试从 ~200 个增长到 1500+，横跨 12 个项目。这些测试不是装饰，它们是 experiments.tsv 里每行"keep"决策的依据。

案例一：story_digest() — 当状态报告本身就是功能

better-ralph 是我们的 PRD 驱动编码框架。问题很具体：要获取当前进展快照，你需要调用 get_status()、get_session_summary()、estimate_remaining()，然后手动拼装结果。没有"一句话告诉我现在怎样了"的方法。

这不是性能问题，是认知负载问题。人类需要一眼看到：做了多少、跳了多少、接下来该做什么。Agent 也一样。

def story_digest(self) -> Dict[str, Any]:
    stories = self.prd_manager.get_all_stories()
    completed = [s for s in stories if s.passes and not s.notes.startswith("[SKIPPED]")]
    skipped = [s for s in stories if s.passes and s.notes.startswith("[SKIPPED]")]
    pending = [s for s in stories if not s.passes]

    top_pending = sorted(pending, key=lambda s: s.priority)[:5]

    digest = {
        "total": len(stories),
        "completed": len(completed),
        "pending": len(pending),
        "skipped": len(skipped),
        "completion_pct": round(len(completed) / max(len(stories), 1) * 100, 1),
        "top_pending": [
            {"id": s.id, "title": s.title, "priority": s.priority}
            for s in top_pending
        ],
        "is_complete": len(pending) == 0,
    }

    if self.current_session_id:
        digest["session"] = {
            "session_id": self.current_session_id,
            "iterations": self.session_stats.total_iterations,
        }

    return digest

几个值得注意的设计选择：

  • 跳过和完成是分开统计的。[SKIPPED] 前缀区分了"做完了"和"决定不做"。混在一起会让完成率失真。
  • top_pending 只取5个，按优先级排序。不是返回所有 pending——那跟直接看列表没区别。5个够了，多一个都是噪音。
  • session 信息是可选的。不在 session 中调用也能用，不返回空值污染结果。

8个测试，307/307 全过，commit fdb9e3a。一个 commit，一个问题，零副作用。

案例二：事件钩子 — 当"用的时候再加"刚刚好

agent-context-store 是一个轻量级的键值存储。之前所有操作都是同步的——你 put 了之后什么也不会发生。问题：消费者想对变更做出反应（比如自动同步、校验），只能轮询或手动包装。

解决方案是最简的 pub/sub，~25 行代码：

def on(self, event: str, callback: callable) -> None:
    """Register callback for event. Events: 'put', 'delete', 'expire'."""
    self._hooks.setdefault(event, []).append(callback)

def off(self, event: str, callback: callable | None = None) -> int:
    """Remove hook. If callback is None, remove all hooks for event."""
    if event not in self._hooks:
        return 0
    if callback is None:
        n = len(self._hooks[event])
        del self._hooks[event]
        return n
    try:
        self._hooks[event].remove(callback)
        return 1
    except ValueError:
        return 0

def _emit(self, event: str, **kwargs) -> None:
    for cb in self._hooks.get(event, []):
        cb(event=event, **kwargs)

这个设计的精髓不在于做了什么，在于没做什么：

  • 没有 Event 类、没有 EventListener 接口、没有优先级排序
  • 没有异步（callback 是同步调用）
  • 没有通配符或模式匹配
  • off() 返回删除数量而不是抛异常——静默失败，因为取消一个不存在的钩子不是错误

有人在 code review 时会说"怎么不用 EventEmitter 模式？"但这就是 Karpathy 说的：当你需要时再添加功能，而不是当你认为可能需要时。现在有人需要通配符吗？没有。那就没有。

58天教会我的三件事

1. 测试是 AI 编程的护栏，不是装饰

每个功能先写测试，意味着你在实现之前就定义了"成功"。对 LLM 来说，"让测试通过"比"实现功能"是更好的指令——前者可验证，后者永远模糊。

2. 小到无聊的改动最安全

58天里没有哪次提交是"激动人心的架构升级"。每一次都是 +8 tests、+12 tests。但累积起来，307 个测试覆盖了一个完整的 PRD 引擎。积少成多不是鸡汤，是工程事实。

3. 约束比能力重要

LLM 能力很强，所以更需要约束。autoresearch 的规则本质上是一个 meta-prompt：不是告诉 AI "写好代码"，而是告诉它"在什么边界内写代码"。边界越清晰，结果越可预测。

可操作建议

如果你也在用 AI Agent 做自主编码，试试这套：

  1. 维护一个 experiments.tsv，记录每次实验的时间、测试数、结果、决策（keep/improve/revert）。这不仅是日志，是训练数据。
  2. 一个 commit 解决一个问题。 tempted to 顺手改？别。写下来，下次循环处理。
  3. 先写测试，定义成功标准。"让 X 测试通过"比"实现 X 功能"更适合 AI 执行。
  4. 功能能少就少。event hooks 用 dict[str, list[callable]] 就够了，不需要 EventEmitter 类。当你真的需要时再加。
  5. 每天检查 experiments.tsv，回滚率的趋势比单次结果更重要。一旦出现连续回滚，说明方法论本身需要调整。

58天。零回滚。不是因为代码写得完美，而是因为不完美的代码根本不会进入代码库。

下一个目标：100天。如果方法论是对的，不需要运气。
