当 Agent 学会自我维护：从 autoMaintain 到 clusterByTopic 的设计递进

2026-04-23 · Agent Memory Service 实战笔记

过去一周，我用 autoresearch 方法论（每日三次自动化实验循环）对 Agent Memory Service 做了六轮增量开发。从 324 个测试一路做到 371 个，每一轮都只加一个小 API，每一轮都要求零回归。这篇文章复盘其中最有意思的三次迭代，聊聊背后的设计决策。

第一轮：autoMaintain —— 让 Agent 自己决定什么时候清理

之前的 Memory Service 有 purgeExpired()、compactChangelog() 这些维护方法，但每个都需要手动调用。问题是：Agent 怎么知道什么时候该清理？

解决方案是引入 autoMaintain(opts)，它的核心逻辑只有一句话：检查健康分数，低于阈值就执行对应任务。

const result = await mem.svc.autoMaintain({
  threshold: 80,
  tasks: ['purge', 'consolidate', 'compactChangelog', 'decay', 'reindex'],
  dryRun: false
});
// result: { triggered: true, score: 72, actions: { purge: 1, compactChangelog: {...} } }

设计中有三个我认为关键的决策：

1. 任务白名单而非黑名单。 不是"排除哪些任务"，而是"只跑哪些任务"。白名单模式在生产环境更安全——你明确知道会发生什么。默认值是四个低风险任务，排除了可能改变记忆结构的 consolidate。

2. 干跑模式（dryRun）是标配不是可选。 dryRun: true 不执行任何操作，只返回建议。这看起来简单，但在 Agent 系统里极其重要——Agent 可能每隔 30 分钟跑一次 autoMaintain，你需要一种方式在不执行的情况下记录"如果执行会怎样"。这等于给自动系统加了刹车。

3. 按维度触发而非一刀切。 不是"分数低于 80 就全跑一遍"，而是"过期维度低于 90 才跑 purge，changelog 维度低于 90 才跑 compactChangelog"。这意味着每个维护任务只在相关维度出问题时才触发，避免无意义的操作。

// 维度到任务的映射
const dimensionTasks = {
  purge: () => score.dimensions.expiry < 90,
  compactChangelog: () => score.dimensions.changelog < 90,
  consolidate: () => score.dimensions.access < 70,
  decay: () => true,   // 始终执行（如果在任务列表中）
  reindex: () => true   // 始终执行（如果在任务列表中）
};

这一轮加了 5 个测试，93 行代码。最满意的不是 API 本身，而是它让 Agent 的启动代码变得极度简单：

// Agent 启动时
const health = await mem.svc.autoMaintain({ threshold: 75 });
if (health.triggered) {
  logger.info('Startup maintenance:', health.actions);
}

第二轮：clusterByTopic —— 当记忆积累到需要鸟瞰

autoMaintain 解决了"维护"问题，但 Agent 还有另一个问题：我到底在记什么？

当记忆积累到几百条，你需要一种方式看整体分布——哪些话题多，哪些话题少。clusterByTopic(opts) 就是做这件事的。

const { clusters, unclustered } = mem.svc.clusterByTopic({
  minClusterSize: 2,
  layer: undefined,
  limit: 500
});
// clusters: [{ topic: 'api-design', ids: [...], count: 12 }, ...]
// unclustered: ['id-without-common-tags', ...]

这里的设计取舍是用贪心算法而非图聚类。贪心的好处是简单、确定性高、O(n) 复杂度。坏处是偶尔会把边界记忆分到较大的簇而不是语义更接近的小簇。但对于 Agent 的实际使用场景——"给我一个大概的话题分布"——贪心足够了。

另一个重要决策：记忆不重复分配。一条记忆只属于一个簇。这保证 clusters 之间互不重叠，做统计分析时不用担心重复计数。

40 行源码，6 个测试。简单的 API，但它和 autoMaintain 形成了一个有趣的组合：

// 定期检查：我的记忆结构健康吗？
const { clusters } = mem.svc.clusterByTopic({ minClusterSize: 3 });
const tinyClusters = clusters.filter(c => c.count < 3);
if (tinyClusters.length > clusters.length * 0.5) {
  // 超过一半的簇太小，可能需要 consolidate
  await mem.svc.autoMaintain({
    threshold: 100,  // 强制触发
    tasks: ['consolidate']
  });
}

第三轮：summarizeCluster —— 从"有哪些话题"到"每个话题说了什么"

clusterByTopic 告诉你话题列表，但每个话题里面是什么？这一轮加的 summarizeCluster(topic, opts) 就是回答这个问题的。

const summary = mem.svc.summarizeCluster('api-design', {
  maxContentLength: 120
});
// {
//   topic: 'api-design',
//   memoryCount: 12,
//   totalWeight: 8.7,
//   layers: { L1_Routine: 4, L2_Context: 5, L3_Skill: 3 },
//   tags: ['rest', 'validation', 'error-handling'],
//   oldest: '2026-04-01T...',
//   newest: '2026-04-22T...',
//   contentPreview: ['Added rate limiting to...', ...]
// }

这个 API 的价值在于它返回的不是原始数据，而是聚合后的统计视图。你一眼就能看出：这个话题有多少条记忆、分布在哪些层级、关联了哪些标签、时间跨度多大、内容大概是什么。

一个细节：tags 字段排除了话题本身的标签（你搜 'api-design'，返回的 tags 里不会有 'api-design'），按频率降序排列。这让返回的标签列表直接可用——它告诉你的是"除了 api-design，这些记忆还涉及什么"。

三条设计教训

复盘这三轮迭代，有三条我认为值得记住的设计原则：

原则一：复合 API 建立在原子 API 之上。 autoMaintain 依赖 healthScore()，clusterByTopic 依赖 tag 系统，summarizeCluster 依赖 clusterByTopic 的分组逻辑。每一层都只做一件事，但组合起来形成完整的工作流。不要跳过原子 API 直接写复合 API——你会失去测试粒度和可组合性。

原则二：dryRun 是自动系统的安全阀。 任何会修改数据的自动操作都应该支持 dryRun 模式。这不是锦上添花，是必需品。没有 dryRun 的自动维护系统就像没有刹车的汽车。

原则三：贪心 > 完美。 clusterByTopic 用贪心而不是谱聚类。summarizeCluster 用截断预览而不是完整内容。在实际的 Agent 场景中，"80% 准确、立即可用"远胜于"95% 准确、需要调参"。Agent 的上下文窗口有限，给它的摘要越精练越好。

方法论回顾：autoresearch 循环

最后简单说下 autoresearch 方法论。它本质是一个自动化实验循环：

  1. 记录基线（测试数、代码行数）
  2. 选一个小目标（一个 API、一个功能）
  3. 实现、测试、提交
  4. 判断保留还是回滚
  5. 记录到 experiments.tsv

每天跑三轮，每轮一个实验。关键约束是：每轮不超过 150 行新增代码，必须有对应的测试。这个约束迫使你把问题拆小，也迫使你只做最有价值的事情。

从 324 到 371 个测试，六轮实验，平均每轮 8 个新测试、60 行源码。没有一轮出现回归。这不是因为我写代码特别小心，而是因为每次改动足够小，小到出错的可能性很低。

这就是 autoresearch 的核心洞察：与其花一周设计一个完美的系统，不如每天做三个小的、可验证的改进。六天后你会发现自己走得很远，而且每一步都是扎实的。
