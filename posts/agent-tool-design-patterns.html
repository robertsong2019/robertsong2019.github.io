AI Agent 编程

为什么工具设计至关重要？

原始模型的局限性

LLM 只有权重和当前上下文。它无法：

  • 访问外部信息（当前时间、实时数据、文件系统）
  • 执行实际操作（发送消息、调用 API、修改文件）
Model + Tools = Agent
模型（智能） + 工具（能力） = 智能体

LangChain 的观点：Harness 是所有不是模型本身的代码、配置和执行逻辑。 原始模型不是 Agent，但当 Harness 给它状态、工具执行、反馈循环和可执行约束时，它就变成了 Agent。

工具设计的三层抽象

Layer 3: Orchestration（编排层）

  • 工具链（Tool Chains）
  • 工具选择策略
  • 错误处理与回退

Layer 2: Interface（接口层）

  • 参数验证
  • 结果解析
  • 错误处理

Layer 1: Implementation（实现层）

  • 核心逻辑
  • 外部 API 调用
  • 文件系统操作

AgentSkills 规范深度解析

AgentSkills 是一个开放的标准，定义了如何为 AI Agent 设计技能/工具。

核心原则

  1. 渐进式披露 - 只加载描述，完整指令按需加载
  2. 相对路径引用 - 从技能目录引用脚本和资源
  3. 前置元数据驱动 - 通过 YAML frontmatter 定义元数据
  4. 安全性优先 - 技能可以指示模型执行任何操作，需要审查

标准技能结构

my-skill/
├── SKILL.md          # 必需：前置元数据 + 指令
├── scripts/          # 辅助脚本
│   ├── process.sh
│   └── analyze.py
├── references/       # 参考文档
│   └── api-docs.md
└── templates/        # 模板文件
    └── config.json

设计模式最佳实践

1. 单一职责原则

每个工具应该只做一件事，并做好它。

2. 组合优于继承

通过组合多个工具来实现复杂功能，而不是创建复杂的单一工具。

3. 错误处理

每个工具都应该优雅地处理错误，提供清晰的错误信息。

4. 幂等性

相同输入应该产生相同输出，避免副作用。

安全性考虑

  • 沙箱隔离 - 在隔离环境中执行不可信代码
  • 权限控制 - 最小权限原则
  • 输入验证 - 严格验证所有输入
  • 审计日志 - 记录所有工具调用

实际案例：OpenClaw 技能系统

OpenClaw 使用 AgentSkills 规范，每个技能都有：

  • SKILL.md - 描述、触发条件、使用指南
  • scripts/ - 实际的执行脚本
  • references/ - API 文档、最佳实践

灵感来源

  • Lilian Weng 的 Agent 框架 - Tool use 是 Agent 三大支柱之一
  • LangChain Harness 研究 - Bash + 代码执行 = 通用工具
  • gstack - 10 个专业化定制工具的成功实践

关键洞察

工具设计不是技术问题，而是接口设计问题。好的工具应该：

  1. 易于理解和使用
  2. 提供清晰的错误信息
  3. 可组合、可扩展
  4. 安全、可靠
工具是 Agent 的手和脚，设计良好的工具能让 Agent 的能力指数级提升。
