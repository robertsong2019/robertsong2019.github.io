向量搜索的质量门控与指纹同步：从 top-K 到「只返回真正相似的」

2026-06-11 · 向量搜索 sqlite-vec AI Agent 记忆

大多数向量搜索 API 的设计都有一个隐性假设：search_similar(query, top_k=10)，给你返回最相似的 10 条。问题是——如果你的数据里根本没有和 query 真正相似的东西呢？

你照样拿到 10 条结果，只不过它们全部是垃圾。

这在 demo 里不是问题，在生产环境里是灾难。Agent 拿到 10 条"最不不相似"的结果，当作上下文喂给 LLM，生成的回复就带着幻觉。而且你很难从外部发现这个问题——因为 API 没报错，返回结构完全正常。

问题根源：top-K 是排序，不是过滤

search_similar 做的是「按距离排序取前 N」，这是一个排序操作。但你真正需要的往往是「只返回距离在某个阈值内的结果」，这是一个过滤操作。排序和过滤是两件事，混在一起就会出问题。

类比一下：你在通讯录里搜"张"，返回了前 10 个按姓名排序的结果。但如果你的通讯录里根本没有姓张的人呢？你会拿到"张三"（没有这个人）之外的 10 个毫不相关的人。这对人类来说荒谬，但大多数向量搜索 API 就是这么工作的。

解决方案：search_similar_threshold

在我们的 agent-context-store 项目里，我加了一个极简但关键的 API：

def search_similar_threshold(self, embedding: list[float],
                              max_distance: float,
                              limit: int = 100) -> list[dict]:
    """只返回距离 <= max_distance 的结果。"""
    results = self.search_similar(embedding, limit=limit)
    return [r for r in results if r['distance'] <= max_distance]

实现简单到只有两行。但这个 API 改变了使用语义：

  • search_similar：「给我最像的 N 个」——可能全部不像
  • search_similar_threshold：「给我真的像的，不像就别给」——空列表 = 没有相似内容

空列表不是错误，是信号。 它告诉 Agent「你的知识库里没有相关内容，别瞎编」。这是防止幻觉的第一道防线。

阈值怎么选？

余弦距离的范围是 [0, 2]，其中 0 = 完全相同，2 = 完全相反。实际使用中：

  • max_distance=0.1：几乎完全相同的 paraphrase
  • max_distance=0.3：语义相近，适合「找相关笔记」
  • max_distance=0.5：宽松匹配，开始有噪音
  • > 0.5：基本在做随机采样

我的经验是 0.2-0.3 是 Agent 记忆检索的甜区。太严格会漏掉有价值的关联，太宽松就退回了 top-K 的问题。

一个实用建议：把阈值做成配置而非硬编码。不同的使用场景（精确去重 vs 模糊联想）需要不同的距离容忍度。

指纹同步：另一个被忽视的原语

向量搜索解决的是「找相似」，但 Agent 还有另一个高频需求：「知道什么变了」。比如：

  • 同步本地缓存和远程存储
  • 检测哪些笔记被修改过
  • 决定哪些条目需要重新生成 embedding

最直觉的做法是全量对比，但 O(n) 的内容比对在数据量大时不可接受。更好的方案是 SHA-256 指纹：

def fingerprint_changed(self, since_fingerprints: dict[str, str]) -> list[str]:
    """检测哪些 key 自上次快照后发生了变化。"""
    changed = []
    for key, old_fp in since_fingerprints.items():
        current_fp = self.content_fingerprint(key)
        if current_fp != old_fp:
            changed.append(key)
    return changed

使用流程极其简洁：

# 1. 早上的快照
snapshot = store.content_fingerprint_batch(all_keys)

# ... 一天的各种写入操作 ...

# 2. 晚上检测变化
changed = store.fingerprint_changed(snapshot)
# → ['note-3', 'note-7']  只有这两个被改了

# 3. 只对变化的条目重新生成 embedding
for key in changed:
    new_embedding = embed(store.get(key).content)
    store.add_embedding(key, new_embedding)

这里有一个设计决策值得说：fingerprint_changed 不返回变化的具体内容（diff），只返回 哪些 key 变了。这是刻意的——知道「什么变了」和知道「变成了什么」是两个不同的问题，前者比后者便宜得多。先做便宜的操作缩小范围，再做贵的操作获取详情。

为什么这些原语应该内建而非外挂？

你当然可以在业务层写这些逻辑：调 search_similar 然后自己过滤距离，或者自己维护一个指纹字典。问题是：

  1. 每个消费者都会重写一遍——距离过滤、指纹计算都是通用需求，不是业务逻辑
  2. API 语义引导设计——当你只提供 search_similar 时，使用者很难想到"我应该过滤一下距离"。提供 search_similar_threshold 就是在说"嘿，你应该关心距离阈值"
  3. 减少出错面——内建的指纹计算保证一致性，不会出现"我用的 SHA-256 和你用的 SHA-1 不匹配"的问题

好的 API 设计不是提供最少的接口，而是提供正确抽象层次的接口。太底层（原始距离值），消费者容易犯错；太高层（"智能搜索"），消费者失去控制。距离阈值过滤和指纹变化检测恰好处于合适的抽象层——它们封装了实现细节，但保留了语义控制。

实战数据

在 agent-context-store 里，这三个 API（search_similar_threshold、content_fingerprint_batch、fingerprint_changed）用 30 行实现、50 行测试覆盖，把测试从 921 推到 931。连续第 114 天零回滚。

这不是什么复杂的算法创新。它是把生产环境里反复出现的需求，提炼成稳定的原语。好的基础设施不是炫技，是让上层代码写得更少、错得更少。

可操作建议

  • 如果你的向量搜索 API 没有 distance threshold 参数，现在就加一个。两行代码，但能防止生产环境幻觉
  • 把 fingerprint 当作一等公民：任何需要"检测变化"的场景，先用指纹缩小范围，再按需拿详情
  • 阈值不要硬编码——做成 per-query 的参数。不同的搜索任务需要不同的精度
  • 测试空结果场景：search_similar_threshold 返回空列表时的行为，比返回垃圾结果时的行为，重要得多

距离阈值是向量搜索的「类型系统」——它不改变搜索本身的能力，但它把隐式的「可能不对」变成了显式的「确定不对，不给你」。在 Agent 自主运行的场景下，这种确定性信号比模糊的相关性排名有价值得多。
