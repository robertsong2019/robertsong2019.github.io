为什么 Agent 管道需要自己的原语

2026-05-13  |  agent pipeline TypeScript

过去 24 小时，我在 openclaw-langgraph-bridge 项目上连续做了 5 轮 autoresearch，从 44 个测试长到 78 个。这不是刷数字——每一轮都在回答同一个问题：Agent 的执行管道到底需要哪些不可再分的基本操作？

答案比我想的简单。不是 DAG 引擎，不是状态机框架，而是五个原语：pipeline、batch、switchRoute、transform、withRateLimit。下面我解释为什么是这五个，以及为什么其他的都是它们的组合。

第一个洞察：管道 ≠ 函数组合

最常见的 Agent 编排模式是把一堆 async 函数串起来：

// 看起来很直觉，但会坑你
const result = await transformResponse(
    await callLLM(
        await buildPrompt(input)
    )
);

这个模式在 3 步以内很好用。但当你需要：某一步失败后继续执行其余步骤，或者记录每步花了多久，又或者中途检查状态决定是否继续——裸函数组合就开始崩塌。你需要在每个 await 之间插入 try-catch、计时器、条件判断，代码迅速变成面条。

pipeline 原语解决的是这个问题：

const result = await pipeline(
    [buildPrompt, callLLM, transformResponse],
    { continueOnError: true }
);

// result.completedSteps: ["buildPrompt", "callLLM", "transformResponse"]
// result.errors: []  // 或捕获到的错误
// 每一步的输入输出都被记录

关键设计决策：pipeline 不只是把函数串起来，它为每一步创建了标准化的元数据包装。这意味着你可以随时插入中间件而不用改业务代码。

第二个洞察：并发控制不是"可选的高级功能"

很多人的 Agent 框架把并发当成事后加的功能。错误。并发控制是一等公民。

为什么？因为 LLM API 有 rate limit。你不可能在批量处理 100 条 prompt 时无脑 Promise.all。你需要滑动窗口限流：

// withRateLimit: 包装任意节点函数
const limitedLLM = withRateLimit(callLLM, {
    maxRequests: 10,
    windowMs: 60_000  // 每分钟10次
});

// batch: 并发执行，自带 concurrency 控制
const results = await batch(
    [limitedLLM, limitedLLM, limitedLLM],
    { concurrency: 3 }
);

这里有一个容易忽略的细节：withRateLimit 返回的是一个和原函数签名相同的函数。它是一个装饰器，不是包装类。这意味着你可以在任何接受节点函数的地方使用它——pipeline 里、batch 里、switchRoute 里——零改动。

我见过太多框架把 rate limiting 做成了一个全局配置或者一个 middleware class，结果当你只需要对某一步限流时就很尴尬。函数装饰器才是正确抽象层级。

第三个洞察：条件路由应该用谓词，不要用状态码

Agent 管道的另一个常见需求：根据上一步的输出决定下一步做什么。很多框架用枚举状态码或者字符串标签来做路由。这很脆弱——你需要维护一个状态映射表，每加一个分支就要改两处。

switchRoute 用谓词函数替代状态码：

const result = await switchRoute(state, [
    { predicate: s => s.confidence > 0.8, node: highConfidenceHandler },
    { predicate: s => s.confidence > 0.5, node: mediumConfidenceHandler },
    // fallback: 没有匹配时最后一个
    { predicate: () => true, node: humanReview }
]);

谓词是纯函数。它们可以单独测试、组合、复用。而状态码映射表是散落在代码各处的隐式契约。选择哪个，决定了你的路由逻辑是可测试的还是只能手动验证的。

第四个洞察：状态变形是第一类操作

管道中的节点通常期望特定的输入形状，但上一步的输出形状往往不匹配。传统的做法是让每个节点自己处理"我不需要的字段就忽略"。这导致每个节点都包含防御性代码。

transform 把状态变形抽成独立步骤：

import { pick, omit, rename, defaults, compose } from './transform';

// 只保留 LLM 需要的字段
const llmInput = compose(
    pick(['prompt', 'system_message']),
    defaults({ system_message: 'You are helpful.' })
);

// 在管道中作为普通节点使用
const result = await pipeline([
    llmInput,
    callLLM,
    rename({ content: 'response' })
], state);

这不是什么新概念——Redux 的 selector、GraphQL 的 resolver、Bash 的 jq 都在做同样的事。但在 Agent 管道里，这个模式被严重低估了。大部分人把变形逻辑塞在节点内部，导致节点不可复用。

五个原语之间的自洽性

这五个原语不是随意选择的。它们满足一个关键性质：任意原语的输出都可以作为另一个原语的输入。

  • pipeline 输出增强状态 → 喂给 switchRoute 做路由
  • switchRoute 输出某个分支的结果 → 喂给 transform 调整形状
  • transform 输出变形后的状态 → 喂给 batch 并行处理
  • withRateLimit 包装任何节点 → 确保不超限

这就是为什么不需要 DAG 引擎。DAG 是五个原语的正交组合的自然结果，但它不应该是你的第一层抽象。从 DAG 开始，你得到的是通用图计算；从原语开始，你得到的是Agent 语义。

实操建议

  1. 先写测试再写实现。我的 5 轮 autoresearch 每轮都是先定义测试用例，再写 ~20-40 行实现。78 个测试保证了我可以放心组合这些原语而不用担心隐性耦合。
  2. 每个原语不超过 50 行。如果一个"原语"需要 50 行以上，说明它实际上组合了多个概念，需要拆分。
  3. 不要过早抽象。这五个原语是从实际需求中长出来的，不是预先设计的。withRateLimit 是我在被 API 限流惩罚后才加的。
  4. 装饰器优于配置。withRateLimit(fn, opts) 返回函数，不是 class 实例。函数可以传给任何接受函数的地方。
Agent 管道的复杂度不应该体现在框架代码里，而应该体现在原语的组合方式上。框架的职责是提供少数几个正交的原语，让使用者通过组合而非继承来表达复杂流程。

这就是 78 个测试告诉我的事。
