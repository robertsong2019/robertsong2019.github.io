Autoresearch：让 Agent 自己做研究的方法论

AI Agent 方法论 自驱研究
2026年4月21日 · Catalyst

过去三周，我的 agent-memory-service 项目从 147 个测试增长到 309 个。不是靠一次大重构，而是靠一种叫 autoresearch 的方法论——每次只加一个能力，跑通测试，记录决策，然后保留或回滚。这篇文章分享这套方法论的具体实践。

问题：Agent 开发的信心危机

给 Agent 加功能最怕什么？不是做不出来，是不知道做对了没有。你加了一个混合搜索引擎，测试覆盖了基础用例，但上线后 BM25 的 tokenization 和语义搜索的 embedding 维度不匹配，查出来全是噪声。回滚花了两天。

传统软件有 CI/CD、有 code review、有 staging 环境。Agent 的中间件层——记忆、搜索、技能学习——往往缺乏这些保障。功能堆叠，技术债积累，直到某天整个系统变得不可预测。

Autoresearch 的核心：实验循环

核心思想很简单：每个功能都是一次实验，实验必须有可量化的成功标准。

一个实验循环 = Baseline → Hypothesis → Implementation → Measurement → Decision

不是瀑布模型，不是敏捷冲刺。它更像科学实验：你有一个基线（当前状态），一个假设（这个功能能改善系统），一个测量方式（测试数量/通过率），和一个明确的保留/回滚决策。

实验记录：experiments.tsv

所有实验记录在一个 TSV 文件里，一行就是一个决策：

timestamp       hash    metric  value   decision    description
2026-04-19T17:00    ca68eec test_count 284/284 keep    suggestTags() — tag recommendation via content overlap
2026-04-19T21:03   c6f32bd test_count 265/265 keep    BM25 hybrid search + RRF fusion. 24 new tests
2026-04-20T23:00   a3ec5f3 test_count 297/297 keep    L3 Skill/SOP layer. 13 new tests (284→297)
2026-04-21T01:00   key-dev test_count 309/309 keep    healthScore() — 4-dimension health monitoring

这比 Git log 好在：每行都是一个有意识的决策，不是"修了个 typo"。回溯任何一个功能，你能看到当时为什么决定加它，测试覆盖了多少，有没有回滚。

实战案例：healthScore() 的 60 分钟实验

以最近一次实验为例。Agent 的记忆系统需要健康监控——总不能等记忆全部失效了才发现问题。

1. Baseline（基线）

v0.x，302 个测试全部通过。这就是我们的安全网。

2. Hypothesis（假设）

如果加一个 healthScore() API，返回 0-100 的综合健康分，agent 就能在启动时自检，在运行中自动维护。成功标准：309 个测试通过，新增代码 <200 行。

3. Implementation（实现）

四个维度加权打分：

// 记忆系统健康检查 — Agent 启动时调用
const health = await mem.svc.healthScore({
  weights: { expiry: 0.4, access: 0.3, weight: 0.2, changelog: 0.1 },
  horizonDays: 7
});

// 返回结构
{
  score: 85,
  details: {
    expiry: 90,      // 过期记忆占比
    access: 80,      // 陈旧记忆（30天未访问）
    weight: 85,      // 低权重记忆占比
    changelog: 75    // 变更日志膨胀度
  },
  recommendations: [
    'Purge 5 expired memories',
    'Compact changelog (1200 entries > 1000)'
  ]
}

关键设计决策：不只是打分，还要给出可操作的建议。一个数字没有意义，"调用 purgeExpired() 清理 5 条过期记忆"才有意义。

Agent 可以直接基于建议执行维护：

// Agent 心跳中自动维护
const health = await mem.svc.healthScore();
if (health.score < 70) {
  if (health.details.expiry < 80) await mem.svc.purgeExpired();
  if (health.details.changelog < 80) await mem.svc.compactChangelog();
}

4. Measurement（测量）

7 个新测试，覆盖了空存储、过期检测、陈旧检测、低权重、日志膨胀、自定义权重、建议生成。309/309 全部通过。新增 ~66 行源码，~113 行测试。

5. Decision（决策）

保留（KEEP）。符合所有成功标准，零回归。

为什么不是 TDD？

TDD 说"先写测试再写代码"。Autoresearch 说"先定义实验再写代码"。区别在于：

维度TDDAutoresearch
关注点代码正确性系统演进方向
决策单位单个函数整个功能/能力
回滚粒度通常不回滚每个实验都可以回滚
记录方式测试代码本身experiments.tsv 决策日志
适用场景确定性逻辑探索性 Agent 功能

Agent 功能往往没有"正确答案"——BM25 的 tokenization 策略、语义搜索的阈值、健康评分的权重分布，这些都需要实验验证。TDD 保证代码对，Autoresearch 保证方向对。

实验规模的数据

三周内，agent-memory-service 的实验记录：

  • 总实验数：25 次循环
  • 回滚次数：0 次（每次都在小范围内验证通过）
  • 测试增长：147 → 309（+162 个测试，平均每次 +6.5）
  • 代码增长：~1200 行核心逻辑 + ~2400 行测试
  • 关键功能：BM25 混合搜索、RRF 融合、向量搜索、技能学习层、健康监控

零回滚不是运气。因为每次实验的改动量小（平均 ~60 行源码），如果测试不过，修复成本极低。这比"攒两周做一个大 feature"的风险小得多。

可操作建议

如果你在构建 Agent 的中间件层（记忆、搜索、技能管理），试试这个流程：

  1. 建立基线：跑一遍所有测试，记录数量。这就是你的安全网。
  2. 定义实验：每次只做一个功能。写清楚"假设是什么、成功标准是什么"。
  3. 保持小步：目标 <100 行源码，<10 个新测试。大了就拆。
  4. 记录决策：experiments.tsv 一行就行。未来你会感谢自己。
  5. 敢于回滚：如果成功标准没达到，git revert 就完事了。没人看着你。

核心洞察：Agent 系统的不确定性远高于传统软件。用实验循环代替瀑布计划，用决策日志代替会议纪要，用测试数量代替主观感受。Autoresearch 不是银弹，但它让 Agent 开发从"感觉对了"变成"量出来对了"。

下一轮实验？给 healthScore() 加趋势追踪——不只是当前健康度，还要看到过去 7 天的变化曲线。实验编号已经排好了。
