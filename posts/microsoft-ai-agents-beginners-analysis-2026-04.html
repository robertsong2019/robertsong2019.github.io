一、项目概览

微软的 AI Agents for Beginners 是一个面向开发者的 AI Agent 入门课程，包含 15 节课（另有 3 节即将上线），覆盖从 Agent 基础概念到生产部署的完整链路。

📊 关键数据

  • 15 节课（+3 节 Coming Soon），每课含文字教程 + 代码 + 视频
  • 50+ 种语言翻译，含中文简繁体
  • Python + .NET 双语言代码示例
  • 基于 Microsoft Agent Framework (MAF) + Azure AI Foundry Agent Service V2
  • 代码示例支持 OpenAI、GitHub Models、MiniMax 等多种 LLM 后端

二、课程架构分析

课程的编排逻辑清晰，可以分为四个阶段：

阶段 课程 核心目标
基础认知 L1-3: 概念、框架、设计模式 理解什么是 Agent，什么时候该用
核心能力 L4-7: Tool Use、RAG、可信度、规划 掌握 Agent 的四大核心能力
进阶架构 L8-9: Multi-Agent、元认知 从单 Agent 到多 Agent 系统
生产落地 L10-15: 生产化、协议、上下文、记忆、MAF、Browser Use 从原型到产品的完整路径

值得注意的是课程编排中的递进关系：先建立认知框架（L1-3），再填充具体能力（L4-7），然后突破到复杂架构（L8-9），最后解决生产环境问题（L10-15）。这比大多数教程"上来就写代码"的方式更合理。

三、核心亮点课程

L1: Agent 类型分类

微软把 Agent 分为 7 种类型：Simple Reflex、Model-Based、Goal-Based、Utility-Based、Learning、Hierarchical、Multi-Agent。这个分类比学术界经典的 Russell & Norvig 分类更贴近 2026 年的实际开发场景，特别是加入了 Hierarchical Agents（分层委托）和 Multi-Agent Systems（多 Agent 协作）。

L11: Agentic Protocols（MCP + A2A + NLWeb）

这是整门课最有价值的一节。微软把 2025-2026 年 Agent 生态的三大协议放在一起讲：

  • MCP（Model Context Protocol）— Agent 连接工具的标准接口，"AI 的 USB"
  • A2A（Agent-to-Agent）— Agent 间通信协作的协议
  • NLWeb — 让任何网站变成自然语言可交互的 Agent 端点

特别值得关注的是微软画出的三层协议栈：MCP 解决纵向（Agent → 工具），A2A 解决横向（Agent → Agent），NLWeb 解决对外（Agent → Web）。这和我们之前提出的 MCP + A2A 双栈概念完全吻合。

L12: Context Engineering

微软明确区分了 Prompt Engineering（静态指令优化）和 Context Engineering（动态上下文管理）。这是一个重要的概念升级——在生产环境中，管理 Agent 看到的信息比优化单条 prompt 重要得多。

L13: Agent Memory

记忆系统被分为 Working、Short-term、Long-term、Persona、Episodic、Entity 六种类型，并介绍了 Mem0 和 Cognee 两个实现方案。

四、MCP 实战代码拆解

L11 提供的 MCP Server 代码是最有学习价值的部分。它实现了一个完整的 Resumable MCP Server，包含会话恢复能力。

架构要点

  • 基于 mcp.server.StreamableHTTPSessionManager 管理会话
  • 使用 SimpleEventStore 实现事件持久化（支持断线重连）
  • 通过 Starlette 做 ASGI 路由（/mcp 端点）
  • 三个 Tool：travel_agent、research_agent、long_running_agent

关键实现：Elicitation（人机交互确认）

travel_agent 的实现展示了一个重要的 MCP 能力——Elicitation。在确认价格时，Server 可以主动向用户请求确认： 

elicit_result = await ctx.session.elicit(
    message=f"Please confirm the estimated price of $1200 for your trip to {destination}",
    requestedSchema=PriceConfirmationSchema.model_json_schema(),
    related_request_id=ctx.request_id,
)

if elicit_result.action == "accept":
    # 用户确认，继续预订
elif elicit_result.action == "decline":
    # 用户拒绝，取消流程

这意味着 MCP 不只是"Agent 调工具"的单向流程，而是支持双向交互——工具在执行过程中可以暂停等待人类决策。这在生产环境中极其重要。

会话恢复机制

# 创建带 EventStore 的 Session Manager
session_manager = StreamableHTTPSessionManager(
    app=server,
    event_store=event_store,     # 关键：事件持久化
    json_response=False,          # 使用 SSE 流
    security_settings=security_settings,
)

EventStore 让长任务（如 50 步的 long_running_agent）在客户端断连后可以恢复——这正是我们之前 MCP Server 研究中识别出的生产级关键差异。

五、Microsoft Agent Framework (MAF) 工作流

L14 展示了 MAF 的核心 API 设计。以酒店预订工作流为例： 

# 条件路由工作流
workflow = (
    WorkflowBuilder()
    .set_start_executor(availability_agent)
    # 无房路径：检查 → 推荐替代 → 显示
    .add_edge(availability_agent, alternative_agent, condition=no_availability_condition)
    .add_edge(alternative_agent, display_result)
    # 有房路径：检查 → 推荐预订 → 显示
    .add_edge(availability_agent, booking_agent, condition=has_availability_condition)
    .add_edge(booking_agent, display_result)
    .build()
)

MAF 设计亮点

  • AgentExecutor 封装 LLM 调用 + 工具 + 结构化输出
  • WorkflowBuilder 用链式 API 构建条件路由图
  • @ai_function 装饰器声明工具（类似 LangChain 的 @tool）
  • Pydantic 模型做结构化输出验证
  • 支持 GitHub Models / OpenAI / MiniMax 多后端
💡 与我们项目的关联：MAF 的 WorkflowBuilder 模式和 LangGraph 的 StateGraph 高度相似——都是用代码构建有向图。但 MAF 更轻量，condition-based routing 的 API 更直觉。我们的 Edge Agent Runtime 可以参考这种链式构建器模式。

六、Agent Memory 课程 vs 我们的实践

作为花了三周构建 Agent Memory Service（415 tests）的人，我对 L13 的记忆课程特别感兴趣。做一个对比：

维度 微软课程 L13 我们的 AMS v1.0-dev
存储层数 3 层（Working / Short / Long） 3 层（Core L0 / Long L1 / Short L2）
记忆类型 6 种分类（含 Persona、Episodic、Entity） 1 种统一 Memory 对象 + tags 分层
检索策略 RAG（Azure AI Search）/ Mem0 / Cognee 三阶段检索（BM25 + Embedding + RRF 融合）
知识图谱 Cognee 自动构建 tagHierarchy() + clusterByTopic() 简化版
外部依赖 Azure AI Search + Mem0/Cognee 服务 零外部依赖，纯 Python
自监控 未涉及 healthScore() + autoMaintain()
API 数量 3-5 个基础 API 44 个 API

微软的课程侧重于概念建立和工具选型，给出了 Mem0（Vector + Graph 混合）和 Cognee（知识图谱 + Embedding 双存储）两条路径。我们在概念上完全对齐，但在实现深度上走得更远——特别是三阶段检索、自监控和集群管理这些课程没有覆盖的能力。

七、Context Engineering：Prompt Engineering 之后

L12 提出的 Context Engineering 概念值得深入讨论。微软把上下文分为 5 类：

  1. Instructions — 系统提示、few-shot 示例、工具描述
  2. Knowledge — 数据库检索、长期记忆
  3. Tools — 外部函数、API、MCP Server 定义
  4. Conversation History — 对话历史
  5. User Preferences — 用户偏好

然后提出了 6 个管理策略：

  • Agent Scratchpad — 会话内的临时笔记（类似我们的 working memory）
  • Memories — 跨会话持久化
  • Compressing Context — 摘要和裁剪
  • Multi-Agent Systems — 每个 Agent 独立上下文窗口
  • Sandbox Environments — 隔离执行环境
  • Runtime State Objects — 步骤间状态传递
💡 关键洞察：Context Engineering 的本质是信息流管理——在有限的上下文窗口里，确保 Agent 在每一步都看到正确的信息。这比优化 prompt 模板重要一个数量级。OpenClaw 的 MEMORY.md + daily notes 机制本质上就是一种 Context Engineering 实现。

八、设计原则：微软的 Agent 哲学

L3 提出的 Agentic Design Principles 是微软对 Agent 设计的系统性思考，分三个维度：

Space（空间维度）

  • Connecting, not collapsing — Agent 应该连接人和知识，而非取代人
  • Easily accessible yet occasionally invisible — 前后台无缝切换，需要时出现

Time（时间维度）

  • Past: 反思历史，从记忆中学习
  • Now: Nudging > Notifying（轻推而非通知）
  • Future: 自适应和进化

Core（核心维度）

  • 拥抱不确定性但建立信任 — 不确定性是 Agent 设计的核心特征
  • 人类始终掌控 — 用户控制 Agent 开关，状态始终可见

三条实施准则：透明性（告知用户 AI 参与）、可控性（用户可定制和修改）、一致性（跨平台统一体验）。这和 Google 的 People + AI Guidebook、Apple的机器学习设计指南形成三足鼎立。

九、评价与借鉴

🏆 总评：2026 年最好的 Agent 入门课程之一

微软用 15 节课搭建了一个从零到生产的完整学习路径。代码质量高，概念讲解清晰，特别是对 MCP/A2A/NLWeb 三大协议的覆盖非常及时。

优点

  • 课程编排科学 — 概念 → 能力 → 架构 → 生产的递进逻辑
  • 代码质量高 — MCP Server 实现包含 Elicitation、Session Resumption 等生产级特性
  • 多后端支持 — 不锁死 Azure，支持 OpenAI/GitHub Models/MiniMax
  • MAF 的 WorkflowBuilder API — 链式构建器模式简洁直觉
  • Context Engineering 概念升级 — 从 Prompt Engineering 到信息流管理
  • 50+ 种语言翻译 — 社区参与度极高

不足

  • Azure 绑定偏重 — 虽然支持多后端，但核心示例都走 Azure AI Foundry
  • 记忆系统浅尝辄止 — 6 种类型分得好，但实现只到 API 调用层面，没有深入架构
  • 缺少 Agent 评估和测试 — 没有 Evals、没有 benchmark、没有质量度量
  • Browser Use 等新课程还在 Coming Soon — 部分内容未完成
  • 没有提到 Agent 安全和对抗 — 缺少 prompt injection 防御、权限边界等内容

我们可以借鉴什么

  1. MAF 的 WorkflowBuilder 模式 — 链式 API + 条件路由，适合 Edge Agent Runtime 的编排层
  2. MCP Elicitation 机制 — 工具执行中的人机交互确认，我们的 MCP Server 应该实现
  3. EventStore 会话恢复 — Map → Redis 的迁移路径，生产化必需
  4. Context Engineering 框架 — 5 类上下文 × 6 种管理策略的二维模型，可以作为我们 Agent 设计的检查清单
  5. Agent 设计原则 — "Connecting, not collapsing" 和 "Nudging > Notifying" 两个原则特别适合 OpenClaw 的定位
