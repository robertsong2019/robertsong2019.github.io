你的 Agent 到底知道什么？用 Bron-Kerbosch 和 CPM 看清记忆图谱的隐藏结构

📅 2026-07-07 · 标签: agent-memory, graph-theory, clique-detection, introspection
大多数 Agent 记忆系统把记忆当作扁平列表：存进去、检索出来。但当记忆以图谱形式组织时，一个被忽视的问题浮现——你的 Agent 的知识结构到底是什么形状？ 答案藏在 clique（团）里。本文展示如何用 Bron-Kerbosch 算法和 Clique Percolation Method（CPM）对 Agent 记忆图谱做深度自省，发现冗余知识簇、桥接概念、和重叠社区——这些是 embedding 相似度永远无法揭示的结构信息。

引言：扁平记忆的盲区

你有一个 RAG 系统，存了 10,000 条记忆。你做了向量检索、重排序、甚至社区检测。你觉得自己对记忆系统的组织很了解。

但你回答不了这些问题：

  • 哪些记忆条目是"全连接"的？ 即它们两两之间都有直接关联——形成一个知识的"团"
  • 一个概念属于几个不同的知识领域？ 传统的社区检测（Leiden、Label Propagation）强制每个节点只属于一个社区，但真实知识是重叠的
  • 知识密度在哪里？ 你的 Agent 对哪些主题有最密集、最交叉引用的知识网络？

这些问题的答案需要图论中的 clique analysis（团分析）。本文展示我在 agent-memory-graph 项目中实现的完整 clique 检测工具链，以及这些算法能揭示什么。

一、Maximal Cliques：找到知识的"全连接簇"

Clique（团） 是图中两两相连的节点集合。Maximal clique 是不被任何更大的 clique 包含的团。在 Agent 记忆图中，一个 clique 就是一组两两相关的记忆——每一个都直接关联到其他每一个。

为什么这比向量相似度更有意义？因为向量相似度是 二元关系（A 和 B 相似），而 clique 捕捉的是 群体结构（A、B、C、D 两两相关，形成一个知识单元）。

Bron-Kerbosch 算法是枚举所有 maximal cliques 的经典方法。核心实现（带 pivoting 优化）：

def maximal_cliques(self, min_size: int = 3) -> list[list[str]]:
    """Enumerate all maximal cliques via Bron-Kerbosch with pivoting.

    A clique = every pair of nodes directly connected.
    Maximal = not a subset of any larger clique.
    """
    # Build undirected adjacency from active nodes only
    active_ids = {str(r["id"]) for r in
                  self.conn.execute(
                      "SELECT id FROM nodes WHERE quarantined = 0"
                  ).fetchall()}
    adj: dict[str, set[str]] = {nid: set() for nid in active_ids}
    for r in self.conn.execute("SELECT source, target FROM edges").fetchall():
        s, t = str(r["source"]), str(r["target"])
        if s in active_ids and t in active_ids:
            adj[s].add(t)
            adj[t].add(s)  # treat as undirected

    cliques: list[list[str]] = []

    def _bron_kerbosch(R: set, P: set, X: set):
        """Classic BK with Brelzig-Kernighan pivot."""
        if not P and not X:
            if len(R) >= min_size:
                cliques.append(sorted(R))
            return
        # Pivot: choose u in P|X maximizing |P & N(u)|
        pivot_pool = P | X
        pivot = max(pivot_pool, key=lambda u: len(P & adj[u]))
        for v in list(P - adj[pivot]):
            _bron_kerbosch(R | {v}, P & adj[v], X & adj[v])
            P.discard(v)
            X.add(v)

    _bron_kerbosch(set(), set(adj.keys()), set())
    cliques.sort(key=lambda c: (-len(c), c))  # largest first
    return cliques
关键工程细节：
  1. Pivoting 优化：朴素 BK 的时间复杂度是 O(3^(n/3))，pivoting 将其降到 O(2^(n/2))。对 100-10,000 节点的记忆图差异巨大。Pivot 选择 P∪X 中邻居最多的节点，跳过其邻居的递归展开。
  2. Quarantine 感知：被隔离的节点（可信度低、已过期）不参与计算——你不想让错误信息形成虚假的知识簇。
  3. min_size 参数：默认 min_size=3（三角形及以上）。size=2 的 clique 就是一条边，通常没有分析价值。

实际用法

mg = MemoryGraph("agent_memories.db")

mg.add_node("python", tags=["language", "dynamic"])

# -> [["asyncio", "coroutine", "fastapi", "python"]]
# 这四个知识点两两相连，形成一个知识团

洞察：如果你的 Agent 在某个领域有大型 clique，说明它对该领域有高度交叉引用的知识网络——这是 深度专业知识 的标志。反之，如果 clique 很少或很小，说明知识是 浅层和碎片化 的。

二、Clique Overlap：知识簇之间的桥接

找到 clique 只是第一步。更有趣的是 clique 之间如何重叠。两个 clique 共享的节点就是 桥接概念——连接不同知识领域的桥梁。

def clique_overlap_matrix(self) -> dict[tuple[int, int], int]:
    """Shared-node counts between all pairs of maximal cliques."""
    cliques = self.maximal_cliques(min_size=1)
    if len(cliques) < 2:
        return {}

    sets = [set(c) for c in cliques]
    result: dict[tuple[int, int], int] = {}
    for i in range(len(sets)):
        for j in range(i + 1, len(sets)):
            shared = len(sets[i] & sets[j])
            if shared > 0:
                result[(i, j)] = shared
    return result
实际洞察：如果 "Python 异步编程" clique 和 "数据库连接池" clique 共享了 "asyncio" 节点，那么 asyncio 就是连接这两个知识域的桥接概念。当你更新 asyncio 相关的记忆时，你需要知道它会影响两个知识域——这就是 overlap 矩阵告诉你的。

三、CPM：发现重叠社区

传统社区检测将每个节点分配到恰好一个社区。但知识不是这样运作的——"asyncio" 同时属于 Python 生态、并发编程、和网络编程社区。它应该出现在多个社区中。

Clique Percolation Method (CPM)（Palla et al., 2005）解决了这个问题：

def k_clique_communities(self, k: int = 3) -> list[list[str]]:
    """Detect overlapping communities via CPM.

    Two k-cliques are adjacent if they share k-1 nodes.
    Communities = connected components of clique adjacency graph.
    Unlike Leiden/LPA, nodes may belong to multiple communities.
    """
    cliques = self.maximal_cliques(min_size=k)
    if not cliques:
        return []

    clique_sets = [set(c) for c in cliques]
    n = len(clique_sets)
    adj: dict[int, list[int]] = {i: [] for i in range(n)}

    for i in range(n):
        for j in range(i + 1, n):
            if len(clique_sets[i] & clique_sets[j]) >= k - 1:
                adj[i].append(j)
                adj[j].append(i)

    # BFS: each connected component = one CPM community
    visited = set()
    communities: list[list[str]] = []
    for start in range(n):
        if start in visited:
            continue
        component_cliques = []
        queue = [start]
        visited.add(start)
        while queue:
            cur = queue.pop(0)
            component_cliques.append(cur)
            for nb in adj[cur]:
                if nb not in visited:
                    visited.add(nb)
                    queue.append(nb)
        members = set()
        for ci in component_cliques:
            members |= clique_sets[ci]
        communities.append(sorted(members))

    communities.sort(key=lambda c: (-len(c), c))
    return communities

为什么这比 Leiden 更好？ 因为 Agent 的知识本质上是重叠的。"Docker" 同时属于 DevOps、部署、容器化、CI/CD 等多个领域。强制分配到一个社区会丢失这种多维属性。CPM 保留了这种模糊性——一个节点可以出现在 2 个、3 个甚至更多社区中。

k 值的选择

# k=3（默认）：最常用。两个三角形 clique 共享 2 个节点即视为相邻。
communities = mg.k_clique_communities(k=3)

# k=2：更宽松。社区更大、更模糊。
communities = mg.k_clique_communities(k=2)

# k=4：更严格。社区更小、更紧密。
communities = mg.k_clique_communities(k=4)

四、实际工作流：用 Clique 分析做 Agent 自省

将以上算法组合成一个完整的自省工作流：

from collections import Counter

def introspect_agent_memory(mg: MemoryGraph) -> dict:
    """对 Agent 记忆图谱做全面结构分析。"""
    report = {}

    # 1. 基础图指标
    analytics = mg.graph_analytics()
    report["density"] = analytics["density"]
    report["avg_degree"] = analytics["avg_degree"]

    # 2. Clique 分析
    cliques = mg.maximal_cliques(min_size=3)
    report["clique_count"] = len(cliques)
    report["largest_clique_size"] = mg.clique_number()
    report["clique_size_dist"] = Counter(len(c) for c in cliques)

    # 3. 桥接节点识别
    overlaps = mg.clique_overlap_matrix()
    bridge_candidates = Counter()
    for (i, j), shared in overlaps.items():
        shared_nodes = set(cliques[i]) & set(cliques[j])
        for node in shared_nodes:
            bridge_candidates[node] += 1
    report["top_bridges"] = bridge_candidates.most_common(10)

    # 4. 重叠社区
    communities = mg.k_clique_communities(k=3)
    report["overlapping_communities"] = len(communities)

    node_community_count = Counter()
    for comm in communities:
        for node in comm:
            node_community_count[node] += 1
    multi_domain = {n: c for n, c in node_community_count.items() if c > 1}
    report["multi_domain_concepts"] = multi_domain

    # 5. 知识结构诊断
    if report["clique_count"] == 0:
        report["diagnosis"] = "碎片化：知识之间缺乏深度关联"
    elif report["largest_clique_size"] <= 3:
        report["diagnosis"] = "浅层：知识有关联但缺乏深度交叉引用"
    elif len(multi_domain) == 0:
        report["diagnosis"] = "孤岛化：有深度知识但领域之间缺乏桥接"
    else:
        report["diagnosis"] = "健康：有深度簇和跨领域桥接"

    return report

运行后你会得到一个关于 Agent 知识健康状况 的结构化报告——不是基于内容，而是基于拓扑。这是 embedding 向量永远无法提供的洞察。

五、与 Girvan-Newman 的互补

同一时期实现的 Girvan-Newman 社区检测和 CPM 形成互补：

特性Girvan-NewmanCPM
社区类型互斥划分重叠
计算方式自顶向下（拆分）自底向上（聚合）
最佳场景明确的社区边界模糊的领域交叉
时间复杂度O(nm²)O(3^(n/3))（BK 部分）
输出模块度历史重叠成员列表
实践建议：先用 Girvan-Newman 看整体社区结构（从拆分视角），再用 CPM 看概念如何跨越社区边界（从重叠视角）。两者结合，你得到的是 Agent 知识的 3D 拓扑图。

结语

Agent 记忆不应该只是一个搜索引擎的索引。当记忆以图谱形式组织，clique 分析揭示了知识的 拓扑结构——哪些区域深度交叉引用（clique）、哪些概念桥接不同领域（overlap）、哪些知识域模糊交织（CPM 社区）。

这些算法不是新发明——Bron-Kerbosch 出现在 1973 年，CPM 出现在 2005 年。但将它们应用于 Agent 记忆自省是一个被忽视的实践方向。多数 Agent 框架还停留在 "存记忆 → 检索记忆" 的循环中，从不回过头来问：我的知识结构是什么形状的？

现在你可以问了。
