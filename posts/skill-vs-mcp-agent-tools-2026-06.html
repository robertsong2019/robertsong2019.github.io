Skill vs MCP：AI Agent 工具集成的两条路径

2026-06-10 • 深度调研 • 15 分钟阅读
AI Agent MCP AgentSkills

问题：你的工具，AI 怎么调用？

你有一堆内部工具和开放 API，想让 AI Agent 来调用。搜了一圈，发现两个主流方案：

  • Agent Skill（SKILL.md）—— 写一份 Markdown 说明书，告诉 Agent 怎么用
  • MCP Server（Model Context Protocol）—— 跑一个标准服务，Agent 通过协议连接调用

该选哪个？还是两个都要？这篇文章帮你搞清楚。

什么是 Agent Skill？

Agent Skill 是一个基于 SKILL.md 文件格式的开放标准，由 Anthropic 在 2025 年 12 月发布。 核心思想：用 Markdown 写一份"操作手册"，AI Agent 读了就知道怎么用你的工具。

截至 2026 年 4 月，SKILL.md 已被 30+ AI 编码工具采用，包括 Claude Code、OpenAI Codex、GitHub Copilot、 Google Gemini CLI、Cursor、Windsurf、JetBrains Junie 等。社区技能数量超过 2,600 个，每季度翻倍。 Anthropic、OpenAI、Google 三方共同将标准捐赠给了 Linux 基金会下的 Agentic AI Foundation（AAIF）。

标准结构

my-skill/
├── SKILL.md          # 必需：YAML frontmatter + Markdown 指令
├── scripts/          # 可选：可执行脚本
├── references/       # 可选：参考文档
└── assets/           # 可选：模板、资源

SKILL.md 格式

---
name: my-weather-tool
description: 查询指定城市的天气信息和未来7天预报
license: MIT
compatibility: 需要 curl 和 jq
---

# 天气查询工具

## 使用场景
当用户询问天气、温度、预报时触发。

## 调用方式
使用 wttr.in API：
\`\`\`bash
curl -s "wttr.in/{city}?format=j1" | jq '.current_condition[0]'
\`\`\`

## 参数
- city: 城市名（拼音或英文）

## 返回格式
JSON，包含温度、湿度、风速等字段。

关键特性：渐进式披露。Agent 先只读 description 判断是否需要激活这个 Skill， 确认后才加载完整的 SKILL.md。这样 2,600 个技能不会同时塞满上下文窗口。

什么是 MCP Server？

Model Context Protocol（MCP）是 Anthropic 在 2024 年 11 月推出的开放协议，用于标准化 AI 系统与外部工具、 数据源的连接方式。2025 年 12 月捐赠给 Linux 基金会 AAIF，由 AWS、Google、Microsoft、OpenAI、Bloomberg、 Cloudflare 联合支持。

截至 2026 年中，MCP 生态数据：

  • 10,000+ 活跃公共 MCP Server
  • 15,926 GitHub 仓库标记 mcp-server topic
  • 86,148 ⭐ 官方 servers 仓库
  • 28% 的财富 500 强已部署 MCP

MCP 本质是一个运行时协议——你需要启动一个 Server 进程（本地或远程），Agent 通过 JSON-RPC 2.0 连接调用。 它定义了三种能力：Resources（读取数据）、Tools（执行操作）、Prompts（提示模板）。

MCP Server 示例

// 最小 MCP Server（Streamable HTTP，2026 RC 规范）
import express from 'express';

const app = express();
app.use(express.json());

app.post('/mcp', (req, res) => {
  const { method, params } = req.body;
  
  if (method === 'tools/list') {
    res.json({
      tools: [{
        name: 'query_weather',
        description: '查询城市天气',
        inputSchema: {
          type: 'object',
          properties: { city: { type: 'string' } },
          required: ['city']
        }
      }]
    });
  }
  
  if (method === 'tools/call') {
    const city = params.arguments.city;
    // 调用实际 API...
    res.json({ content: [{ type: 'text', text: `${city}: 28°C, 晴` }] });
  }
});

app.listen(3000);

它们不是竞争关系，是互补关系

Auth0 的技术分析说得很清楚：三者是不同层次的抽象——

  • Tool（工具）：单个可调用函数，最底层的能力单元
  • MCP Server：标准化协议服务，暴露一组工具、数据和提示
  • Skill（技能）：最高层，指导 Agent 如何编排多个工具完成复杂目标
Skill 是 Agent 的内部剧本，MCP Server 是 Agent 的外部神经系统。

一个 Skill 可以指导 Agent 调用多个 MCP Server。比如一个"客户分析"Skill 可能同时调用 CRM 的 MCP Server（查客户数据）、PostgreSQL 的 MCP Server（查订单历史）， Skill 负责编排整个多步骤工作流。

选择指南：三个问题帮你决定

Q1：你的工具需要实时数据还是静态知识？

→ 实时数据（数据库、API 调用）→ MCP Server

→ 静态知识（API 文档、使用模式）→ Skill

Q2：Agent 是读取信息，还是执行操作？

→ 只是读取/理解 → Skill

→ 需要写入/修改/执行 → MCP Server

Q3：你愿意投入多少运维成本？

→ 零运维（丢个文件就行）→ Skill

→ 可以维护服务进程 → MCP Server

🚀 快速决策：

只用 Skill：工具已有 HTTP API，Agent 只需知道怎么调。写个 SKILL.md 描述清楚 API 端点、参数、返回值，Agent 自己用 curl/fetch 调。

只用 MCP Server：需要实时连接数据库、执行写入操作、维护会话状态。

两者结合：Skill 描述工作流逻辑，MCP Server 提供具体的工具调用能力。这是最强大的组合。

实战场景对比

场景 1：调用已有的开放 API（如天气、汇率）

推荐：Skill

API 已经在那了，写个 SKILL.md 告诉 Agent 怎么调就行。不需要额外跑服务，Agent 用 curl 或 web_fetch 直接请求。

场景 2：连接内部数据库

推荐：MCP Server

数据库连接需要管理连接池、权限、查询安全。MCP Server 可以在服务端处理这些，只暴露安全的查询接口给 Agent。

场景 3：多步骤业务工作流

推荐：Skill + MCP Server

用 Skill 编排工作流（先查 CRM，再查订单，最后生成报告），每个步骤通过 MCP Server 执行具体操作。

场景 4：给团队分享最佳实践

推荐：Skill

把团队的开发规范、部署流程、代码审查标准写成 Skill，所有 Agent 都能读到。零运维，版本控制跟着 Git 走。

场景 5：需要维护会话状态

推荐：MCP Server

比如长时间的数据库迁移任务，需要追踪进度、支持暂停恢复。MCP Server 可以在服务端维护状态。

常见误区

❌ "Skill 是某个 Agent 的私有格式"

不是。SKILL.md 是开放标准，由 AAIF 治理。30+ 工具原生支持，一份 Skill 到处运行。

❌ "有了 Skill 就不需要 MCP 了"

Skill 是文档（告诉 Agent 怎么做），MCP 是服务（帮 Agent 做事）。两者解决不同层次的问题。

❌ "MCP 总是更好的选择"

MCP 需要运行和维护服务进程。如果你的 API 已经是 HTTP 了，写个 Skill 让 Agent 直接调，比再套一层 MCP Server 更简单。

❌ "必须二选一"

最佳实践是两者结合。Skill 管编排逻辑和知识传递，MCP Server 管工具执行和状态管理。

我的建议

从 Skill 开始。成本最低，见效最快。把你的 API 文档写成 SKILL.md， 当天就能让 30+ AI Agent 用起来。

需要时再加 MCP。当你发现 Skill 描述的某些操作需要服务端执行（比如数据库写入、 长时间任务、实时数据流），再为这些操作搭建 MCP Server。

最终目标是 Skill + MCP 的组合。Skill 是 Agent 的"大脑"（知道该做什么）， MCP Server 是 Agent 的"双手"（能实际做事）。两者结合，才是完整的 AI Agent 工具集成方案。

先写 Skill，跑通再上 MCP。别把简单问题搞复杂了。
