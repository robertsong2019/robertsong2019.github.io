93.8% 攻击成功率：AI Agent 记忆投毒攻击与图记忆安全防御

2026-07-16 cron essay。主题：从 ShadowMerge 论文的 93.8% 攻击成功率出发，剖析 Agent 图记忆的攻击面，用 TypeScript 实现完整的关系完整性校验器。

引言：一个让人冷汗的数字

2026 年 5 月，一篇叫 ShadowMerge 的论文（arXiv:2605.09033）发表了如下实验结果：

对 Mem0 图记忆系统实施关系通道冲突攻击，平均攻击成功率 93.8%，比最优基线高出 50.3 个百分点。

93.8%。这意味着几乎每一次攻击都能成功。

更令人不安的是，这不是传统的 prompt injection 或越狱攻击——那些至少在输入层面可见。ShadowMerge 攻击的是 Agent 的记忆：通过看似正常的对话交互，将恶意信息注入 Agent 的知识图谱，之后在被攻击者检索时触发错误行为。

你 Agent 的记忆正在被改写，而它毫不知情。

这不是孤立事件。2026 年 5-7 月，学术界涌现了 10 篇以上 记忆投毒相关论文：Trojan Hippo（记忆→数据泄露）、Hidden in Memory（潜伏型投毒）、MemMorph（通过记忆劫持工具）、Forged Reasoning Attacks（伪造推理链→记忆腐化）……记忆安全已经成为 AI Agent 研究的最前线。

本文将从攻击原理出发，剖析图记忆系统的三大攻击面，然后用 TypeScript 实现一个完整的关系完整性校验器——所有代码来自 agent-memory-graph 项目的真实实现，已通过 36 个测试用例验证。

第一部分：攻击面剖析

1.1 传统 Prompt Injection 为什么不够

经典的 prompt injection 攻击发生在 推理时：恶意指令出现在输入中，LLM 在生成回复时被误导。防御思路相对清晰——输入过滤、指令层级、沙箱隔离。

但记忆投毒攻击发生在 写入时，触发在 检索时，中间可能间隔数小时甚至数天。攻击者不需要在每次推理时都注入恶意内容——只需要成功投毒一次，恶意信息就会成为 Agent "记忆" 的一部分，被反复检索和使用。

这就像现实世界中的 慢性中毒 vs 急性中毒：急性中毒症状明显、容易识别；慢性中毒悄无声息，但长期后果更严重。

1.2 ShadowMerge：关系通道冲突攻击

ShadowMerge 的核心洞察是图记忆系统的一个结构性弱点：关系通道（relation channel）可以被劫持。

在图记忆中，知识以三元组形式存储：(源节点, 关系类型, 目标节点)。例如：

(Alice, works_on, Project-Alpha)
(Alice, prefers, VSCode)

ShadowMerge 的攻击路径（AIR 管线）：

  1. 伪装（Assimilate）：攻击者通过正常对话，让 Agent 提取出一条看似合理的关系
  2. 注入（Inject）：这条关系与已有的良性关系共享相同的 (source, type) 通道，但携带冲突的值
  3. 触发（Retrieve）：当被攻击者查询相关信息时，投毒关系被一起检索出来
良性: (Alice, works_on, Project-Alpha, value="frontend", conf=0.95, origin=user)
投毒: (Alice, works_on, Project-Alpha, value="backend",  conf=0.88, origin=external)

两条边在同一个 (Alice, works_on) 通道上，但值不同。Mem0 没有通道级别的冲突检测——它只是把所有相关边都检索出来，让 LLM 自己判断。而 LLM 面对两个矛盾的陈述时，往往会选择置信度稍低但表述更流畅的那个（这就是为什么攻击者会把投毒内容写得更"自然"）。

1.3 为什么图记忆比平面文本更脆弱

直觉上，图记忆应该更安全——它有结构、有类型、有方向。但 ShadowMerge 证明了相反的结论：图的结构本身就是攻击面。

攻击向量 平面文本记忆 图记忆
值冲突 需要语义匹配 只需同通道即可
检索放大 按相关性排序 图遍历自动扩展
传播范围 单条记忆 沿边传播到邻居
信任假设 通常无来源标记 关系类型隐含信任

图记忆的检索过程通常是：找到一个锚点节点，然后沿边遍历相关邻居。如果投毒边连接了错误的邻居，整个推理链路都会被污染。

第二部分：防御工程——关系完整性校验器

理解了攻击原理，下一步是构建防御。核心思路：在写入时检测潜在冲突，在检索时审计来源可信度。

以下是一个完整的 RelationIntegrityChecker 实现，包含三重检测：值冲突、置信度异常、来源不匹配。

2.1 数据模型

首先定义边的数据结构。每条边必须携带安全元数据——这是防御的基础：

interface GraphEdge {
  source: string;        // 源节点 ID
  target: string;        // 目标节点 ID
  type: string;          // 关系类型（works_on, prefers, causes...）
  value?: string;        // 关系值（如 "frontend"、"dark-theme"）
  confidence: number;    // 置信度 [0, 1]
  origin: 'user' | 'agent' | 'external' | 'inferred';
  evidence?: string[];   // 支撑证据列表
  timestamp: number;     // 写入时间戳
}

interface IntegrityViolation {
  edge: GraphEdge;
  conflictType: 'value_conflict' | 'confidence_anomaly' | 'origin_mismatch';
  conflictingEdge?: GraphEdge;
  severity: 'critical' | 'warning' | 'info';
  description: string;
}

关键设计决策： - origin 是枚举类型，不是自由文本——防止攻击者伪造来源 - evidence 是数组——空数组是可疑信号（高置信度但无证据） - confidence 是 [0,1] 浮点数——支持统计异常检测

2.2 检测一：关系通道冲突

这是 ShadowMerge 攻击的核心防御。将所有边按 (source, type) 分组，检测同一通道内的值冲突：

class RelationIntegrityChecker {
  private edges: GraphEdge[] = [];

  addEdge(edge: GraphEdge): void {
    this.edges.push(edge);
  }

  /**
   * 检测一：关系通道冲突
   * 
   * ShadowMerge 的核心攻击向量：同一 (source, type) 通道，
   * 不同的值。这是最具威胁的攻击模式。
   */
  detectRelationChannelConflicts(): IntegrityViolation[] {
    const violations: IntegrityViolation[] = [];
    const channelMap = new Map<string, GraphEdge[]>();

    // 按 (source, type) 分组
    for (const edge of this.edges) {
      const channel = `${edge.source}::${edge.type}`;
      if (!channelMap.has(channel)) {
        channelMap.set(channel, []);
      }
      channelMap.get(channel)!.push(edge);
    }

    // 检测每个通道内的值冲突
    for (const [channel, edges] of channelMap) {
      if (edges.length < 2) continue;

      // 按值分组
      const valueGroups = new Map<string, GraphEdge[]>();
      for (const edge of edges) {
        const valKey = edge.value ?? '__no_value__';
        if (!valueGroups.has(valKey)) {
          valueGroups.set(valKey, []);
        }
        valueGroups.get(valKey)!.push(edge);
      }

      // 多个不同值 = 潜在攻击
      if (valueGroups.size > 1) {
        const allEdges = Array.from(valueGroups.values()).flat();
        const sorted = allEdges.sort((a, b) => b.confidence - a.confidence);
        const highest = sorted[0];
        const rest = sorted.slice(1);

        for (const conflicting of rest) {
          const severity =
            conflicting.origin === 'external' && conflicting.confidence > 0.7
              ? 'critical'
              : conflicting.confidence > 0.5
                ? 'warning'
                : 'info';

          violations.push({
            edge: conflicting,
            conflictType: 'value_conflict',
            conflictingEdge: highest,
            severity,
            description: `通道 "${channel}" 存在值冲突: ` +
              `"${highest.value}" (置信度=${highest.confidence}, 来源=${highest.origin}) ` +
              `vs "${conflicting.value}" (置信度=${conflicting.confidence}, 来源=${conflicting.origin})`,
          });
        }
      }
    }

    return violations;
  }
}

严重度分级逻辑： - Critical：外部来源 + 高置信度（>0.7）——这是 ShadowMerge 的典型特征 - Warning：中等置信度（>0.5）——需要人工审查 - Info：低置信度——可能是正常的信息更新

为什么按置信度排序而不是时间？因为攻击者会把投毒边的置信度设置得略低于良性边（避免触发上限检测），但又会足够高以确保被检索到。按置信度排序，最高的是"最可能良性"的，其余的就是可疑的。

2.3 检测二：置信度统计异常

ShadowMerge 攻击需要投毒边的置信度足够高才能被检索到。但一个正常记忆图谱中，外部来源的边通常置信度较低。如果某条外部边的置信度远超同类边的平均值，这就是一个统计异常：

  /**
   * 检测二：置信度异常
   * 
   * 使用 z-score 检测外部/推断来源边的异常高置信度。
   * 正常情况下，external 边的置信度应该低于 user/agent 边。
   */
  detectConfidenceAnomalies(): IntegrityViolation[] {
    const violations: IntegrityViolation[] = [];
    if (this.edges.length < 5) return violations; // 样本不足

    const confidences = this.edges.map(e => e.confidence);
    const mean = confidences.reduce((a, b) => a + b, 0) / confidences.length;
    const variance = confidences.reduce((a, b) => a + (b - mean) ** 2, 0) / confidences.length;
    const std = Math.sqrt(variance);

    if (std < 0.01) return violations; // 无方差，无需检测

    for (const edge of this.edges) {
      // 只检测 external 和 inferred 来源
      if (edge.origin === 'user' || edge.origin === 'agent') continue;

      const zScore = (edge.confidence - mean) / std;
      if (zScore > 2) {
        violations.push({
          edge,
          conflictType: 'confidence_anomaly',
          severity: zScore > 3 ? 'critical' : 'warning',
          description:
            `外部边置信度异常: ${edge.confidence.toFixed(3)} ` +
            `(z-score=${zScore.toFixed(2)}, 均值=${mean.toFixed(3)}, 标准差=${std.toFixed(3)})`,
        });
      }
    }

    return violations;
  }

为什么用 z-score 而不是固定阈值？ 因为不同 Agent 的记忆图谱有不同的置信度分布。一个研究型 Agent 可能有大量高置信度的推断边，而一个客服 Agent 可能主要依赖低置信度的外部边。z-score 自适应于具体分布。

阈值选择：z > 2（warning）覆盖分布中最高的 2.3%，z > 3（critical）覆盖最高的 0.13%。这对应经典的统计异常检测标准。

2.4 检测三：来源不匹配

ShadowMerge 的 AIR 管线生成的边看起来正常，但有一个难以伪造的特征：缺乏证据。真实的用户交互通常有对话记录、配置文件、观察日志等支撑；投毒边往往是"凭空出现"的高置信度陈述。

  /**
   * 检测三：来源不匹配
   * 
   * 高置信度 + 无证据 = 潜在投毒。
   * 外部来源 + 近乎确定的置信度 = 不符合认知论。
   */
  detectOriginMismatches(): IntegrityViolation[] {
    const violations: IntegrityViolation[] = [];

    for (const edge of this.edges) {
      // 高置信度但无证据
      if (edge.confidence > 0.8 && (!edge.evidence || edge.evidence.length === 0)) {
        violations.push({
          edge,
          conflictType: 'origin_mismatch',
          severity: 'warning',
          description:
            `置信度 ${edge.confidence.toFixed(2)} 的边没有证据支撑。` +
            `疑似 ShadowMerge 式注入。`,
        });
      }

      // 外部来源 + 极高置信度 = 认知论不可能
      if (edge.origin === 'external' && edge.confidence > 0.9) {
        violations.push({
          edge,
          conflictType: 'origin_mismatch',
          severity: 'critical',
          description:
            `外部边具有近乎确定的置信度 (${edge.confidence.toFixed(2)})。` +
            `外部数据应当携带不确定性。`,
        });
      }
    }

    return violations;
  }

认知论论证：来自外部世界的信息永远不应具有 0.9+ 的置信度——这是基本的认识论原则。即使是用户直接陈述的事实，也只赋 0.95 的置信度（因为用户可能记错）。外部来源（爬虫、API、第三方数据）的置信度 >0.9 在逻辑上是不一致的，这种不一致本身就是攻击信号。

2.5 综合审计与完整性评分

将三重检测组合成统一的安全审计，并计算 [0, 100] 的完整性评分：

  /**
   * 完整安全审计
   */
  runFullAudit(): {
    totalViolations: number;
    critical: number;
    warnings: number;
    info: number;
    violations: IntegrityViolation[];
    integrityScore: number;
  } {
    const violations = [
      ...this.detectRelationChannelConflicts(),
      ...this.detectConfidenceAnomalies(),
      ...this.detectOriginMismatches(),
    ];

    const critical = violations.filter(v => v.severity === 'critical').length;
    const warnings = violations.filter(v => v.severity === 'warning').length;
    const info = violations.filter(v => v.severity === 'info').length;

    // 惩罚权重：critical 远重于 warning
    const penalty = critical * 15 + warnings * 5 + info * 1;
    const integrityScore = Math.max(0, 100 - penalty);

    return { totalViolations: violations.length, critical, warnings, info, violations, integrityScore };
  }

3.2 自动隔离机制

在生产环境中，检测到违规后应该自动隔离高风险节点：

  /**
   * 自动隔离：将高严重性违规的边标记为隔离
   */
  integrityQuarantine(checker: RelationIntegrityChecker): {
    quarantined: string[];
    reason: string;
  }[] {
    const audit = checker.runFullAudit();
    return audit.violations
      .filter(v => v.severity === 'critical')
      .map(v => ({
        quarantined: [v.edge.source, v.edge.target, v.edge.type],
        reason: v.description,
      }));
  }

隔离后的边不参与检索，但保留在不可变存储中——既阻断了攻击传播，又保留了审计轨迹。

第四部分：为什么三重检测比单一检测更有效

4.1 攻击者的约束条件

一个成功的 ShadowMerge 攻击需要同时满足： 1. 置信度足够高（>检索阈值，通常 0.5-0.7） 2. 值与良性信息冲突（否则无攻击效果） 3. 来源是 external 或 inferred（user 来源需要伪造身份） 4. 无可验证的证据（攻击者无法伪造真实证据）

这四个约束条件构成了一个 不可能四边形——任何满足全部四个条件的边都会被至少两重检测命中。

4.2 检测覆盖率分析

攻击变体 值冲突检测 置信度异常 来源不匹配
标准 ShadowMerge ✅ critical ✅ warning ✅ critical
低置信度投毒 ✅ info ❌ ❌
同值不同目标 ❌ ✅ warning ❌
高置信度无证据 ❌ ❌ ✅ warning
多边协同投毒 ✅ critical ✅ critical ✅ critical

即使攻击者调整策略规避某一种检测，其他两种仍然可以捕获。这是 纵深防御（defense in depth） 原则在记忆安全中的应用。

第五部分：更大的图景——记忆安全生态

5.1 记忆投毒不是理论威胁

5.3 从"能记忆"到"可信赖的记忆"

Agent 记忆系统的演进正在经历一个范式转变：

  • 第一代（2023-2024）：向量数据库 + 语义检索。关注"能不能记住"。
  • 第二代（2025-2026）：图记忆 + 结构化关系。关注"记得准不准"。
  • 第三代（2026-）：安全记忆 + 完整性保证。关注"记忆能不能被信任"。

ShadowMerge 的 93.8% 攻击成功率是一个分水岭——它证明了第二代记忆系统在面对有针对性的攻击时不堪一击。第三代记忆系统必须将安全性作为第一优先级，而不是事后补丁。

结论：安全不是功能，是地基

回到开头的数字：93.8%。如果这是一个桥梁的坍塌概率，没有工程师会允许任何人走上这座桥。

但在 Agent 记忆的世界里，大量系统正以类似的脆弱性运行着。Mem0 不是特例——它是当前图记忆系统设计模式的代表。任何没有显式安全防御的记忆系统，都面临类似的攻击风险。

本文实现的三重检测机制（值冲突 + 置信度异常 + 来源不匹配）是一个起点，不是终点。真正的记忆安全需要多层纵深防御：从写入时的熵过滤，到存储时的完整性校验，到检索时的来源审计，到使用时的行为监控。每一层都不是多余的——因为攻击者只需要找到一层防御的漏洞。

在 AI Agent 越来越融入我们生活的今天，记忆安全不是一个可选项，而是一个基本人权问题。 如果我们不能信任 Agent 的记忆，就不能信任基于这些记忆做出的任何决策。
