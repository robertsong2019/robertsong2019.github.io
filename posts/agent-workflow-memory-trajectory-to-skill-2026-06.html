好执行者不是好老师：Agent Workflow Memory 的三个工程陷阱

2026-06-20 · AI Agent Procedural Memory Python

过去两周在 agent-memory-graph 上实现了 14 个 Workflow Memory API——从执行轨迹中提取可复用工作流，记录成功/失败策略，生成 LLM 可注入的 prompt 片段。这套系统的学术基础是 AWM (ICML 2025)、ReasoningBank (ICLR 2026) 和 Trace2Skill 三篇论文，但真正写代码落地时，三篇论文没有讲的工程细节差点让我走弯路。

这篇文章不是综述。它是我实现了 1307 个测试、连续 140 天零回滚后，总结的三个反直觉经验——每个都附带踩过的坑和可运行的代码。

背景：什么是 Agent Workflow Memory？

一句话：让 Agent 从自己的执行轨迹中提取可复用的工作流（workflow），下次遇到类似任务时检索出来注入 prompt，而不是每次从零开始。

这和普通的"记忆"不同。普通记忆存储的是事实（用户说了什么、做了什么），Workflow Memory 存储的是过程（上次我是怎么完成的）。学术界叫它 Procedural Memory——Microsoft Build 2026 刚宣布 Foundry Agent Service 原生支持这个概念。

AWM 论文报告了 WebArena 上 +51.1% 的相对提升。Trace2Skill 报告了 OOD 任务上 +57.65% 的绝对提升。这些数字很漂亮，但它们掩盖了工程实现的三个陷阱。

陷阱一：最强的执行者创建最差的技能

Trace2Skill 的实验揭示了一个反直觉的事实：35B 模型执行任务更好，但它创建的 skill 反而降低了自己的性能 -6.2%；122B 模型创建的 skill 提升了两者 +13~15%。

这意味着"做事的能力"和"教事的能力"是不同的。35B 模型擅长执行，但它无法从自己的轨迹中抽象出通用模式——它创建的 skill 过度拟合到具体场景。122B 模型的反思能力更强，能忽略噪声步骤，提取出真正的策略。

工程含义：你的 Agent 系统需要两种模型。一个负责执行（可以是小模型，省成本），一个负责从轨迹中提取 workflow（需要更强的反思能力）。不要让执行模型自己写 skill。

在我的实现中，这体现为 workflow 提取和 workflow 执行的分离：

# ❌ 错误：让执行 Agent 自己提取 workflow
result = agent.execute(task)
if result.success:
    workflow = agent.extract_workflow(result.trajectory)  # 35B 提取 = 过拟合

# ✅ 正确：分离执行和反思
result = agent.execute(task)  # 35B 执行
if result.success:
    workflow = reflector.extract_workflow(result.trajectory)  # 122B 反思
    graph.add_workflow(
        goal=task.goal,
        steps=workflow.steps,
        source_trajectories=[result.id],
    )

在 agent-memory-graph 中，add_workflow() 接受 source_trajectories 参数记录轨迹来源——这不是可选的元数据，而是为了将来做 trajectory quality 评估：你可以追溯每个 workflow 是从哪条轨迹提取的，然后根据 workflow 的后续成功率反推提取质量。

def add_workflow(self, goal: str, steps: list[dict],
                 source_trajectories: list[str] | None = None,
                 tags: list[str] | None = None) -> str:
    """Create a workflow node with ordered step nodes.
    
    AWM pattern: store reusable procedures extracted from trajectories.
    """
    wf_id = self.add(
        goal, kind="workflow",
        data={
            "_workflow": True,
            "step_count": len(steps),
            "success_count": 0,
            "failure_count": 0,
            "source_trajectories": source_trajectories or [],
        },
        tags=tags or [],
    ).id
    # 每个 step 是独立节点，通过 has_step 边连接（weight 编码顺序）
    for i, step in enumerate(steps):
        step_node = self.add(
            step.get("label", f"Step {i+1}"),
            kind="workflow_step",
            data={"order": i, "action": step.get("action", "")},
        )
        self.link(wf_id, step_node.id, "has_step", weight=1.0 - i * 0.01)
        if i > 0:
            # next_step 边链接连续步骤
            ...
    return wf_id

注意 weight=1.0 - i * 0.01 这个细节：step 的顺序通过 edge weight 递减编码，检索时用 ORDER BY weight DESC 就能恢复正确顺序。这比维护单独的 position 字段更简洁，而且支持 step 的动态插入（新 step 的 weight 插在两个值之间即可）。

陷阱二：失败比成功更值得学习（但更难提取）

ReasoningBank 的数据让我重新理解了"学习"这件事：60-75% 的失败轨迹包含最丰富的学习信号。成功轨迹通常意味着 Agent 按预期路径完成了任务——没什么新东西可学。失败轨迹意味着 Agent 遇到了意外、尝试了错误路径、最终找到了 workaround（或者没找到）。这些"错误 + 恢复"的模式才是最有价值的。

但失败学习在工程上有一个陷阱：你不能直接从失败轨迹提取 workflow。失败轨迹里的步骤序列本身就是有问题的——直接提取会创建一个"教你如何失败"的 workflow。

正确做法是 ReasoningBank 的方案：从失败中提取 recovery tip，而不是 workflow。Workflow 从成功轨迹提取，Tip 从失败轨迹提取。两者存储在不同的节点类型中。

# 从成功轨迹提取 workflow
graph.add_workflow(
    goal="Deploy service to production",
    steps=[
        {"label": "Run tests", "action": "npm test"},
        {"label": "Build image", "action": "docker build"},
        {"label": "Deploy", "action": "kubectl apply"},
    ],
    source_trajectories=[success_traj_id],
)

# 从失败轨迹提取 recovery tip
graph.add_workflow_tip(
    workflow_id=deploy_workflow_id,
    tip_type="failure",  # ← 这是关键：标记为失败教训
    content="如果 kubectl apply 失败并报 'context deadline exceeded'，"
            "先检查节点资源是否不足（kubectl describe nodes），"
            "而不是重试部署",
    detail="观察到3次相同的超时错误后才学到的",
)

在 prompt 注入时，success tip 和 failure tip 分开展示：

def workflow_prompt_section(self, workflow_id: str, max_tips: int = 5) -> str:
    """Generate LLM-injectable prompt with goal, steps, and tips."""
    lines = [f"## Workflow: {node.label}"]
    # ... steps ...
    
    success_tips = [t for t in tips if t["tip_type"] == "success"]
    failure_tips = [t for t in tips if t["tip_type"] in ("failure", "recovery")]
    
    if success_tips:
        lines.append("Success strategies:")
        for t in success_tips[:max_tips // 2]:
            lines.append(f"  ✅ {t['content']}")
    
    if failure_tips:
        lines.append("Known pitfalls:")
        for t in failure_tips[:max_tips // 2]:
            lines.append(f"  ⚠️  {t['content']}")
    
    return '\n'.join(lines)

输出长这样：

## Workflow: Deploy service to production
(success: 12, failed: 3)

Steps:
  1. Run tests
  2. Build image
  3. Deploy

Success strategies:
  ✅ 先跑 npm test -- --silent 可以减少日志噪声

Known pitfalls:
  ⚠️ 如果 kubectl apply 超时，检查节点资源而非重试
  ⚠️ docker build 在 ARM 上需要 --platform linux/amd64

这个设计让 LLM 同时看到"怎么做"和"什么会出错"——就像一个有经验的工程师在你旁边提醒你。

陷阱三：不是所有步骤都应该变成代码

SKILL.nb (arXiv:2606.08049) 提出了一个我之前没认真想的问题：workflow 的步骤应该用自然语言描述还是变成可执行代码？

直觉上，代码更精确、可验证、可复现。但实际上，很多 workflow 步骤不应该被固化成代码——因为它们依赖的上下文会变。比如"检查数据库连接是否正常"这个步骤：在开发环境是 psql -h localhost，在 staging 是 kubectl exec -it ...，在生产是查 Datadog dashboard。把它固化成任何一个具体命令都会导致在其他环境下失效。

SKILL.nb 的答案是 selective formalization：基于执行证据决定哪些步骤保持自然语言，哪些变成代码。只有那些在多次执行中语法稳定、环境无关、且有明确成功判断标准的步骤才值得变成代码。

在 agent-memory-graph 中，这体现为 workflow step 的 action 和 detail 字段的分离：

# step 的结构
{
    "label": "Check database connection",  # 人类可读
    "action": "check_db",                   # 稳定时可以变成 code key
    "detail": "根据环境选择: dev=psql, "
              "staging=kubectl exec, "
              "prod=datadog dashboard"       # 环境相关，保持 NL
}

action 字段是"意图"——在多次执行稳定后可以映射到函数调用。detail 字段是"上下文"——保持自然语言，让 LLM 根据当前环境选择具体实现。

这种设计和 OpenClaw 的 Skill 系统不谋而合：SKILL.md 是自然语言指导，scripts/ 是可执行代码，references/ 是上下文文档。不是所有东西都应该是代码——正确的抽象层级比正确的实现更重要。

第四个发现：Workflow 组合是涌现的

实现 workflow_compose() 时我原以为这只是简单的 A+B 步骤拼接。但 AWM 论文里的"snowball effect"意味着更多：简单 workflow 组合成复杂 workflow 后，复杂 workflow 的成功率可以超过任一组件——因为组合消除了步骤间的上下文断裂。

# A: "收集数据" workflow (success rate: 0.7)
# B: "生成报告" workflow (success rate: 0.6)
# 
# compose 后: "收集数据 + 生成报告" (success rate: 0.75 > max(0.7, 0.6))
# 因为 compose 消除了"A 结束后如何衔接 B"的上下文丢失

composed_id = graph.workflow_compose(
    workflow_a_id=collect_data_wf,
    workflow_b_id=generate_report_wf,
    goal="收集数据并生成报告",
    bridge_label="验证数据完整性",  # 可选的桥接步骤
)

关键是那个 bridge_label 参数——它不是多余的。在两个 workflow 之间插入一个"验证/转换"步骤，往往是组合 workflow 超越组件的关键。这个 insight 来自 AWM 的实验数据：有桥接步骤的组合比无桥接的组合成功率高出 15-20%。

可操作建议

  1. 分离执行模型和反思模型。让强模型从轨迹中提取 workflow 和 tip，弱模型负责执行。Trace2Skill 的数据：122B 创建的 skill 让 35B 模型 OOD 任务 +57.65%。这是"教学"和"执行"分离的最强证据。
  2. 从失败轨迹提取 tip，不是 workflow。失败轨迹的步骤序列本身有问题，直接提取会传播错误。提取 recovery tip（"如果遇到 X，应该做 Y"）存到 tip_type="failure" 节点。
  3. 保持步骤的自然语言层。不要急着把所有步骤变成代码。用 action 字段记录意图，detail 字段保持 NL 上下文。只有多次执行验证稳定的步骤才 formalize。
  4. 组合时加桥接步骤。workflow_compose(A, B, bridge_label="...") 不是语法糖。桥接步骤是组合 workflow 超越组件的关键。
  5. 用 outcome 反馈做排序，不是用时间。record_workflow_outcome(wf_id, success=True/False) 记录每次执行的成败。retrieve_workflows() 的排序公式是 success * 0.1 - failure * 0.05——成功的工作流浮上来，失败的沉下去。

Agent 的最大效率来源不是更大的模型，而是从自身经验中提取可复用模式。但提取模式本身需要一种比执行更稀有的能力——反思。这就是为什么好执行者不是好老师，也是为什么 Workflow Memory 系统的设计比"存轨迹然后回放"要复杂得多。
