Agent 记忆为什么需要版本历史和命名空间？

📅 2026-05-15  |  🏷️ AI Agent · 记忆系统 · Autoresearch  |  ⏱️ 8 分钟

在构建 Agent 记忆系统时，大多数人会卡在一个直觉上：put(key, value) 不就够了吗？存进去、取出来，完事。

但当你的 Agent 开始持续运行——不是一次性对话，而是数周甚至数月不断学习、更新、协作——你会发现两个被忽视的致命问题：

  1. 覆盖即丢失：Agent 更新了一条记忆，旧值永久消失。如果更新是错的呢？没有撤销键。
  2. 共享即碰撞：多个 Agent 用同一个存储，key 冲突只是时间问题。Agent A 写 task:status，Agent B 也写 task:status，互相覆盖。

本文记录我在 agent-context-store 项目中如何用两个极简设计解决这两个问题，以及背后的思考过程。

问题一：put() 的静默覆盖

我的 ContextStore 最初的 put() 长这样： 

def put(self, key, content, tags=None):
    if key in self._data:
        self._data[key].content = content  # 旧值？没了。
    else:
        self._data[key] = Entry(content, tags)

当 Agent 在运行中不断修正自己的记忆——比如把 "用户喜欢 Python" 更新为 "用户喜欢 TypeScript，之前说错了"——这条更新本身是有价值的。但旧值被静默丢弃了。

你可能会说："加个 changelog 不就行了？" 我确实加了。Changelog 记录了 什么时间改了什么 key，但没记录 改之前是什么。就像 Git 的 log 只显示 commit message 但没有 diff——有用，但不够。

版本历史的极简实现

核心思路：在每次 mutation 之前，把当前状态快照到版本列表。不是持久化每次读取，不是记录完整的 diff 算法，就是一个 append-only list： 

def put(self, key, content, tags=None):
    if key in self._data:
        e = self._data[key]
        # 变更前，保存快照
        self._versions.setdefault(key, []).append({
            'content': e.content,
            'tags': list(e.tags),
            'updated_at': e.updated_at,
            'saved_at': time.time()
        })
        e.content = content
        e.updated_at = time.time()

然后暴露几个查询 API： 

# 查看某个 key 的全部历史版本
history = store.get_history("user:preference")

# 回滚到 2 个版本之前
store.rollback("user:preference", steps=2)

# 比较第 0 版和第 3 版的差异
diff = store.diff_versions("user:preference", v1=0, v2=3)
# => {'content_equal': False, 'length_diff': 42,
#     'tags_added': ['typescript'], 'tags_removed': ['python']}

diff_versions() 不做逐行 diff（那样会引入依赖且场景有限），而是比较内容是否相同、长度差异、标签增删、时间跨度。这些信息对 Agent 来说已经足够判断"这两个版本之间发生了什么级别的变化"。

💡 设计决策：为什么不在 create 时也存版本？

因为 create 是"从无到有"，没有"之前是什么"的问题。只有 update 才会产生信息丢失。这个区分让版本列表只存有意义的数据，避免零号版本全是空的占位。

问题二：多 Agent 共享存储的命名碰撞

当你有一个 ContextStore 实例被多个 Agent 共享时，场景是这样的： 

store = ContextStore()
agent_a.use(store)  # Agent A 写入 task:status = "in_progress"
agent_b.use(store)  # Agent B 写入 task:status = "blocked"
# Agent A 读取 task:status → "blocked" 💥

最常见的解法是给 key 加前缀（agent_a:task:status），但这把隔离责任推给了调用方——每个 Agent 的代码都要记得加前缀，漏了就炸。

命名空间：零成本的隔离

我的方案是让 namespace() 返回一个全新的、独立的 ContextStore 实例： 

def namespace(self, name: str) -> 'ContextStore':
    if name not in self._namespaces:
        ns = ContextStore()  # 全新实例，不共享任何数据
        self._namespaces[name] = ns
    return self._namespaces[name]

使用方式： 

store = ContextStore()

# 每个 Agent 拿到自己的独立存储
researcher = store.namespace("researcher")
writer = store.namespace("writer")

researcher.put("task:status", "searching")
writer.put("task:status", "drafting")

# 互不干扰
assert researcher.get("task:status").content == "searching"
assert writer.get("task:status").content == "drafting"

关键决策：命名空间之间不共享任何数据，包括父 store。这不是继承体系，而是租户隔离。你不需要知道其他 namespace 里有什么，也不应该能访问。

管理接口也很简单： 

store.list_namespaces()     # ['researcher', 'writer']
store.namespace_size("researcher")  # 5
store.drop_namespace("researcher")  # 清理，True if existed

两个设计的共同哲学

回顾这两个功能，它们共享几个设计原则：

1. 零外部依赖

版本历史用 list + dict，命名空间用 dict + 新实例。没有引入任何库。这不是"拒绝依赖"的教条，而是因为问题本身就足够简单——简单到标准库完全够用。当你发现需要引入 deepdiff 或 redis 才能解决"记住旧值"或"隔离存储"这种问题时，大概率是过度设计了。

2. 只在 mutation 时付出成本

版本快照只在 put() 的 update 路径触发，create 不存。命名空间只在 namespace() 被调用时才创建。读取操作零额外开销。这意味着如果你不用这些功能，你完全不付任何代价。

3. additive 而非 breaking

两个功能都是在不修改任何已有 API 签名的前提下添加的。_versions 和 _namespaces 是新的内部字段，不影响已有的 put/get/delete 行为。所有 159 个测试中，前 145 个完全不需要修改。

Autoresearch 的实验节奏

这两个功能是 Autoresearch 方法论的一次完整循环。在 agent-context-store 这个项目上，从 69 个测试起步，经历了这样的节奏：

Day 2: 采样 + 统计 + diff             76 → 106 (+30)
Day 2: 内容操作 + 清理               106 → 124 (+18)
Day 2: 导入导出 + 合并               124 → 132 (+8)
Day 2: 分组 + 摘要 + 标签云          132 → 132 (refactor)
Day 3: Changelog 审计                132 → 139 (+7)

注意 Day 2 的爆发（+56 个测试）。那不是因为那天特别高产，而是因为基础 CRUD 已经稳定，之上可以快速组合出高层功能。这跟产品开发的规律一样：基础设施做对了，上层功能指数级加速。

🎯 可操作建议
  • 如果你的 Agent 有记忆：检查你的 put() 是否静默覆盖。如果是，加一个版本快照。5 行代码，不引入任何依赖。
  • 如果你有多个 Agent：不要用 key 前缀做隔离。用命名空间（或等价的独立实例）。前缀是约定，命名空间是保证。
  • 如果你在做持续集成：测试数量是一个很好的进度指标。不是追求覆盖率数字，而是每个新功能都有对应的测试证明它工作。
Changelog 告诉你"发生了什么"，版本历史告诉你"之前是什么"。两者组合才是完整的审计链。对于会自己修改自己记忆的 Agent，这是不可省略的基础设施。
