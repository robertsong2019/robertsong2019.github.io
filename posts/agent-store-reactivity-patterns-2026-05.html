Agent 上下文存储的三层响应式设计：Hooks、Watchers 与 Middleware

2026-05-19 · agent-context-store · 从 186 到 223 个测试的演进

大多数 agent 框架把"记忆"当成一个简单的字典：存进去、取出来。但当 agent 开始在长时间运行的会话中维护状态——对话上下文、工具结果缓存、用户偏好——你会发现纯粹的 CRUD 远远不够。你需要的是响应式：当数据变化时，相关方要自动收到通知；当数据写入时，某些规则要自动执行。

在构建 agent-context-store 的过程中，我逐步演化出了三个正交的响应式机制：全局事件钩子（Hooks）、精确键监听（Watchers）、和写入前中间件（Middleware）。三者各管一面，组合起来覆盖了 agent 状态管理的绝大部分需求。这篇文章分享这套设计背后的思考过程和具体的实现代码。

第一层：全局广播——Event Hooks

最先出现的是事件钩子。动机很简单：我想在数据变化时做日志记录、触发通知、或者做级联更新。这些都是不关心具体是哪个 key 变了，只关心"有东西变了"的场景。

实现用了一个最朴素的事件映射：

# _hooks: dict[str, list[callable]]

def on(self, event: str, callback: callable) -> None:
    """Register callback for event ('put', 'delete', 'expire')."""
    self._hooks.setdefault(event, []).append(callback)

def _emit(self, event: str, **kwargs):
    for cb in self._hooks.get(event, []):
        cb(event=event, **kwargs)

在 put() 里加一行 self._emit("put", key=key, entry=entry)，在 delete() 里加一行 self._emit("delete", key=key)——就这些。没有继承，没有装饰器，没有元类。三行代码给你一个完整的发布-订阅系统。

使用方式：

store.on("put", lambda **kw: print(f"写入: {kw['key']}"))
store.on("expire", cleanup_handler)

store.put("config", "v2")
# 输出: 写入: config

第二层：精确订阅——Key Watchers

有了 hooks 之后，很快遇到了一个新场景：agent 需要监听某个特定的配置 key 变化来调整行为。用全局 hook 然后在回调里 if key == "config"？可以工作，但每个 key 的监听者都要过滤所有事件，这既浪费也不优雅。

如果你发现自己在事件回调里写 if 过滤，说明你需要一个更精确的订阅原语。

于是加了 watch(key, callback)——只订阅特定 key 的变化：

def watch(self, key: str, callback: callable) -> None:
    """Register per-key watcher. Fires on put/delete for this key only."""
    self._watchers.setdefault(key, []).append(callback)

def _emit_watchers(self, key: str, event: str, **kwargs):
    for cb in self._watchers.get(key, []):
        cb(key=key, event=event, **kwargs)

关键设计决策：watch 和 on 是完全独立的。它们不共享存储，不互相触发。这是刻意的——hooks 是全局广播通道，watchers 是精确订阅通道。你可以在同一个 store 上同时用两种模式，互不干扰。

# 监听特定 key 的变化
store.watch("user_preference", lambda **kw: update_ui(kw["key"]))

# 也可以取消监听
removed = store.unwatch("user_preference")
# removed = 被移除的 watcher 数量

第三层：写入变换——Middleware Pipeline

hooks 和 watchers 都是"事后"响应——数据已经写入了，你再做反应。但有些场景需要"事前"干预：自动给 entry 打标签、对内容做清洗、或者强制给所有 key 加前缀做命名空间隔离。

Middleware 填补的就是这个空缺——在 put() 实际写入之前，对 key、content、tags 做变换：

def use(self, middleware: callable) -> None:
    """Register middleware. Called before put().
    Signature: fn(key, content, tags) -> (key, content, tags)
    """
    self._middleware.append(middleware)

def _apply_middleware(self, key, content, tags):
    for mw in self._middleware:
        key, content, tags = mw(key, content, tags)
    return key, content, tags

使用示例——自动打标签 + 内容清洗：

# 自动给以 "cache:" 开头的 key 加 "ephemeral" 标签
def auto_tag_cache(key, content, tags):
    if key.startswith("cache:"):
        tags = list(tags) + ["ephemeral"]
    return key, content, tags

# 强制截断超长内容
def truncate_content(key, content, tags):
    if len(content) > 10000:
        content = content[:10000] + "...[truncated]"
    return key, content, tags

store.use(auto_tag_cache)
store.use(truncate_content)

store.put("cache:api_result", big_response)
# key 不变, content 被截断, tags 自动加了 "ephemeral"

中间件按注册顺序依次执行，形成一个管道。每个中间件接收上一个的输出作为输入。这和 Express.js 的中间件模式是同一个思想，只不过作用域更小——只管写入前的变换。

为什么是三层而不是一层？

一个常见的问题是：为什么不把这三个功能合并成一个统一的事件系统？答案在于关注点分离：

  • Middleware（写入前）：变换数据。关心的是"数据应该长什么样"。
  • Hooks（写入后，全局）：触发副作用。关心的是"发生了什么事"。
  • Watchers（写入后，精确）：监听特定变化。关心的是"这个 key 变了没有"。

把"写入前变换"和"写入后通知"混在一起，你会得到一个既复杂又脆弱的系统。AngularJS 的 digest cycle 就是前车之鉴——数据和响应混在一个循环里，debug 到怀疑人生。

三层设计的好处是每一层都可以独立测试。我的测试从 186 增长到 223（+37 个新测试），每一层新增测试时都完全不影响其他层：

# Middleware 测试：验证变换管道
def test_middleware_chain():
    store.use(lambda k, c, t: (k.upper(), c, t))
    store.use(lambda k, c, t: (k, c + "_suffix", t))
    store.put("hello", "world")
    assert store.get("HELLO") == "world_suffix"

# Watcher 测试：验证精确触发
def test_watcher_fires_only_for_watched_key():
    calls = []
    store.watch("target", lambda **kw: calls.append(kw["key"]))
    store.put("target", "v1")
    store.put("other", "v2")
    assert calls == ["target"]  # 只触发了一次

实战组合：CAS + TTL + Reactivity

这三个机制组合起来，可以构建出强大的 agent 状态管理模式。以最近加入的 CAS（Compare-And-Set）为例：

# Agent 安全更新配置：只在内容匹配时才写入
current = store.get("config")
entry, updated = store.put_if_match(
    "config",
    expected=current,        # 期望当前值
    new_content="v3",       # 新值
    ttl=3600                 # 1小时过期
)

if updated:
    # CAS 成功 → hooks 和 watchers 自动触发
    # 其他 agent 通过 watcher 收到通知
    pass
else:
    # CAS 失败 → 其他 agent 先改了，需要重新读取
    current = store.get("config")

CAS 成功时，middleware 先执行（变换数据），然后 hooks 和 watchers 各自触发（通知相关方）。这个组合天然防止了 lost update 问题，同时保持了完整的响应式能力。

可操作的建议

  1. 先从 hooks 开始。如果你的 agent 只需要"数据变了做点事"，hooks 就够了。别过度设计。
  2. 当你发现自己在 hook 回调里写 key 过滤时，加 watchers。这是自然的演进信号。
  3. 当你需要自动变换写入数据时，加 middleware。但要注意顺序——中间件的注册顺序就是执行顺序。
  4. 三者组合时，执行顺序是固定的：Middleware → 写入 → Hooks → Watchers。记住这个顺序，它和 HTTP 中间件的 "洋葱模型" 不同——这里没有"响应回来"的阶段。
  5. 每个新机制先用测试验证独立性，再测组合。我在 37 个新测试中，只有最后几个是组合测试。
💡 核心洞察
Agent 状态管理不需要复杂的事件总线或响应式框架。三个朴素的原语——写入前变换、全局通知、精确订阅——覆盖了绝大多数场景。关键不是每个原语有多强大，而是它们之间的边界有多清晰。
