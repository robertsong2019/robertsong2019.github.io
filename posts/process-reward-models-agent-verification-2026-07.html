AI Agent 怎么知道自己错了？过程奖励模型正在重定义验证

2026-07-14 · 罗嵩 Process Reward Model Agent Verification RLVR
你让 AI Agent 帮你改一个 bug。它写了 20 步操作，改了 8 个文件，最后跑测试——挂了。你知道结果错了，但到底哪一步开始错的？Agent 自己不知道，你也不知道。这就是 2025 年之前 AI Agent 验证的现状。

一场被低估的革命

2023 年底，OpenAI 发了一篇论文叫《Let's Verify Step by Step》。当时没太多人注意——大家都在讨论 GPT-4 的上下文窗口又长了多少。但这篇论文埋下了一颗种子：过程奖励模型（Process Reward Model, PRM）。

种子在 2025-2026 年长成了大树。从数学推理到代码生成，从 GUI 操作到多 Agent 协作，PRM 已经成为 AI Agent 领域最活跃的研究方向之一。2026 年上半年就有 15+ 篇重量级论文发表，覆盖金融、软件工程、视觉推理等几乎所有 Agent 场景。

为什么 PRM 突然变得这么重要？因为Agent 的任务越来越长，而传统验证方法在长任务上彻底失效。

一、足球比分问题：为什么"只看结果"行不通

传统的 AI 验证用的是结果奖励模型（Outcome Reward Model, ORM）。它只看最终结果：答案对不对？代码能不能跑？测试过没过？

🎯 一个类比
想象你在看一场足球比赛的比分：3:0。你知道谁赢了，但你不知道：
• 哪个进球最关键？
• 防守哪里出了问题？
• 什么时候比赛走势开始逆转？
比分告诉你结果，但对改进毫无帮助。

对于 3 步的简单任务，这没什么问题——错了就重来。但当一个 SWE Agent 要执行 50 步操作（搜索代码、理解依赖、修改文件、运行测试、修复编译错误……），ORM 就变成了"黑箱里的黑箱"：

  • 成功了你不知道为什么成功（无法复现）
  • 失败了你不知道从哪步开始失败（无法修复）
  • 训练时，50 步只有 1 个信号（最终通过/不通过），梯度极其稀疏

SWE-TRACE（2026 年 4 月）给出了量化证据：在 50+ 步的 SWE 任务上，PRM 和 ORM 的性能差距超过 20 个百分点。任务越长，差距越大。

二、PRM 是什么：给每一步打分

过程奖励模型的核心思想极其简单：不要只看最终答案，要给每一步推理打分。

# 传统 ORM：只有一个信号
trajectory = [step_1, step_2, ..., step_50]
outcome = run_tests(final_code)  # 0 or 1

# PRM：每一步都有信号
step_scores = [prm(step_i) for i in range(50)]
# [0.9, 0.85, 0.3, 0.1, ...]
#          ↑ 从第 3 步开始出问题了

知道了哪一步出错，就能做三件以前做不到的事：

  1. 精确归因——第 27 步的文件修改引入了类型错误，直接定位
  2. 搜索引导——在生成时用 PRM 分数做 beam search，自动避开低质量路径（这就是 test-time scaling 的核心机制）
  3. 高效训练——RL 训练时，把奖励信号分配到关键步骤，而不是均匀撒在 50 步上

OREAL（上海 AI Lab，2025）用这个思路做到了一件震撼的事：7B 参数的模型，经过 outcome RL + token 级信用分配训练，在 MATH-500 上达到了 94.0 pass@1——之前只有 32B 模型能做到。相当于用更精确的奖励信号，把小模型的能力拉到了大模型水平。

三、2026 年的三大突破

突破 1：生成式 PRM——会思考的验证器

早期的 PRM 是一个分类器：输入推理步骤，输出一个 0 到 1 的分数。简单粗暴，但有个致命问题：它不能解释为什么打了这个分。

2025 年开始，研究者提出了生成式 PRM（GenPRM）。它不直接打分，而是先生成一段评估推理，然后给出分数：

步骤："要解 x² + 5x + 6 = 0，使用求根公式..."

GenPRM 的评估：
"该步骤正确识别了方程为二次方程。
但也可以直接因式分解：(x+2)(x+3) = 0，
更为简洁。求根公式方法正确但非最优。
步骤质量：正确但次优。
评分：0.7"

"PRMs That Think"（Khalifa et al., 2025）的数据：生成式 PRM 在 ProcessBench 上达到 78.1% 准确率，比分类器 PRM 高 6 个百分点。差距在更难的问题上还会拉大。

更妙的是，生成式 PRM 的评估文本可以直接给人类看——它天然就是一个可审计系统。当 Agent 出错时，你可以读 PRM 的评估来理解为什么错，而不是对着一个 0.23 的分数发呆。

突破 2：无监督 PRM——不需要人工标注

PRM 最大的推广瓶颈一直是数据。给每一步打标签需要专家标注，成本高昂。Math-Shepherd（2024）用自动化的方法生成步骤标签取得了一定进展，但 2026 年的两篇论文从根本上解决了这个问题。

Unsupervised PRMs（Gadetsky et al., 2026 年 5 月）完全不需要标签。它的思路是：让模型对同一问题生成多个解答轨迹，然后看哪些步骤在正确和错误的轨迹中出现频率差异最大——这些步骤就是关键步骤。通过这种"投票"机制，无监督 PRM 的性能竟然追平了监督训练的 PRM。

SWE-TRACE（Han et al., 2026 年 4 月）走了另一条路：用执行轨迹自动生成评分标准（rubric）。它不需要人写评判标准，而是从代码执行结果中自动提取多维度标签（正确性、效率、安全性），然后用这些标签训练 PRM。

💡 关键洞察
PRM 的数据问题在 2026 年基本被解决了。无监督训练 + rubric 自动生成， 意味着任何领域都可以快速构建 PRM。 这就像 2023 年 RLHF 的 DPO 简化——一个技术门槛的消除会引爆应用。

突破 3：自我进化的验证器——SEVA

所有 PRM 都有一个隐含假设：验证器本身是好的。但如果 Agent 在变强，而验证器还停留在旧水平呢？

SEVA（Yuan et al., 2026 年 6 月，被 ICML 2026 接收）解决了这个问题。它构建了一个自我进化循环：

  1. Agent 生成答案 + 事实引用
  2. 验证器检查每条引用是否准确
  3. 验证失败的信息被回收为训练数据
  4. 验证器随之进化，与 Agent 同步提升

这相当于一个"陪练"角色——验证器不是裁判，而是和 Agent 一起变强的训练伙伴。

四、领域特化：为什么通用 PRM 不够用

2026 年最明确的研究趋势之一：每个领域都在建自己的 PRM。

领域 系统 "正确"意味着什么
数学推理 OREAL, GroundedPRM 逻辑自洽
软件工程 SWE-TRACE 测试通过 + 代码可维护
GUI 操作 VisCritic, StainFlow 视觉状态按预期变化
金融分析 Fin-PRM 符合监管要求
多 Agent 系统 MAS-ProVe 协作轨迹一致性
工具调用 ToolPRMBench 参数正确 + 工具选择合理

VisCritic 的思路尤其巧妙：它不检查 Agent 的"行为"，而是检查行为之后屏幕画面的变化。如果 Agent 说"我点击了提交按钮"，但屏幕截图显示提交按钮还在——VisCritic 就知道这步有问题。这比分析 Agent 的推理文本可靠得多。

这背后的哲学值得品味：最好的验证不是检查过程，而是检查结果的状态变化。对于 GUI Agent，屏幕就是真理。对于 SWE Agent，测试就是真理。对于数学 Agent，形式化证明就是真理。每个领域都需要找到自己的"验证锚点"。

五、从理论到实践：你现在能做什么

好，PRM 很厉害。但大多数读者不是在做 AI 研究，而是在构建产品。以下是可以立即采取的行动：

1. 开始记录步骤级日志（零成本）

这是最简单的起点。不管你用什么 Agent 框架，确保每次工具调用都被记录：

{
  "step_id": 27,
  "tool": "edit_file",
  "input": {"path": "src/auth.py", "old": "...", "new": "..."},
  "output": {"success": true},
  "timestamp": "2026-07-14T20:15:32Z"
}

即使你现在不做 PRM，这些日志将来就是训练 PRM 的原材料。没有步骤级数据，一切都是空谈。

2. 用 LLM-as-Judge 做步骤评估（低成本）

在有了步骤日志后，用一个 LLM 来评估每一步质量。你不需要训练任何模型——直接用 GPT-4 或 Claude 给每步打分并解释原因。这本质上就是一个"穷人的生成式 PRM"。

关键提示：让 LLM 不仅打分，还要解释为什么。这些解释本身就是未来训练更小 PRM 的数据。

3. 在关键步骤插入验证关卡（中成本）

不是每一步都需要验证，但某些关键操作值得"停一下检查"：

  • 文件写入前：变更是否符合意图？
  • 工具调用前：参数是否合理？是否有副作用风险？
  • 任务完成时：结果是否满足原始需求？

在这些关卡放一个验证器（哪怕只是规则检查 + LLM 审查），能捕获 80% 的问题。这比只在最后检查结果有效得多。

4. 探索 RLVR 训练（高投入高回报）

如果你在训练自己的 Agent 模型，开始关注 RLVR + PRM 的训练范式。OREAL 的代码已经开源，SWE-TRACE 的方法也可以适配到其他领域。

关键是：不要一次性引入完整的 PRM 训练流程。先用 LLM-as-Judge 生成步骤标签，训练一个小的 PRM，然后逐步迭代。这比从零开始构建标注流水线快得多。

六、更大的图景：验证与生成的融合

回看 2023 到 2026 年的演进，一条主线越来越清晰：验证正在从"事后检查"变成"过程中的实时护栏"。

2023 年：Agent 先生成完整个答案，然后检查对不对。
2025 年：Agent 在生成过程中用 PRM 做 beam search，避开差的路径。
2026 年：PRM 和 Agent 一起进化，验证信号直接用于训练，Agent 在生成时同步验证。

这与认知科学中的 System 1 / System 2 理论形成了有趣的呼应。System 1 是快速直觉式的生成，System 2 是缓慢审慎的验证。人类在解决复杂问题时，这两个系统是交替工作的——直觉给假设，审验来确认或推翻。

PRM 正在成为 AI Agent 的 System 2。一个没有 PRM 的 Agent 就像一个只有 System 1 的人——反应快，但经常自信地犯错。

当 SEVA 这样的系统让验证器与 Agent 同步进化，当无监督训练让任何领域都能低成本构建 PRM，当生成式 PRM 能像同事一样跟你解释"为什么这步有问题"——我们正在接近一个临界点：AI Agent 真正可以信任的临界点。

不是因为它不会犯错，而是因为它知道自己在什么时候可能犯错。这比永远不出错更重要。
