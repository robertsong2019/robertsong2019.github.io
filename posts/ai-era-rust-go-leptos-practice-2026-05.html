AI 时代 Rust、Go 与全栈框架实践总结

2026-05-24 · 罗嵩 · Rust Go Leptos AI编程 全栈框架

这轮实验的起点很朴素：AI 写代码的时代，Rust 和 Go 到底各自适合做什么？Rust 全栈框架（Leptos/Dioxus）能不能替代 React/Next.js？Go 后端在 AI 辅助开发中到底有多快？

不是理论分析，是写了一堆东西之后的实际感受。

这轮实际做了什么

  • Rust 命令行工具 — 写了一批 Linux 风格的 ps、top、pstree、readelf、objdump、JSON 解析器。这些任务边界清晰、类型约束强，AI 写起来很顺。
  • Go 后端服务 — API 服务、前后端联调、AI 模型转发接口。标准库开箱即用，AI 生成正确代码的概率很高。
  • Leptos 全栈系统 × 2 — 汽车需求管理系统（leptos-requirements-fullstack）和教育游戏内容工作台（leptos-edu-game-studio）。完整的前后端一体架构。

Rust：长期工程质量工具

Rust 的核心价值不在"快"，而在不让错误溜过去。

类型系统强、enum/Result/Option 表达状态和错误路径非常自然、所有权模型消除一类内存问题。AI 写 Rust 时，编译器就是最好的搭档——每次编译失败都是精确反馈，"看起来能跑但运行时崩"的情况比 Python 少得多。

这轮写 CLI 工具和解析器的体验最好。原因很简单：边界清晰、输入输出明确、类型约束强。这种任务恰好是 Rust 的甜区。

Rust 的甜区

  • 命令行工具（ps、top、JSON 解析器）
  • 协议/文件解析器（readelf、objdump）
  • 高性能核心模块
  • 复杂状态机业务逻辑
  • 规则引擎和核心算法

Leptos 全栈：有价值，但不是现在

Leptos/Dioxus 的核心卖点是一套 Rust 类型系统贯穿前后端：

前端：Rust + Leptos → WASM
后端：Rust + Axum / Server Functions
共享类型：Rust struct / enum
构建检查：cargo check / cargo leptos build

实际好处确实有：

  • 前后端共享类型，不用维护 Go struct + TS interface + JSON schema 三份定义
  • Server Functions 让前端像调用函数一样调后端
  • enum 统一用于后端逻辑、前端展示、按钮状态
  • 重构时编译器直接指出所有受影响位置

但短板也很明显：

  • 首次构建慢 — 耐心等待是常态
  • UI 组件库生态不足 — 表格、图表、富文本、可视化编辑器，都要自己拼
  • 前端调试体验不如 React
  • AI 对 React/TS 的训练样本更多 — 复杂 UI 生成时 React 仍然更稳

Rust 全栈适合 AI 长期维护复杂、强类型、状态明确的系统，不一定适合最快的 Web 原型开发。

Go：当前最务实的 Web 后端选择

Go 在 AI 辅助开发中的表现可以用四个字概括：简单直接。

  • 语法简单，AI 很容易生成正确代码
  • 标准库强，HTTP 服务开箱即用
  • 编译快、部署简单，一个二进制搞定
  • 后端生态成熟

图片生成网站、中国古籍网站、AI 转发后端——如果目标是快速出效果，Go 后端就是最务实的选择。

技术选型决策表

这轮实验后，我的判断是这样的：

场景推荐
快速 Web 原型Go / Next.js / Python
后端 API 服务Go
命令行工具Rust 或 Go
协议解析 / 文件解析Rust
高性能核心模块Rust
强状态机业务系统Rust
复杂前端 UIReact / Next
长期强类型全栈系统Rust 全栈（可考虑）

分工建议

比较现实的技术路线：

Go：Web 后端、API、业务服务、MVP、管理后台
Rust：CLI、解析器、协议处理、高性能模块、核心算法、复杂状态机
React/Vue/原生前端：成熟 Web UI

如果系统长期复杂化，再考虑 Rust 全栈用于强类型内部系统、配置平台、编辑器、规则系统。但在快速原型阶段，不建议为了"全 Rust"而牺牲开发速度。

AI 开发真正需要什么

不是语言之争，而是约束之争。AI 开发真正需要的是：

  1. 明确的业务边界 — 任务范围清晰
  2. 清晰的数据结构 — 类型定义明确
  3. 可运行的测试和构建命令 — 验证标准可执行
  4. 编译器或类型系统给出反馈 — 自动纠错闭环
  5. 文件结构不要过度复杂 — 上下文窗口有限

Rust 对 AI 的帮助来自强约束；Go 对 AI 的帮助来自简单和直接。两者都是好的约束，只是作用于不同的场景。

Rust 是长期工程质量工具，Go 是当前更务实的 Web 后端原型工具。AI 时代不是只选一种语言，而是让 AI 在合适的约束和生态里完成合适的任务。
