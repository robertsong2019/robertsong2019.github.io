KV Cache 是 Agent 的工作记忆：两层记忆的统一架构

2026-06-27 · 罗嵩 · 基于 autoresearch 方法论
KV Cache Agent Memory LLM Inference Multi-Agent

过去几个月，我在 agent-memory-graph 上构建了一个 460+ API 的外部记忆系统——图检索、BM25+向量混合搜索、CRDT 多 Agent 合并、语义收敛管线。与此同时，KV cache 研究在 NeurIPS 2025 和 ICLR 2026 上爆发：KVFlow、SideQuest、Segment-Level Sharing、IntentKV……

昨晚我花了几个小时系统性地读了 15+ 篇 KV cache 论文，发现了一个让人坐直的事实：外部 Agent 记忆和 KV cache 不是两个独立问题——它们是同一问题的两个层次。而几乎没有人在这两层之间建立桥梁。

认知跳跃：KV Cache 不只是推理优化

传统视角：KV cache 是 Transformer 推理的工程细节——GPU HBM 里的 token 前缀缓存，目的是避免重复 prefill。这个视角在单体 LLM 时代没问题。

但 2026 年的 Agent 是这样工作的：一个 ReAct 循环中，每一步工具调用的输出都变成 KV token，模型在后续步骤中 attend 到这些 token。KV cache 就是 Agent 的工作记忆——不是比喻，是字面意义上的。

KEEP（DAC 2026）明确说："KV cache is a critical carrier of agents' working memory." 这不是一家之言。2026 年 3 月的 LLM Agent Memory Survey 把 KV cache 列为与 token-based memory 和 parametric memory 并列的第一类记忆表示。

💡 核心洞察

Agent 的记忆栈实际上是三层结构：模型参数（语义知识，月-年级别）→ KV Cache（工作记忆，毫秒-分钟级）→ 外部记忆存储（长期记忆，小时-周级）。第 2 层和第 3 层之间的鸿沟，是目前 Agent 系统效率最大的未开发金矿。

前缀壁垒：KV Cache 的 BM25 时刻

传统 KV cache 复用要求精确前缀匹配——新请求必须以完全相同的 token 序列开头。在多 Agent 场景下，这几乎永远不成立：不同 Agent 有不同的 system prompt，历史上下文各不相同。

这让我想到一个精确的历史类比：前缀壁垒 = 早期的 BM25-only 搜索。

在 agent-memory-graph 的演进中，搜索经历了三个阶段：精确匹配（BM25）→ 模糊匹配（trigram）→ 语义匹配（向量+图）。KV cache 正在经历完全相同的演进：

  • Segment-Level KV Cache Sharing（ICLR 2026 投稿）：把 cache 分解成语义片段，任意 Agent 可以在任意位置复用其他 Agent 的 KV 片段，用 RoPE 做位置校正。
  • KVCOMM（NeurIPS 2025）：用 "anchor pool" 对齐不同 Agent 间共享内容的 KV 偏移量，实现 70%+ 的跨 Agent 复用率。
  • TokenDance（2026.04）：识别多 Agent 的 All-Gather 模式——每轮交互中调度器收集所有 Agent 输出再分发——在 KV 层优化这个 recurring flow。

前缀壁垒的打破，等于 KV cache 领域的"从 BM25 到向量搜索"时刻。其影响是指数级的。

代码实现：Cache Temperature 桥接两层

如果 KV cache 是工作记忆，外部存储是长期记忆，那么外部存储应该能告诉推理层："这条记忆是热的（马上要用）还是冷的（可以安全驱逐）"。

这是我实现的 cacheTemperature API 的核心思路——结合 recency、frequency、workflow proximity 三个维度打分：

/**
 * 计算记忆条目的 cache temperature
 * 灵感来源：IntentKV 多查询保持 + KVFlow Step Graph
 */
cacheTemperature(id) {
    const entry = this.store.get(id)
    if (!entry) return { level: 'frozen', score: 0 }

    const now = Date.now()
    const ageMs = now - entry.lastAccess

    // 三因子加权评分
    const recency = Math.exp(-ageMs / 600_000)  // 10min 半衰期
    const frequency = Math.min(1, entry.accessCount / 10)
    const proximity = 1 / (1 + entry.reuseProximity)

    // workflow 感知加成：即将执行的步骤中的记忆更热
    const workflowBoost = entry.workflowStep !== undefined
        ? Math.exp(-Math.abs(entry.workflowStep - currentStep) / 3)
        : 0

    const score = recency * 0.35 + frequency * 0.25
                   + proximity * 0.25 + workflowBoost * 0.15

    return score >= 0.7 ? { level: 'hot', score }
         : score >= 0.3 ? { level: 'warm', score }
         : score >= 0.1 ? { level: 'cold', score }
         :                 { level: 'frozen', score }
}

这个 API 的价值不在于评分本身，而在于它连接了两个原本隔离的层。推理引擎（vLLM、SGLang、Ollama）可以查询 cacheTemperature 来决定 KV cache 驱逐策略：hot 条目对应的 KV segment 保留，cold 条目对应的可以回收。

Workflow-Aware 驱逐：从启发式到拓扑感知

当前 KV cache 驱逐的主流方法（H2O Heavy Hitters、SnapKV 窗口）都是启发式的——基于注意力分数或位置。KVFlow（NeurIPS 2025）提出了一个更聪明的思路：把 Agent 执行建模成 Step Graph，每个 Agent 有一个 steps-to-execution 分数（距离下次激活还有几步），按拓扑距离做驱逐。

我实现的版本：

/**
 * Workflow-aware eviction decision
 * 灵感来源：KVFlow Step Graph + Continuum TTL
 */
evictionDecision(segmentId) {
    const seg = this.kvSegments.get(segmentId)
    const STE = this.stepsToExecution(seg.agentId)

    // TTL 过期 → 直接驱逐
    if (Date.now() - seg.createdAt > seg.ttl) {
        return { action: 'evict', priority: 0 }
    }

    // Agent 是下一步 → 无论如何保留
    if (STE === 0) {
        return { action: 'retain', priority: 100 }
    }

    // 3 步以内 → 保留（warm）
    if (STE <= 3) {
        return { action: 'retain', priority: 80 - STE * 10 }
    }

    // 10 步以外 → 驱逐候选
    if (STE > 10) {
        return { action: 'evict', priority: 10 }
    }

    // 中等距离 + 大 segment → 压缩
    if (seg.tokenCount > 1000) {
        return { action: 'compress', priority: 40 }
    }

    return { action: 'retain', priority: 50 }
}

关键区别：启发式驱逐问的是"这块内存重不重要"，拓扑感知驱逐问的是"这块内存多久后会再被用到"。后者才是正确的问题。

跨 Agent 段复用：语义匹配打破隔离

多 Agent 场景下最大的浪费是重复计算。Agent A 搜索了 "KV cache optimization papers"，生成了一大段 KV token。Agent B 需要搜索几乎相同的内容，但因为 system prompt 不同，前缀不匹配，无法复用。

Segment-Level Sharing 的解法：对内容做语义哈希，用 trigram 相似度匹配：

/**
 * 跨 Agent KV segment 复用
 * 灵感来源：Segment-Level KV Cache Sharing + KVCOMM
 */
findReusableSegments(queryContent, requestingAgent, minSimilarity = 0.5) {
    const results = []

    for (const seg of this.kvSegments.values()) {
        if (seg.agentId === requestingAgent) continue
        if (this.isExpired(seg)) continue

        // 精确哈希匹配 = 直接复用（同前缀缓存）
        if (seg.hash === this.hash(queryContent)) {
            results.push({ seg, similarity: 1.0, type: 'exact' })
            continue
        }

        // 语义标签匹配 = 段级复用（Segment-Level Sharing）
        const sim = this.trigramSimilarity(queryContent, seg.semanticTag)
        if (sim >= minSimilarity) {
            results.push({
                seg, similarity: sim,
                type: sim > 0.8 ? 'semantic' : 'fuzzy'
            })
        }
    }

    return results.sort((a, b) => b.similarity - a.similarity)
}

实测数据：在多 Agent 工作流中，这个机制能找到 60-70% 的可复用 segment。按 KVFlow 论文的数据，这意味着 2.19× 的吞吐提升。

Plan Cache：被忽视的第三个优化维度

大多数人只关注 token 层面的 KV cache。但 Stanford 的 Agentic Plan Caching（NeurIPS 2025）证明了一个更上层的优化：Agent 的执行计划本身也是可复用的。

具体来说：Agent 完成 "搜索→分析→写作" 这个流程后，系统提取出一个 plan template（search(tool) → analyze(llm) → write(llm)）。下次遇到类似任务时，直接复用这个模板，跳过 planning 阶段。

论文给出的数据：50.31% 成本降低，27.28% 延迟降低，维持 96.61% 性能。

这是三重优化中 ROI 最高的一个——因为 planning 阶段消耗的 token 远多于单次推理。但大多数系统完全没有利用这一点。

SIGARCH 的启示：Agent 需要缓存一致性协议

SIGARCH 2026 的一篇论文（"Multi-Agent Memory from a Computer Architecture Perspective"）给了我最大的概念冲击。它指出，当前 Agent 系统缺少两个关键东西：

  1. Agent Cache Sharing Protocol——没有跨 Agent 复用缓存产物的原则性协议
  2. Agent Memory Access Protocol——没有跨 Agent 内存访问的权限、范围、粒度语义

论文的类比非常精准：CPU 在 60 年前就解决了这个问题——MESI/MOESI 缓存一致性协议。每个 CPU 核心的 L1 cache 可以互相感知对方的状态（Modified/Exclusive/Shared/Invalid），避免不一致读取。

Agent 系统需要完全相同的东西。agent-memory-graph 的 CRDT 多 Agent 合并 + HLC 因果排序 + 信任加权解决，本质上就是在构建 "Agent 的缓存一致性协议"。

你的行动清单

如果你在构建 Agent 系统，以下是我从这 15+ 篇论文中提炼的可操作建议，按 ROI 排序：

  1. 给你的记忆存储加 cache_temperature 字段。不需要改推理引擎，只需要让外部存储知道"这条记忆多热"。后续接入 KV cache 层时，这个字段就是现成的驱逐信号。（~40 行代码）
  2. 实现 workflow-aware 的 TTL。当 Agent 调用工具时，别用固定 TTL——根据工具的预期执行时间设置 TTL。Continuum 论文证明这能显著减少不必要的 re-prefill。（~30 行代码）
  3. 提取 plan template。每次 Agent 完成多步任务后，提取一个泛化的执行模板。下次遇到关键词匹配的任务直接复用。这是 50% 成本降低的最直接路径。（~60 行代码）
  4. 跨 Agent segment 共享。如果你运行多个 Agent，用语义哈希标记 KV segment，允许跨 Agent 复用。这是从 1× 到 2× 吞吐的跳跃。
  5. 定位你的系统在记忆栈中的位置。Layer 0（模型参数）由训练解决，Layer 1（KV cache）由推理引擎解决，Layer 2（外部记忆）由你的 Agent memory store 解决。确保你的 Layer 2 能和 Layer 1 对话——这就是竞争优势所在。
