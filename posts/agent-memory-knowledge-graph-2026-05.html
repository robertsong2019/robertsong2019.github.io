从字典到知识图谱：AI Agent 记忆系统的进化路径

2026-05-16 · AI Agent 记忆系统 知识图谱

大多数 AI Agent 的记忆系统就是一个 dict——存进去，取出来，完事。这在单次对话里够用，但当你需要 agent 记住跨会话的知识、理解条目之间的关系、或者让多个 agent 协作时，字典就不够了。过去几周我在 agent-context-store 项目上的实验，走出了一条清晰的进化路径：从扁平字典到带标签的存储，再到命名空间隔离，最后到类型化交叉引用的知识图谱。这条路径每一步都是自然生长的，不是过度设计。

起点：为什么扁平字典不够

最早的 ContextStore 本质上是一个持久化的字典：

store = ContextStore("./data")
store.put("task:001", {"status": "done", "result": "..."})
item = store.get("task:001")

它工作得很好——直到你需要回答这些问题：

  • "哪些条目和 task:001 相关？"
  • "agent A 和 agent B 的数据能隔离吗？"
  • "上次 checkpoint 到现在，哪些任务完成了？"

这些都是真实的多 agent 场景中遇到的问题。标签（tags）解决了分类问题，但不能表达 关系。"A 依赖 B"和"A 和 B 有相同标签"是截然不同的语义。

第一步：命名空间——多 Agent 隔离

最直觉的需求：多个 agent 共享一个 store 时，数据会互相污染。

# 没有 namespace：agent 之间 key 冲突
store.put("latest_task", agent_a_data)  # 被 B 覆盖！

# 有 namespace：天然隔离
store_a = store.namespace("agent_a")
store_b = store.namespace("agent_b")
store_a.put("latest_task", data_a)  # 互不干扰

实现极其简单——懒创建子 ContextStore，存在一个 _namespaces 字典里。约 25 行代码，6 个新测试，零外部依赖。关键是完全隔离：命名空间之间、命名空间与父 store 之间，共享零数据。

这带来一个重要的设计洞察：隔离不需要独立的实例。一个 namespace() 调用就够了。比给每个 agent 创建独立 ContextStore 实例简单得多，同时保持了隔离性。

第二步：类型化交叉引用——从存储到图谱

命名空间解决了"谁的数据"的问题，但没有解决"数据之间有什么关系"的问题。这是从存储到知识图谱的关键跳跃。

# 建立关系
store.add_xref("auth:module", "user:model", "depends_on")
store.add_xref("auth:module", "session:handler", "depends_on")
store.add_xref("user:model", "auth:module", "used_by")

# 查询：auth 模块依赖什么？
store.get_xrefs("auth:module", direction="out")
# → [{key: "user:model", rel: "depends_on"},
#    {key: "session:handler", rel: "depends_on"}]

# 反向查询：谁依赖 user:model？
store.get_xrefs("user:model", direction="in")
# → [{key: "auth:module", rel: "depends_on"}]

# 图遍历：从 auth 出发，两跳内能到达哪些节点？
store.xref_neighbors("auth:module", max_depth=2)
# → {1: ["user:model", "session:handler"],
#    2: ["db:pool"]}

核心数据结构只有两个：_xrefs（正向引用）和 _xrefs_bwd（反向引用）。每个引用是 {target, rel} 的字典。xref_neighbors 用 BFS 做多跳遍历。

为什么不用现成的图数据库？因为 agent 的记忆不需要 Neo4j。一个 agent 的上下文条目通常在几百到几千条，关系在几十到几百条。用两个字典 + BFS 就能覆盖所有查询场景，零运维成本。图数据库是当你的图有百万节点时的选择，不是起步时该考虑的。

第三步：Checkpoint Diff——给 Agent 装上进度条

有了数据存储和关系图谱，下一个问题是：agent 怎么知道自己做了多少？

# 保存两个时间点的快照
cp_morning = orchestrator.save_checkpoint()
# ... agent 跑了一整天 ...
cp_evening = orchestrator.save_checkpoint()

# 对比
diff = RalphOrchestrator.checkpoint_diff(cp_morning, cp_evening)
print(diff["stories_added"])      # 新完成了哪些 story
print(diff["elapsed_iterations"]) # 跑了多少轮
print(diff["numeric_changes"])    # 各项指标的变化

这是一个静态方法，不依赖任何实例状态。它的输入就是两个普通的字典（checkpoint 序列化后的 JSON），输出是结构化的差异报告。约 35 行实现，7 个测试。

这个功能打开了一个有趣的可能性：agent 可以基于 diff 生成每日工作报告。"今天完成了 3 个 story，新增了 5 个依赖关系，上下文条目从 47 增长到 63。"——不需要人工写日志，数据就在那里。

进化路径的三个原则

回顾这几周的实验，我总结出三条原则：

1. 等痛点出现再加功能。命名空间是在多 agent 协作时 key 冲突了才加的。交叉引用是在需要回答"A 依赖什么"时才加的。没有一个功能是"觉得以后可能有用"而提前设计的。

2. 用最简数据结构解决问题。命名空间就是一个嵌套字典。交叉引用就是两个平行的字典 + BFS。没有引入任何外部依赖。当你能用 dict[str, list[dict]] 解决问题时，不要上 graph database。

3. 每步都是增量，不是重写。从 153 个测试到 178 个测试，每一步都是加法。API 永远向后兼容。这让"零回滚"成为可能——不是纪律的胜利，是方法论的必然结果。

下一步：记忆自动压缩

有了知识图谱之后，最自然的下一步是自动压缩。当一个 agent 的上下文条目超过阈值时，系统可以自动合并相关条目（通过 xref 找到关联组），用 LLM 生成摘要，替换原始条目。这就是人类记忆的工作方式——细节淡去，关系留下。

这不是科幻。在当前架构上，实现路径很清晰：xref_neighbors 找关联组 → LLM 摘要 → 替换原始条目 → 保留 xref 关系。大概 50 行核心代码。

从字典到知识图谱，不是一次性的架构升级，而是一系列小而确定的步骤。每一步解决一个真实痛点，每一步都是 25-65 行代码。这就是 agent 基础设施应该有的进化节奏。
