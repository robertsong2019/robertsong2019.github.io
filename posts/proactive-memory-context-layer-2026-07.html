记忆不该等提问：为 Agent 构建主动感知层

2026-07-14 · 罗嵩 · Agent Memory Proactive Context 知识图谱 认知架构
所有 Agent 记忆系统都在做同一件事：等用户提问，然后搜索。但人类记忆不是这样工作的——相关的上下文会自动浮现，不需要你"搜索"。本文展示如何用三层架构——社区检测 → 意图结晶 → 无查询推送——让 Agent 记忆从被动检索进化到主动感知。

一个反直觉的发现

Apple 在 ACL 2026 发表了一个令人意外的实验结果：

在多轮对话任务中，全量记忆注入（71% 准确率）反而比无记忆基线（79%）更差。

原因不复杂——无关记忆就是噪声。当你把所有历史都塞进上下文窗口，Agent 会被淹没在信息中，反而抓不住重点。

Apple 的解法是"选择性持久化"：写入时只保留四类记忆。但这里有一个更深层的问题：写入时重要的记忆，读取时未必重要。反过来，写入时不起眼的细节，可能在特定上下文中成为关键线索。

我们需要的不只是写入时过滤，而是读取时的主动组装。

检索范式的盲区

2026 年的 Agent 记忆系统几乎都是检索驱动的：

用户 Query → Embedding/Keyword → Top-K 结果 → 拼接 Prompt

这在"事实查询"场景下没问题。但 Agent 不是搜索引擎。Agent 需要持续地感知环境，而不是只在被问到时才翻记忆。

考虑这个场景：你想知道"上周的设计评审有没有遗漏"。Agent 检索到了评审记录，回复了你。但你接着想："等等，那个评审和上个月的架构变更有没有冲突？"——这次检索可能空手而归，因为你不知道该搜什么关键词。

问题的本质不是检索能力不够，而是检索这个范式本身有盲区。

从人类认知到 Agent 架构

人脑的海马体-皮层系统有一个关键机制：记忆不是被搜索的，而是被激活的。

  • 你看到自行车，骑车的记忆自动激活——不需要"搜索"
  • 你走进办公室，同事的名字自然涌现——不需要"检索"
  • 这种激活是前认知的——它发生在你意识到之前

CogniFold（2026）在 Agent 记忆中模拟了这个机制，提出三层扩展：事件层 → 概念层 → 意图层。上层自动从下层涌现，形成越来越抽象的记忆表征。在 LoCoMo 基准上达到 81.23%，LongMemEval 达到 93.0%。

基于这个思路，我构建了一个三层主动感知架构。

三层架构设计

Layer 3: Intent Space (意图空间) ↑ crystallize Layer 2: Concept Community (概念社区) ↑ detect Layer 1: Event Memory (事件记忆) ↑ write Raw Memory Entries

第一层：事件记忆

标准记忆层。每个条目有内容、时间戳、标签：

到这里和普通记忆系统没有区别。但下一步开始发生变化。

第二层：社区自组织

系统自动运行 Label Propagation Algorithm，发现记忆的聚类结构：

# 自动发现社区结构
communities = mg.detect_communities(algorithm="lpa")

for cid, members in communities.items():
    print(f"社区 {cid}: {members}")
# 社区 0: [微服务拆分, 服务网格风险, Istio建议, 微服务监控]
# 社区 1: [OAuth集成]
# 社区 2: [API网关选型]

社区检测不需要预定义分类——它基于记忆之间的共现关系（共享标签、引用同一实体、时间邻近）自动发现概念簇。系统自己发现"微服务"是一个活跃主题，而不靠人工打标。

但社区本身是隐式的。我们需要的下一步，是让这些隐式概念变成显式的记忆节点。

第三层：意图结晶

crystallize_intents() 在社区密度超过阈值时，自动创建一个 intent 节点，把整个社区"结晶"为可引用的实体：

# 自动结晶：社区密度 > 阈值时创建 intent 节点
result = mg.crystallize_intents(
    density_threshold=0.3,
    intent_prefix="intent:"
)

print(result)
# {
#   "crystallized": ["intent:微服务"],
#   "skipped": [],
#   "details": {
#     "intent:微服务": {
#       "community_id": 0,
#       "member_count": 4,
#       "density": 0.42,
#       "abstracted_keys": ["微服务拆分", "服务网格风险", ...]
#     }
#   }
# }

结晶后，intent 节点通过 abstracts 边链接所有成员：

[intent:微服务] ──abstracts──→ [微服务拆分方案] ──abstracts──→ [服务网格风险] ──abstracts──→ [Istio建议] ──abstracts──→ [微服务监控]

这个操作是幂等的——重复运行不会创建重复节点：

# 再次运行——已结晶的社区会被跳过
result2 = mg.crystallize_intents()
# {"crystallized": [], "skipped": ["intent:微服务"]}

无查询上下文推送

最后一步是真正的"主动感知"——在没有用户查询的情况下组装并推送上下文：

# 无需 query！系统主动组装上下文
context = mg.read_proactive_context(
    active_intents=["intent:微服务"],
    top_k=5,
    min_score=0.2,
    deduplicate=True
)

print(context)
# {

评分机制：Cache Temperature

每个记忆条目有一个 0-1 的 cache_temperature 分数，控制其在主动推送中的优先级：

def compute_cache_temperature(entry):
    """记忆的'温度'——越高越容易被主动推送"""
    age_score = temporal_score(entry)
    access_score = min(entry.access_count / 10, 1.0)
    validity_score = 1.0 if entry.valid else 0.0
    
    # 加权组合
    return (
        age_score ** 0.40 *
        access_score ** 0.35 *
        validity_score ** 0.25
    )

其中 temporal_score 使用连续衰减函数，而不是二元的"新/旧"判断：

import math

def temporal_score(age_days, half_life_days=30, alpha=0.5):
    """
    连续的时间评分 [0, 1]
    
    alpha=0:   静态事实（永不衰减）
    alpha=0.5: 平衡衰减（默认）
    alpha=1:   激进衰减（高度时效性）
    """
    if alpha == 0:
        return 1.0
    return math.exp(-alpha * age_days / half_life_days)
关键洞察：不同类型记忆需要不同衰减策略。用户偏好几乎是永久的（α≈0），今天的会议纪要高度时效（α≈1）。系统允许每条记忆指定自己的 α 值。

实战：完整三层管线

mg.add("用户反馈登录页面加载慢，P95 延迟 3.2s",

# Agent 现在拥有"前认知"上下文：
# 知道性能优化在推进、深色模式在讨论中
# 不需要用户提问就能感知到

对比：有 vs 没有主动感知

# ❌ 传统方式：用户必须问对问题
answer = mg.search("登录页面性能问题")

# ❌ 全量注入：研究表明更差
all_memories = mg.search("", top_k=999)

# ✅ 主动感知：系统自己推送
context = mg.read_proactive_context(
    active_intents=["intent:性能优化"]
)

为什么这是范式转换

维度传统检索主动感知
触发方式用户 query系统自驱动
上下文组装关键词匹配意图 + 社区 + 温度
抽象层级平坦的 top-K事件 → 概念 → 意图
时间感知二元（新/旧）连续相位衰减
信号密度低（靠用户精炼 query）高（系统预过滤）

这个架构的深层启示是：Agent 记忆系统不应该只是一个数据库，而应该是一个认知层——它主动感知、自我组织、预判需求。

工程细节

幂等性

三层操作都是幂等的，可以安全重复执行：

mg.detect_communities()           # 结果确定
mg.crystallize_intents()          # 跳过已结晶
mg.read_proactive_context(...)    # 只读，不修改状态

你可以在每次写入后、每次会话开始时运行它们，没有副作用。

性能

# 复杂度（N = 记忆数，E = 边数，C = 社区数）
detect_communities:     O(N + E)
crystallize_intents:    O(C × avg_community_size)
read_proactive_context: O(K × log N)   # K = 活跃意图数

# 10,000 条记忆规模：完整管线 < 50ms

与检索管线共存

主动感知不替代检索——它补充检索：

# 1. 启动时：主动感知铺底
proactive = mg.read_proactive_context(active_intents=current_intents)

# 2. 用户提问：检索增强
reactive = mg.retrieve(query=user_input, top_k=10)

# 3. 合并：背景 + 焦点
full_context = merge_and_deduplicate(proactive, reactive)

主动上下文提供"背景"，检索结果提供"焦点"。两者结合，Agent 既不会被无关记忆淹没，也不会错过跨主题联想。

哲学小结

这篇文章的核心不是某个 API，而是一个思维方式的转变：

记忆不是存储。记忆是持续进行的认知过程。

当你把记忆从"写一次、读一次"的存储模型，升级到"自我组织、主动涌现、持续感知"的认知模型时，Agent 的行为会发生质变——它不再是一个有记忆的搜索引擎，而是一个有认知的协作伙伴。

CogniFold 的作者说得很好："记忆的不完美不是 bug，而是特性。" 正是因为记忆会遗忘、会抽象、会重组，它才能产生洞察而不是回放。主动感知层，就是这个"不完美但有用"哲学的工程实现。
