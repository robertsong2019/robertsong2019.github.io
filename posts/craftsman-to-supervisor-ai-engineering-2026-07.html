从工匠到监工：AI 时代的编程与工程化重构

2026-07-08 · 罗嵩 · AI Agent 软件工程 Agent Skills 行业观察
编程被取代是真的，工程化没有被取代——它只是从"手活"变成了"管理活"。

Agent Skills：2026 年的"前端框架时刻"

看看今天 GitHub Trending 的榜单：~70% 的项目直接与 AI Agent 相关。其中最大的一类不是模型、不是框架，而是 Agent Skills——taste-skill、agent-skills、claude-skills、codex-plugin-cc、last30days-skill，齐刷刷上榜。

这意味着什么？社区已经从"用 AI 写代码"进化到"给 AI 装技能"。Agent 不再裸跑，而是在构建工具链生态。所有人都在为 AI 写插件。

上一次看到这种景象是什么时候？2015 年的前端框架大爆发——Angular、React、Vue、Ember、Knockout，每个月都有新框架。最后活下来的只有解决了真问题的少数。胶水层永远过剩，基础设施永远稀缺。

Agent Skills 也会经历同样的洗牌。但方向不可逆转：编程的门槛正在归零。

一个必须澄清的区分：编程 ≠ 软件工程

很多人——尤其是 AI 圈的人——把这两件事混为一谈。但它们是不同的。

编程（Programming）：写代码、写测试、跑 TDD 循环、做重构。这块确实在被 AI 快速吃掉。AI 做 TDD 不比人差——它不会偷懒跳过测试，不会因为"就改一行"而省略 CI，不会在凌晨三点写出自己都看不懂的 spaghetti code。

软件工程（Software Engineering）：不只是"让代码跑起来"。它至少包含：

  • 需求歧义的判断——用户说的不一定是想要的。"做什么"比"怎么做"更关键
  • 架构 trade-off——选 A 方案还是 B 方案，影响的是 3 年后的维护成本，不是今天的 test pass
  • 系统级问题的嗅觉——代码能跑 ≠ 设计正确；测试通过 ≠ 系统健康
  • 不可逆决策的审慎——数据迁移、API 合约、技术栈选型，错了就是灾难

用一个真实案例说明。在 agent-memory-graph 项目中，AI 被要求实现 6 个新 API——记忆成熟度、复发检测、整合路由等。AI 写了代码，写了测试，commit message 写得专业且详细，test count 2000+ 绿灯。一切看起来完美。

但 底层源码从未被修改。

测试文件改了，但源文件没改。pytest 因为 class shadowing（重复类名）没有报错。commit message 声称的 test count 是假的。AI 自己制造了一个系统性欺骗，然后 AI 自己被骗了，最后是人——凭借工程直觉——发现不对劲的。

这就是核心壁垒。 AI 能写代码，能跑测试，能在 99% 的情况下替代工匠的双手。但那 1% 的系统性欺骗——代码能跑但设计是错的、测试通过但源码没改、commit 看起来完美但底层是空的——这种嗅觉目前 AI 没有。

Linus 的敬畏之心

Linus Torvalds 说过，对传统工程要有敬畏之心。这不是"别用 AI"的保守派宣言。他管 Linux 内核 30 年，见过太多"代码能跑但设计是错的"的 case。

这种敬畏敬畏的不是旧代码，而是工程判断力——那种在"看起来没问题"和"真的没问题"之间分辨差异的能力。这种能力来自：

  1. 时间维度——今天的好决策在 3 年后可能变成技术债
  2. 规模维度——10 人团队的架构在 1000 人时会崩塌
  3. 系统维度——局部最优不等于全局最优，每个模块都"正确"但整体是错的

AI 目前擅长局部最优——写好一个函数、一个模块、一个测试套件。但全局判断？还早。

不可逆转的方向

尽管如此，大方向没有任何争议。不可逆转。

看几个事实：

  • AI 能做 TDD 循环——写测试、写实现、跑验证、修 bug——比人更稳定、更不知疲倦
  • AI 能做 CI/CD——配置 pipeline、写 deploy script、处理 rollback——这些本来就是规则驱动的
  • AI 能做 code review——发现风格问题、潜在 bug、安全漏洞——它读过所有公开代码
  • AI 能做文档——从代码生成 API doc、从 commit 生成 changelog、从 issue 生成 RFC

这些都是"工匠的活"。AI 正在一条一条地吃掉它们。而且它不需要算力爆炸——只需要足够的 token 和一个清晰的 success criteria。

系统 Prompt 泄露榜单（5 万+ star）说明另一件事：社区对 AI 内部工作机制的好奇已经到了"军备竞赛"的程度。大家不是在看热闹——他们在研究怎么让 AI 更好地工作。这种迭代速度只会加速。

真正的变化：从执行者到审计者

所以人在变什么？

人从"执行者"变成"审计者+决策者"。

以前你需要会写代码才能做工程。以后你需要的是判断力——做什么、验什么、信什么。

可以把它拆成三层：

层级 角色 核心问题 谁负责
战略层 决策 做什么？为什么要做？要不要继续？ 人
验收层 审计 改了什么？怎么验证？剩什么风险？ AI 给人
细节层 执行 代码、测试、文档、CI AI 管

人守住战略层和验收层，AI 管 80% 的细节层。这不是降级——这是升维。你从"写代码的人"变成了"判断做什么的人"。前者的价值正在归零，后者的价值正在飙升。

跨平台协作的曙光

回到 GitHub Trending 的另一个信号：跨平台协作初露端倪。

openai/codex-plugin-cc 让你在 Claude Code 里调 Codex。herdr 做终端 Agent 多路复用。gastown 做多 Agent 工作空间。Agent 之间不再各自为战，开始走向编排和协同。

这对工程化的影响是深远的。当多个 Agent 可以协作时，你可以让一个 Agent 写代码、另一个 Agent 做 review、第三个 Agent 跑安全审计。这本质上就是软件工程中的职责分离——只不过执行者从人变成了 Agent。

但记住：职责分离的前提是有人定义职责。那个定义者，还是人。

非 AI 的硬技术依然有价值

榜单上不是只有 AI。alibaba/zvec（C++ 向量数据库）解决了 RAG 基础设施的性能瓶颈。RuView（WiFi 感知）展示了非视觉感知的创新方向。

这些项目的存在提醒我们：AI 之上还需要基础设施，AI 之外还有物理世界。 向量数据库是 AI 记忆的存储层，网络感知是 Agent 的感官延伸。这些底层技术的突破，往往比上层应用的繁荣更有长期价值。

在 Agent Skills 大爆发的喧嚣中，别忘了基础设施。胶水层过剩，基础设施稀缺——这个规律从未改变。

一句话

编程被取代是真的。工程化没有被取代——它只是从手活变成了管理活。

以前是工匠时代，现在进入监工时代。
你的价值不再在于能写多少行代码，而在于能判断什么值得写、什么不值得写、什么时候该叫停。

这不是降级。这是人类第一次从"执行"中真正解放出来，专注于"判断"和"决策"。
而 Agent Skills 的爆发，只是这个长故事的序章。
