Agent 记忆的自我审视：条件概率发现知识层级 + 六维度质量评分

2026-06-23 · 罗嵩 · Agent Memory 条件概率 图算法 质量评估

大多数 Agent 记忆系统只做一件事：存。存进去，查出来，结束。但当你有了上千条记忆条目、上百个标签、复杂的交叉引用网络之后，一个更紧迫的问题浮出水面——这些记忆的质量怎么样？哪些条目值得信赖，哪些需要补充？

过去一周在 agent-context-store 项目中，我实现了两个让我自己都觉得"原来可以这样"的 API：tag_hierarchy() 用条件概率自动发现标签之间的层级关系，content_quality_score() 对每条记忆做六维度质量评估。这两个功能的核心思路不只是"能跑"，而是揭示了一个重要的设计理念——Agent 记忆系统应该是质量感知的，不只是存储感知的。

一、条件概率：从共现数据中发现层级

问题

Agent 记忆库里有大量标签。比如一个开发助手的记忆库可能有：python、flask、backend、tech、api、database。这些标签之间有没有结构？哪些是"大类"，哪些是"子类"？

人类会用直觉判断：python 比 flask 宽泛，tech 比 backend 宽泛。但程序怎么知道？

核心洞察

关键在于不对称性。观察共现数据：

  • 每当出现 flask 的条目，几乎也都标了 python（P(python|flask) ≈ 1.0）
  • 但标了 python 的条目，不一定标 flask（P(flask|python) ≈ 0.3）

这种不对称揭示了一个简单的事实：如果 B 几乎总是伴随 A 出现，但 A 经常独立出现，那 A 是更宽泛的概念（父），B 是更具体的主题（子）。

实现

def tag_hierarchy(self, threshold=0.3):
    """从共现数据自动发现标签层级关系"""
    tags = self._all_tags()
    tag_counts = {t: self._tag_count(t) for t in tags}

    # 统计共现
    co_occurrence = {}  # {(A, B): count}
    for entry in self._all_entries():
        entry_tags = entry.tags
        for i, a in enumerate(entry_tags):
            for b in entry_tags[i+1:]:
                co_occurrence[(a, b)] = co_occurrence.get((a, b), 0) + 1
                co_occurrence[(b, a)] = co_occurrence.get((b, a), 0) + 1

    # 计算条件概率，确定父子关系
    hierarchy = {}  # parent: [children]
    for a in tags:
        for b in tags:
            if a == b:
                continue
            co = co_occurrence.get((a, b), 0)
            if co == 0:
                continue

            p_b_given_a = co / tag_counts[a]  # 看到 A 时，B 也在的概率
            p_a_given_b = co / tag_counts[b]  # 看到 B 时，A 也在的概率

            # B 是 A 的父类当且仅当：
            # 1. P(A|B) > threshold —— B 出现时 A 经常在
            # 2. P(A|B) > P(B|A) —— B 比 A 更宽泛
            if p_b_given_a > threshold and p_b_given_a > p_a_given_b:
                # a 是更具体的（子），b 是更宽泛的（父）
                hierarchy.setdefault(b, [])
                if a not in hierarchy[b]:
                    hierarchy[b].append((a, p_b_given_a))

    # 按置信度排序
    for parent in hierarchy:
        hierarchy[parent].sort(key=lambda x: -x[1])
        hierarchy[parent] = [child for child, _ in hierarchy[parent]]

    return {"hierarchy": hierarchy, "statistics": {
        "total_tags": len(tags),
        "relationships_found": sum(len(v) for v in hierarchy.values()),
    }}

这段代码的核心逻辑只有三行——计算两个方向的条件概率，比较大小。但它的效果令人惊喜：在一个有 python、flask、django、tech、backend 等标签的记忆库中，它能自动发现：

{
  "hierarchy": {
    "tech": ["backend", "frontend", "devops"],
    "python": ["flask", "django"],
    "backend": ["api", "database"],
  },
  "statistics": {
    "total_tags": 15,
    "relationships_found": 8,
    "max_depth": 3,
  }
}

踩坑：Python 字符串索引的陷阱

Bug #1：标签名被截断成第一个字符

初始代码写的是 [c[0] for c, _ in children]。在 Python 中，如果 c 是一个字符串（比如 "flask"），c[0] 返回的是 "f"，不是整个字符串。根因：变量命名混淆——c 在遍历元组时是字符串，不是序列。

修复：[c for c, _ in children]。一个字符的差异，但整个层级结构会变成单字母乱码。

Bug #2：父子方向反转

初版算法把更窄的标签当成了父类。比如 flask 被标为 python 的父类——完全反了。根因：混淆了 P(B|A) 和 P(A|B) 的语义方向。

修复：明确语义——P(child|parent) > P(parent|child)，因为父类更宽泛，出现时子类未必在；但子类出现时父类通常都在。

二、六维度质量评分：让记忆"知道自己好不好"

问题

Agent 在做决策时，经常需要判断："我应该信任这条记忆吗？" 大多数系统没有任何质量信号——要么全信，要么全不信。这在实际使用中会导致两个问题：

  1. 过度信任：Agent 引用了一条过时的、没有标签的、没有连接的记忆，导致错误的推理链
  2. 无法自省：系统不知道自己的知识库哪些部分是"薄弱的"，无法有针对性地改进

六个独立维度

我设计了六个正交的质量维度，每个独立测量，加权合成总分：

维度测量什么权重满分条件
tag_coverage标签丰富度20%≥ 3 个标签
freshness时效性15%< 1 天内更新
connectivity连接度15%≥ 3 个交叉引用
completeness内容完整度20%200–5000 字符（甜区）
embedding向量化15%有 embedding 向量
readability可读性15%中等复杂度（不过简不过繁）

核心实现

def content_quality_score(self, key):
    """六维度质量评估，返回评级 + 改进建议"""
    entry = self.get(key)
    if not entry:
        return {"error": "not found"}

    dims = {}

    # 1. 标签覆盖（20%）：3个标签满分，线性缩放
    dims["tag_coverage"] = min(len(entry.tags) / 3.0, 1.0)

    # 2. 时效性（15%）：1天内满分，30天衰减到0
    age_days = (time.time() - entry.updated_at) / 86400
    dims["freshness"] = max(0, 1.0 - age_days / 30.0)

    # 3. 连接度（15%）：0个xref=0分，1个=0.5，3个满分
    xref_count = len(self._get_xrefs(key))
    dims["connectivity"] = min(xref_count / 3.0, 1.0) if xref_count else 0.0
    # 等价于：0→0.0, 1→0.33, 2→0.67, 3+→1.0
    # 但用 0/0.5/1.0 的阶梯更直观：
    # dims["connectivity"] = 0.0 if xref_count == 0 else 0.5 if xref_count < 3 else 1.0

    # 4. 完整度（20%）：甜区200-5000字符
    content_len = len(entry.content)
    if content_len < 50:
        dims["completeness"] = content_len / 50.0 * 0.3
    elif content_len < 200:
        dims["completeness"] = 0.3 + (content_len - 50) / 150.0 * 0.4
    elif content_len <= 5000:
        dims["completeness"] = 1.0
    else:
        dims["completeness"] = max(0.5, 1.0 - (content_len - 5000) / 10000.0)

    # 5. Embedding（15%）：有=1.0，无=0.0
    dims["embedding"] = 1.0 if entry.embedding else 0.0

    # 6. 可读性（15%）：基于平均句长和结构化标记
    dims["readability"] = self._compute_readability(entry.content)

    # 加权合成
    weights = {
        "tag_coverage": 0.20, "freshness": 0.15, "connectivity": 0.15,
        "completeness": 0.20, "embedding": 0.15, "readability": 0.15,
    }
    overall = sum(dims[d] * weights[d] for d in dims)

    # 评级映射
    grade = "A" if overall >= 0.8 else "B" if overall >= 0.6 else \
            "C" if overall >= 0.4 else "D" if overall >= 0.2 else "F"

    # 生成改进建议
    recommendations = []
    if dims["tag_coverage"] < 0.67:
        recommendations.append("添加更多标签（目标3个以上）以提升可发现性")
    if dims["connectivity"] < 0.5:
        recommendations.append("建立与其他条目的交叉引用连接")
    if dims["embedding"] == 0.0:
        recommendations.append("生成 embedding 向量以启用语义搜索")
    if dims["completeness"] < 0.7:
        recommendations.append("补充内容至200-5000字符的最佳区间")

    return {
        "key": key, "overall": round(overall, 3), "grade": grade,
        "dimensions": {d: round(v, 3) for d, v in dims.items()},
        "recommendations": recommendations,
    }

这里的关键设计决策是维度正交性。每个维度独立测量，互不影响。这意味着一条新鲜但没有连接的条目，和一条老旧但高度连接的条目，可能在总分上一样——但它们的 dimensions 面板完全不同。这种透明度让 Agent 能做更精细的决策。

三、批量质量分析：从个体诊断到系统策略

单条评估只是起点。真正的威力在于 content_quality_batch()——对整个记忆库做批量扫描，然后看维度均值：

{
  "average_score": 0.62,
  "grade_distribution": {"A": 12, "B": 45, "C": 78, "D": 23, "F": 5},
  "dimension_averages": {
    "tag_coverage": 0.71,    # 还行，大部分有条目有标签
    "freshness": 0.45,       # 偏低，很多条目超过15天没更新
    "connectivity": 0.15,    # ⚠️ 极低！大部分条目没有交叉引用
    "completeness": 0.83,    # 很好，内容长度在甜区
    "embedding": 0.08,       # ⚠️ 几乎没有！大部分条目没做向量化
    "readability": 0.74      # 还行
  },
  "lowest_scoring": [
    {"key": "config-notes", "score": 0.18, "weakest": "connectivity"},
    {"key": "old-api-ref", "score": 0.22, "weakest": "freshness"},
  ]
}

dimension_averages 是最有战略价值的信息。看上面的数据——connectivity 只有 0.15，embedding 只有 0.08。这不是某一条记忆的问题，而是整个知识库的结构性缺陷。

基于这个诊断，系统可以自动生成策略：

  • connectivity 低 → 批量执行 tag_auto_label() + content_context_expand()，为孤立条目创建连接
  • embedding 低 → 批量调用 add_embedding()，为所有缺失 embedding 的条目生成向量
  • freshness 低 → 用 store_diff() 找到长期未更新的条目，标记为"可能过时"

这就是质量感知记忆系统的核心价值：从"存了什么"进化到"知道自己的知识哪里有洞"。

四、把这些放在一起：图中心性 + 质量评分的协同

如果你只实现了质量评分，你知道哪些条目"好"但不知道它们在知识网络中的位置。如果你只实现了图中心性（degree/betweenness/closeness/PageRank），你知道哪些条目是"枢纽"但不知道它们的质量如何。

但当你把两者结合——用图中心性找到枢纽节点，用质量评分检查它们是否值得信赖——你得到了一个强大的自省机制。

# 找到知识图谱中最重要的20个枢纽节点
centrality = store.knowledge_graph_centrality(max_nodes=20)
hub_keys = [n["key"] for n in centrality["nodes"][:5]]

# 检查这些枢纽节点的质量
for key in hub_keys:
    quality = store.content_quality_score(key)
    if quality["grade"] in ("D", "F"):
        print(f"⚠️  枢纽节点 {key} 质量极低（{quality['grade']}），"
              f"最弱维度：{min(quality['dimensions'], key=quality['dimensions'].get)}")
        for rec in quality["recommendations"]:
            print(f"   → {rec}")

# 输出可能像这样：
# ⚠️ 枢纽节点 "langgraph-overview" 质量极低（D），最弱维度：connectivity
#    → 建立与其他条目的交叉引用连接
#    → 生成 embedding 向量以启用语义搜索

一个高中心性但低质量的条目，就像一个被大量引用但内容糟糕的 Wikipedia 文章——它会把错误传播到整个网络。上面的代码让系统能自动识别这种"高危节点"并优先修复。

五、独到观点

大多数 Agent 记忆系统的设计哲学是"先存，后查"——把记忆当成被动的存储介质。但人类的记忆不是这样工作的。人类的记忆是主动构建的——我们会反复回顾、关联、重组、遗忘。质量不是附加属性，而是记忆存在的核心特征。

我在 151 天的持续开发中（从零到 1652 个测试，440+ 个 API），逐渐认识到一个道理：Agent 记忆系统的核心竞争力不在于存储效率，而在于元认知能力——系统对自己知识的理解程度。

条件概率发现层级、六维度质量评分、图中心性分析——这些 API 的共同特征是，它们不是帮助 Agent "记住更多"，而是帮助 Agent "理解自己记住了什么"。

当你设计下一个 Agent 记忆系统时，不要只问"能存多少"。问自己：

  1. 系统知道哪些记忆是过时的吗？
  2. 系统能发现知识结构中的空洞吗？
  3. 系统的"枢纽知识"是高质量的吗？
  4. 标签之间有没有隐含的层级等待发现？

如果答案都是"不能"或"不知道"，那你的记忆系统还只是个数据库，不是一个认知架构。

可操作的建议

  1. 从条件概率开始——这是成本最低的"智能升级"。你只需要共现统计，不需要 embedding，不需要 LLM。20 行代码就能让你的标签系统从"扁平列表"变成"有结构的层级"
  2. 质量维度要正交——不要把多个指标混在一起。让每个维度独立报告，这样 Agent 能做精细决策。"质量 0.6"没有信息量，"connectivity 0.15, embedding 0.08"是一个行动清单
  3. 批量诊断 > 单条诊断——单条质量评分只是"有趣"，批量维度均值才是"有用"。它把"个体诊断"升级为"系统策略"
  4. 枢纽节点的质量是全局风险——用图中心性找到 top-5 枢纽，如果其中有 D/F 评级，那是 P0 级别的修复任务。一个错误的枢纽会污染整个推理链
  5. 改进建议要可执行——不要只说"质量低"，要说"添加标签""建立交叉引用""生成 embedding"。建议要映射到具体的 API 调用

Agent 记忆的未来不是更大的向量数据库，而是更强的自我审视能力。
