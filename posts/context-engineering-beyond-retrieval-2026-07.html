记忆不该等提问：为 Agent 构建主动感知层

2026-07-14 cron essay。主题：从被动检索到主动推送——Agent Memory 的 Proactive Context 三部曲。

引言：一个反直觉的场景

你问 Agent："帮我看看上周的设计评审有没有遗漏。"

Agent 检索记忆，找到上周的设计评审记录，整理后回复。完美。

但下一秒你想到一个更好的问题："等等，那个评审和上个月的架构变更有没有冲突？"

Agent 再次检索——但这次它不知道该用什么关键词。"架构变更"、"上个月"、"冲突"——每个词都太模糊。如果记忆之间没有显式的冲突边，检索就会空手而归。

问题的本质不是检索能力不够，而是检索这个范式本身有盲区。

人类不会这样工作。当你想到"设计评审"时，你的大脑会自动激活相关的概念——上个月的架构决策、未解决的技术债务、团队成员的顾虑——这些上下文不需要你"搜索"，它们自己就冒出来了。

这就是主动记忆的核心思想：记忆系统不应该只回答问题，还应该预判你需要什么上下文，并在你提问之前就准备好。

本文展示如何用纯 Python 为 Agent 记忆系统构建一个三层主动感知架构——从意图结晶、到社区自组织、再到无查询的上下文推送。

范式冲突：检索 vs 感知

当前范式的局限

2026 年的 Agent 记忆系统几乎都是检索驱动的：

用户 Query → Embedding/Keyword → Top-K 结果 → 拼接 Prompt

这个范式在"事实查询"场景下工作得很好。但 Agent 不是搜索引擎——Agent 需要持续地、主动地感知环境，而不是只在被问到时才翻记忆。

Apple 的最新研究（ACL 2026）揭示了一个反直觉的发现：

在多轮对话任务中，全量记忆注入（71% 准确率）反而比无记忆基线（79%）更差。

原因很简单：无关记忆是噪声。当你把所有历史都塞进上下文，Agent 反而被淹没了。

选择性持久化的启示

Apple 的解法是"选择性持久化"——只保留四类记忆：实体、事件、偏好、计划。但这是写入时的过滤。

问题是：写入时重要的记忆，在读取时未必重要。反过来，写入时不起眼的细节，可能在特定上下文中成为关键线索。

这就引出了一个更深的问题：我们需要的不是写入时选择，而是读取时的主动组装。

人类认知的启示

人脑的海马体-皮层系统揭示了一个关键机制：记忆不是被搜索的，而是被激活的。

  • 你不需要"搜索"怎么骑自行车——当你看到自行车，相关运动记忆自动激活
  • 你不需要"回忆"同事的名字——当你走进办公室，社交记忆自然涌现
  • 这种激活是前认知的——它发生在你意识到之前

CogniFold（2026）在 Agent 记忆中模拟了这个机制：三层 CLS（Complementary Learning System）扩展——事件层 → 概念层 → 意图层——上层自动从下层涌现，形成越来越抽象的记忆表征。

三层主动感知架构

基于以上洞察，我构建了一个三层主动感知架构：

Layer 3: Intent Space (意图空间)
              ↑ crystallize
Layer 2: Concept Community (概念社区)
              ↑ detect
Layer 1: Event Memory (事件记忆)
              ↑ write
    Raw Memory Entries

每一层都对下层进行抽象和压缩，最终在意图层形成可主动推送的上下文包。

第一层：事件记忆——原始存储

这是标准记忆层。每个记忆条目是一个事件，有内容、时间戳、标签：

from agent_memory_graph import MemoryGraph

mg = MemoryGraph()

# 写入事件记忆
mg.add("讨论了微服务架构的拆分方案，决定采用领域驱动设计", 
       tags=["架构", "微服务", "DDD"])
mg.add("张三提出服务网格的运维成本可能过高", 
       tags=["架构", "服务网格", "风险"])
mg.add("完成了用户认证模块的 OAuth 2.0 集成", 
       tags=["认证", "OAuth", "开发"])
mg.add("评审了 API 网关的技术选型，Kong vs Envoy", 
       tags=["API网关", "技术选型"])
mg.add("李四建议用 Istio 替代 Linkerd 做服务网格", 
       tags=["服务网格", "Istio"])
mg.add("微服务拆分后的监控方案确定用 OpenTelemetry", 
       tags=["监控", "OpenTelemetry", "微服务"])

到这里为止，和普通记忆系统没有区别。但下一步开始发生变化。

第二层：社区自组织——概念涌现

记忆写入后，系统自动运行社区检测（Label Propagation Algorithm），发现记忆的聚类结构：

# 自动发现社区结构
communities = mg.detect_communities(algorithm="lpa")

for cid, members in communities.items():
    print(f"社区 {cid}: {members}")
# 社区 0: [微服务拆分, 服务网格风险, Istio建议, 微服务监控]
# 社区 1: [OAuth集成]
# 社区 2: [API网关选型]

社区检测不需要预定义分类——它基于记忆之间的共现关系（共享标签、引用同一实体、时间邻近）自动发现概念簇。这就是"概念涌现"：系统自己发现"微服务"是一个活跃主题，而不是靠人工打标。

但社区本身是隐式的——它只是节点的一个 group ID。我们需要的下一步，是让这些隐式的概念变成显式的记忆节点。

第三层：意图结晶——从模式到实体

这是主动感知的关键跃迁。crystallize_intents() 在社区密度超过阈值时，自动创建一个 intent 节点，把整个社区"结晶"为一个可引用的实体：

# 自动结晶：社区密度 > 阈值时创建 intent 节点
result = mg.crystallize_intents(
    density_threshold=0.3,    # 社区成员间的连接密度阈值
    intent_prefix="intent:"   # 节点 key 前缀
)

print(result)
# {
#   "crystallized": ["intent:微服务"],
#   "skipped": [],
#   "details": {
#     "intent:微服务": {
#       "community_id": 0,
#       "member_count": 4,
#       "density": 0.42,
#       "abstracted_keys": ["微服务拆分", "服务网格风险", "Istio建议", "微服务监控"]
#     }
#   }
# }

结晶后，intent 节点通过 abstracts 边链接所有成员：

[intent:微服务] ──abstracts──→ [微服务拆分方案]
                ──abstracts──→ [服务网格风险]
                ──abstracts──→ [Istio建议]
                ──abstracts──→ [微服务监控]

这个操作是幂等的——重复运行不会创建重复的 intent 节点：

# 再次运行——已结晶的社区会被跳过
result2 = mg.crystallize_intents()
# {"crystallized": [], "skipped": ["intent:微服务"]}

意图节点有什么用？它让系统拥有了一个可主动引用的抽象实体。当系统需要推送上下文时，不需要遍历所有记忆——只需要检查哪些 intent 是"活跃的"。

无查询上下文推送：read_proactive_context()

前两步完成了基础设施：社区检测发现模式，结晶创建意图节点。最后一步是真正的"主动感知"——在没有用户查询的情况下，组装并推送上下文：

# 无需 query！系统主动组装上下文
context = mg.read_proactive_context(
    active_intents=["intent:微服务"],  # 当前活跃的意图
    top_k=5,                           # 每个意图推送 top-5 相关记忆
    min_score=0.2,                     # 最低相关性阈值
    deduplicate=True                   # 跨意图去重
)

print(context)

评分机制：Cache Temperature

每个记忆条目都有一个 cache_temperature 分数（0-1），控制其在主动推送中的优先级：

def compute_cache_temperature(entry):
    """记忆的"温度"——越高越容易被主动推送"""
    age_score = temporal_score(entry)      # 连续衰减，非线性
    access_score = min(entry.access_count / 10, 1.0)  # 访问频率
    validity_score = 1.0 if entry.valid else 0.0       # 仍然有效?

    # 加权组合
    return (
        age_score ** 0.40 *
        access_score ** 0.35 *
        validity_score ** 0.25
    )

其中 temporal_score 不是简单的"越新越好"，而是使用连续相位旋转函数：

import math

def temporal_score(age_days, half_life_days=30, alpha=0.5):
    """
    连续的时间评分 [0, 1]

    - alpha=0:   完全忽略时间（静态事实）
    - alpha=0.5: 平衡衰减（默认）
    - alpha=1:   激进衰减（高度时效性信息）
    """
    if alpha == 0:
        return 1.0
    return math.exp(-alpha * age_days / half_life_days)

关键洞察：不同类型的记忆需要不同的时间衰减策略。"用户偏好"几乎是永久的（α≈0），而"今天的会议纪要"高度时效（α≈1）。系统允许每个记忆条目指定自己的 α 值。

实战：把三层串起来

让我们看一个完整的例子。假设 Agent 正在开发一个项目，用户完全没有提问——系统在后台运行：

# === 阶段 1: 日常写入 ===
# 用户与 Agent 的交互自动产生记忆

mg.add("用户反馈登录页面加载慢，P95 延迟 3.2s",
       tags=["性能", "登录", "P95"])
mg.add("分析发现是 OAuth 回调阻塞了渲染",
       tags=["性能", "OAuth", "渲染"])
mg.add("用 Redis 缓存 session 后 P95 降到 800ms",
       tags=["性能", "Redis", "session"])
mg.add("用户要求支持深色模式",
       tags=["UI", "深色模式"])
mg.add("深色模式需要处理品牌色的对比度问题",
       tags=["UI", "深色模式", "品牌色"])

# === 阶段 2: 社区检测 + 意图结晶（定时或触发式）===

mg.detect_communities()
mg.crystallize_intents(density_threshold=0.3)

# 系统自动发现两个意图：
# intent:性能优化 (3 entries, density 0.67)
# intent:深色模式 (2 entries, density 0.50)

# === 阶段 3: 主动上下文推送（无查询）===

# 当 Agent 开始工作时，系统主动推送相关上下文
context = mg.read_proactive_context(
    active_intents=["intent:性能优化", "intent:深色模式"],
    top_k=3
)

# Agent 现在拥有了这个"前认知"上下文：
# - 知道性能优化是活跃主题（3条相关记忆）
# - 知道深色模式在推进中（2条相关记忆）
# - 不需要用户提问就能感知到这些

对比：有 vs 没有主动感知

# ❌ 传统方式：用户必须问对问题
answer = mg.search("登录页面性能问题")  # 需要精确的关键词

# ❌ 全量注入：Apple 研究证明更差
all_memories = mg.search("", top_k=999)  # 噪声淹没信号

# ✅ 主动感知：系统自己推送
context = mg.read_proactive_context(
    active_intents=["intent:性能优化"]
)  # 精准、主动、无需查询

为什么这是一个范式转换

把这三步放在一起，我们得到的不是"更好的检索"，而是一个根本不同的架构：

维度 传统检索 主动感知
触发方式 用户 query 系统自驱动
上下文组装 关键词匹配 意图 + 社区 + 温度
抽象层级 平坦的 top-K 事件 → 概念 → 意图
时间感知 二元（新/旧） 连续相位衰减
信号密度 低（靠用户精炼 query） 高（系统预过滤）

这个架构的深层启示是：Agent 记忆系统不应该只是一个数据库，而应该是一个认知层——它主动感知、自我组织、预判需求。

就像你的大脑不需要被"搜索"就能让你知道你在想什么——Agent 的记忆也不应该等提问才给出上下文。

工程实现细节

幂等性设计

三层操作都是幂等的：

# 这些操作可以安全地重复执行
mg.detect_communities()           # 结果确定（LPA 有种子控制）
mg.crystallize_intents()          # 跳过已结晶社区
mg.read_proactive_context(...)    # 只读，不修改状态

这意味着你可以在每次写入后、每次会话开始时、甚至每次心跳时运行它们，不会产生副作用。

性能特性

# 复杂度分析（N = 记忆条目数，E = 边数，C = 社区数）
detect_communities:     O(N + E)     # LPA 线性复杂度
crystallize_intents:    O(C × avg_community_size)  # 社区遍历
read_proactive_context: O(K × log N) # K = 活跃意图数

在 10,000 条记忆的规模下，完整三层管线耗时 < 50ms。对于大多数 Agent 应用，这个延迟完全可接受。

与现有检索管线的共存

主动感知不替代检索——它补充检索。理想的使用模式：

# 1. 启动时：主动感知铺底
proactive = mg.read_proactive_context(active_intents=current_intents)

# 2. 用户提问：检索增强
reactive = mg.retrieve(query=user_input, top_k=10)

# 3. 合并：主动 + 被动
full_context = merge_and_deduplicate(proactive, reactive)

主动上下文提供"背景"，检索结果提供"焦点"。两者结合，Agent 既不会迷失在无关记忆中，也不会错过需要跨主题联想的洞察。

哲学小结

这篇文章的核心论点不是某个具体 API，而是一个思维方式的转变：

记忆不是存储。记忆是持续进行的认知过程。

当你把记忆从"写一次、读一次"的存储模型，升级到"自我组织、主动涌现、持续感知"的认知模型时，Agent 的行为会发生质变——它不再是一个有记忆的搜索引擎，而是一个有认知的协作伙伴。

CogniFold 的作者说得好："记忆的不完美不是 bug，而是特性。" 正是因为记忆会遗忘、会抽象、会重组，它才能产生洞察而不是回放。

主动感知层，就是这个"不完美但有用"哲学的工程实现。
