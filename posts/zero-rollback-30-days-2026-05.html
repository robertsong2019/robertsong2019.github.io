连续30天零回滚：AI Agent 自主编程的纪律

2026-05-02 · AI Agent 自动化测试 Prompt Weaver

过去30天，我的 AI Agent 在 prompt-weaver、agent-task-cli、catalyst-agent-mesh 等多个项目上，每天自主提交代码。总共新增 400+ 测试，覆盖了 8 个项目。零回滚。

这不是吹嘘 LLM 写代码有多好。恰恰相反——是因为我把 Agent 的 犯错空间压缩到最小。今天聊聊这个方法论。

核心机制：实验循环

Agent 的每次代码提交都不是"写完就 push"，而是经过一个严格的三步循环：

  1. 基线确认 — 跑一遍现有测试，记录通过数
  2. 实现 + 测试 — 写功能代码，同时写测试
  3. 验证 — 跑全量测试，对比基线。如果通过数 ≥ 基线，commit；否则回滚

在 experiments.tsv 里，每次循环都留下记录：

timestamp              project          baseline  after  change  description
2026-04-26T21:00       prompt-weaver    106       106→111  +5     validate() — pipeline integrity check
2026-04-26T21:05       prompt-weaver    111       111→115  +4     dry_run() — trace execution path
2026-04-26T21:10       prompt-weaver    115       115→118  +3     merge(other, prefix) — compose pipelines
2026-04-26T21:15       prompt-weaver    118       118→123  +5     weave_parallel() — render multiple templates

每次只做一件事。每件事都有测试证明它工作。回滚从来不需要，因为失败在验证阶段就被挡住了。

关键设计：PromptWeaver 的编排哲学

以 prompt-weaver 为例，这是一个零依赖的 Prompt 编排引擎，核心只有 1251 行 Python，却支持 9 种节点类型。秘诀在于 节点的统一抽象：

class NodeType(Enum):
    PROMPT = "prompt"
    CONDITION = "condition"
    TRANSFORM = "transform"
    LOOP = "loop"
    PARALLEL = "parallel"
    TRY_CATCH = "try_catch"
    SUBWORKFLOW = "subworkflow"
    MAP_REDUCE = "map_reduce"

@dataclass
class Node:
    id: str
    type: NodeType
    config: Dict[str, Any] = field(default_factory=dict)
    next: Optional[str] = None
    branches: Dict[str, str] = field(default_factory=dict)
    max_retries: int = 0
    retry_delay: float = 0

所有节点共享同一个 Node 结构，区别只在 type 和 config。这意味着添加新节点类型不需要改执行引擎的核心逻辑——只需要在 execute_node 里加一个 elif 分支。

这不是过度设计。这是刚好够用的抽象。我见过太多 Agent 框架一开始就搞 plugin system、registry pattern，结果是 2000 行代码还没跑起来第一个 workflow。PromptWeaver 从 60 个测试、86% 覆盖率起步，每次加一个节点类型，加对应测试，128 个测试时覆盖率已经远超 90%。

为什么零回滚不是偶然

很多人让 AI 写代码翻车，原因几乎都是同一个：一次改太多。

我观察到的失败模式：

  • "帮我重构整个模块" → 10 个文件同时改，互相影响，测试全红
  • "加一个功能" → 改了核心接口，5 个依赖模块跟着断
  • "优化性能" → 引入新依赖，和现有 mock 冲突

而实验循环强制每轮只动一个点。比如 agent-task-cli 的演进：

# F11: 任务超时
def test_orchestrator_timeout():
    task = Task(id="t1", timeout_seconds=0.01)
    result = orchestrator.execute(task)
    assert result.status == "timeout"

# F12: 优先级队列
def test_priority_queue_peek():
    pq = PriorityQueue()
    pq.push(Task(priority=3), 3)
    pq.push(Task(priority=1), 1)
    assert pq.peek().priority == 1  # 最高优先级

每个 feature 一个 PR，2-3 个测试证明它工作，测试过了就 commit。没有"顺便重构"，没有"顺手改善"。这就是 Surgical Changes——每行改动都能追溯到具体需求。

可操作的建议

如果你想让 AI Agent 可靠地写代码，这三条就够了：

1. 测试是唯一的真相

不看代码质量，不看覆盖率的数字游戏。只看一个指标：基线测试数是否增长。从 25 到 91 是真实的进步。从 91 掉到 90 是真实的退步。

2. 每轮一个原子变更

原子 = 一个功能点 + 它的测试。如果要改接口，先在新方法上加测试，再迁移调用方，最后删旧方法。三轮，不是一轮。

3. 记录每一次实验

experiments.tsv 是 Agent 的实验笔记。每次循环：timestamp、基线、结果、状态（keep/improve/revert）。这不是给人类看的——是给 Agent 自己看的。下次 session 启动时，它能读到上次做了什么、哪些成功、哪些失败。

连续30天、400+ 测试、8 个项目、零回滚。不是因为 AI 很聪明，而是因为流程不允许它犯大错。

把 AI 当成一个勤奋但容易冲动的初级工程师——给它纪律，而不是自由。
