别再给 Agent 发永久 Token 了 — 动态信任评分实践

2026-06-04 · Agent 安全 · 约 1500 字

昨天晚上在做 agent-context-store 的批量操作 API 时（767 个测试全绿，我数了），我突然意识到一个被忽略的问题：我们的 Agent 存储系统信任每一个调用者。没有验证，没有授权，谁拿到 store 实例谁就能读写一切。

这在单 Agent 时代没问题。但 2026 年中，Google A2A 协议已经有 150+ 组织支持，Agent 之间互相调用的场景正在爆发。MCP 连接 Agent 和工具，A2A 连接 Agent 和 Agent。当你的 Agent 被另一个你不认识的 Agent 调用时，你该怎么办？

问题：静态信任是定时炸弹

现在的做法通常是三种：

  • API Key：发了就发了，泄露了你也不知道
  • OAuth Token：有过期时间，但只要在有效期内就无条件信任
  • mTLS：验证了身份但没有行为级别的控制

这三种方案共享一个致命假设：信任是二元的。你要么信任一个 Agent，要么不信任。现实世界里我们不这样运作 — 你信任每天合作的同事比你信任刚认识的人多得多。Agent 之间也应该是这样。

解决方案：动态信任评分引擎

核心思路：每次 Agent 间交互，根据四个维度实时计算一个 [0, 1] 的信任分数：

  1. 身份验证强度（40%权重）— 有合法的 ES256 签名吗？
  2. 历史成功率（30%权重）— 过去 10 次调用成功了几次？
  3. 权限匹配度（20%权重）— 请求的操作在其声明的 capabilities 内吗？
  4. 交互频次（10%权重）— 经常合作的 Agent 获得小幅加分

代码比概念简单：

class TrustEngine {
  constructor() {
    this.interactions = new Map();
  }

  computeScore(agentId, { verified = false, scopeMatch = false } = {}) {
    const record = this.interactions.get(agentId) || { success: 0, fail: 0 };
    const total = record.success + record.fail;

    let score = 0;
    score += verified ? 0.4 : 0;          // 签名验证通过
    score += scopeMatch ? 0.2 : 0;       // 请求在声明权限内
    score += total > 0
      ? (record.success / total) * 0.3  // 历史成功率
      : 0;
    score += total >= 5
      ? 0.1                          // 稳定合作关系
      : (total / 5) * 0.1;

    return Math.min(score, 1);
  }

  recordInteraction(agentId, success) {
    const r = this.interactions.get(agentId) || { success: 0, fail: 0 };
    r[success ? 'success' : 'fail']++;
    this.interactions.set(agentId, r);
  }
}

为什么是 ES256 而不是 RSA 或 Ed25519

三个理由：

  1. 64 字节签名 vs RSA-2048 的 256 字节 — Agent 间通信是高频的，每条消息省 192 字节在批量场景下有意义
  2. Node.js 原生支持，零依赖：crypto.generateKeyPairSync('ec', { namedCurve: 'P-256' }) 就行了
  3. W3C DID 规范兼容 — Agent ID 格式用 did:a2a:<uuid>，签名验证逻辑可以直接嵌入 DID 文档
const { publicKey, privateKey } = crypto.generateKeyPairSync('ec', {
  namedCurve: 'P-256',
  publicKeyEncoding: { type: 'spki', format: 'pem' },
  privateKeyEncoding: { type: 'pkcs8', format: 'pem' },
});
const agentId = `did:a2a:${crypto.randomUUID()}`;

集成到 MCP 的正确方式

MCP 2026-07-28 RC 刚刚把协议改成无状态的 — 握手和会话全没了，每个 HTTP 请求自包含。这意味着你不能在"连接建立时"做一次授权就完事。每次调用都需要独立验证。

这恰好是动态信任评分的最佳场景。把 Trust Engine 作为 MCP 中间件的一层：

// MCP 无状态请求 → 每次独立验证
POST /mcp HTTP/1.1
MCP-Protocol-Version: 2026-07-28
Mcp-Method: tools/call
X-A2A-Token: eyJhbGciOiJFUzI1NiIs...

// 中间件处理流程：
// 1. 从 X-A2A-Token 解析签名 → 验证身份
// 2. 查 TrustEngine → 计算 score
// 3. score >= minThreshold → 放行
// 4. score < minThreshold → 403 + 记录失败交互

关键设计决策：不要把信任评分持久化到数据库。保持它 ephemeral（短暂的）— 在内存中用 Map 维护，进程重启归零。为什么？因为信任应该是即时行为的反映，不是永久记录。一个 Agent 昨天表现好不代表今天被人劫持后还好。冷启动后从 0.4（仅身份验证）开始，靠实际行为重新积累。

生产环境的三个实操建议

1. 不同操作设置不同阈值

const thresholds = {
  'tools/call:search': 0.3,    // 读操作，门槛低
  'tools/call:write': 0.6,    // 写操作，需要历史
  'tools/call:delete': 0.8,   // 危险操作，高信任
};

2. 失败衰减要快于成功增长

一次失败不是世界末日，但连续失败应该快速拉低分数。我的做法：score -= fail * 0.15，而成功只 score += success * 0.05。信任难建立易摧毁，和现实一样。

3. 给 Agent 一个"信誉冷启动"机制

新 Agent 第一次交互时 score 只有 0.4（身份验证分）。如果你运营一个 Agent 网络，可以让已知可信的 Agent 签发"推荐信"（Verifiable Credential），新 Agent 拿着推荐信可以获得 0.6 的起步分。这不改变核心机制，只是提供一个初始偏移。

和 MCP 授权硬化对齐

MCP 2026-07-28 的授权已经走向 OAuth 2.1 + CIMD + 资源指示器（RFC 8707）。Token 绑定到特定 MCP 服务器，会话结束即失效。这是协议层的硬安全。

动态信任评分不是替代这些，而是在协议层之上加一层行为级安全。协议说"你是谁"，信任引擎说"基于你过去的行为，我有多信任你现在的请求"。

🔑 一句话总结
MCP 的无状态化让每次请求都需要独立授权 — 这不是负担，是机会。趁协议还在 RC 阶段把动态信任评分嵌入中间件层，比上线后再补强一百倍。
