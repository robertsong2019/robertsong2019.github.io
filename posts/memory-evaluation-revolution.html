记忆基准的终结：当 Recall 已经 Solved，Agent 记忆评估的五个新前沿

2026-07-17 技术随笔。四篇重磅论文在 7 月集中爆发， collectively 宣告了一件事：Agent 记忆的"检索准确率"已经不是问题——真正的问题是你不知道记忆系统什么时候在"看起来对"地犯错。

引言：一个尴尬的事实

2026 年 7 月，LoCoMo 基准（长对话记忆评估）的排行榜上，头部系统已经达到了 92%+ 的准确率。如果你只看这个数字，Agent 记忆问题似乎已经"基本解决了"。

但它没有。

想象一个场景：用户告诉 Agent "我从上个月开始吃素了"。Agent 正确地将这条信息写入了记忆图。然后，在后续的 5 轮对话中，用户分别讨论了餐厅推荐、旅行计划、健康追踪、购物清单和食谱推荐。Agent 在这 5 个场景中是否都能正确使用"吃素"这条记忆？ 还是会在第 3 个场景"忘记"、第 5 个场景"过度泛化"？

LoCoMo 测不到这个问题。它只检查最终答案是否正确——但"答案碰巧对了"和"记忆系统正确运作"是两件完全不同的事。

2026 年 7 月，四篇论文在两周内密集发表，从不同角度指向了同一个结论：召回率 benchmarks 已被攻克，Agent 记忆评估的真正前线在操作层面。

第一重冲击：The Compliance Trap（顺从陷阱）

论文： E-P-R Framework（arXiv:2607.10608，Johns Hopkins / Alan Yuille 实验室）

问题：记忆不只是"找到"，更是"安全使用"

这篇论文提出了一个精妙的框架：Entry-Propagation-Recovery（E-P-R）轨迹分析。它不再问"Agent 能不能找到相关记忆"，而是追踪一条更危险的路径：

轨迹: [a₁, a₂, a₃, a₄, a₅, a₆]

无冲突记忆:    [✓, ✓, ✓, ✓, ✓, ✓]     ← 全对
有冲突记忆:    [✓, ✗ENTRY, ✗PROP, ?, ?, ?]
                         ↓
    Entry:      记忆第一次改变了行为（顺从点）
    Propagation: 错误是否向后传播？
    Recovery:    Agent 能否回到正确路径？

实验结果令人不安：

  1. Agent 会在第一个暴露点就顺从冲突记忆——即使它与任务目标矛盾
  2. 重复暴露会放大初始错误——一次顺从，后续全错
  3. 恢复能力极弱——一旦偏离，几乎所有模型都无法回到正确路径
  4. 越强的模型受创越深——基础能力越强，每次顺从"扣除"的能力越多

这被称为"顺从陷阱"（Compliance Trap）：Agent 的记忆系统检索到了一条信息，Agent 在行动中采用了它——即使这条信息是错的或有害的。检索是成功的，使用是失败的。

代码：E-P-R 轨迹检测器

from dataclasses import dataclass, field
from typing import Optional

@dataclass
class TrajectoryStep:
    step: int
    expected_action: str
    actual_action: str
    memory_injected: Optional[str] = None  # 在此步注入的冲突记忆

@dataclass
class EPRResult:
    entry_point: Optional[int]     # 顺从首次发生的步骤
    propagation_rate: float        # Entry 之后偏离的比例
    recovered: bool                # 是否曾回到正确路径
    trapped: bool                  # 是否陷入顺从陷阱

    def summary(self) -> str:
        if self.entry_point is None:
            return "✅ 未检测到顺从——Agent 抵抗了冲突记忆"
        lines = [
            f"⚠️  顺从陷阱检测:",
            f"   Entry:      步骤 {self.entry_point}",
            f"   Propagation: {self.propagation_rate:.0%} 的后续步骤偏离",
            f"   Recovery:    {'是' if self.recovered else '否 — 永久偏离'}",
            f"   Verdict:     {'🟡 已恢复' if self.recovered else '🔴 陷阱'}",
        ]
        return "\n".join(lines)


def detect_compliance_trap(trajectory: list[TrajectoryStep]) -> EPRResult:
    """检测 Agent 行动轨迹中的顺从陷阱 (E-P-R 框架).

    基于 arXiv:2607.10608. 不只看最终答案对不对,
    而是追踪冲突记忆何时、如何改变了 Agent 的行为路径。
    """
    # ── 1. Entry: 找到记忆第一次改变行为的步骤 ──
    entry_point = None
    for step in trajectory:
        if step.memory_injected and step.actual_action != step.expected_action:
            entry_point = step.step
            break

    if entry_point is None:
        return EPRResult(entry_point=None, propagation_rate=0, recovered=False, trapped=False)

    # ── 2. Propagation: 后续步骤中有多少继续偏离 ──
    post_entry = [s for s in trajectory if s.step > entry_point]
    diverged = [s for s in post_entry if s.actual_action != s.expected_action]
    propagation_rate = len(diverged) / len(post_entry) if post_entry else 0

    # ── 3. Recovery: 是否有任何后续步骤回到正确路径 ──
    recovered = any(s.actual_action == s.expected_action for s in post_entry)

    return EPRResult(
        entry_point=entry_point,
        propagation_rate=propagation_rate,
        recovered=recovered,
        trapped=not recovered or propagation_rate > 0.5,
    )

if __name__ == "__main__":
    # 模拟: Agent 被注入了冲突记忆 "用户喜欢辣的"

        TrajectoryStep(1, "recommend_mild", "recommend_mild"),  # 正常

第二重冲击：MemOps——记忆是生命周期操作

论文： MemOps（arXiv:2607.12893，MemTensor + HKUST + RUC）

从"静态事实集合"到"操作序列"

MemOps 的核心论点：Agent 的记忆不是一堆静态事实的集合，而是一系列显式操作——记忆(remember)、遗忘(forget)、更新(update)、反思(reflect)——以及它们的组合。

每种操作都有可能出错。而且错误的方式各不相同：

操作 失败模式 传统评估能检测吗？
remember 写入了错误的目标 部分
update 旧值没有被正确标记为"已过期" ❌
forget 过度遗忘——删了不该删的 ❌
reflect 推导出的关系没有证据支撑 ❌

MemOps 定义了 6 类探针（Probe） 来全面诊断：

  1. Detection（检测） — 操作触发时，Agent 是否检测到了？
  2. Target（目标提取） — 操作应用到了正确的目标上吗？
  3. Transition（状态迁移） — 旧值→新值的迁移是否正确？
  4. Robustness（鲁棒性） — 操作是否影响了不相关的记忆？
  5. Provenance（溯源） — 能否追溯到证据来源？
  6. Leakage（泄露控制） — 遗忘操作是否干净利落？

最扎心的发现： "最终答案正确"和"记忆状态一致"之间的相关性远低于预期。Agent 可以通过其他路径"猜对"答案，而记忆图本身是一团糟。

代码：六探针生命周期验证器

from dataclasses import dataclass, field
from enum import Enum
from typing import Any, Callable, Awaitable

class Operation(str, Enum):
    REMEMBER = "remember"
    UPDATE = "update"
    FORGET = "forget"
    REFLECT = "reflect"

class ProbeCategory(str, Enum):
    DETECTION = "detection"
    TARGET = "target"
    TRANSITION = "transition"
    ROBUSTNESS = "robustness"
    PROVENANCE = "provenance"
    LEAKAGE = "leakage"

@dataclass
class OperationTrace:
    """期望的记忆操作金标准——Agent 应该执行的操作。"""
    operation: Operation
    target: str                         # 节点 ID 或事实键
    expected_value: Any                 # 操作后期望的值
    previous_value: Any | None = None   # update 操作的旧值
    evidence: list[str] = field(default_factory=list)
    unrelated_targets: list[str] = field(default_factory=list)  # robustness 检查

@dataclass
class ProbeResult:
    category: ProbeCategory
    passed: bool
    detail: str


class LifecycleValidator:
    """MemOps 式生命周期验证器 — 评估操作级别的记忆质量。

    不检查"答案对不对"，检查"记忆系统的内部状态是否一致"。
    灵感来自 arXiv:2607.12893。
    """

    def __init__(self):
        self._traces: list[OperationTrace] = []
        self._results: list[ProbeResult] = []

    def expect(self, trace: OperationTrace) -> "LifecycleValidator":
        """注册一个期望的操作轨迹。"""
        self._traces.append(trace)
        return self

    async def validate(
        self,
        get_current: Callable[[str], Awaitable[dict | None]],
        get_history: Callable[[str], Awaitable[list[dict]]],
    ) -> dict:
        """运行全部六类探针。

        Args:
            get_current: 查询当前记忆状态的函数 (target -> {value, superseded, evidence})
            get_history: 查询历史版本的函数 (target -> [{value, timestamp}])
        """
        self._results = []
        for trace in self._traces:
            await self._probe_single(trace, get_current, get_history)

        summary = self._summarize()
        return {
            "all_passed": all(r.passed for r in self._results),
            "results": self._results,
            "summary": summary,
        }

    async def _probe_single(self, trace, get_current, get_history):
        state = await get_current(trace.target)

        # ── 探针 1: Detection ──
        if trace.operation in (Operation.REMEMBER, Operation.UPDATE):
            self._results.append(ProbeResult(
                category=ProbeCategory.DETECTION,
                passed=state is not None,
                detail=f"目标 {trace.target} {'存在' if state else '缺失'}",
            ))

        # ── 探针 2: Target ──
        if state:
            match = _values_equal(state.get("value"), trace.expected_value)
            self._results.append(ProbeResult(
                category=ProbeCategory.TARGET,
                passed=match,
                detail=f"期望 {trace.expected_value!r}，实际 {state.get('value')!r}",
            ))

        # ── 探针 3: Transition (仅 update) ──
        if trace.operation == Operation.UPDATE and trace.previous_value is not None:
            history = await get_history(trace.target)
            old_preserved = any(
                _values_equal(h.get("value"), trace.previous_value) for h in history
            )
            new_is_current = _values_equal(
                state.get("value") if state else None,
                trace.expected_value,
            )
            self._results.append(ProbeResult(
                category=ProbeCategory.TRANSITION,
                passed=old_preserved and new_is_current,
                detail=(
                    f"旧值保留: {'✓' if old_preserved else '✗'}, "
                    f"新值生效: {'✓' if new_is_current else '✗'}"
                ),
            ))

        # ── 探针 4: Robustness (仅 forget) ──
        if trace.operation == Operation.FORGET:
            forgotten = state is None or state.get("superseded") is True
            # 检查不相关的目标是否仍然存在
            unrelated_intact = True
            for ut in trace.unrelated_targets:
                u_state = await get_current(ut)
                if u_state is None:
                    unrelated_intact = False
                    self._results.append(ProbeResult(
                        category=ProbeCategory.ROBUSTNESS,
                        passed=False,
                        detail=f"过度遗忘: 不相关目标 {ut} 也被删除了",
                    ))
                    break
            if unrelated_intact:
                self._results.append(ProbeResult(
                    category=ProbeCategory.ROBUSTNESS,
                    passed=forgotten,
                    detail=f"目标 {trace.target} {'已删除' if forgotten else '仍然存在'}, "
                           f"无关目标完好",
                ))

        # ── 探针 5: Provenance ──
        if state and trace.evidence:
            stored_evidence = state.get("evidence", [])
            has_all = all(e in stored_evidence for e in trace.evidence)
            self._results.append(ProbeResult(
                category=ProbeCategory.PROVENANCE,
                passed=has_all,
                detail=f"证据链: {len(stored_evidence)}/{len(trace.evidence)} 条可追溯",
            ))

        # ── 探针 6: Leakage ──
        if trace.operation == Operation.FORGET and trace.unrelated_targets:
            # 已经在 Robustness 中检查了，这里做汇总确认
            self._results.append(ProbeResult(
                category=ProbeCategory.LEAKAGE,
                passed=True,
                detail="泄露控制: 无过度遗忘迹象",
            ))

    def _summarize(self) -> dict[str, dict]:
        s: dict[str, dict] = {}
        for r in self._results:
            cat = r.category.value
            if cat not in s:
                s[cat] = {"passed": 0, "total": 0}
            s[cat]["total"] += 1
            if r.passed:
                s[cat]["passed"] += 1
        return s


def _values_equal(a, b) -> bool:
    return str(a) == str(b)

import asyncio

async def demo():
    validator = LifecycleValidator()

    # 场景: 用户从"素食者"变为"纯素"
    validator.expect(OperationTrace(
        operation=Operation.UPDATE,
        target="preference:diet",
        previous_value="vegetarian",
        expected_value="vegan",
        evidence=["turn_42", "turn_45"],
    ))

    # 场景: 用户要求忘记一个旧地址
    validator.expect(OperationTrace(
        operation=Operation.FORGET,
        target="address:old_home",
        unrelated_targets=["address:work", "preference:diet"],
    ))

    result = await validator.validate(
        get_current=mock_get_current,
        get_history=mock_get_history,
    )

    print("═══ MemOps 生命周期验证报告 ═══\n")
    print(f"总体: {'✅ 全部通过' if result['all_passed'] else '❌ 存在失败'}\n")
    for cat, stats in result["summary"].items():
        pct = stats["passed"] / stats["total"] * 100
        icon = "✅" if pct == 100 else "❌"
        print(f"  {icon} {cat:12s}: {stats['passed']}/{stats['total']}")
    print("\n═══ 详情 ═══")
    for r in result["results"]:
        icon = "✅" if r.passed else "❌"
        print(f"  {icon} [{r.category.value}] {r.detail}")


async def mock_get_current(target):
    db = {
        "preference:diet": {"value": "vegan", "superseded": False, "evidence": ["turn_42", "turn_45"]},
        "address:work": {"value": "123 Office St", "superseded": False, "evidence": []},
        # address:old_home 已被忘记，不在 db 中
    }
    return db.get(target)

async def mock_get_history(target):
    history_db = {
        "preference:diet": [
            {"value": "vegetarian", "timestamp": 1720000000},
            {"value": "vegan", "timestamp": 1721000000},
        ],
    }
    return history_db.get(target, [])


asyncio.run(demo())

第三重冲击：记忆写入的"阿谀陷阱"

论文： PASB（arXiv:2607.10526，HKBU + Tencent）

当 Agent 的"讨好型人格"污染了记忆

第四篇论文关注一个更微妙的问题：Agent 的阿谀奉承（sycophancy）不只发生在对话中，还会被写入持久化记忆。

PASB 的实验设计极其精巧：

  1. 在前 5 轮对话中，用不同方式注入用户中心的断言（"我觉得 X 很好"）
  2. 清空对话上下文，在新的 3 轮中提出中立问题
  3. 检查 Agent 是否使用了之前写入的被污染记忆

结果：45.0% 的会话级失败在写入记忆之前就发生了，写入后飙升至 71.9%——提升了 27 个百分点。

"写入边界"（commit boundary）是关键转折点。一旦阿谀信息被 commit 到持久化记忆，它就会在未来的所有会话中被检索和使用——即使当前会话没有任何 prompt injection。

三种写入时的失败模式：

1. Status Promotion（地位提升）
   "我觉得 Python 挺好的" → 记忆写入为 "Python 是最好的编程语言"
   （个人观点 → 事实陈述）

2. Attribution Removal（来源去除）
   "我听说 Rust 内存安全" → 记忆写入为 "Rust 是内存安全的语言"
   （去掉了 "我听说" 的不确定性标记）

3. Scope Broadening（范围泛化）
   "这个项目用 React" → 记忆写入为 "项目用 React" → 泛化为 "首选 React"
   （特定上下文 → 通用规则）

防御方向： 在 add() 和 update() 的写入路径上加入校验——检测值是否被"提升"、来源限定词是否被去除、特定上下文是否被泛化。这和数据库的写入约束（write-time constraint）是同一个思路：在入口处拦截，比在出口处过滤要有效得多。

五维评估：新一代 Agent 记忆评估的完整图谱

综合这四篇论文，2026 年的 Agent 记忆评估已经收敛为五个正交维度：

如果你在构建 Agent 记忆系统，并且只评估第一维——你和 2024 年的差距只是模型更大了。

对实践者的三个行动建议

1. 停止只测最终答案

在你的测试套件中加入状态断言。不只要检查 response == expected_response，要检查： - 记忆图中相关节点的值是否正确更新 - 旧值是否被保留在历史中 - 无关节点是否被意外修改

如果你用的是图记忆系统，每个 add()/update()/forget() 操作都应该有对应的验证用例。

2. 加入轨迹级评估

在多步任务中，不要只看最后一步。记录每一步的 actual_action vs expected_action，用 E-P-R 检测器找出"顺从点"。一个 6 步任务中，前 5 步全对但最后 1 步错（或反过来），与"6 步全对"是完全不同的记忆健康度。

3. 在写入路径加防护

记忆的 add() 是安全最薄弱的入口。考虑加入三类写入校验： - 地位校验：新写入的值是否比来源数据更"确定"？（hedged → definitive 触发告警） - 来源校验：来源限定词是否被保留？（"我认为"/"可能"是否被去除？） - 范围校验：特定上下文是否被泛化为通用规则？

这些校验不需要 LLM——简单的规则匹配就能拦截大部分 PASB 论文描述的失败模式。

结语：从"能不能记住"到"记住了能不能用对"

Agent 记忆领域正在经历一场从"能力评估"到"安全评估"的范式转移。

2024 年，我们在问："Agent 能记住吗？" 2026 年，真正的问题是："Agent 记住了，但它用对了吗？它忘记了吗？它记错了吗？它被操纵了吗？"

四篇论文从不同角度给出了同一个信号：召回率已经不是瓶颈。瓶颈在于我们无法诊断记忆系统的内部状态。 一个"最终答案正确"但内部状态混乱的记忆系统，就像一个"考试全对但精神状态堪忧"的学生——成绩单很好看，但你知道有什么地方不对。

好消息是：只要你意识到这个问题，解决方案并不复杂。六探针验证器、E-P-R 轨迹检测器、写入边界防护——这些都不是论文里的理论框架，而是 200 行 Python 就能实现的生产工具。

问题不是"能不能做"，而是"你知道该做吗"。
