Autoresearch：让 Agent 自己做实验的开发方法论

📅 2026-04-24 · 标签：Autoresearch, AI Agent, 方法论, 自动化开发

过去 24 小时，我的 agent-memory-service 项目从 371 个测试涨到了 395 个，新增了 7 个 API 方法，写了大约 350 行源码。零回滚。全是 AI Agent 自己干的。

这不是那种"让 AI 写个 hello world"的玩具实验。每个功能都有测试覆盖、有设计决策、有记录在案的教训。我想把这套方法论——我称之为 Autoresearch——分享出来，因为它改变了我对"AI 辅助开发"的认知。

核心循环：30 分钟一轮的实验

Autoresearch 的基本单位是一个 实验循环（Experiment Cycle），每轮大约 30 分钟：

  1. 选择目标：从候选列表中选下一个最有价值的功能点
  2. 实现 + 测试：写代码，写测试，确保全部通过
  3. 决策：KEEP（保留）还是 ROLLBACK（回滚）
  4. 记录：写入 experiments.tsv，包括提交 hash、测试数、状态、描述

关键约束：每轮只做一件事。不是"顺便重构一下"，不是"既然打开了就改改那个"。一个 cycle，一个功能，一组测试。

为什么是 30 分钟？ 因为这是"足够写一个小功能 + 测试"但"不够搞砸整个项目"的时间窗口。如果 30 分钟还没写完，说明功能拆得太大了。

实战案例：24 小时，7 个功能

让我用实际数据说话。这是 agent-memory-service 项目过去 24 小时的实验记录：

时间功能新增测试新增代码状态
04-23 19:00compareMemories(id1, id2)+5~35 行✅ keep
04-23 19:10tagHierarchy(opts)+4~55 行✅ keep

注意一个规律：每个功能都是前一个的自然延伸。compareMemories 发现标签体系有层级关系 → tagHierarchy。层级多了需要权重调整 → rebalance。有了层级和权重 → summarizeCluster 生成摘要。有了摘要 → autoTag 批量标注。标注完了 → mergeClusters 合并同类。

这不是我规划的路线图。这是 Agent 自己在每一轮结束后，根据代码现状推理出的"下一步最合理的事情"。

experiments.tsv：不可篡改的实验日志

整个方法论的核心基础设施是一个简单的 TSV 文件：

timestamp                commit   metric         value     status  description
2026-04-24T01:00         212574f  test_count     394/395   keep    mergeClusters(topics, opts) — merge multiple topic clusters into unified tag.
2026-04-24T00:00         be8a86a  test_count     389/389   keep    autoTag(opts) — automatically apply suggestTags to untagged memories.
2026-04-23T01:00         e3a6637  test_count     371/371   keep    summarizeCluster(topic, opts) — aggregate summary of memories in a topic cluster.

为什么用 TSV 而不是数据库？因为 git 友好。每一行变更都是一个 diff，可以回溯、可以 code review、可以作为 CI 的 checkpoint。Agent 在每次 commit 前读取这个文件，知道项目到了哪一步。

这个文件的格式我强烈推荐你抄走：

// experiments.tsv 列定义
timestamp   — ISO 8601 时间戳，精确到分钟
commit      — git hash（可验证）
metric      — 你关心的指标（test_count, test_pass_rate, 等）
value       — 当前值（371/371 表示 371 total, 371 passing）
status      — keep | rollback | improve
description — 人可读的功能描述

教训捕获：每个 Bug 都是一次学习

Autoresearch 不是只记录成功。每个 cycle 的开发笔记里都有"教训"部分。举几个真实例子：

教训 1： new MemoryService(tmp) vs new MemoryService({ dbPath: tmp }) — 传字符串参数不会设置 dbPath。构造函数只解构对象。

教训 2： tagHierarchy 的 minCoOccurrence=2 时，低频共现标签不会建立关系。这不是 bug，是设计选择，但测试必须显式覆盖这个边界条件。

教训 3： totalTags 是唯一标签数，不是输入标签数。去重后的计数。测试里如果放重复标签会得到比预期小的数字。

这些教训会被写进当天的 memory 日志。当同一个教训出现第三次时，它会升级成代码层面的约束（比如构造函数参数校验），而不是继续靠文档提醒。

代码示例：mergeClusters 的完整实现

让我展示一个典型的 autoresearch cycle 产出——mergeClusters。这个功能解决的问题是：当你有 "machine-learning" 和 "neural-networks" 两个独立的知识集群，想把它们合并成一个：

/**
 * Merge multiple topic clusters into one unified cluster.
 * @param {string[]} topics - Array of topic tag names to merge
 * @param {object} opts - Options
 * @param {string} opts.targetTag - Unified tag name (default: first topic)
 * @param {boolean} opts.removeSourceTags - Remove source tags (default: true)
 * @returns {{ targetTag: string, mergedCount: number, sourceTopics: string[] }}
 */
mergeClusters(topics, opts = {}) {
    if (!Array.isArray(topics) || topics.length < 2) {
        throw new Error('mergeClusters requires at least 2 topics');
    }

    const targetTag = opts.targetTag || topics[0];
    const removeSourceTags = opts.removeSourceTags !== false;
    let mergedCount = 0;

    for (const topic of topics) {
        const matches = this.#store.query({ tag: topic });
        for (const mem of matches) {
            const tags = mem.tags || [];
            const hasTarget = tags.some(t =>
                t.toLowerCase() === targetTag.toLowerCase()
            );

            if (!hasTarget) {
                tags.push(targetTag);
            }

            if (removeSourceTags) {
                const filtered = tags.filter(t =>
                    t.toLowerCase() !== topic.toLowerCase()
                );
                mem.tags = hasTarget && !removeSourceTags ? tags : filtered;
                if (!hasTarget && removeSourceTags) {
                    mem.tags = filtered.includes(targetTag)
                        ? filtered
                        : [...filtered, targetTag];
                }
            } else {
                mem.tags = tags;
            }
            this.#store.update(mem.id, { tags: mem.tags });
            mergedCount++;
        }
    }

    this.#store.reindex();
    return {
        targetTag,
        mergedCount,
        sourceTopics: topics.filter(t =>
            t.toLowerCase() !== targetTag.toLowerCase()
        )
    };
}

~40 行源码。6 个测试覆盖：正常合并、默认移除源标签、保留源标签、默认 targetTag、少于 2 个 topic 报错、已有 targetTag 跳过。每个边界条件一个测试。

为什么这套方法有效？三个反直觉的洞察

1. 限制反而加速

"每轮只做一件事"听起来效率低下，但实际效果相反。因为 Agent 不需要在多个上下文间切换，不需要处理"改了 A 导致 B 挂了"的连锁反应。每个 cycle 的 diff 通常小于 60 行，code review（即使是人类回看）只需要 2 分钟。

2. 测试是最好的设计文档

我没有为这些功能写过一行设计文档。但看测试用例：

// 这个测试比任何文档都清楚地说了 API 的行为
test('dryRun does not modify memories', async () => {
    await svc.store('m1', 'content about AI', { tags: ['ai'] });
    await svc.store('m2', 'content about cooking');  // untagged

    const result = await svc.autoTag({ dryRun: true });

    // dryRun=true: 推荐但不写入
    expect(result.tagged).toBe(0);
    const mem = await svc.get('m2');
    expect(mem.tags).toEqual([]);  // 原始数据未修改
});

测试是可执行的规格说明。它们不会过时（因为 CI 会跑），不会撒谎（因为断言会失败），不需要维护者去同步。

3. Agent 不需要大局观

这是最重要的洞察。我没有给 Agent 一个"v2.0 路线图"。我只是给了它：当前代码 + 候选功能列表 + 每轮选最合理的一个。

结果？7 个功能自然形成了一条连贯的功能链。compareMemories → tagHierarchy → rebalance → summarizeCluster → autoTag → mergeClusters。每一步都是上一步的逻辑延伸。

这不是巧合。这是涌现式设计——好的架构不是规划出来的，是约束条件下一步步长出来的。就像生物进化不需要"理解"整个生态系统，只需要每一步做出局部最优的选择。

可操作建议：如何在你的项目中启动 Autoresearch

  1. 创建 experiments.tsv：第一行写表头，第二行写你的 baseline（当前测试数）。把它加入 git 追踪。
  2. 列一个候选功能清单：不需要完整规划，只需要 5-10 个"下一步可以做的小功能"。每个功能应该能在 30-40 行内实现。
  3. 设定 cycle 约束：每轮只做一个功能，写完必须通过全部测试，必须记录到 TSV。不合格就 rollback。
  4. 捕获教训：每个 cycle 结束后写下"如果重来我会怎么做"。这些教训比代码更有价值。
  5. 定期回看：每 5-10 个 cycle 后，停下来看 experiments.tsv 的整条链。你会发现功能自然聚集成主题，形成你最初没想到的架构。
核心原则： 不要规划 3 个月的路标。规划下一个 30 分钟的实验。让架构从约束中涌现，而不是从白板上的框图中规划。

agent-memory-service 现在有 395 个测试，7 天前是 195 个。没有回滚过一次。不是因为代码完美——是因为每个改动足够小，小到错误在下一个 cycle 开始之前就能被发现和修复。

这就是 Autoresearch 的力量：不是让 AI 替你写代码，而是让 AI 替你做实验。代码是实验的副产物。
