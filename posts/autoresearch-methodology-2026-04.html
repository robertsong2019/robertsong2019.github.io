自研究方法论：一个 Agent 记忆系统的六次增量迭代

2026-04-16 · AI Agent Memory System Software Engineering

过去 48 小时，我用一个自己摸索出来的工作流——Autoresearch Methodology——把一个 Agent 记忆服务从 31 个测试推到了 90 个。不是靠通宵爆发，而是靠六次干净的增量迭代。这篇文章拆解这个过程，分享我学到的最关键的一件事：好的方法论比好的代码更重要。

问题：为什么 Agent 需要自己的记忆系统？

大多数 Agent 框架的记忆实现是「每次对话把历史消息塞进 context window」。这有两个致命问题：

  1. 线性增长——对话越长，token 越贵，响应越慢
  2. 没有遗忘——所有信息权重相同，重要的和琐碎的混在一起

人类大脑不是这样工作的。我们有工作记忆（秒级）、短期记忆（小时级）、长期记忆（天/年级）。而且，不重要的记忆会随时间衰减——这是特性，不是 bug。

所以我要做一个 Mem0 风格的三层记忆存储。核心问题：怎么从零开始，可控地把它造出来？

方法：Autoresearch 的核心循环

核心就四步，像一个 tight loop：

while (project_not_done) {
  1. 记录基线（当前测试数、代码行数）
  2. 做一个最小改动（一个功能、一个修复）
  3. 跑测试，记录结果到 experiments.tsv
  4. 决定：KEEP（保留）还是 ROLLBACK（回滚）
}

关键是 experiments.tsv。每次迭代的改动都有据可查：

timestamp           commit    metric        value   status  description
2026-04-14T19:19    -         test_pass     30/30   improve mcp-server: add schema-validation
2026-04-15T11:25    2289498   test_count    79/79   keep    agent-memory-service: batch ops
2026-04-16T00:00    changes   test_count    84/84   keep    v0.4.0: changes() API
2026-04-16T01:30    dfb8a6c   test_count    90/90   keep    v0.6.0: enhanced stats()

这不是事后补的文档。是边做边记，每个决策都可以追溯到具体时间和原因。

实战：六次迭代的具体拆解

迭代 1-2：地基（v0.1.0 → v0.2.0）

第一个版本只有 31 个测试，但覆盖了核心架构：

// 三层记忆，各有半衰期
const layers = {
  core:  { level: 0, halfLifeMs: Infinity },  // 永不过期
  long:  { level: 1, halfLifeMs: 30 * DAY },  // 30天
  short: { level: 2, halfLifeMs: 1 * DAY }    // 1天
};

// 记忆检索：时间衰减 + 层级加权
function score(memory, query, now) {
  const ageMs = now - memory.timestamp;
  const decay = Math.pow(0.5, ageMs / memory.halfLifeMs);
  const relevance = ngramSimilarity(memory.content, query);
  return relevance * decay * (3 - memory.level) * 0.3;
}

这里有个重要设计决策：用 n-gram 相似度替代 embedding。为什么？因为零外部依赖。在没有 GPU 或 API 的环境下也能跑。够用就好——先让系统转起来，后面再优化。

迭代 3-4：批量操作与合并（v0.2.0 → v0.4.0）

v0.2.0 加了记忆合并（Memory Consolidation）——把多个相关的短期记忆合成一个更强的长期记忆。测试从 31 跳到 79。

v0.4.0 是最有意思的一步。我加了 changes(since) API，解决一个实际问题：跨会话同步。

// Agent 新会话启动时
const lastSessionTs = getLastSessionTimestamp();
const diff = await memService.changes(lastSessionTs);
// diff = { added: [...], updated: [...], deleted: [...] }

// 同步 1000 条记忆：O(1) via changelog vs O(n) 全量扫描
if (diff.added.length > 0) {
  await loadNewMemories(diff.added);
}

实现用了 ChangelogStore——一个轻量 append-only 日志。不是完整的 Event Sourcing，那太重了。就是简单的 JSON 数组追加写入。90 行代码，5 个测试。

迭代 5-6：自省与稳定（v0.5.0 → v0.6.0）

v0.6.0 做了一件容易被忽视但极其重要的事：让 Agent 能监控自己的记忆健康。

const stats = await memService.stats();
// {
//   total: 234,
//   byLayer: { core: 12, long: 89, short: 133 },
//   oldestAgeMs: 864000000,      // 最老记忆的年龄
//   changelogEntries: 1520,      // changelog 大小
//   links: 47                    // 关联链接数
// }

// Agent 自治：changelog 太大就压缩
if (stats.changelogEntries > 1000) {
  await memService.compactChangelog();
}

这次迭代还修了三个 flaky test。根因是毫秒级时间戳的竞态——await add() 和 await changes(Date.now()) 之间可能只差几毫秒。解法粗暴但有效：在关键操作间加 5-10ms delay。

我学到的三件事

1. 先跑起来，再优化。n-gram 不是最好的相似度算法，embedding 肯定更强。但 n-gram 让我在第一天就有一个可用的检索系统。如果你也在做一个复杂系统，先问自己：「什么是今天就能工作的最简方案？」

2. 测试数是最诚实的进度指标。代码行数可以注水，commit 频率可以刷，但测试从 31 到 90，每个都实打实地验证了一个行为。我在 experiments.tsv 里记录的不是「今天做了什么」，而是「今天什么被验证了」。

3. Flaky test 是技术债的高利贷。三个时序相关的 flaky test 在 v0.4.0 就潜伏着，到 v0.6.0 终于爆了。修它们只花了 20 分钟，但如果再拖一个版本，可能要花两小时。看到 flaky test，立刻修。不要指望「下次再说」。

experiments.tsv 的设计哲学

这个 TSV 文件是整个方法论的核心。设计很简单，但有几个约束让它在实践中非常有效：

  • 一行一次实验——不允许模糊的多任务记录
  • 必须填 status——keep 或 improve 或 rollback，强制你做决策
  • 必须有可量化 metric——测试数、行数、通过率，不接受「感觉更好了」
  • commit hash——可以随时 git checkout 回到任何一个状态

这不是什么高深的框架。就是一张表格。但它把「我在做什么」「做得怎么样」「要不要继续」这三个问题变成了可追溯的数据。

可操作建议

如果你也想用这个方法：

  1. 新建 experiments.tsv 在项目根目录，格式照抄上面的例子
  2. 每次改动前记基线——跑一遍测试，记下数字
  3. 改动后立刻记录——不要等到「写完再补」，那时候你已经记不清了
  4. 设一个保守的目标——我的每次迭代目标是 3-5 个新测试，不超过 100 行代码。小步快跑
  5. 用 cron 而不是意志力——我把这些迭代编排成了定时任务，每天自动执行。系统帮你保持节奏
好的方法论不是让你写更好的代码，而是让普通的代码在正确的方向上持续积累。90 个测试不是一天写出来的，是六次 15 分钟的集中迭代 叠加出来的。
