Autoresearch：用实验循环替代瀑布开发

方法论 Agent开发 实验驱动
2026-04-17 · 罗嵩

过去 24 小时，我用一个叫 autoresearch 的方法论把 agent-memory-service 从 v0.4.0 推到了 v0.8.0。测试数从 84 涨到 133。发现并修掉了一个真实 bug。零回滚。这篇文章是我对这个方法论的一次完整复盘。

核心思想：每轮实验都是一个原子单元

Autoresearch 的结构极其简单：

  1. 记录基线（测试数、代码行数、版本号）
  2. 做一个增量改动（单功能，不超过 100 行）
  3. 跑全部测试
  4. 记录结果到 experiments.tsv，决定 KEEP 或 ROLLBACK

就这么简单。没有冲刺计划，没有迭代回顾，没有 JIRA 票。就是一轮又一轮的实验循环。

24 小时实战记录

以下是实际的实验日志：

四轮循环，版本从 v0.4.0 → v0.8.0。每轮都有明确的基线和结果。注意最后一轮——它修掉了一个真实存在的 bug。

被实验发现的 Bug

第三轮实现了 scheduledMaintenance()——一个让 agent 在心跳时做内存整理的便捷方法。第四轮决定加 reindex()，让 agent 可以重建内存索引。

在实现 reindex 的过程中，我发现了一个隐蔽的 bug：put() 方法在更新已有 memory 时，会往 tag/entity 索引里加新条目，但永远不会清理旧条目。

场景是这样的：假设一条 memory 的 tags 从 ['old'] 更新为 ['new']，那么 'old' 的索引里仍然指向这条 memory ID。索引和实际数据不一致。

这个 bug 不做 reindex 根本不会暴露。正是因为实验循环驱动我往这个方向探索，才发现了它。

修复只用了三行代码：

async put(m) {
  const existing = this.store.get(m.id);
  if (existing) this.#deindexMemory(existing);  // 先清理旧索引
  this.#indexMemory(m);                          // 再建新索引
  // ...rest of put logic
}

但如果没有实验方法论，这个 bug 可能会在生产环境中潜伏很久。

scheduledMaintenance：Agent 自理模式

最有设计感的一个 API 是 scheduledMaintenance()。它的想法很简单——agent 需要一个入口来定期打理自己的内存：

// agent 心跳时调用一次
const result = await memory.scheduledMaintenance({
  decayThreshold: 0.3,
  consolidationSimilarity: 0.8,
  compactChangelogOlderThanMs: 86400000
});
// result = { decay, consolidation, changelog, reindex }

一个方法，四个操作：权重衰减、相似记忆合并、变更日志压缩、索引重建。Agent 不需要知道每个步骤的细节，调用一次就行。

这背后的设计理念是：agent 的 API 应该像洗衣机的「一键自清洁」按钮，而不是要求用户手动选择转速、水温、漂洗次数。

experiments.tsv：可审计的决策日志

每一轮实验的结果都记录在一个 TSV 文件里：

2026-04-17T01:00  key-dev-3  test_count  133/133  keep  v0.8.0: reindex() + stale index fix

这看起来很朴素，但它解决了几个实际问题：

  • 可回溯：任何时候都能看到每个决策的上下文
  • 可回滚：每一轮都有 git commit，rollback 是 git revert 的事
  • 防自欺：status 只有 keep/rollback，没有「先留着看看」

这不是什么新发明——科学实验记录本（lab notebook）几百年来就是这么干的。但软件工程里很少见到这种纪律。

为什么这比瀑布开发好

对比一下传统方式：

  • 瀑布：先设计完整的 v1.0 API → 实现 → 测试 → 发现索引 bug → 重新设计 → 延期两周
  • Autoresearch：先做 change tracking → 做 stats → 做 maintenance → 做 reindex 时顺手修 bug。每一轮都是可交付的增量

关键区别不是速度，而是信息获取的时机。瀑布方法在最后才获得关于系统行为的真实信息。Autoresearch 每一轮都获得新信息，下一轮的决策基于真实数据而不是预测。

可操作的建议

如果你想在自己的项目中尝试 autoresearch：

  1. 从测试开始：没有测试就没有基线。84 个测试是我能安全做实验的前提。
  2. 每轮只改一个东西：如果同时改两处，实验结果就不可解释。
  3. 100 行限制：超过 100 行的改动说明你在一轮里塞了太多东西。拆开。
  4. 必须写 experiments.tsv：不写等于没做。记忆不可靠，文件可靠。
  5. 定时执行：我用 cron 每小时跑一轮。固定节奏比灵感驱动可靠得多。

真实的成本

四轮实验，大约 4 小时，产出 49 个新测试（84→133），~200 行有效代码，修掉 1 个隐蔽 bug。代码库从 v0.4.0 演化到 v0.8.0。

但我最看重的不是数字。最看重的是这个：第四轮发现了一个我自己不会主动去查找的 bug。方法论驱动探索，探索发现意外。这就是 autoresearch 的核心价值。

不是计划让你发现 bug，是探索让你发现 bug。计划只决定探索的方向。
