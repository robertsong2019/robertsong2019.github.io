Autoresearch：让 AI Agent 自己做实验的方法论

2026-04-22 · AI-Agent Self-Evolution Testing TypeScript

过去十天，我的 AI Agent 在我睡觉的时候自己做了 15 轮实验。测试从 66 个涨到 329 个，代码从零增长到完整功能——全程没有任何人工干预。这不是科幻，而是一套我称之为 "Autoresearch" 的方法论的实际产出。

这篇文章分享这套方法的核心循环、关键设计决策，以及从 experiments.tsv 数据中提炼的实操建议。

一、核心循环：实验不是一次性的

传统开发是「写代码 → 测试 → 部署」。Autoresearch 加了一个维度：记录、评估、留存或回滚。每一轮实验都是独立的原子单元：

# experiments.tsv 格式
date       | project            | baseline | after | change    | notes
2026-04-17 | github-creative    | 0/0      | 36/36 | +36 tests | Fixed TS compilation bugs
2026-04-19 | catalyst-agent-mesh| 98/98    | 148/148| +50 tests| TaskScheduler + HealthMonitor
2026-04-22 | agent-memory       | 324/324  | 329/329| +5 tests | autoMaintain() API

关键洞察：baseline 和 after 必须同时记录。没有 baseline，你不知道是进步还是退步；没有 after，你无法判断实验是否成功。这个看似简单的 TSV 文件，是整个方法论的地基。

二、增量构建：每个实验都是上一个的垫脚石

看 catalyst-agent-mesh 的增长曲线：

04-16: 68 → 83 tests (+15) PipelineStage, CRUD
04-18: 83 → 98 tests (+15) MeshConfig, workflows
04-19: 98 → 148 tests (+50) TaskScheduler, HealthMonitor
04-19: 148 → 171 tests (+23) HealthMonitor 边界用例
04-22: 171 → 182 tests (+11) health_monitor + pipeline_executor

这不是随机的——每轮实验都直接复用上一轮的产出。HealthMonitor 测试为 autoMaintain 提供了基础；MeshConfig 的 workflow 测试为 PipelineExecutor 铺了路。就像搭积木，但每块积木放下去之前都经过测试验证。

原则：永远不要在不确定的地基上建新楼层。每一轮实验结束时，所有已有测试必须通过。

三、代码实战：autoMaintain 的设计

以最新一轮实验为例——为 Agent 记忆服务添加自动维护 API。核心问题：Agent 的记忆会过期、碎片化、膨胀，但没人记得手动清理。

// 自动维护：基于健康分数触发
async autoMaintain(opts = {}) {
  const {
    threshold = 80,      // 健康分低于此值触发维护
    tasks = ['purge', 'compactChangelog', 'decay', 'reindex'],
    dryRun = false        // 预览模式，只报告不执行
  } = opts;

  // 1. 获取当前健康分数
  const { score, dimensions } = await this.healthScore();

  // 2. 分数健康则跳过
  if (score >= threshold) {
    return { triggered: false, score, actions: {} };
  }

  // 3. 按维度决定执行哪些任务
  const actions = {};
  for (const task of tasks) {
    if (task === 'purge' && dimensions.expiry < 90) {
      actions.purge = dryRun
        ? { recommendation: 'purge expired' }
        : await this.purgeExpired();
    }
    if (task === 'compactChangelog' && dimensions.changelog < 90) {
      actions.compactChangelog = dryRun
        ? { recommendation: 'compact changelog' }
        : await this.compactChangelog();
    }
    // ... 其他任务同理
  }

  return { triggered: true, score, actions };
}

这个 API 的设计有三个值得说的地方：

第一，健康分数驱动的决策。不是"每 N 小时跑一次"，而是"记忆不健康了才跑"。这避免了无意义的定时任务，也让维护频率自适应——数据多时维护更频繁，数据少时几乎不触发。

第二，维度级的精细控制。健康分数被拆成多个维度（expiry、changelog、access），每个维度独立评估。过期问题严重就跑 purge，变更日志膨胀就跑 compact，不需要一刀切。

第三，dry-run 作为一等公民。生产环境最怕的不是没有功能，而是功能跑出了意外后果。dry-run 让你在执行前预览，这不是锦上添花——是必需品。

四、经验数据：10 天 15 轮实验的统计

从 experiments.tsv 的真实数据中：

  • 测试增长：跨 5 个项目，从 66 个测试增长到 329+ 个，零回归
  • 平均每轮：+15-20 个测试，约 50-100 行代码
  • Bug 发现：2 个 NameError、1 个 Set import 缺失、1 个 missing self——都是小问题，但不写测试永远不会发现
  • 回滚次数：0。因为每轮变更小、边界清晰

最反直觉的发现：变更越小的实验，越不容易出问题，也越容易积累出大成果。04-19 那天 catalyst-agent-mesh 一天做了 3 轮实验（+73 tests），不是一口气写的——是三轮独立的、每轮都通过全部测试的增量叠加。

五、可操作建议

如果你想在项目中应用 Autoresearch 方法论：

  1. 建一个 experiments.tsv。不需要数据库，一个 TSV 文件就够了。每轮实验一行，记录日期、项目、baseline/after 测试数、变更描述。
  2. 设定硬性约束：每轮不超过 150 行代码，所有已有测试必须通过。违反约束就回滚。
  3. 让 Agent 在空闲时自主运行。用 cron 触发，选一个有明确 baseline 的项目，让它自己选下一步做什么。人类只需要看 experiments.tsv 的增量行。
  4. 重视 dry-run 和预览。任何写操作都应该有预览模式。代码里的 if (dryRun) 分支不是负担，是安全网。
  5. 复用而非重写。每轮实验应该站在上一轮的肩膀上。autoMaintain 用了 healthScore 的输出；HealthMonitor 用了 MeshConfig 的基础设施。

六、反思

Autoresearch 不是万能的。它最适合的场景是：有明确测试标准、可以增量构建的项目。对于探索性研究或架构级重构，这个循环的粒度太细了。

但它解决了一个真实问题：个人开发者如何在没有团队的情况下维持高质量的持续产出。答案不是更努力地工作，而是建立一个系统——让实验自动发生、自动记录、自动验证。

最终，Autoresearch 的核心信念是：测试数字不会说谎。329/329 比 324/324 好，这不需要任何人的判断。当你把决策交给数据，Agent 就能在你睡觉的时候替你做实验。
