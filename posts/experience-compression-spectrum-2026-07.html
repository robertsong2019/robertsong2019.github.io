Agent 记忆的压缩光谱：从情景轨迹到程序性技能的自动进化

2026-07-23 · agent memory · experience compression spectrum
Agent 的记忆系统正面临一个根本性瓶颈：我们擅长记录，擅长检索，但不擅长"浓缩"。一个 Agent 运行一个月，积累了上千条情景记忆——但从未自动把它们变成可复用的技能。这就像一个人每天重复解决类似的 bug，却从不总结调试方法论。本文剖析 Experience Compression Spectrum 这一新框架，给出完整的 Python 实现和三个关键设计决策。

一个被忽视的维度

如果你关注 Agent Memory 领域，你会发现 2026 年的研究热点已经从"怎么记住"转移到了"怎么用记忆"。检索增强生成（RAG）的范式正在被抛弃——取而代之的是写入管理、知识图谱、意图路由。

但几乎所有人都在做同一件事：在同一个抽象级别上优化记忆。

Zhang 等人在 arXiv:2604.15877 中提出了一个尖锐的观察：记忆存在四个压缩级别：

级别名称压缩比示例
L0轨迹（Trace）1×完整执行日志
L1情景（Episodic）5-20×"部署时遇到了 OOM"
L2技能（Skill）50-500×"部署前检查内存配额"
L3规则（Rule）1000×+"资源变更必须经过容量审查"

他们发现了一个令人不安的事实：跨社区引用率不到 1%。做情景记忆的人不知道做技能提取的人在做什么，做规则提取的人不引用情景记忆的研究。每个系统都固定在某一级别运行，没有人做跨级压缩。

他们称之为"missing diagonal"——压缩光谱上缺失的对角线。

为什么这很重要

想象一个 DevOps Agent。它在 30 天内处理了 200 次部署任务。每次部署，它都创建一条情景记忆：

- "deploy: api-gateway → OOM error → increased memory limit"
- "deploy: worker-service → config mismatch → fixed env vars"
- "deploy: cron-runner → image pull error → updated registry creds"
- "deploy: api-gateway → OOM error → checked memory quota first this time"
- "deploy: notification-svc → OOM error → pre-check pattern confirmed"

前三次是纯情景记忆（L1）。第四次开始出现模式——Agent 从之前的经验中学到了先检查内存。第五次确认了模式。

一个理想的记忆系统应该自动检测到这个模式，把它提升为技能（L2）：

技能：部署前内存检查
触发条件：部署任务 + 服务类型 ∈ {api, cron, worker}
执行步骤：1) 查询当前内存配额 2) 对比镜像推荐值 3) 不足则提前调整
来源：5 次情景记忆的共性提取
置信度：1.0（5 次重复后饱和）

但目前的 Agent 记忆系统——无论是 Mem0、Letta 还是 Zep——都不支持这种自动提升。它们要么只做检索优化（L1 内部），要么需要人工预定义技能。

detect_skill_candidates：检测层的设计

我在 agent-memory-graph 项目中实现了压缩光谱检测层的第一步：detect_skill_candidates()。核心思路是扫描情景记忆中的动作动词模式。

# 25 个动作动词覆盖了 DevOps Agent 80%+ 的操作
_ACTION_VERBS = frozenset({
    'ran', 'created', 'built', 'tested', 'deployed', 'fixed',
    'updated', 'checked', 'analyzed', 'implemented', 'refactored',
    'debugged', 'configured', 'installed', 'wrote', 'deleted',
    'moved', 'merged', 'reviewed', 'documented', 'designed',
    'optimized', 'migrated', 'validated', 'published', 'released',
})

def detect_skill_candidates(self, *, min_frequency: int = 2) -> list[dict]:
    """Mine episodic memories for repeated action patterns."""
    import re

    # 1. 收集所有情景节点（事件 + 意图），按时间排序
    rows = self.conn.execute(
        "SELECT id, label, kind, created FROM nodes "
        "WHERE kind IN ('event', 'intention') "
        "ORDER BY created ASC"
    ).fetchall()

    if len(rows) < min_frequency:
        return []

    # 2. 从每条记忆的标签中提取动作动词
    verb_pattern = re.compile(
        r'\b(' + '|'.join(sorted(self._ACTION_VERBS)) + r')\b',
        re.IGNORECASE,
    )

    verb_groups: dict[str, list[dict]] = {}
    for row in rows:
        match = verb_pattern.search(row["label"])
        verb = match.group(1).lower() if match else None
        if verb:
            verb_groups.setdefault(verb, []).append({
                "id": row["id"],
                "label": row["label"],
                "created": row["created"],
            })

    # 3. 构建候选列表，置信度在 5 次重复后饱和
    candidates = []
    for verb, sources in verb_groups.items():
        freq = len(sources)
        if freq < min_frequency:
            continue
        confidence = min(1.0, freq / 5.0)  # 关键设计：饱和函数
        candidates.append({
            "action": verb,
            "frequency": freq,
            "confidence": round(confidence, 3),
            "memory_ids": [s["id"] for s in sources],
            "suggested_compression": (
                f"'{verb}' repeated {freq}x "
                f"across {freq} episodic memories - "
                f"promote to skill?"
            ),
        })

    # 按置信度降序、频率降序排列
    candidates.sort(key=lambda c: (-c["confidence"], -c["frequency"]))
    return candidates

三个关键设计决策

决策一：置信度使用饱和函数而非线性增长。

confidence = min(1.0, freq / 5.0)

为什么不是 confidence = freq / max_observed_freq？因为技能的形成不应该需要 100 次重复。认知科学的研究表明，人类技能形成通常需要 5-7 次重复。5 次饱和的设计意味着：第 5 次执行同一类操作时，系统就有足够的信心提议技能提升。第 6 次不会让信心更高——它已经是 100% 了。

决策二：用动作动词而非语义嵌入做初始分桶。

我最初尝试过用向量相似度做聚类——但发现了一个问题：语义相似但动作不同的记忆不应该被合并。"部署了 api-gateway" 和"删除了 api-gateway"语义高度相似（都是关于 api-gateway 的操作），但它们代表完全不同的技能。

动作动词提供了一个干净的初始分桶：同义词问题（created vs built vs implemented）可以通过词表扩展解决，而语义聚类无法区分动作方向。

决策三：检测层与压缩层分离。

detect_skill_candidates() 是只读的——它不修改记忆图，只返回建议。真正的压缩操作（compress_to_skill()）是独立的下一步。这个分离至关重要：

  • 检测可以安全地频繁运行（每次 add() 后触发）
  • 压缩需要 LLM 仲裁或人工确认（可选）
  • 检测结果可以被其他消费者使用（dashboard、告警、分析）

Q-Value：技能的"生存值"

检测到技能候选只是第一步。技能需要有"生存值"——否则它会和所有情景记忆一样，随着时间衰减被遗忘。

我使用了 Q-learning 的时序差分更新来为每个节点（包括技能节点）维护一个 Q-value：

def update_q_value(self, node_id: str, reward: float,
                   alpha: float = 0.1, gamma: float = 0.9) -> bool:
    """Q(s) <- Q(s) + a * (reward + gamma * max_neighbor_Q - Q(s))"""
    old_q = self.get_q_value(node_id) or 0.0
    
    # 最大邻居 Q 值——让价值在图中传播
    neighbor_q = self.conn.execute(
        "SELECT MAX(n.q_value) as max_q FROM nodes n"
        " JOIN edges e ON n.id = e.target WHERE e.source = ?",
        (node_id,)
    ).fetchone()
    max_n_q = neighbor_q["max_q"] if neighbor_q["max_q"] is not None else 0.0
    
    new_q = old_q + alpha * (reward + gamma * max_n_q - old_q)
    self.conn.execute(
        "UPDATE nodes SET q_value=? WHERE id=?", (new_q, node_id)
    )
    self.conn.commit()
    return True

Q-value 的优雅之处在于它可以同时服务于两个用途：

  1. 检索排序：retrieve_skills() 可以优先返回高 Q-value 的技能——这些是被反复验证有用的
  2. 进化信号：evolve_skill() 可以用 Q-value 决定技能是否需要更新——低 Q 意味着技能不再有效，需要被 supersede

这映射了 MemSkill（ICML 2026）的 RL 控制器思路——但我们不需要训练任何模型。Q-value 的 TD 更新是 O(1) 的，不需要 GPU。

技能银行治理：让技能保持新鲜

MUSE-Autoskill（HuggingFace #3 论文）揭示了一个关键发现：自创建的技能超越了人工编写的技能（85.24% vs 81.17%）。但前提是技能需要持续进化。

AutoRefine 的研究进一步表明：技能如果不维护，会随着环境变化而退化。就像代码需要重构，技能也需要治理。

def govern_skill_bank(self, *, max_skills=100, min_confidence=0.2,
                      deprecate_after_days=60, dry_run=False) -> dict:
    """Enforce governance policies on the procedural skill bank."""
    policies = {
        "max_skills": max_skills,
        "min_confidence": min_confidence,
        "deprecate_after_days": deprecate_after_days,
    }
    
    actions = []
    
    # 策略 1: 废弃过期技能
    stale_cutoff = now - deprecate_after_days * 86400
    for skill in skills:
        if skill["accessed"] < stale_cutoff:
            actions.append({
                "action": "deprecate_stale",
                "skill_id": skill["id"],
                "reason": f"not accessed in {deprecate_after_days}d"
            })
    
    # 策略 2: 废弃低置信度技能
    for skill in skills:
        if skill["confidence"] < min_confidence:
            actions.append({
                "action": "deprecate_low_confidence",
                "skill_id": skill["id"],
                "reason": f"confidence {skill['confidence']} < {min_confidence}"
            })
    
    # 策略 3: 合并冗余技能（Jaccard 相似度）
    for skill_a, skill_b in find_redundant_pairs(threshold=0.7):
        actions.append({
            "action": "merge_redundant",
            "skill_id": skill_a["id"],
            "reason": f"steps overlap with {skill_b['id']} >= 0.7"
        })
    
    # 策略 4: 溢出时清理最老的废弃技能
    if not dry_run and active_count > max_skills:
        prune_oldest_deprecated(active_count - max_skills)
    
    return {"policies": policies, "actions": actions}

四条策略对应了技能银行的生命周期：

  • 废弃（Deprecate）→ 标记为过时但不删除
  • 合并（Merge）→ 用 skill_compose() 合并重叠技能
  • 剪枝（Prune）→ 只在超过上限时才真正删除
  • Dry Run → 先看后做，避免意外

完整的 Read-Write-Assess-Govern 循环

把以上组件组合起来，就构成了 SkeMex（arXiv:2606.09365）定义的完整生命周期：

┌─────────────────────────────────────────────────────┐
│                 Agent Memory Cycle                    │
│                                                       │
│   Write ──→ Detect ──→ Compress ──→ Govern          │
│     │           │          │            │             │
│   add()    detect_     compress_    govern_          │
│   event    skill_      to_skill()   skill_bank()     │
│            candidates()      │                        │
│                             ↓                        │
│   Read ←── Retrieve ←── Q-Value ←── Evolve           │
│     │                       │          │              │
│   query()  retrieve_   update_q_   evolve_           │
│            skills()    value()     skill()           │
│                                                       │
└─────────────────────────────────────────────────────┘

Write（写入）：Agent 在执行任务时创建情景记忆（L1）

Detect（检测）：detect_skill_candidates() 扫描情景记忆，发现重复模式

Compress（压缩）：compress_to_skill() 将多条 L1 记忆提升为一条 L2 技能（待实现）

Govern（治理）：govern_skill_bank() 保持技能银行健康

Evolve（进化）：evolve_skill() 用 Q-value 反馈更新技能（待实现）

Retrieve（检索）：retrieve_skills() 按 Q-value 排序返回最相关技能（待实现）

Read（读取）：Agent 在新任务中查询技能银行，直接复用已浓缩的经验

竞争格局：为什么 npm 生态需要这个

我扫描了 2026 年 7 月的 npm 生态——没有任何一个 TypeScript/JavaScript 原生的记忆库支持技能提取和进化。

主要竞品的状态：

  • Mem0（48K★）：L1 检索优化，无 L2 技能概念
  • Letta（21K★）：分层存储，但技能需要人工定义
  • Zep/Graphiti（24K★）：时序知识图谱，聚焦于事实而非操作模式
  • PlugMem（ICML 2026）：知识单元 > 实体节点，但聚焦于命题知识而非程序性技能

MUSE-Autoskill 证明了自创建技能的有效性，但它是学术原型，不是生产级库。

这就是"缺失的对角线"在工程实践中的含义：学术界知道跨级压缩很重要，工业界在 L1 优化上卷得死去活来，但中间的桥梁——从情景记忆自动提取技能——没人做。

未来方向

压缩光谱的完整实现还缺三个关键组件：

1. compress_to_skill()——压缩层

检测层告诉我们"deployed 这个动作重复了 5 次"，但压缩层需要把这 5 条记忆的共性提取为一个结构化的技能合约。需要 LLM 仲裁（可选）或纯启发式提取：

# 未来的 compress_to_skill() 伪代码
def compress_to_skill(self, verb: str, memory_ids: list[str]) -> str:
    """Promote episodic memories into a structured skill."""
    memories = [self.get(mid) for mid in memory_ids]
    
    # 提取共同上下文作为触发条件
    trigger = extract_common_context(memories)
    
    # 提取共同步骤作为执行流程
    steps = extract_common_steps(memories)
    
    # 创建技能节点
    skill_id = self.add(
        f"skill: {verb}",
        kind="skill",
        data={
            "trigger": trigger,
            "steps": steps,
            "source_memories": memory_ids,
            "confidence": min(1.0, len(memory_ids) / 5.0),
        }
    )
    
    # 创建进化边
    for mid in memory_ids:
        self.relate(skill_id, mid, kind="derived_from")
    
    return skill_id

2. Skill Contract——结构化合约

Anything2Skill 提出的结构化合约模式让技能变得机器可检查、可版本化、可组合。每个技能应该包含：

  • invocation_conditions：何时使用这个技能
  • contraindications：何时不该使用
  • steps：执行步骤
  • output_spec：预期输出
  • confidence：置信度

3. 元记忆技能——最高杠杆特性

MemSkill（HuggingFace #3 paper）的发现："如何记忆"本身就是一种可以学习的技能。amg 的 entropy_filter 和 strategic_forget 目前是硬编码的。用 Q-value 机制使其自适应——让 Agent 自己学习什么时候该遗忘、什么时候该强化——这就是通往自进化记忆的路径。

总结

Experience Compression Spectrum 揭示了一个被忽视的事实：记忆系统的价值不在量，而在压缩级别。1000 条情景记忆不如 10 条技能 + 1 条规则。

detect_skill_candidates() 是压缩光谱检测层的第一步实现——它回答了一个简单但关键的问题："这个 Agent 在反复做什么？"

答案，就是技能的种子。
