从轨迹到规则：Agent 经验压缩的四个层级与缺失的对角线

2026-07-18 · 罗嵩 · Agent Memory 技能压缩 Experience Compression Procedural Memory
所有 AI Agent 记忆系统都卡在单一压缩级别——要么只存原始日志，要么只存抽象规则。2026 年 6 月，Zhang 等人形式化了经验压缩谱（L0-L3），发现跨级自适应压缩几乎是一片空白——"缺失的对角线"。本文用 Python 实现 Skill Contract 结构化合约和 compress_to_skill() 跨级压缩管线，展示如何让 Agent 的记忆从"记住"进化到"学会"。

引言：Agent 记住了，但没学会

一个典型的 AI Agent 今天是这样工作的：它记住你的偏好（"用户喜欢简洁的回复"），记住事实（"项目用 TypeScript"），记住上下文（"刚才在讨论数据库设计"）。然后在新任务中检索这些记忆。

但如果你观察人类的学习方式，会发现一个根本差异：人类不只是一个事件一个事件地记住——人类会从经验中提取模式，把模式压缩成技能，再把技能组合成规则。 一个程序员不会每次写代码都从头推理，而是调用"如何写单元测试"这个技能——这个技能是从数百次写测试的经验中压缩出来的。

2026 年 6 月，Zhang 等人在 arXiv:2604.15877 中形式化了这个过程，提出了 Experience Compression Spectrum（经验压缩谱）：一个从 L0 到 L3 的四级框架。他们分析了几十个 Agent 记忆系统后发现了一个令人惊讶的事实：

几乎所有现有系统都只在单一压缩级别运行，没有系统实现了跨级自适应压缩。跨社区引用率不到 1%——这个方向几乎是空白的。

Zhang 等人称之为"缺失的对角线"（the missing diagonal）。本文展示这个对角线是什么，为什么它重要，以及如何用代码实现它。

四级压缩谱：从原始轨迹到抽象规则

L0：原始轨迹（Raw Trace）

每一次 Agent 执行任务的完整记录。包括输入、推理链、工具调用、输出、耗时、成功/失败。

@dataclass
class L0Trace:
    """原始执行轨迹——不丢失任何细节"""
    trace_id: str
    task: str
    steps: list[dict]  # [{action, input, output, latency_ms}, ...]
    outcome: str       # "success" | "partial" | "failure"
    context_snapshot: dict  # 执行时的完整上下文
    timestamp: float

L0 的问题是信息过载。一次复杂任务可能产生 50+ 步、数万 token 的轨迹。下次遇到类似任务时，把整个 L0 轨迹塞进 prompt 等于什么都没做——Agent 还是要在海量细节中找到相关部分。

L1：片段记忆（Episodic Memory）

从 L0 轨迹中提取的关键事实和事件。这是绝大多数 Agent 记忆系统（Mem0、Letta、Zep、amg）所在的级别。

@dataclass
class L1Episode:
    """片段记忆——从轨迹中蒸馏出的关键信息"""
    episode_id: str
    summary: str          # "用户偏好简洁回复"
    entities: list[str]   # ["user", "communication_style"]
    source_trace: str     # 指向 L0 的引用
    weight: float         # 重要性权重
    created_at: float

L1 的压缩比通常在 5-20 倍：一条 10000 token 的轨迹压缩成 500-2000 token 的片段。但 L1 有一个结构性弱点：它是"记住发生了什么"，不是"学会了怎么做"。

L2：技能（Skill）

从多个 L1 片段中抽象出的可复用程序性知识。技能回答的不是"发生了什么"，而是"怎么做"。

@dataclass
class SkillContract:
    """技能合约——机器可检查、可版本化、可组合"""
    skill_id: str
    name: str

    # 什么时候该用这个技能？
    invocation_conditions: list[str]
    # ["任务涉及数据库迁移", "用户提到 schema 变更"]

    # 什么时候不该用？
    contraindications: list[str]
    # ["数据库为空（无需迁移）", "跨引擎迁移（需专用工具）"]

    # 具体步骤
    steps: list[str]
    # ["1. 导出当前 schema", "2. 生成 diff", "3. 创建迁移脚本", ...]

    # 约束条件
    constraints: list[str]
    # ["必须先备份", "迁移脚本必须可回滚"]

    # 输出规格
    output_spec: str
    # "一个可执行的 SQL 迁移文件 + 回滚脚本"

    # 置信度（从多少次成功经验中提炼？）
    confidence: float                 # 0.0 - 1.0

    # 来源片段
    source_episodes: list[str]        # L1 episode_ids

这就是 Anything2Skill 论文提出的结构化合约格式。与自由文本技能描述相比，结构化合约有三个优势：

  1. 机器可检查——Agent 可以在执行前验证条件是否满足
  2. 可版本化——技能 v2 可以 supersede v1，带有变更日志
  3. 可组合——技能 A 的输出可以作为技能 B 的输入

L2 的压缩比通常在 50-500 倍。100 次数据库迁移的 L1 经验（可能 200,000 token）压缩成一个 400 token 的技能合约。

L3：规则（Rule）

从多个 L2 技能中提取的跨领域原则。规则是最抽象的记忆形态，压缩比可达 1000 倍以上。

@dataclass
class L3Rule:
    """抽象规则——跨技能的通用原则"""
    rule_id: str
    statement: str  # "破坏性操作必须有回滚路径"
    scope: list[str]  # ["database", "deployment", "refactoring"]
    confidence: float
    derived_from: list[str]  # skill_ids

规则的价值在于跨域迁移：从数据库迁移中学到的"破坏性操作需要回滚"，可以应用到代码重构、配置变更甚至消息发送。

"缺失的对角线"

下图展示了压缩谱的现状。每个圆圈代表一个系统，圆圈位置表示它运行的压缩级别：

压缩级别 L0 ──────────────────────────────────────── ● (OpenTelemetry, LangSmith — 只记录) L1 ──────────────────────────────────────── ● ● ● ● ● (Mem0, Letta, Zep, amg, Hindsight — 只记片段) L2 ──────────────────────────────────────── ● (AutoRefine, SkillRL — 只用技能) L3 ──────────────────────────────────────── ● (Voyager — 只提取规则) ────────────────────────────────────── ↑ 这条对角线是空的

问题不在于缺少某个级别的系统——问题在于没有系统跨越所有四个级别。 一个只有 L1 的系统永远在"记住事实"，无法进化为"掌握技能"。一个只有 L2 的系统有技能但无法从新经验中学习（因为它不存 L0/L1）。

Zhang 等人的论文用网络分析证实了这一点：四个级别的研究社区之间的跨引用率不到 1%。每个社区都在自己的级别上优化，没有人搭建跨级的桥梁。

代码：compress_to_skill() 的设计

缺失的对角线需要一个能从 L1 片段自动压缩到 L2 技能的函数。以下是 agent-memory-graph 项目中即将实现的 compress_to_skill() 的设计草案：

def compress_to_skill(
    self,
    episodes: list[str],  # L1 episode_ids
    name: str,
    *,
    min_confidence: float = 0.6,
    max_steps: int = 10,
    merge_threshold: float = 0.75,  # Jaccard 相似度
) -> SkillContract:
    """从多个 L1 片段中压缩出一个 L2 技能合约。
    
    算法：
    1. 按时间排序所有片段
    2. 提取共同实体和共同操作序列
    3. 用最长公共子序列（LCS）找到稳定的步骤模式
    4. 从失败片段中提取 contraindications
    5. 与现有技能比较，Jaccard ≥ threshold 则合并
    """
    # 1. 加载并排序片段
    eps = [self.get(eid) for eid in episodes]
    eps.sort(key=lambda e: e.created_at)
    
    # 2. 提取共同实体
    entity_freq = Counter()
    for ep in eps:
        entity_freq.update(ep.entities)
    common_entities = [
        e for e, count in entity_freq.items() 
        if count >= len(eps) * 0.5  # 出现在 ≥50% 的片段中
    ]
    
    # 3. 提取操作序列并用 LCS 找稳定模式
    action_sequences = self._extract_action_sequences(eps)
    stable_steps = self._longest_common_subsequence(action_sequences)
    
    # 4. 从失败片段中提取禁忌条件
    failed = [ep for ep in eps if ep.outcome == "failure"]
    contraindications = self._extract_failure_patterns(failed)
    
    # 5. 计算置信度
    success_count = sum(1 for ep in eps if ep.outcome == "success")
    confidence = success_count / len(eps)
    if confidence < min_confidence:
        raise ValueError(
            f"置信度 {confidence:.2f} 低于阈值 {min_confidence}。"
            f"需要更多成功经验才能形成技能。"
        )
    
    # 6. 检查是否与现有技能重复
    for existing in self._skills.values():
        similarity = self._jaccard(
            set(stable_steps), set(existing.steps)
        )
        if similarity >= merge_threshold:
            # 合并而非创建新技能
            return self._merge_skill(existing, stable_steps, contraindications)
    
    # 7. 创建技能合约
    skill = SkillContract(
        skill_id=f"skill_{int(time.time())}",
        name=name,
        invocation_conditions=self._infer_conditions(common_entities),
        contraindications=contraindications,
        steps=stable_steps[:max_steps],
        constraints=self._infer_constraints(eps),
        output_spec=self._infer_output(eps),
        confidence=confidence,
        source_episodes=episodes,
    )
    self._skills[skill.skill_id] = skill
    return skill

关键设计决策

1. 为什么用 LCS 而不是 LLM 来提取步骤？

最长公共子序列（LCS）是确定性算法——同一段输入永远产生同一段输出。LLM 提取可能每次不同，导致技能不稳定。正确做法是：用确定性算法提取骨架步骤，用 LLM 补充自然语言描述。

2. 为什么用 Jaccard 而非余弦相似度？

Jaccard 是集合操作，不需要 embedding 计算。当比较的是步骤列表（字符串集合）时，Jaccard 比 cosine 更精确——它直接衡量"有多少步骤重合"，而不是"语义有多接近"。

3. 为什么 confidence < min_confidence 时报错而不是降级？

一个 30% 成功率的经验序列不值得压缩成技能。报错迫使系统收集更多数据，这比创建一个低质量技能更好。这是"verification beats advice"原则——不要用软警告替代硬约束。

自创技能 > 人类编写：MUSE 的发现

2026 年 5 月，MUSE-Autoskill 论文（HuggingFace #3 trending paper）报告了一个让很多人意外的实验结果：

Agent 从自身经验中自动创建的技能，性能（85.24%）超过了人类手动编写的技能（81.17%）。

差异来源是什么？研究者做了消融实验：

因素贡献
技能内容与实际任务分布更匹配+2.8pp
技能粒度更细（人类倾向于写"大"技能）+1.5pp
技能数量更多（自动创建覆盖更广）+0.9pp
持续更新（人类写完就不管了）+0.7pp

这告诉我们：技能质量不取决于编写者的水平，取决于技能与 Agent 实际任务分布的匹配度。 人类专家写的技能是基于"我认为 Agent 会遇到什么任务"，而 Agent 自创的技能是基于"我实际遇到了什么任务"。数据胜过直觉。

闭环：读-写-评-治

有了 compress_to_skill()，Agent 的记忆系统就从单向的"记记记"升级为一个完整的技能生命周期：

读 (Read):   retrieve_skills()  →  按任务匹配技能
写 (Write):  compress_to_skill() →  从经验中创建技能
评 (Assess): skill_bank_health() →  评估技能库质量
治 (Govern): evolve_skill()     →  版本进化 + 废弃过时技能

这四个步骤对应 SkeMex 论文提出的 Read-Write-Assess-Govern 框架。其中 Govern 步骤是最容易被忽略的：

def govern_skill_bank(
    self,
    *,
    stale_days: int = 90,
    low_confidence_threshold: float = 0.3,
    merge_redundant_threshold: float = 0.75,
    max_skills: int = 200,
    dry_run: bool = True,
) -> dict:
    """治理技能库——清理、合并、修剪。"""
    report = {"deprecated": [], "merged": [], "pruned": []}
    
    # 1. 废弃过时技能
    now = time.time()
    for sid, skill in list(self._skills.items()):
        age_days = (now - skill.created_at) / 86400
        if age_days > stale_days and skill.confidence < 0.5:
            report["deprecated"].append({
                "skill": skill.name,
                "reason": f"stale ({age_days:.0f} days) + low confidence",
            })
            if not dry_run:
                skill.status = "deprecated"
    
    # 2. 合并冗余技能
    skills_list = list(self._skills.values())
    for i, a in enumerate(skills_list):
        for b in skills_list[i+1:]:
            if a.status == "deprecated" or b.status == "deprecated":
                continue
            sim = self._jaccard(set(a.steps), set(b.steps))
            if sim >= merge_redundant_threshold:
                merged = self._merge_skill(a, b.steps, b.contraindications)
                report["merged"].append({
                    "a": a.name, "b": b.name, 
                    "similarity": round(sim, 2),
                })
                if not dry_run:
                    self._skills[merged.skill_id] = merged
    
    # 3. 修剪溢出
    active = [s for s in self._skills.values() if s.status == "active"]
    if len(active) > max_skills:
        active.sort(
            key=lambda s: s.confidence * s.usage_count,
            reverse=True,
        )
        for skill in active[max_skills:]:
            report["pruned"].append(skill.name)
            if not dry_run:
                skill.status = "pruned"
    
    return report

dry_run 模式是必须的——治理操作是破坏性的，应该先审查再执行。这就像 git diff 之于 git apply——永远先看清楚再动手。

反直觉：为什么不应该用 LLM 做所有事

在构建压缩管线的早期，我尝试过让 LLM 做全流程：输入 L1 片段，让 LLM 输出 L2 技能合约。结果发现了三个问题：

1. 幻觉步骤。 LLM 会编造片段中不存在的步骤。两个不同任务的 L1 片段经过 LLM 压缩后，技能中出现了"检查网络连接"——但原始片段从未涉及网络操作。

2. 不稳定性。 同一组片段，两次压缩产生不同的步骤列表。技能应该像函数签名一样稳定——调用者需要知道技能的接口不会变。

3. 不可解释的置信度。 LLM 给出 confidence=0.85，但无法解释为什么是 0.85 而不是 0.42。而基于成功/失败计数的确定性置信度是 100% 可解释的。

解决方案是职责分离：确定性算法负责提取结构（LCS、Jaccard、计数），LLM 负责生成自然语言描述（步骤的可读版本、条件的语义化）。这就像编译器架构——解析器是确定性的，优化器可以做启发式搜索，但最终输出必须可验证。

可操作建议

  1. 审计你的记忆系统处于哪个级别。 如果只有 L0，你只是在做日志。如果只有 L1，你只是在做数据库。加上 L2 技能层，记忆系统才开始"学习"。
  2. 不要跳级。 从 L0 直接到 L3 看似高效，但缺少了 L1 和 L2 的中间表示，规则变成了无根之木——无法追溯到具体经验。source_episodes 和 source_traces 字段不是装饰，是可审计性的基础。
  3. 让置信度驱动行动。 confidence < 0.5 的技能不应该自动执行——它需要人类确认或更多数据积累。confidence > 0.9 的技能可以考虑自动化执行。
  4. 技能会退化。 AutoRefine 论文证明，技能如果不持续更新，性能会随时间下降。govern_skill_bank() 的 stale_days 参数不是可选的——它是技能库健康的基本保障。
  5. 跨级引用是审计基础。 每个技能合约的 source_episodes 字段让你可以追溯："这个技能是从哪些经验中学到的？"当技能出错时，你可以回溯到原始片段，找到学习过程的缺陷。

本文基于 agent-memory-graph 项目的前瞻设计。当前实现：L1 片段记忆（3945 个测试验证）+ 评估五重奏 + 知识缝隙检测。L2 技能层（compress_to_skill + retrieve_skills + evolve_skill + govern_skill_bank）是下一个里程碑——约 140 个测试，预计 2 个开发周期完成。代码 MIT 协议开源。
