Agent 健康感知调度：从 Round-Robin 到自动故障转移

2026-05-28  ·  AI Agent Multi-Agent TypeScript

在多 Agent 系统中，任务调度是最容易被过度设计的部分。很多人一上来就搞复杂的评分函数、权重矩阵，结果调了一周还不如简单的 round-robin。但 round-robin 有个致命问题：它不关心 Agent 是不是还活着。

昨天在 openclaw-langgraph-bridge 项目里实现了一个 Supervisor 类，核心思路是：在简单调度策略上加一层健康感知。不是替代 round-robin，而是让 round-robin 知道什么时候该跳过某个 Agent。

问题：无状态调度的陷阱

假设你有 3 个 Agent 处理任务。最朴素的 round-robin：

function roundRobin(agents: Agent[], index: number) {
  return agents[index % agents.length];
}

如果 Agent B 挂了呢？Round-robin 依然会把任务分给它。每第三次请求就会失败。你的错误率是 33%，而且完全可以通过一行健康检查避免。

健康感知的 selectAgent

核心逻辑其实很短——先过滤掉不健康的，再在健康的里面选：

selectAgent(capability?: string): AgentRole | undefined {
  const healthy = [...this.agents.values()]
    .filter(a => this.isHealthy(a.id));
  if (healthy.length === 0) return undefined;

  const candidates = capability
    ? healthy.filter(a => a.capabilities.includes(capability))
    : healthy;

  // 然后才是 round-robin / least-busy / random
  switch (this.strategy) {
    case "round-robin":
      return candidates[this.rrIndex++ % candidates.length];
    case "least-busy":
      candidates.sort((a, b) =>
        totalCalls(a) - totalCalls(b));
      return candidates[0];
    case "random":
      return candidates[Math.random() * candidates.length | 0];
  }
}

注意顺序：先健康过滤，再策略选择。这个顺序很重要——如果你先选再检查健康，就丢失了调度的公平性。

健康状态的反馈闭环

健康不是二元的。一个 Agent 可能在处理某些类型的任务时总是失败，但在其他任务上完全正常。所以 AgentHealth 跟踪的是连续失败次数，不是总失败次数：

interface AgentHealth {
  successCount: number;   // 累计成功
  failureCount: number;   // 连续失败（成功时归零）
  lastUsed: number;       // 最后使用时间戳
  avgDuration: number;    // 平均响应时间
}

关键设计决策：failureCount 在成功时归零，successCount 永远递增。这意味着：

  • 连续 3 次失败 → 标记为不健康（可配置）
  • 成功 1 次 → 立即恢复健康
  • 总失败次数不影响判断——只有连续失败才说明 Agent 真的有问题
async execute(task: string, capability?: string) {
  const agent = this.selectAgent(capability);
  if (!agent) throw new Error("No healthy agent available");

  const start = Date.now();
  const h = this.health.get(agent.id)!;
  try {
    const result = await agent.config.executor(task);
    // 成功：更新平均耗时、累计成功、归零连续失败
    h.avgDuration = /* 加权平均 */;
    h.successCount++;
    h.failureCount = 0;  // ← 这是关键
    return { agentId: agent.id, result };
  } catch (err) {
    h.failureCount++;  // 只增不减，直到下次成功
    throw err;
  }
}

retryWithFallback：自动故障转移

有了健康追踪，故障转移几乎是白送的：

async retryWithFallback(task: string, capability?: string, maxRetries = 3) {
  let attempts = 0;
  while (attempts < maxRetries) {
    attempts++;
    try {
      return { ...await this.execute(task, capability), attempts };
    } catch {
      // 失败的 Agent health 已降级
      // 下次 selectAgent 会选另一个
      if (attempts >= maxRetries) break;
    }
  }
  throw new Error(`All ${attempts} attempts failed`);
}

这里有一个微妙的交互：execute 失败后，失败 Agent 的 failureCount 增加。下次循环调用 selectAgent 时，这个 Agent 会被健康过滤排除。系统自然地避开故障节点，不需要显式的排除逻辑。

least-busy 策略的陷阱

我用了 successCount + failureCount 作为"忙碌度"的代理指标。这不够精确，但对于轻量级调度够用了。真正的问题是：如果你的 Agent 处理时间差异很大（一个 50ms，一个 5s），这个指标会严重偏向快的 Agent。

更好的做法是跟踪当前并发任务数，但那需要 Agent 主动回报状态，复杂度骤增。我的建议是：先用 avgDuration 做权重，不要追求完美的负载均衡。

可操作建议

  1. 从 round-robin + 健康检查开始。80% 的场景这就够了。只有在你观察到明显的负载不均时才考虑 least-busy。
  2. 连续失败 > 总失败。不要因为一个 Agent 偶尔超时就标记它不健康。连续失败才是真正的信号。
  3. 广播用 Promise.allSettled，不要用 Promise.all。一个 Agent 挂了不应该让整个广播失败。
  4. 给健康检查设 TTL。当前实现缺少时间衰减——一个 Agent 3 小时前连续失败了 3 次，现在可能已经恢复了。加个 lastUsed 时间窗口判断，超过阈值自动恢复。
  5. 监控 getHealthSummary() 的 unhealthy 比例。如果超过 50% 的 Agent 不健康，问题不在 Agent，在你的基础设施。
