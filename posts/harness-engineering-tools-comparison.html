Harness Engineering 三大工具深度对比：gstack、GSD 与 Better Ralph

#AI #Agent #Harness #gstack #GSD #BetterRalph

随着 LLM 能力的提升，AI Agent 编码从"对话式生成"进化到"工程化生产"。Harness Engineering —— 这个正在形成的新兴工程学科，关注的是如何将多个 AI Agent session 编排成可重复、可靠的工作流。

最近，我深入研究了三个代表性工具：Garry Tan 的 gstack、TÂCHES 的 GSD (Get Shit Done)，以及社区驱动的 Better Ralph。它们代表了 Harness Engineering 的不同方向。

核心定位对比

工具 核心定位 作者/背景 驱动方式
gstack 虚拟工程团队（23个角色技能） Garry Tan (YC CEO) 命令式 /skill 调用
GSD Spec-driven 项目全流程工具 TÂCHES Phase 驱动（需求→交付）
Better Ralph PRD 迭代引擎 社区 PRD stories 循环

工作模型对比

gstack：按需调用的工具箱

gstack 将 Claude Code 变成一个虚拟工程团队，提供 23 个 slash commands，每个代表一个专业角色。你不需要走完整流程，而是根据需要调用特定技能。

/land-and-deploy

/design-consultation
/plan-eng-review
/freeze
/guard

状态管理在 Claude Code session 内部，不持久化到文件。适合临时任务（审计、review、QA）和完整功能开发。

GSD：固定的全流程框架

GSD 采用固定的项目工作流：

New Project
    ↓
For Each Phase:
    /gsd-discuss-phase → /gsd-ui-phase → /gsd-plan-phase
    → /gsd-execute-phase → /gsd-verify-work → /gsd-ship
    ↓
Next Phase? — No → /gsd-audit-milestone → /gsd-complete-milestone

每个 phase 生成多个文件（RESEARCH.md、PLAN.md、VALIDATION.md 等），状态分散在 .planning/ 目录。特点是强 quality gates：

  • Schema drift detection — ORM 变了但没加 migration？拦截
  • Security enforcement — 验证锚定到威胁模型
  • Scope reduction detection — 防止 planner 默默砍掉需求
  • Nyquist validation — 测试覆盖映射，每个 requirement 都有自动验证

Better Ralph：轻量级 PRD 循环

Better Ralph 是最简单的：读 prd.json → 挑下一个 story → 实现 → 测试 → 标记 passed。所有状态在单个 prd.json 文件中。

Read PRD
    ↓
Pick next story
    ↓
Implement + Check
    ↓
Commit + Mark passed
    ↓
Loop

适合有明确 PRD 的迭代开发，零学习成本。

详细功能对比

维度 gstack GSD Better Ralph
状态管理 Session 内存，无持久化 .planning/ 目录多文件 prd.json 单文件
质量门控 专项审计（/review、/cso、/qa） 3 层 gates + Nyquist validation 测试通过 + checks
上下文管理 每个 skill 独立上下文 Phase 文件 + Knowledge graph PRD + 代码
适用场景 临时任务 + 完整功能开发 从 0 到 1 项目 PRD 迭代
复杂度 高（23 个命令） 中（固定工作流） 低（单循环）
学习成本 中（按需学习） 低（跟着流程走） 最低

场景适用性

场景 最佳工具 理由
临时安全审计 gstack (/cso) OWASP + STRIDE 审计开箱即用
PR review gstack (/review) 专门找生产 bug 的 reviewer
QA 测试 gstack (/qa) 真实浏览器测试
从 0 做完整项目 GSD 全流程 + quality gates
迭代实现 PRD Better Ralph PRD stories 循环
产品战略评审 gstack (/plan-ceo-review) 4 种 scope 模式
性能基准 gstack (/benchmark) 专门的性能测试
架构评审 gstack (/plan-eng-review) 工程视角的架构锁定

互补性与组合使用

关键洞察：三者不是竞争关系，而是互补的

它们覆盖了 Harness Engineering 的不同层次：

  • gstack = 工具箱层 —— 23 个角色技能，按需调用
  • GSD = 框架层 —— 项目全流程，质量门控
  • Better Ralph = 引擎层 —— PRD 迭代，轻量循环

实际组合示例：

新建项目：
    用 GSD 做规划 + 用 gstack 的 /plan-ceo-review 做战略评审

迭代开发：
    用 Better Ralph 实现 stories + 用 gstack 的 /review 做 PR review

发布前：
    用 gstack 的 /qa + /cso 做测试和安全审计

总结

维度 gstack GSD Better Ralph
本质 工具箱（23 个角色技能） 项目框架（全流程） 迭代引擎（PRD 循环）
优势 覆盖全生命周期，开箱即用 强 quality gates，适合完整项目 简单轻量，快速迭代
劣势 需学习 23 个命令 对临时任务过重 只适合有 PRD 的场景
适用 临时任务 + 完整功能开发 从 0 到 1 项目 PRD 迭代

实践建议

  1. 从 gstack 开始 — 先用最常用的 5-6 个命令（/office-hours、/review、/qa、/cso、/ship），按需学习其他
  2. 新项目用 GSD — 从 0 到 1 做完整项目时，GSD 的 quality gates 能显著提升可靠性
  3. 迭代用 Better Ralph — 有明确 PRD 时，用 Better Ralph 快速迭代 stories
  4. 组合使用 — GSD 做项目框架 + gstack 做专项审计 + Better Ralph 做 PRD 迭代 = 完整工具链

结语

Harness Engineering 仍在快速演化中，gstack、GSD、Better Ralph 代表了三个不同方向：工具箱式、框架式、引擎式。它们不是竞争关系，而是互补的。

随着 AI Agent 能力的提升，Harness Engineering 将成为 2026 年软件工程的核心竞争力。选择合适的工具，建立自己的工作流，将是每个工程师的必修课。

"The model is commodity. The harness is moat." — Harness Engineering 核心洞察
