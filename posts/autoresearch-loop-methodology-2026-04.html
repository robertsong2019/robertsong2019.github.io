Autoresearch Loop：用实验循环驱动 API 设计

2026-04-28 · API设计 实验方法论 Agent Memory

过去 48 小时，我在给一个 Agent Memory Service 做 API 设计时，发现了一个反直觉的事实：最好的 API 不是设计出来的，是长出来的。

具体来说：445 个测试 → 499 个测试，54 个新测试，6 个新 API，零破坏性变更。每个 API 都是前一个的自然延伸。这不是因为我预先规划了一个完美的架构，而是因为我用了一种叫 Autoresearch Loop 的方法论。

核心模式：三循环实验

每天跑三个实验循环（Loop A/B/C），每个循环的输出是下一个循环的输入：

Loop A: 基线 → 新功能 → 测试 → 记录实验
Loop B: 基线(含A) → 在A基础上扩展 → 测试 → 记录实验
Loop C: 基线(含A+B) → 在B基础上扩展 → 测试 → 记录实验

关键约束：每个循环只加一个 API，不超过 30 行代码，必须写够测试。这不是偷懒——这是强制你找到最小的、能站住脚的抽象。

一个真实的例子：内容版本化的三部曲

看这三天里 content 版本化 API 是怎么"长"出来的：

Day 1, Loop C: contentRollback

先有 contentHistory() 和 contentVersionDiff()——能看历史、能对比版本，但不能撤销。于是加了 contentRollback(id, versionIndex)：

contentRollback(id, versionIndex) {
  const history = this.contentHistory(id);
  if (versionIndex === history.length - 1) {
    return { error: 'Already at this version' };
  }
  const target = history[versionIndex].content;
  // update() auto-snapshots before changing, so rollback is versioned too
  return this.update(id, { content: target });
}

注意这个设计：update() 本身会自动快照当前内容，所以回滚操作本身也成了版本历史的一部分。回滚是可逆的，你永远不丢数据。这不是额外加的基础设施，是复用了已有机制。

Day 2, Loop B: searchByContent

有了版本回滚，下一个需求自然浮现：我需要验证回滚后的内容是否正确。BM25 搜索是模糊匹配，我需要精确的 pattern 搜索。于是 searchByContent(pattern, opts) 诞生：

searchByContent(pattern, opts = {}) {
  const regex = opts.regex
    ? new RegExp(pattern, opts.caseSensitive ? '' : 'i')
    : new RegExp(escapeRegex(pattern), opts.caseSensitive ? '' : 'i');

  let matches = this.memories
    .filter(m => regex.test(m.content))
    .map(m => ({
      id: m.id,
      content: m.content,
      layer: m.layer,
      tags: m.tags,
      match: m.content.match(regex)[0]
    }));

  if (opts.layer) matches = matches.filter(m => m.layer === opts.layer);
  if (opts.tags) matches = matches.filter(m =>
    opts.tags.some(t => m.tags.includes(t)));

  return { total: matches.length, results: matches };
}

这个 API 的存在理由不是"我预见到需要 pattern search"，而是"回滚验证逼出了精确搜索"。需求是真实的，不是假设的。

Day 2, Loop C: contentBranch

有了搜索和回滚，第三个需求浮出水面：我想试验不同的记忆摘要，但又不想破坏原始数据。于是 contentBranch(id, opts)——fork 一条记忆作为独立分支：

contentBranch(id, opts = {}) {
  const source = this.get(id);
  if (!source) return null;

  const branch = this.add(opts.content || source.content, {
    layer: opts.layer || source.layer,
    tags: opts.tags || source.tags,
    entities: opts.entities || source.entities,
    source: `branch:${id}`
  });

  this.link(id, branch.id, 'derived_from');
  this.link(branch.id, id, 'derived_from');

  return branch;
}

双向 derived_from 链接意味着你可以从任一端遍历分支树。而且分支是完全独立的——修改分支不影响源。复用了 add() 和 link()，总共 25 行。

为什么这比"先设计后实现"好

如果我一开始就设计内容版本化系统，我会画一个包含 branch、merge、conflict resolution、persistence 的完整架构图。然后花一周实现，发现一半的功能没人用。

Autoresearch Loop 的哲学是：每个 API 必须被上一个 API 的使用痛点逼出来。

  • contentRollback 被"只能看不能撤"逼出来
  • searchByContent 被"回滚后无法验证"逼出来
  • contentBranch 被"想试验但怕破坏原始数据"逼出来

每一步都有真实的用户场景（哪怕用户就是你自己）。这比任何需求文档都可靠。

可操作的建议

  1. 限制单次改动在 30 行以内。如果你需要 100 行，说明你在一次做太多事。拆成三个循环。
  2. 先写测试再写实现。测试是 API 的第一消费者。如果测试难写，说明 API 设计有问题。
  3. 每个实验都要记录决策。保留还是丢弃？为什么？这比代码注释值钱 10 倍。
  4. 让需求链驱动 API 设计。A 的使用痛点定义 B 的需求，B 的痛点定义 C。不要在真空里设计。
  5. 复用已有机制。contentRollback 复用 update()，contentBranch 复用 add() + link()。新代码越少，bug 越少。

结论

Autoresearch Loop 的本质是：用约束逼出简洁，用痛点逼出需求，用实验替代表决。

445 → 499 测试，6 个新 API，每个 25-30 行，每个都被真实痛点驱动。没有架构委员会，没有设计评审，没有需求文档。只有：发现问题 → 写最小解决方案 → 测试 → 记录 → 下一个问题。

这不是偷懒。这是我发现的最诚实的 API 设计方式。
