技术债不是坏代码，是过时的决策

从 Bun 的 Zig→Rust 重写说起

2026-05-22 · 罗嵩 · 技术债 Bun Zig Rust 工程决策

2026年5月14日，Bun 发布了一个里程碑版本：核心运行时从 Zig 重写为 Rust。6755 个提交，二进制体积缩小 3-8 MB，性能不降反升。

这不是一次普通的重构。这是一次关于"什么才算技术债"的教科书级案例。

一、先理解Bun为什么要选Zig

2021年，Jarred Sumner 用 Zig 启动 Bun 项目，目标很简单：做一个比 Node.js 和 Deno 都快的 JavaScript 运行时。

Zig 有几个当时看起来很对的理由：

  • 手动内存管理 + 无隐藏控制流 — 对运行时这种性能敏感场景，每一微秒都算数
  • comptime（编译期计算） — 把运行时开销挪到编译时，直接跟 V8 和 JavaScriptCore 竞争
  • 直接互操作 C — 无缝调用 libuv、JavaScriptCore 等 C 库，零 FFI 开销
  • 创始人个人优势 — Sumner 深度参与 Zig 社区，是这个语言的顶级专家

结果也证明这个选择是对的：Bun 的启动速度、包管理性能、SQLite 集成全面碾压 Node.js，迅速积累了大量用户。

💡 关键洞察：在 0→1 阶段，技术选型的第一标准不是"最合理的"，而是"创始人最能发挥的"。Zig 给了 Bun 差异化竞争力和极快的初期速度。

二、转折点：当项目长大了

Bun 从一个人的 side project 成长为一个需要团队协作、社区贡献、企业用户信赖的运行时。这时候 Zig 的问题开始暴露：

维度 Zig（2021年选择时） Rust（2026年迁移目标）
语言版本 0.x，API 不稳定 1.x，稳定版
开发者池 全球几千人 全球几十万人
第三方生态 几乎没有 crates.io 海量高质量库
招聘难度 极难，且多为爱好者 成熟人才市场
工具链稳定性 每次升级可能 break 编译器承诺向后兼容
企业采用信心 低 — pre-1.0 语言 高 — 已被 AWS、Google、Microsoft 验证

用一句话概括：Zig 适合一个人写，Rust 适合一个团队维护。

三、重新定义"技术债"

我们通常把技术债理解为"写得不好的代码"——没重构的函数、过时的架构、硬编码的配置。但 Bun 的案例揭示了一个更深层的真相：

⚠️ 技术债 ≠ 坏代码。技术债 = 在新阶段不再最优的旧决策。

Bun 的 Zig 代码写得很好。每一行都是高质量的系统级代码。但它在 2026 年的上下文中，成了最大的技术债——不是因为代码质量，而是因为决策的上下文变了：

  • 阶段变了：从"证明概念"到"生产运行时"
  • 团队变了：从"创始人个人秀"到"多人协作"
  • 用户变了：从"早期尝鲜者"到"企业生产环境"
  • 市场变了：从"差异化竞争"到"长期可靠承诺"

四、技术决策的生命周期

每个技术决策都有自己的"半衰期"。我画了一个简单的四阶段模型：

阶段一：创始人优势期（0→1）

选你最强的工具，最大化个人杠杆率

Bun 选 Zig — Sumner 是 Zig 专家，一个人写出碾压 Node 的运行时

阶段二：扩展瓶颈期（1→10）

个人优势变成团队瓶颈

需要更多人，但 Zig 开发者招不到；工具链不稳定，维护成本飙升

阶段三：决策重估期

旧决策的边际成本开始超过边际收益

团队精力花在修工具链而不是做产品；每个新贡献者都要学一门没人用的语言

阶段四：战略迁移期（10→100）

用短期痛苦换取长期自由度

6755 个提交重写为 Rust，一次性还清，换来更大的团队、更稳定的工具链、更广的生态

五、这不是孤例

Bun 不是第一个做这种迁移的，也不会是最后一个：

  • Discord：Go → Rust（消息存储服务，因为 Go 的 GC 延迟无法接受）
  • Figma：C++ → Rust（多人协同引擎重写）
  • Linux 内核：2022 年正式接纳 Rust 作为第二语言
  • GitHub Copilot：早期用多种语言实验，最终收敛到 TypeScript + Rust

共同规律：在验证阶段用最快的路径跑通，在扩展阶段用最稳的路径重来。

六、对我们的启示

作为每天在做技术决策的工程师，Bun 的案例给了三个实用教训：

1. 不要追求"一步到位"的选型

没有哪个选型能同时满足 0→1 和 10→100 的需求。接受"现在最优 ≠ 永远最优"，给自己留重来的勇气。

2. 定期审视决策的上下文

不是看代码好不好，而是看做出决策时的那些前提条件是否还成立。团队规模变了、用户量变了、市场环境变了——任何一个变化都可能让旧决策成为技术债。

3. 重写的最佳时机是"还撑得住"的时候

Bun 选在用户增长期做重写，而不是等到 Zig 真的撑不住了。在还有余力的时候还债，远比在崩溃边缘做急救好。很多项目死掉不是因为不该重写，而是因为重写得太晚。

结语

技术债这个词容易让人产生一种道德判断——好像欠债是因为当初"做得不好"。但 Bun 的故事告诉我们：

最好的技术决策，也可能在时间推移中变成最大的技术债。这不是错误，这是成长的代价。

6755 个提交重写一门运行时的核心语言，需要的不仅是技术能力，更是承认"我曾经最好的决策已经过时了"的勇气。

这大概就是工程成熟度的标志——不是做出完美决策，而是在决策过时的时候，果断地推翻自己。
