让本地模型100%输出合法JSON：Constrained Decoding实战

2026-05-11 · LLM Structured Output Edge AI TypeScript

做 Agent 的人都会撞上同一堵墙：模型输出的 JSON 经常坏掉。少个括号、多了个逗号、字段名拼错、嵌套层级不对——每次解析失败就意味着重试、超时、用户体验崩塌。云 API（OpenAI、Anthropic）靠 function calling 基本解决了这个问题，但在本地部署场景（Ollama、llama.cpp、vLLM），我们过去只能祈祷 + 正则兜底。

这篇随笔分享我在 Edge Agent 项目中研究 constrained decoding 后得到的实战方案：一套 Zod + Ollama JSON Mode + 递归重试 的 TypeScript 工具链，在本地模型上实现可靠的结构化输出。

三级输出控制：从"请返回JSON"到数学保证

结构化输出有三个层级，理解它们是选对方案的前提：

  • Level 1 — Prompt Engineering：在 prompt 里写"请返回JSON格式"。可靠性 80-95%，适合内部原型。
  • Level 2 — Function Calling / Tool Use：通过 schema hint 引导模型。可靠性 95-99%，依赖模型是否支持。
  • Level 3 — Constrained Decoding：JSON Schema 编译成有限状态机（FSM），每个 token 生成时屏蔽所有不符合 FSM 状态的 token。可靠性 100% schema-valid——不是"大概率对"，是数学保证。

Level 3 的原理很优雅：schema 被编译成 FSM，在每一步生成时，只有能沿着 FSM 继续前进的 token 才被允许。无效 token 的 logit 被设为负无穷，概率归零。输出在生成时就被约束，不存在"生成完了发现不对"的情况。

被低估的 Ollama JSON Mode

很多人不知道 Ollama 从 0.5+ 版本就支持 format: "json" 参数，底层调用 llama.cpp 的 GBNF 引擎——这就是 Level 3 的 FSM 约束，不需要额外库。

对 TypeScript/Node.js 边缘场景，这意味着：

  • 不需要 Python 依赖
  • 不需要单独的推理引擎
  • 一个 HTTP 调用就得到 schema-valid JSON

实战代码：Zod + Ollama + 递归重试

Constrained decoding 只保证结构合法，不保证语义正确（字段对但值不合理）。所以真正的生产方案需要两层防御：

import { z } from "zod";

// 定义 Agent Action 的 schema
const ToolCallSchema = z.object({
  name: z.string().describe("Function name"),
  args: z.record(z.unknown()).describe("Function arguments"),
  confidence: z.number().min(0).max(1),
});

const AgentActionSchema = z.object({
  thought: z.string(),
  toolCalls: z.array(ToolCallSchema),
  shouldRespond: z.boolean(),
  response: z.string().optional(),
});

type AgentAction = z.infer<typeof AgentActionSchema>;

// JSON 提取器：处理 markdown code block 包裹的情况
function extractJSON(text: string): string | null {
  const trimmed = text.trim();
  if (trimmed.startsWith("{")) {
    let depth = 0;
    for (let i = 0; i < trimmed.length; i++) {
      if (trimmed[i] === "{") depth++;
      if (trimmed[i] === "}") depth--;
      if (depth === 0) return trimmed.slice(0, i + 1);
    }
  }
  const match = trimmed.match(/```(?:json)?\s*\n?([\s\S]*?)\n?\s*```/);
  return match ? match[1].trim() : null;
}

// 核心函数：带 Zod 验证 + 错误反馈重试
async function structuredGenerate<T extends z.ZodType>(
  schema: T,
  prompt: string,
  config: { model: string; baseUrl: string; maxRetries: number }
): Promise<z.infer<T>> {
  const schemaStr = JSON.stringify(schema._def, null, 2);
  const systemPrompt = `You are a structured data generator.
JSON Schema:
${schemaStr}
Output valid JSON only. No markdown. No explanation.`;

  let lastError = "";

  for (let attempt = 0; attempt < config.maxRetries; attempt++) {
    const fullPrompt = attempt === 0
      ? `${systemPrompt}\n\nUser: ${prompt}`
      : `${systemPrompt}\n\nUser: ${prompt}\n\nPrevious errors:\n${lastError}\nFix and output valid JSON.`;

    // Ollama JSON mode — Level 3 FSM 约束
    const res = await fetch(`${config.baseUrl}/api/generate`, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        model: config.model, prompt: fullPrompt,
        stream: false, format: "json",
        options: { temperature: 0.1 },
      }),
    });
    const raw = (await res.json()).response;
    const jsonStr = extractJSON(raw);
    if (!jsonStr) { lastError = "No JSON found"; continue; }

    const result = schema.safeParse(JSON.parse(jsonStr));
    if (result.success) return result.data;

    // 把 Zod 验证错误喂回模型 — 这是关键
    lastError = result.error.issues
      .map(i => `Field "${i.path.join(".")}" ${i.message}`)
      .join("; ");
  }
  throw new Error(`Failed after ${config.maxRetries} retries: ${lastError}`);
}

关键设计决策：把 Zod 的验证错误原样反馈给模型做重试。这不是 hack——模型能理解"Field 'confidence' must be between 0 and 1"这种错误信息，并在下一轮修正。在我的测试中，第一轮通过率约 85%，加上一轮重试就到 98%+。

Schema 缓存：高频场景的性能优化

在 Agent 循环中，同一个 schema 会被反复使用。每次序列化 schema 定义到 prompt 是浪费：

class SchemaCache<T extends z.ZodType> {
  private cache = new Map<string, { schema: T; compiledPrompt: string }>();

  get(key: string, factory: () => T, promptBuilder: (s: T) => string) {
    if (!this.cache.has(key)) {
      const schema = factory();
      this.cache.set(key, { schema, compiledPrompt: promptBuilder(schema) });
    }
    return this.cache.get(key)!;
  }
}

// Agent 循环中复用
const cache = new SchemaCache();
const action = cache.get("agent-action-v1",
  () => AgentActionSchema,
  (s) => `JSON Schema:\n${JSON.stringify(s._def)}`
);
// 后续调用零开销

一个反直觉的发现：约束可能更快

直觉上，constrained decoding 会拖慢生成——毕竟每步多了一层过滤。但实际测试发现相反：语法约束修剪了搜索空间，减少了模型在无效路径上浪费的 token。XGrammar 的 benchmark 显示某些场景下 structured output 吞吐量高于 unconstrained output。

XGrammar 的核心创新是 Adaptive Token Mask Caching：将 grammar state 分为"上下文无关"（约 99% 词汇表）和"上下文相关"（约 1%），前者预计算缓存，后者实时计算。在 batch decoding 下效率极高，这也解释了为什么 vLLM V1 的 structured output 比 V0 快一个数量级。

2026年的 SLM 已经准备好了

另一个重要发现：Qwen3.5-0.8B、Phi-4-mini、Ministral-3B 这些小型模型已经原生支持 function calling。不需要 hack，不需要 few-shot，开箱即用。这意味着 Edge Agent 可以在不到 4GB 显存/内存上运行可靠的 tool-use agent——这在一年前还不可想象。

可操作建议

  1. 如果你用 Ollama：立刻加上 format: "json" 参数。零成本，可靠性从 ~90% 跳到 ~99%。
  2. 如果你用 vLLM 自部署：升级到 V1，默认使用 XGrammar 后端。structured output 在 V1 上比 V0 快 10x。
  3. 无论什么引擎：都加一层 Zod 验证 + 重试。结构合法不等于语义正确，只有 Zod 能帮你抓到"confidence: 1.5"这种错。
  4. 不要用正则提取 JSON：用括号匹配提取器（见上面代码），能处理嵌套和 code block 包裹的情况。
  5. 缓存你的 schema prompt：在 Agent 循环中，同一个 schema 反复编译是纯浪费。

Constrained decoding 不是"更好的 prompting"——它是 generation-time 的硬约束。理解这一点，你的 Agent 架构就有了可靠性的根基。
