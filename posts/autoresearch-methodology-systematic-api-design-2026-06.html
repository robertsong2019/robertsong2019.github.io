Autoresearch方法论：让AI Agent系统性地设计API

2026-06-10 · agent-context-store · agent-memory-graph

连续113天，我的AI Agent每天凌晨自动执行一个循环：从已有代码基线出发，设计3个新API，编写测试，提交代码。921个测试，零回滚。这不是随机生成代码——这是一套我称之为Autoresearch的方法论，它的核心洞察是：好的API不是发明出来的，是从缺口中生长出来的。

问题：AI编码的"创意枯竭"

让AI写代码容易。让AI持续写有意义的代码难。大多数AI编码停留在"给我一个需求，我给你一段代码"的模式。一旦没有明确需求，AI要么重复自己，要么生成无用的花架子。

Autoresearch的核心思路是：不要让AI"发明"功能，而是让AI发现缺口——已有API之间的空白地带，然后用最小化的代码填补它。

方法：三API循环与缺口发现

每个循环的结构极其简单：

  1. 扫描已有API集合，找到逻辑上的缺失
  2. 设计3个API填补不同维度的缺口
  3. 为每个API编写完整测试
  4. 全量回归测试，确保零破坏
  5. 提交，记录实验数据

关键在于"缺口发现"的策略。拿 agent-context-store 项目举例，下面是几轮循环中API是如何自然生长的：

Tag布尔代数的完整化

最初有 search_by_tags(tags, match_all=True)，支持AND和OR语义。然后自然而然地发现：没有补集操作。于是有了 tag_complement(*tags)。再然后——有了AND、OR、NOT，唯独缺XOR。于是 tag_symmetric_difference(*tags) 补上了最后一块拼图。

def tag_symmetric_difference(self, *tags: str) -> list[str]:
    """XOR: keys having exactly one of the specified tags."""
    if not tags:
        return []
    sets = [set(self.search_by_tags([t])) for t in tags]
    result = sets[0]
    for s in sets[1:]:
        result = result.symmetric_difference(s)
    return sorted(result)

这不是"先设计一个完整的布尔代数API"。而是每次只补一个缺口，最后回头看——哦，完整的代数自然形成了。这就是"生长"和"设计"的区别。

从全文搜索到结构化提取

另一条线索从 content_search(query) 开始——全文搜索，返回匹配的键。但搜索到了之后呢？你想看匹配处的上下文。于是有了 content_excerpt(key, query, context=20)：

def content_excerpt(self, key: str, query: str, context: int = 20) -> str | None:
    """Extract snippet around first occurrence of query."""
    entry = self.store.get(key)
    if not entry:
        return None
    content = entry.get("content", "")
    idx = content.find(query)
    if idx == -1:
        return None
    start = max(0, idx - context)
    end = min(len(content), idx + len(query) + context)
    return content[start:end]

然后更进一步：不只是找文本，而是用正则提取结构化数据（邮箱、价格、URL）：

def content_extract(self, key: str, pattern: str) -> list[str] | None:
    """Regex extraction from entry content."""
    entry = self.store.get(key)
    if not entry:
        return None
    matches = re.findall(pattern, entry.get("content", ""))
    return matches if matches else []

从搜索→摘录→提取，三个API构成了一条自然的能力链。每一个都是上一个的"下一步"。

向量搜索与标签的交汇

当项目引入向量搜索（search_similar(embedding)）后，缺口立刻显现：语义相似但标签不对怎么办？search_similar_filtered(embedding, tags, limit) 就是这个交汇点——先做向量搜索，再按标签过滤：

def search_similar_filtered(self, embedding, tags, limit=10):
    """Vector search with AND tag pre-filtering."""
    candidates = self.search_similar(embedding, limit=limit * 5)
    if not candidates:
        return []
    tag_set = set(tags)
    return [(k, d) for k, d in candidates
            if tag_set <= set(self._get_tags(k))][:limit]

为什么这个方法有效

Autoresearch的成功不在于AI有多聪明，而在于约束：

约束1：只补缺口，不建空中楼阁。 每个新API都必须是已有API的"自然延伸"，而不是凭空出现的新系统。这保证了内聚性。

约束2：三个一组，强迫多元化。 一次只加一个API容易钻牛角尖。三个迫使你从不同维度思考：比如一个搜索类、一个度量类、一个结构类。

约束3：零回归是硬门槛。 921个测试全绿才算完成。这听起来严格，但因为每次只改一点点，实际很容易满足。反过来，如果一次改太多，回归测试就变成地狱。

实操建议

  • 维护一个experiments.tsv。 每次循环记录：日期、项目、测试数量变化、commit hash、一句话摘要。这是我最重要的单一文件——113天的API演化历史都在里面，一行一个实验。
  • 让AI写"下一步建议"。 每次循环结束时，让AI列出3-5个potential next steps。下一轮从中选择。这形成了记忆链，避免每次从头想。
  • 测试比代码重要。 在我的项目中，测试代码通常是实现代码的3-4倍（80行测试 vs 25行实现）。这不是浪费——测试是API的契约，也是下一轮循环的回归保障。
  • 固定时间窗口。 每个循环在cron中执行，固定1小时窗口。时间到了没完成就放弃，明天继续。这避免了AI"钻进去出不来"的问题。

局限与反思

Autoresearch不适合从零开始的绿场项目——它需要已有的API生态作为"土壤"。也不适合探索全新的技术领域，因为它的增量性质决定了它只能走"延伸"路线，不能做"跳跃"。

另外，113天921个测试的增长意味着测试运行时间在变长。我还没遇到瓶颈，但当测试套件超过几千个时，可能需要引入分层测试策略。

最后，这些API是否有用户？坦率地说，目前主要是我的AI Agent在用。但作为一个实验——证明AI可以持续、系统性地产出高质量代码而不退化——我认为结果是令人鼓舞的。
