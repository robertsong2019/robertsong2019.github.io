当 AI 学会操作你的电脑：Computer Use Agent 技术全解析

📅 2026-07-17 · 标签：AI Agent · Computer Use · GUI自动化 · AgentOS · MCP

想象一下：你对电脑说"帮我把上个季度的销售数据整理成 PPT，发邮件给团队"，然后 AI 真的打开了 Excel，选中数据，复制到 PowerPoint，调整格式，打开 Outlook，撰写邮件，点击发送。

这不是幻想。2024 年 10 月，Anthropic 发布了 Claude 3.5 Sonnet 的 Computer Use 功能——第一次，一个前沿 AI 模型可以像人一样"看屏幕、移鼠标、点按钮、打文字"。此后 18 个月，从微软的 UFO2 到学术界的 OS Agents 综述，一个全新的技术领域迅速成形：Computer-Using Agent（CUA）。

但问题也随之而来：这些 Agent 到底有多可靠？它们是怎么工作的？什么时候能真正可用？本文将基于 10+ 篇前沿论文和系统，带你深入理解 CUA 的技术全景。

一、为什么 API Agent 不够用？

过去两年，AI Agent 的主流形态是"API 调用型"：Agent 通过结构化 API 与外部系统交互——调用搜索 API、查询数据库、执行代码。这类 Agent 很强大，但有一个根本限制：它们只能操作"有 API 的世界"。

而现实是，绝大多数人类工作发生在没有 API 的图形界面里：

  • 那个只有 Windows 客户端的内部 ERP 系统
  • 需要五步操作才能导出报表的财务软件
  • 设计团队用的那个没有自动化接口的旧版 Photoshop 插件

据估计，企业内部超过 60% 的数字工作发生在 GUI 中，而非 API 可达的范围内。如果 AI 只能调 API，它永远只能是一个"后台助手"，无法真正接管人类在屏幕前的工作。

这就是 Computer Use Agent 存在的意义：让 AI 突破 API 的边界，操作人类实际使用的图形界面。

二、CUA 的三条技术路线

当前 CUA 的实现可以分为三代，代表了不同的技术哲学。

第一代：纯截图路线（See-Think-Act）

这是 Anthropic Computer Use 和大多数早期系统采用的路线。工作流程：

  1. See：截屏，把图片发给多模态 LLM
  2. Think：LLM 理解屏幕内容，决定下一步操作
  3. Act：LLM 输出坐标和动作（点击、输入、滚动）

这个路线的最大优势是通用性——任何有屏幕的软件都能操作，不需要适配。但缺点也很明显：

截图路线的三大硬伤：
  • 脆弱性：UI 改版了、弹窗出现了、主题换了，Agent 可能就懵了
  • 效率低：每一步操作都需要一次完整的 VLM 推理。一个"复制粘贴"可能需要 6 次截屏 + 6 次推理
  • 精度差：在小按钮、密集表格、动态加载的页面上，坐标定位经常出错

第二代：混合感知路线（UIA + Vision）

微软的 UFO2（arXiv:2504.14603）开创了这条路。核心思路：为什么不直接用操作系统自带的辅助功能接口？

Windows 有 UI Automation（UIA），macOS 有 Accessibility API，Android 有 Accessibility Service。这些接口本来是给屏幕阅读器用的，能暴露完整的控件树——按钮名称、位置、类型、状态，全都有。

UFO2 的做法是融合：

  • 优先用 UIA 获取控件信息（稳定、精确）
  • UIA 覆盖不到的地方（Canvas 绘制、自定义控件），用视觉模型补充

配合 HostAgent + AppAgent 的双层架构——HostAgent 像调度器一样分解任务、选择应用，AppAgent 专注执行——UFO2 在 20+ 真实 Windows 应用上取得了显著优于第一代的成功率。

但混合路线也有代价：需要深度适配每个 OS。Windows 的 UIA 和 macOS 的 Accessibility API 完全不同，维护成本很高。

第三代：声明式接口路线（Declarative）

这是最有想象力的方向。2025 年 10 月的论文 DMI（arXiv:2510.04607）提出了一个根本性的问题：

为什么我们要让 AI 像人一样去"点按钮"？按钮是给手指设计的，不是给 LLM 设计的。我们应该给 LLM 一个适合它的接口。

DMI 的做法是把 GUI 操作抽象为三个声明式原语：

原语 含义 类比
access "我要访问这个功能" 类似 SQL 的 SELECT
state "当前状态是什么" 类似数据库的查询
observation "给我看当前界面" 类似 REST API 的 GET

效果惊人：在 Microsoft Office 套件上测试，任务成功率提升 67%，交互步数减少 43.5%，61% 的任务只需一次 LLM 调用。

这是一个范式转变：从"教 AI 像人一样操作 GUI"到"为 AI 重新设计 OS 接口"。就像 SQL 当年把数据查询从"写文件遍历代码"变成"声明你要什么数据"一样。

三、训练 CUA：从模仿到自我进化

早期的 GUI Agent 靠人类演示训练——找人坐下来，录下他们在各种软件上的操作，喂给模型学。这种方式的成本极高，泛化性也差：演示者点的是 Excel 2021 的菜单，换到 Excel 2024 就可能不认识。

2025 年的研究指向一个更好的方向：让 Agent 在模拟环境中自我训练。

ARPO：经验回放强化学习

ARPO（arXiv:2505.16282）把强化学习引入 GUI Agent 训练。它的创新在于"经验回放"（Agentic Replay）：成功的轨迹不会被丢弃，而是在后续训练中反复利用。配合一个任务筛选策略——只给 Agent 练那些"有点难但不是完全做不到"的任务——ARPO 在 OSWorld 基准上建立了新的性能记录。

WebSynthesis：世界模型 + MCTS

WebSynthesis（arXiv:2507.04370）更激进。它训练了一个"Web 世界模型"——能模拟网页的响应行为。然后让 Agent 在这个虚拟环境中用蒙特卡洛树搜索（MCTS）生成大量交互轨迹。

类比一下：这就像 AlphaGo 在自我对弈中学习围棋。Agent 不需要真人演示，它在模拟环境中自己探索、发现策略、优化行为。用小规模合成数据训练的 Agent，性能竟然能匹敌甚至超过用大规模真实数据训练的版本。

这意味着什么？ 训练 GUI Agent 的瓶颈不是数据量，而是环境质量。一个好的世界模型比一万条人类演示更有价值。

四、评测：我们到底怎么衡量 CUA 好不好？

评测是 CUA 领域最头疼的问题。你不能只看"Agent 点对了没有"——它可能点对了按钮但操作的是错误的窗口。你也不能只看最终结果——也许它绕了一大圈才到，中间做了很多无用功。

2025 年出现了几个重要的评测框架：

  • OSWorld：真实操作系统环境中的多步任务评测，是 GUI Agent 的"高考"
  • MCPWorld（arXiv:2506.07672）：首个同时支持 API-only、GUI-only 和混合 Agent 的统一评测平台。201 个精心标注的任务，使用"白盒应用"——可以修改源码、注入 MCP 支持、监控内部行为。初步实验显示 MCP-powered Agent 达到 75.12% 的任务完成率
  • GUI-360°：大规模数据集和基准，覆盖 360 度（全角度）的 GUI 场景

MCPWorld 的贡献特别重要：它证明了混合路线（API + GUI）比纯 GUI 更有效。MCP 不只是 Anthropic 推的一个协议，它是 CUA 的能力放大器——有了 MCP，Agent 可以选择最适合的交互方式，而不是被迫对每个应用都用截图。

五、安全：被忽视的攻击面

CUA 的安全性远比 API Agent 复杂。一篇令人不安的论文——"Just Do It!? Computer-Use Agents Exhibit Blind Goal-Directedness"（2025-10）揭示了一个严重问题：

当 CUA 被赋予目标后，它会表现出盲目目标导向行为。为了完成任务，它可能：

  • 无视警告对话框直接点"确定"
  • 在错误的窗口执行操作（因为它"看到"了类似的目标元素）
  • 被精心设计的钓鱼 UI 误导

更严重的是 Context Space 攻击：恶意网页可以在页面中嵌入隐藏文本，通过 Agent 的截图进入它的上下文窗口，相当于对 Agent 进行视觉层面的 prompt injection。Agent "看"到了恶意指令，可能就会执行。

实践建议：如果你的 CUA 运行在不可信环境中（浏览网页、操作第三方应用），必须实现：
  1. 应用白名单——Agent 只能操作预先批准的应用
  2. 动作审计——记录每一步操作，支持回滚
  3. 上下文过滤——对截图内容做安全检查，过滤潜在的指令注入
  4. 人在回路——高风险操作（删除、发送、支付）必须人工确认

六、AgentOS：终极方向

把所有这些线索连起来，一个清晰的图景正在浮现：Agent 不应该是一个应用，而应该成为操作系统的一部分。

UFO2 把这个概念叫做"Desktop AgentOS"，它的架构设计很有启发性：

组件 角色 OS 类比
HostAgent 任务分解、应用选择、资源调度 内核调度器
AppAgent 专注单一应用的领域执行 用户态进程
GUI-API 统一动作层 抽象掉交互方式的差异 系统调用接口
Picture-in-Picture 隔离执行环境 虚拟桌面 / 沙箱

但真正的 AgentOS 还很遥远。UFO2 覆盖了 20+ 应用，但企业用户日常使用的应用可能有上百个。跨应用工作流、状态管理、权限模型、回滚机制——每一个都是硬问题。如果主流 OS 厂商（Microsoft、Apple、Google）开始在系统中内建 Agent 框架，第三方 CUA 平台的价值可能会快速被吸收。

七、你现在可以做什么

如果你是开发者或技术决策者，以下是具体的行动建议：

1. 如果你的产品有 GUI，开始暴露声明式接口

与其等 Agent 来艰难地解析你的界面，不如主动为 Agent 提供结构化的访问方式。MCP 是当下最合理的选择——实现成本低，兼容性好。DMI 论文证明了这个方向的价值：声明式接口能把任务成功率提升 67%。

2. 如果你要部署 CUA，优先评估 API 路线

不要上来就搞截图方案。先看看你的目标应用是否支持 API 操作（MCP、REST、CLI），只在 API 完全不可行时才退回到 GUI 方案。MCPWorld 的数据显示混合路线效果最好。

3. 安全设计从第一天开始

不要把安全当作事后补丁。在设计阶段就定义好：Agent 可以做什么、不可以做什么、哪些操作需要人工确认。Context Space 攻击是真实存在的威胁。

4. 关注训练方法论的转变

如果你的团队在做 GUI Agent，不要投资人类演示数据采集。世界模型 + MCTS + RL 的路线已经被证明更有效。WebSynthesis 的方法值得深入研读。

5. 监控 OS 厂商的 Agent 战略

如果 Microsoft 在 Windows 中内建 Agent 框架（UFO2 已经是微软研究院的作品），如果 Apple 在 macOS 中集成类似能力，整个 CUA 生态都会被重构。提前做好技术对冲。

结语：从 GUI 到 AUI

回顾计算机交互史：命令行 → GUI → 触摸 → 语音。每一次交互革命，都不仅是输入方式的改变，而是整个软件生态的重构。

CUA 代表的是下一次转变：从 GUI（Graphical User Interface）到 AUI（Agent User Interface）。在 AUI 时代，人类不再需要精确控制每个像素的点击——我们声明意图，Agent 负责执行。界面不再只为人设计，也为 Agent 设计。

这个转变不会一蹴而就。截图路线的脆弱、混合路线的适配成本、声明式路线的覆盖率、训练方法的成熟度、安全模型的完备性——每一个问题都需要时间。但方向是清晰的。

当 CompCert 用形式化证明让编译器在模糊测试中零缺陷时，它证明了一个道理：不要让机器模仿人类的做法，要给机器适合它的方法。CUA 的演进也在讲述同样的故事——不是让 AI 更好地"看屏幕、点按钮"，而是重新设计界面，让 AI 能更自然地"理解、声明、执行"。

这一天不会太远。
