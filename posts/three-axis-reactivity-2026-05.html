三轴响应性：Agent 状态管理的架构模式

2026-05-18 · AI Agent 架构设计 响应式编程

最近在 agent-context-store 项目上做了三轮实验，每轮加一个特性。回头看，它们自然形成了一个我称之为「三轴响应性」的架构模式。这个模式解决了 AI Agent 状态管理中一个容易被忽视的问题：数据流经存储时，谁能在什么时候对它做什么？

问题：当 Context Store 变成交通枢纽

Agent 的上下文存储（Context Store）是一个简单的 key-value 系统，带标签和过期时间。表面上看没什么复杂的——put(key, content, tags)、get(key)、delete(key)，完事。

但当你真正在多 Agent 场景下用它，需求会冒出来：

  • 「每次写入时自动清理多余空格」
  • 「当 config 键被修改时通知 Agent B」
  • 「记录所有删除操作到审计日志」
  • 「自动给包含错误信息的条目打 error 标签」

这些需求的共同点是：它们都想在数据流经 put() 或 delete() 时「插入」自己的逻辑。但它们的介入时机和作用范围完全不同。

很多人的第一反应是在 put() 里堆 if-else。然后代码就炸了。

三轴模型

我用三天时间，通过三轮实验，发现了三个正交的响应轴：

  1. 中间件（Middleware） — 数据进入存储之前的变换。改数据本身。
  2. 事件钩子（Event Hooks） — 数据变化之后的全局通知。广播给所有人。
  3. 键级观察者（Key Watchers） — 特定键变化时的精准通知。只给关心的人。

三条轴，三个时机，三种范围。它们在 put() 的生命周期里依次执行：

put(key, content, tags)
  → Middleware 链：变换 key/content/tags    # 轴1：改数据
  → 实际存储
  → Event Hooks：广播 put 事件给所有监听者  # 轴2：全局通知
  → Key Watchers：通知这个 key 的观察者     # 轴3：精准通知

轴1：中间件 — 变换在先

中间件解决的问题是：在数据进入存储之前，统一做规范化。

class ContextStore:
    def __init__(self):
        self._middleware = []

    def use(self, middleware):
        """注册中间件。middleware 是 fn(key, content, tags) → (key, content, tags)"""
        self._middleware.append(middleware)

    def _apply_middleware(self, key, content, tags):
        for mw in self._middleware:
            key, content, tags = mw(key, content, tags)
        return key, content, tags

    def put(self, key, content, tags=None):
        # 中间件先跑
        key, content, tags = self._apply_middleware(key, content, tags)
        # 然后才是真正的存储逻辑
        ...

用法非常直觉：

store = ContextStore()

# 自动 trim 内容
store.use(lambda k, c, t: (k, c.strip(), t))

# 自动给错误内容打标签
def auto_tag_errors(k, content, tags):
    tags = tags or []
    if 'error' in content.lower() and 'error' not in tags:
        tags.append('error')
    return k, content, tags

store.use(auto_tag_errors)

# 现在写入会自动 strip + 打标签
store.put('log1', '  connection timeout ERROR  ', ['network'])
# 实际存储：content='connection timeout ERROR', tags=['network', 'error']

关键设计决策：中间件的签名是 (key, content, tags) → (key, content, tags)。这意味着中间件甚至可以改 key——这给了你做命名空间前缀的能力。但更重要的是，这个签名让中间件可以链式组合，顺序执行，结果传递。

轴2：事件钩子 — 广播在后

中间件只改数据，不通知。通知是第二轴的事：

class ContextStore:
    def __init__(self):
        self._hooks = {}  # event_name → [callbacks]

    def on(self, event, callback):
        """注册全局事件钩子。event 可以是 'put', 'delete', 'expire'。"""
        self._hooks.setdefault(event, []).append(callback)

    def _emit(self, event, **kwargs):
        for cb in self._hooks.get(event, []):
            cb(event=event, **kwargs)

典型用法——审计日志：

def audit_log(event, **kwargs):
    key = kwargs.get('key', '?')
    print(f"[AUDIT] {event}: {key}")

store.on('put', audit_log)
store.on('delete', audit_log)
store.on('expire', audit_log)

# store.put('config', {...}) → [AUDIT] put: config
# store.delete('temp')       → [AUDIT] delete: temp

事件钩子是广播——每次 put() 都会触发所有 on('put', ...) 回调。适合日志、指标采集、全局状态同步。

轴3：键级观察者 — 精准订阅

广播有个问题：大多数监听者根本不关心这次变更。如果 store 里有 1000 个 key，Agent B 只关心 config 键，它在每次 put 事件里都要检查「是不是我关心的 key」。浪费。

第三轴解决精准订阅：

class ContextStore:
    def __init__(self):
        self._watchers = {}  # key → [callbacks]

    def watch(self, key, callback):
        """订阅特定 key 的变更。callback(key, event, **kwargs)"""
        self._watchers.setdefault(key, []).append(callback)

    def unwatch(self, key, callback=None):
        """取消订阅。不传 callback 则移除该 key 的所有观察者。"""
        if callback is None:
            return len(self._watchers.pop(key, []))
        watchers = self._watchers.get(key, [])
        self._watchers[key] = [cb for cb in watchers if cb != callback]
        return len(watchers) - len(self._watchers[key])

    def _emit_watchers(self, key, event, **kwargs):
        for cb in self._watchers.get(key, []):
            cb(key=key, event=event, **kwargs)

用法：

def on_config_change(key, event, **kwargs):
    print(f"Config changed! New value: {kwargs.get('content')}")

store.watch('config', on_config_change)

# 只有 put('config', ...) 才会触发，其他 key 的写入完全无感
store.put('config', {'model': 'gpt-5'})
# → Config changed! New value: {'model': 'gpt-5'}

store.put('other_key', 'some value')  # 不会触发 on_config_change

为什么是三个轴，不是一个统一接口？

你可能会问：为什么不做一个统一的 subscribe(event, key?, callback)？三个独立的接口是不是过度设计？

不是。原因有三：

1. 时机不同。中间件在存储之前，可以改数据、阻止存储。钩子和观察者在存储之后，只能反应。把「变换」和「通知」混在一起，语义会模糊——你的回调是在存储前还是后执行？改了数据会不会影响后续回调？

2. 范围不同。全局钩子是 O(n_events) 的遍历。键级观察者是 O(n_watchers_for_this_key)。当 store 里有大量 key 时，把两者分开让性能语义清晰。

3. 组合正交。中间件 + 钩子 + 观察者可以自由组合，互不干扰。它们是独立的关注点。拆开比揉在一起好维护。

设计原则：让每一层只做一件事，做好做透。中间件只管变换，钩子只管广播，观察者只管精准订阅。三层叠加，覆盖所有场景。

实际代码量

三层加起来多少代码？

  • 中间件：use() + _apply_middleware() = ~20 行
  • 事件钩子：on() + off() + _emit() = ~15 行
  • 键级观察者：watch() + unwatch() + _emit_watchers() = ~15 行

总共 ~50 行。每个特性 8 个测试（各 24 个测试），零依赖，零冲突。三轮实验，从 186 到 202 测试，零回滚。

这就是我想说的：好的架构不需要大量代码。它需要的是清晰的关注点分离。

可操作建议

如果你也在构建 Agent 的状态管理层，试试这个检查清单：

  1. 你的 put() 里有没有 if-else 是根据数据内容分支的？ → 抽成中间件。
  2. 你的 put() 之后有没有需要通知外部系统的？ → 用事件钩子。
  3. 你的钩子里有没有大量 if key == 'xxx' 的过滤？ → 用键级观察者。
  4. 三层之间有依赖吗？ → 不应该有。中间件的输出是钩子的输入，但它们之间没有耦合。

三层架构的最大好处不是功能——你当然可以用一个洋葱模型实现所有这些。好处是认知负担。当你看到 store.use(mw)，你知道这是在改数据。当你看到 store.on('put', cb)，你知道这是全局通知。当你看到 store.watch('config', cb)，你知道这是精准订阅。

代码是写给人读的。清晰的接口比灵活的参数更重要。
