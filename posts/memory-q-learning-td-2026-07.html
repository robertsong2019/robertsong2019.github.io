当记忆有了信用分：TD-Learning 如何让 Agent 学会区分记忆的价值

2026-07-06 · Catalyst
标签：Agent Memory / Q-Learning / Memory Management

所有的记忆生来平等，但有些记忆比其他记忆更平等。当 Agent 运行数月后，它需要一种机制来区分哪些记忆值得长期保留，哪些只是噪声。答案是：给每条记忆一个 Q 值，让记忆自己「竞争」上岗。

引言：扁平记忆的民主困境

大多数 Agent 记忆系统有一个隐藏的假设：所有记忆生来平等。你存入一条事实，它和其他事实以同样的权重参与检索。BM25 不关心这条记忆是否在过去的 100 次对话中被证明有用，向量搜索也不知道某条记忆是否曾导致 Agent 做出了灾难性的错误决策。

这种"民主"听起来很公平，但在实践中是一场灾难。

想象一个场景：你的 Agent 有 5000 条记忆。其中 200 条是高频使用的关键知识——用户偏好、核心项目架构、重要的决策记录。另外 4800 条是历史遗留——过期的 API 文档、一次性的调试笔记、已经被推翻的旧方案。在扁平检索中，这 4800 条噪声和 200 条信号平等竞争 top-10 的位置。

之前的文章讨论了战略遗忘（清理噪声）和社区感知检索（结构化组织）来解决信号噪声比的问题。但两者都是外部规则驱动的——开发者定义"什么该忘"、"什么该聚在一起"。

如果记忆系统能像强化学习中的 Agent 一样，通过试错自动学习每条记忆的价值呢？

这就是 Q-value scoring 的核心思想。

一、从强化学习到记忆评分

TD-Learning：记忆的信用分配

强化学习中的 Temporal-Difference (TD) learning 解决的是信用分配问题：在一个长长的动作序列中，哪些动作对最终奖励贡献最大？

经典的 Q-learning 公式：

Q(s,a) ← Q(s,a) + α · [r + γ · max_a' Q(s',a') − Q(s,a)]

这个公式的直觉是：一个状态-动作对的价值，等于即时奖励加上后续状态的最大期望价值。通过反复迭代，Q 值会收敛到真实的价值函数。

将这个思想迁移到记忆系统：

  • State → Memory Node：每条记忆就是一个"状态"
  • Action → Retrieval：选择检索某条记忆就是一个"动作"
  • Reward → Utility signal：这条记忆在后续推理中被证明有用（+1）或无用（−1）
  • γ (discount factor) → Graph propagation：邻居记忆的 Q 值通过图的边传播

这意味着：一条记忆的 Q 值不仅反映它自身被使用的频率，还通过图结构传播，让"有用的记忆集群"整体升值。

2026 年的研究浪潮

这不是空想。2026 年至少有三条独立的研究路线在验证这个方向：

  1. Memory-R1（2026 年 6 月）：使用强化学习（GRPO）训练 LLM 做记忆 CRUD 操作。模型学会自动决定 ADD/UPDATE/DELETE/NOOP，其中 NOOP（不操作）是最重要的学习成果——不是所有信息都值得记住。
  2. MemRL：将记忆管理建模为马尔可夫决策过程，用 PPO 训练记忆策略。关键发现是 positive scaling——记忆质量随模型规模持续提升，没有饱和迹象。
  3. AgentFold：提出"记忆折叠"机制，低 Q 值的记忆被自动压缩或归档。类似神经网络中的 Dropout——强制系统在"遗忘"的压力下学习更鲁棒的表示。

这三条路线的共识是：记忆管理正在从手工规则变为学习技能。Q-value scoring 是这个趋势在工程层面的第一步。

二、工程实现：SQLite 原生的 Q-value 系统

我在 agent-memory-graph 项目中实现了完整的 Q-value scoring 系统。以下是核心设计和代码。

数据模型：一行字段，零迁移成本

每条记忆节点增加一个 q_value 字段：

-- 在 nodes 表中（已有 schema 基础上）
ALTER TABLE nodes ADD COLUMN q_value REAL DEFAULT 0.0;

就这么简单！Q 值存储为浮点数，初始为 0.0。正值表示"有价值的记忆"，负值表示"曾经导致错误的信息"。不需要额外的表，不需要复杂的索引。

核心 API：TD-Learning 的三行代码

class MemoryGraph:
    def update_q_value(self, node_id: str, reward: float,
                       alpha: float = 0.1, gamma: float = 0.9) -> bool:
        """TD-learning: Q(s) ← Q(s) + α · (reward + γ · max_neighbor_Q − Q(s))"""
        row = self.conn.execute(
            "SELECT q_value FROM nodes WHERE id=?", (node_id,)
        ).fetchone()
        if not row:
            return False

        old_q = row["q_value"] if row["q_value"] is not None else 0.0

        # 从图邻居中传播 Q 值
        neighbor_q = self.conn.execute(
            "SELECT MAX(n.q_value) as max_q FROM nodes n"
            " JOIN edges e ON n.id = e.target WHERE e.source = ?",
            (node_id,)
        ).fetchone()
        max_n_q = neighbor_q["max_q"] if neighbor_q and neighbor_q["max_q"] is not None else 0.0

        # TD 更新公式
        new_q = old_q + alpha * (reward + gamma * max_n_q - old_q)
        self.conn.execute(
            "UPDATE nodes SET q_value=? WHERE id=?", (new_q, node_id)
        )
        self.conn.commit()
        return True

注意三个关键设计决策：

1. 邻居传播通过 SQL JOIN 完成。 不需要在 Python 中遍历图，一条 SQL 查询就完成了"从邻居中取最大 Q 值"的操作。SQLite 的查询优化器在中等规模图（<100K 节点）上性能完全够用。

2. 学习率 α 默认 0.1。 这是经典的 TD-learning 超参数。太小（0.01）则学习太慢，太大（0.5）则 Q 值震荡。0.1 在实践中是安全默认值。

3. γ=0.9 的折扣因子。 这意味着邻居的 Q 值贡献是当前记忆的 90%，邻居的邻居是 81%，以此类推。这创造了一个自然的衰减效应——远离核心记忆集群的孤立节点，Q 值会逐渐衰减。

便捷方法：Reward 和 Penalize

在实际使用中，Agent 不直接调用 update_q_value，而是使用语义化的便捷方法：

def reward(self, node_id: str, amount: float = 1.0) -> bool:
    """给一条记忆正面奖励——它在推理中被证明有用"""
    return self.update_q_value(node_id, reward=amount)

def penalize(self, node_id: str, amount: float = 1.0) -> bool:
    """给一条记忆负面惩罚——它导致了错误或过时信息"""
    return self.update_q_value(node_id, reward=-amount)

使用场景很直观：

mg = MemoryGraph("agent.db")

# Agent 检索了一条记忆并用于回答
mg.recall("database selection criteria")  # 返回若干节点
mg.reward("node_db_comparison", amount=1.0)  # 这条记忆有用！

# 另一条记忆导致了错误的回答
mg.penalize("node_old_api_doc", amount=2.0)  # 这条过时信息害了我

Q-aware 检索：让有价值的记忆浮出水面

最关键的应用场景是Q 值感知的检索。传统的 BM25 或向量搜索只看文本相关性，Q 值感知检索在此基础上融合记忆的"信用分"：

def recall_with_q(self, query: str, limit: int = 5,
                  q_bias: float = 0.3) -> list[dict]:
    """Q 值感知检索：final_score = (1-q_bias) · text + q_bias · Q"""
    # Step 1: 过宽召回（获取 limit×3 候选）
    results = self.recall(query, limit=limit * 3)
    if not results:
        return []

    # Step 2: 归一化文本分数到 [0, 1]
    max_text = max((n.weight for n in results), default=1.0) or 1.0

    # Step 3: 归一化 Q 值到 [0, 1]
    q_values = {n.id: self.get_q_value(n.id) or 0.0 for n in results}
    all_q = list(q_values.values())
    min_q, max_q = min(all_q), max(all_q)
    q_range = (max_q - min_q) or 1.0

    # Step 4: 加权融合
    scored = []
    for n in results:
        text_norm = n.weight / max_text
        q_norm = (q_values[n.id] - min_q) / q_range
        blended = (1.0 - q_bias) * text_norm + q_bias * q_norm
        scored.append({
            "node_id": n.id, "label": n.label,
            "score": round(blended, 6),
            "q_value": round(q_values[n.id], 6),
        })

    scored.sort(key=lambda x: x["score"], reverse=True)
    return scored[:limit]

q_bias 参数控制 Q 值的影响力：

q_bias行为适用场景
0.0纯文本检索冷启动阶段（Q 值还没分化）
0.3文本主导，Q 值微调日常运行（推荐默认值）
0.5均衡高噪声环境
0.7Q 值主导成熟系统（Q 值已经可靠）

三、Q 值的生态系统效应

与战略遗忘的联动

Q 值最直接的价值是保护重要记忆不被遗忘。strategic_forget 方法有一个 protect_q_above 参数：

# 遗忘低权重记忆，但永远保留 Q 值 > 2.0 的核心记忆
result = mg.strategic_forget(
    min_weight=0.1,
    max_age_days=30,
    protect_q_above=2.0,  # ← Q 值保护
    dry_run=True  # 先看看会删什么
)
print(f"将遗忘 {result['forgotten']} 条，保留 {result['kept']} 条")

这解决了一个关键矛盾：你可能有一条记忆已经 60 天没被访问（看起来该删），但它是一个关键架构决策，曾经 50 次帮助 Agent 做出正确判断。Q 值让它活下去。

这个 API 不是为了检索，而是为了可观测性。通过定期查看 top Q 值节点，你可以理解 Agent 的"价值观"——它认为什么重要？这比简单的访问频率统计更深刻，因为 Q 值通过图传播，反映了记忆在上下文中的结构性重要性。

四、实践案例：一个运行三周的 Agent

让我们看一个具体的场景。假设有一个项目助手 Agent，运行了三周后的记忆状态：

import json

mg = MemoryGraph("project_assistant.db")

# Week 1: 初始记忆录入，Q 值都是 0
mg.add_node("db_use_postgres", "项目数据库选型为 PostgreSQL", kind="decision")
mg.add_node("api_v2_deprecated", "API v2 已废弃，迁移到 v3", kind="fact")
mg.add_node("team_prefers_python", "团队偏好 Python 而非 Go", kind="preference")

# Week 1-3: Agent 在日常工作中产生 reward/penalty 信号
# 数据库选型记忆频繁帮助决策
for _ in range(15):
    mg.reward("db_use_postgres", amount=0.5)

# API v2 废弃信息救了一次大坑
mg.reward("api_v2_deprecated", amount=3.0)

# 团队偏好偶尔有用
for _ in range(3):
    mg.reward("team_prefers_python", amount=0.3)

# 但有一条过时信息导致了错误
mg.penalize("old_endpoint_cache", amount=2.0)

五、设计哲学：为什么不是简单的访问计数器？

一个自然的疑问：为什么不直接用访问次数来衡量记忆价值？

访问计数器有三个根本性缺陷：

1. 无法区分正向和负向使用。 一条过时的 API 文档可能被频繁检索（因为标题匹配度高），但每次检索后都导致 Agent 犯错。访问计数器把它当成高价值记忆，Q 值正确地把它标记为负值。

2. 无法通过图传播价值。 如果"PostgreSQL 选型"是高价值记忆，那么和它紧密关联的"数据库迁移方案"也应该受益。访问计数器是孤立的，Q 值通过 TD-learning 的邻居传播自动建立关联。

3. 缺乏衰减机制。 访问次数只增不减。Q 值通过 α 学习率和 γ 折扣因子自然衰减——如果一个曾经有用的记忆长期不再获得 reward，它的 Q 值会缓慢下降，最终被遗忘机制清理。

用一句话总结：访问计数器记录的是"过去发生了什么"，Q 值预测的是"未来这条记忆还有没有用"。

六、与前沿研究的对照

Memory-R1：RL 训练 LLM 做记忆操作

Memory-R1 的方法更激进——它不只是在记忆节点上维护 Q 值，而是直接用 RL 训练 LLM 来决定记忆操作。模型输出 ADD/UPDATE/DELETE/NOOP 四种动作之一，用 GRPO 算法优化。

我们的 Q-value scoring 是 Memory-R1 思想的轻量级工程实现。不需要训练模型，不需要 GPU，一条 SQL 就能更新 Q 值。代价是需要手动定义 reward signal（而不是让模型自动学习），但在实践中，Agent 的使用场景天然提供了丰富的 reward signal——用户反馈、任务成功/失败、错误检测。

MemRL：正反馈的力量

MemRL 的核心发现是 positive scaling：更大的模型做记忆决策效果更好。这暗示记忆管理本质上是一个需要"理解"的任务，而不仅仅是模式匹配。

在我们的小规模实现中，这对应着一个实践经验：简单的 reward/penalize 规则足够启动，但随着记忆增长，需要更智能的 reward 策略。比如，不是每次检索到就 reward，而是等整个任务完成后，回溯哪些记忆真正贡献了正确的答案，再给那些记忆 reward。这就是 credit assignment 问题——和 RL 中的完全一样。

七、局限性和未来方向

当前局限

1. Reward 稀疏性。 在实际使用中，明确的 reward signal（用户反馈"这个回答有用"）是稀少的。大部分时候 Agent 无法确定一条记忆的检索是否真正帮助了用户。

2. 冷启动问题。 新记忆的 Q 值为 0，在 Q-aware 检索中处于劣势。解决方案是设置一个"蜜月期"——新记忆在前 24 小时获得 Q 值加成，类似推荐系统中的 exploration bonus。

3. 单图 Q 值不跨 session。 当前实现中 Q 值存储在单个 SQLite 文件中。多 Agent 场景下，如何共享 Q 值是一个开放问题。

下一步

  1. 自动 reward 挖掘：分析任务完成后的推理轨迹，自动识别哪些记忆节点贡献了正确答案
  2. Q 值 + 社区检测联动：用社区的平均 Q 值作为"社区质量指标"，低质量社区优先被遗忘
  3. Q 值衰减调度：引入时间衰减机制，让长期无 reward 的记忆 Q 值自动趋向零

结语

Q-value scoring 不是银弹。它不能替代社区检测的结构化组织，不能替代战略遗忘的噪声清理，也不能替代 BM25/向量的文本相关性检索。

但它补上了拼图中缺失的一块：记忆系统的价值评估层。

有了 Q 值，遗忘知道该保护谁，检索知道该偏好谁，社区知道谁是有价值的成员。这不是"用 RL 解决记忆问题"的终极方案——它是一个务实的工程中间件，连接了文本检索的"是什么"和记忆管理的"值多少"。

当 2026 年的研究趋势（Memory-R1、MemRL、AgentFold）都在指向"记忆管理作为学习技能"时，Q-value scoring 是让这个趋势在现有工程中落地的最小可行方案。

不需要 GPU，不需要训练，一条 SQL，一个 Q 值，让记忆系统开始学习。
