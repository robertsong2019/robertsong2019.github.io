缓存驱逐的三重奏：LRU + TTL + Compact 实战

2026-05-06 · 系统设计 Embedding AI Agent

昨天凌晨，我的 AI Agent 在自主迭代中给 EmbeddingProvider 加了一层 TTL 缓存驱逐。这是继 LRU 驱逐和 Compact 恢复之后的第三层缓存策略。三种机制各管一件事，合在一起意外地优雅。值得展开聊聊。

问题：向量缓存为什么需要驱逐？

Embedding 计算不便宜——每次调用 API 都要花 token 和延迟。所以几乎所有做语义搜索的系统都会缓存：同一个文本只 embed 一次，下次直接取向量。

但缓存不管理就是内存泄漏。你的 agent 持续运行，记忆库不断膨胀，缓存跟着涨。两个具体问题：

  • 容量问题 — 缓存条目无限增长，内存爆掉
  • 新鲜度问题 — 文本已更新，但缓存的向量还是旧的

第一个问题 LRU 能解。第二个问题，LRU 完全无能为力——如果一个条目频繁访问但源数据已经变了，LRU 反而会把它钉在缓存里。

第一层：LRU — 管容量

经典做法，没什么好说的。设置 maxCacheSize，超出时踢掉最久没用的：

embed(text) {
  if (this.#cache.has(text)) {
    this.#lruOrder.delete(text);
    this.#lruOrder.add(text);  // 移到末尾
    return this.#cache.get(text);
  }
  const vector = await this._computeEmbedding(text);
  this.#setCache(text, vector);  // 内部检查 LRU 淘汰
  return vector;
}

注意实现细节：lruOrder 用 Map 的插入顺序语义，比维护一个双向链表简单得多。删除是 O(1)，重排也是 O(1)。在条目数不超过几万的场景下完全够用。

第二层：TTL — 管新鲜度

这是昨天加的。核心想法：每个缓存条目记录插入时间戳，读取时检查是否过期：

embed(text) {
  if (this.#cache.has(text)) {
    const entry = this.#cache.get(text);
    // TTL 检查：过期则透明重新 embed
    if (this.#cacheTTL > 0 && Date.now() - entry.ts > this.#cacheTTL) {
      this.#cache.delete(text);
      // fall through to re-embed below
    } else {
      this.#touchLRU(text);
      return entry.vector;
    }
  }
  const vector = await this._computeEmbedding(text);
  this.#setCache(text, { vector, ts: Date.now() });
  return vector;
}

几个设计决策值得说：

  1. 默认关闭（cacheTTL=0）。不打破现有用户的行为。opt-in 比默认开启安全。
  2. 透明驱逐。调用方不需要知道缓存过期了——重新 embed 后自动更新缓存。API 表面零变化。
  3. 运行时可调。setCacheTTL(ms) 让你在测试或不同场景中动态切换策略。

还有一个 evictExpired() 做批量清理，适合在心跳或低峰期调用：

evictExpired() {
  if (this.#cacheTTL === 0) return 0;
  const now = Date.now();
  let evicted = 0;
  for (const [key, entry] of this.#cache) {
    if (now - entry.ts > this.#cacheTTL) {
      this.#cache.delete(key);
      this.#lruOrder.delete(key);
      evicted++;
    }
  }
  return evicted;
}

第三层：Compact — 管恢复

缓存持久化到磁盘后，文件可能损坏（进程中断、磁盘满）。Compact 策略是在 loadCache() 时验证每个条目：

loadCache() {
  const raw = JSON.parse(fs.readFileSync(this.#cachePath, 'utf8'));
  let loaded = 0;
  for (const [key, val] of Object.entries(raw)) {
    const vector = val.vector ?? val;  // 兼容新旧格式
    if (Array.isArray(vector) && vector.length > 0) {
      this.#cache.set(key, {
        vector,
        ts: val.ts ?? Date.now()
      });
      loaded++;
    }
  }
  return loaded;
}

注意那个 val.vector ?? val — 这是向后兼容的关键。旧格式是 {key: [0.1, 0.2, ...]}，新格式是 {key: {vector: [...], ts: 12345}}。一行代码解决迁移，不需要 migration script。

三种策略如何配合

画个决策树：

请求 embed(text)
  ├─ 缓存命中？
  │   ├─ TTL 过期？ → 重新 embed（透明更新）
  │   └─ 未过期 → 返回缓存（刷新 LRU 位置）
  ├─ 缓存未命中？
  │   ├─ 缓存已满？ → LRU 淘汰最旧条目
  │   └─ 有空间 → 直接加入
  └─ 持久化时 → Compact 验证 + 存储时间戳

低峰期/心跳：
  └─ evictExpired() → 批量清理过期条目

每一层独立工作，不互相依赖，但组合起来覆盖了缓存生命周期的所有阶段。这是我认为最值得学的部分：好的系统设计不是一个大而全的机制，而是几个正交的小机制叠加。

集成测试发现的真实 Bug

写完 TTL 驱逐后，Agent 接着写了 session 生命周期的集成测试。测的是 initialize_session → get_iteration_context → add_iteration_result → 持久化 这条完整链路。

集成测试发现了一个单元测试不可能抓到的 bug：Path 对象无法 JSON 序列化。

# memory_manager.py 中的 _save_project_context()
# before (有 bug)
def _save_project_context(self):
    data = asdict(self.project_context)  # 包含 Path 对象!
    json.dump(data, f)  # PosixPath 不是 JSON serializable → 静默失败

# after (修复)
def _save_project_context(self):
    data = asdict(self.project_context)
    data = json.loads(json.dumps(data, default=str))  # Path → str
    json.dump(data, f)

为什么单元测试没抓到？因为单元测试 mock 了文件系统，而 asdict() 在单元测试中返回的路径碰巧是字符串（测试用的相对路径）。只有真实文件系统中才出现 PosixPath。

经验法则：单元测试验证逻辑，集成测试验证胶水。两者都要有。

可操作的建议

  1. 缓存设计从三个维度思考：容量（LRU）、新鲜度（TTL）、恢复（Compact/校验）。不要只做一个。
  2. 向后兼容是强制要求。用 val.vector ?? val 这种模式，一行代码兼容新旧格式，比 migration script 可靠。
  3. 集成测试不是奢侈品。这次一个简单的端到端测试就抓到了数据类 + 序列化的真实 bug。写 6 个集成测试花的时间，比排查线上数据丢失少得多。
  4. 默认关闭新行为。cacheTTL=0 表示"不启用"。这让你的新功能可以合并进主分支而不影响任何人。
  5. 透明优于显式。TTL 过期时自动重新 embed，调用方无感知。如果你需要调用方手动处理过期，你的抽象层错了。

最终数据：48 个 embed 测试全部通过，156 个项目测试全部通过，零回归。三层缓存策略各约 30 行代码，总计不到 100 行，覆盖了缓存从创建到驱逐到持久化的完整生命周期。

有时候好的工程不是加功能，而是在正确的维度上加正确的薄层。
